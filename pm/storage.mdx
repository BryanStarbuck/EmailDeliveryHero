# EmailDeliveryHero — Local Storage, Config & Settings

This is the spec for how EmailDeliveryHero **stores things on disk**: configuration, per-user
settings, the monitored-domains list and their sync flags, audit history, the auth session, and
logs. There is **no database** — runtime state we control lives in small, hierarchical **YAML**
files (config/settings) and machine-written **JSON** collection files (domains, audits), all under a
single **state root** outside the repo.

It applies in two deployment modes and is written so the **same code path works in both**:

* **Local mode** — one person runs the app on their own machine (localhost). Usually one user.
* **Server mode** — the app runs on a shared host and several teammates log in to it.

The only things that change between the two are *where* the state root resolves and *who* is logged
in. The layout, the file formats, and the read/write code are identical — a state dir made locally
and one made on a server have the same shape and can be copied between machines.

> **Learned from the house pattern.** The reference internal app (`~/BGit/all/app/`) specifies the
> same idea in its `pm/local_storage.mdx`: an app-level `config.yaml` plus **per-logged-in-user**
> config under `users/<email>/config.yaml`, all under one state root, fronted by a typed store
> layer. EmailDeliveryHero follows that model, adapted to this app's state root
> (`~/.email_delivery_hero/`), its OpenAuthFederated auth, and its domain/audit collection stores.

Related specs:

* [Engineering](./engineering.mdx) — §7 "On-disk store (no database)" and §8 "Periodic checks"; the

* [Settings](./settings.mdx) — the Settings **page**: every field, which are admin-only, and the

* [Error handling & logging](./errors.mdx) — the behavior of the two log files this document lays

## 1. The state root

Everything the app persists lives under a single **state root directory**:

```
~/.email_delivery_hero/
```

* Default `~/.email_delivery_hero/`, overridable with the **`EDH_STATE_DIR`** environment variable
  (server mode points it at a service-owned dir, e.g. `/var/lib/email_delivery_hero`).
* The directory is created on first use (`mkdirSync(dir, { recursive: true })`, best-effort) — the
  app never crashes because the folder is missing.
* Implemented in `code/packages/backend/src/shared/state-dir.ts` — `resolveStateDir()`,
  `stateSubdir(...parts)` (mkdir-on-demand subdirectories, used by the blacklists store §7A), and
  `resolveLogDir()` (**`EDH_LOG_DIR`** points the two log files elsewhere without moving the rest
  of the state — tests use this so specs never pollute the real `error.err`).
* **Code today still defaults to the older `~/T/_emaildeliveryhero/`.** Moving the code default to
  `~/.email_delivery_hero/` (with a one-time adopt of an existing old-path state dir) is drift item
  **D1** (§16); this document and all 24 `checks/*.mdx` specs use `~/.email_delivery_hero/`.

### Resolution rule

```
stateDir =
  process.env.EDH_STATE_DIR                          // explicit override (server mode)
  ?? join(os.homedir(), ".email_delivery_hero")      // default (local mode)
  ?? "/tmp/_emaildeliveryhero"                       // last resort if homedir() throws
```

Nothing under the state root is ever committed to the repo — it is the boundary between code and
runtime data (`~/BGit/Bryan_git/EmailDeliveryHero/CLAUDE.md`).

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/state-dir.ts` — `resolveStateDir()` (the `EDH_STATE_DIR` → homedir → temp resolution + legacy-dir adopt), `stateSubdir()`, and `resolveLogDir()`.

## 2. Directory layout

```
~/.email_delivery_hero/
├── config.yaml              # APP-LEVEL settings — install-wide, hierarchical YAML (§3) [planned]
├── domains.yaml             # the monitored-domains list + each domain's sync on/off flag (§5) [built]
├── domains.json             # LEGACY — one-time migration source for domains.yaml; never written (§5)
├── audits.json              # latest audit result keyed by domain id — the fast cache (§6) [built]
├── runs/                    # RUN HISTORY — one YAML file per run, per domain (§7) [target model]
│   ├── whitehatengineering.com/
│   │   ├── 2026_06_30_03_00AM.yaml
│   │   └── 2026_07_01_09_14AM.yaml
│   └── act3ai.com/
│       └── 2026_07_01_09_14AM.yaml
├── runs.json                # CURRENT CODE's run history (flat JSON, 50/domain) — migrates into runs/ (§7.5)
├── blacklists/              # per-domain Blacklists DEEP STORE, keyed by domain name (§7A) [built]
│   └── <domain-name>/
│       ├── latest.yaml      #   newest full run (snake_case)
│       ├── runs/<audit_id>.yaml   # one file per run, pruned to 50
│       └── portals.yaml     #   provider-portal checklist — USER-EDITABLE
├── blacklist_zones.yaml     # operator overrides over the bundled blocklist-zone registry (§7A) [built]
├── reports/                 # ingested DMARC / TLS-RPT report emails (emails.mdx) [planned]
├── .auth_session_secret     # stable HS256 signing secret (auto-created; never in .env) (§8) [built]
├── users/                   # PER-LOGGED-IN-USER tree, keyed by verified email (§4)
│   └── <email>/
│       ├── config.yaml      # that user's personal settings (theme, notif prefs, last view) [planned]
│       └── sessions/        # OpenAuthFederated FileSessionStore — one JSON per session (§8) [built]
├── log.log                  # INFO/DEBUG/VERBOSE — the normal event trail (§11, errors.mdx) [built]
└── error.err                # WARN/ERROR/FATAL — the fault trail (§11, errors.mdx) [built]
```

`[built]` = in `code/` today (module named in each section); `[planned]` = spec'd contract, not yet
implemented; the §15 code map and §16 drift ledger track the difference precisely.

Three tiers:

* **App-level** (`config.yaml`) — one copy for the whole install; the global, mostly admin-owned
  settings that drive every scheduled run and every domain's score. In local mode that is "this
  machine"; in server mode "this server".
* **User-level** (`users/<email>/config.yaml`) — one folder per person who has ever logged in,
  created lazily on first write. Personal, cosmetic, per-user preferences.
* **Machine-written collections & runtime** (`audits.json`, `runs/`, `sessions/`,
  `.auth_session_secret`, `log.log`, `error.err`) — not hand-edited; the app owns them. The
  **domains list** (`domains.yaml`) is app-owned too but deliberately kept in **YAML** so an
  operator can read and hand-edit it (§5). The **run history** (`runs/`, §7) is app-written but
  also deliberately **YAML** — each run file is a human-readable record an operator can open,
  diff, and copy between machines.

### Why YAML for config + the domain list, JSON for the audit collection

The product rule is **"keep human-facing data in YAML unless the data type makes that a bad fit."**

| Need | Goes in | Why |
| --- | --- | --- |
| Settings, preferences, toggles, small key/values | `config.yaml` (app) / `users/<email>/config.yaml` | Human-readable, hot-editable, **hierarchical**. |
| The **monitored-domains list** — small, operator-facing, hand-editable on localhost | `domains.yaml` | A short sequence a person actually reads and edits; the store rewrites it atomically. |
| The **latest-audit collection** — machine-written, high-churn, never hand-edited | `audits.json` | A keyed map the app owns and rewrites atomically; JSON matches the current code. |
| The **run history** — one immutable record per run, read by humans and the UI alike | `runs/<domain>/<timestamp>.yaml` | Written once at run completion, never rewritten; YAML so an operator can open any run and read every test result (§7). |
| Secrets (Google client id/secret) | `~/.credentials/email_delivery_hero.json` **or** env | Never in the state store, never in the repo (§8, [Authentication](./authentication.mdx)). |
| Append-only logs | `log.log` / `error.err` | Flat append streams; appending to YAML/JSON is wrong. |

## 3. App-level `config.yaml` — the settings store (hierarchical)

`~/.email_delivery_hero/config.yaml` is the install-wide settings file that the
[Settings page](./settings.mdx) reads and writes. It is **default-everything**: the app boots on
sensible defaults and writes the file only when a value is set, so a fresh install runs with no file
present. Every value is nested under a top-level group so the tree stays **hierarchical** and one
concern is one block.

```yaml
# ~/.email_delivery_hero/config.yaml — app-level settings (install-wide)
schema_version: 1
updated_at: 2026-07-01T14:32:15.123Z

# Where the app is reachable (mirrors the ports in Engineering / overview)
server:
  frontend_port: 4444        # React 19 + Vite
  backend_port: 9312         # NestJS REST API under /api
  mode: local                # local | server

# ── §2 Checks configuration (admin-only) — drives the audit engine ──
checks:
  enabled: [spf, dkim, dmarc, blacklists, dns_infra]   # subset of the six categories;
                                                       # spam_content optional/off by default
  spf:
    maxLookups: 10           # RFC 7208 limit before SPF is flagged
  dkim:
    defaultSelectors: [google, selector1, selector2, k1]  # fallback when a domain sets none
  dnsbl:
    zones:                   # the blocklist zones the Blacklists category queries (ordered)
      - zen.spamhaus.org
      - b.barracudacentral.org
      - bl.spamcop.net
  thresholds:                # 0–100 audit score → colour
    green: 90                # score >= green → green
    amber: 70                # green > score >= amber → amber; below → red
  weights:                   # points deducted per finding severity (feeds score derivation)
    critical: 40
    warning: 15
    info: 0

# ── §3 Scheduled checks (admin-only) — full model in scheduled_checks.mdx ──
schedule:
  enabled: true              # the dashboard on/off toggle (master switch)
  cadence: daily             # interval | daily | weekly
  everyHours: 6              # used only when cadence: interval
  times: ["06:00", "18:00"]  # used by daily/weekly — supports multiple runs per day
  weekdays: [mon, wed, fri]  # used only when cadence: weekly
  timezone: America/Los_Angeles   # IANA tz; defaults to system tz
  domains: all               # "all" | ["<domainId>", …] — subset to include
  runner: in-process         # in-process | os — which scheduling layer is active
  os:
    kind: launchd            # launchd | cron | systemd | schtasks (detected)
    installed: false         # whether the OS artifact is currently installed
    label: com.emaildeliveryhero.scheduler

# ── §4 Notifications — SHARED channels here (admin-only); per-user prefs in users/<email>/ ──
notifications:
  webhook:                   # admin-only: shared Slack-compatible incoming webhook
    enabled: false
    url: ""
  smtp:                      # admin-only: shared relay used to send notification email (swaks)
    host: ""
    port: 587
    from: "edh@whitehatengineering.com"

# ── §5 Storage & data (admin-only) ──
storage:
  retentionDays: 90          # how long to keep audit history before pruning

# ── §6 Tools & environment (admin-only) ──
tools:
  preferCli: false           # use dig/swaks instead of node:dns/promises when available
  resolvers: []              # DNS resolver IPs; [] = system default
  timeoutMs: 5000            # per-lookup timeout

# ── §7 Access — DISPLAY of the OpenAuthFederated-enforced policy (admin-only) ──
access:
  allowedDomains: [whitehatengineering.com, act3ai.com]  # applied on the OAF side, shown here

# ── Defaults applied to every user until their own users/<email>/config.yaml overrides them ──
defaults:
  theme: system              # system | light | dark
  density: comfortable       # comfortable | compact
  notifications:
    desktop: true
    email: false
    minSeverity: warning     # info | warning | critical
    mode: immediate          # immediate | daily
```

Secrets never go here — Google client credentials come from env or the out-of-repo credentials file
([Authentication](./authentication.mdx) §credentials); the auth signing secret is its own file
(§8). `config.yaml` holds only non-secret ports, flags, and PM/admin-editable settings.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/config-store.ts` — `readAppConfig()` (validated deep-merge over `defaultAppConfig()`) and `updateAppConfig()` (read-modify-write, stamps `schema_version`/`updated_at`, atomic YAML write).

## 4. Per-logged-in-user storage — the per-user requirement

This is the piece that makes storage **based on the current logged-in user**. App-level `config.yaml`
holds the shared, admin-owned settings; each person's **personal** preferences live in their own file:

```
~/.email_delivery_hero/users/<email>/config.yaml
```

* `<email>` is the user's **verified** email from the OpenAuthFederated JWT (the Google Workspace
  identity — [Authentication](./authentication.mdx)) when signed in, or the reserved literal
  **`default`** when logged out. Login is **optional** ([Security](./security.mdx)): there is always
  a current user, so every request resolves to a folder key — a trusted verified email, or `default`.
  Verified emails always contain `@`, so a real identity can never collide with the `default` key.
  (The stable `sub` claim is an acceptable alternative key for signed-in users; email is chosen so
  the tree is human-browsable.)
* In **server mode** this cleanly separates every teammate's settings on one shared install. In
  **local mode** there is usually one user, but the same folder is used, so the shape is identical.
* The folder is created **lazily** on the first write for that user. Reading a user who has no folder
  yet returns the `defaults` block from the app `config.yaml` — never an error, never a first-run
  write.

### `users/<email>/config.yaml` shape

```yaml
# ~/.email_delivery_hero/users/bryan@whitehatengineering.com/config.yaml
schema_version: 1
updated_at: 2026-07-01T14:32:15.123Z

# §8 Appearance (per-user, cosmetic)
ui:
  theme: dark                # overrides defaults.theme
  density: compact           # overrides defaults.density
  last_route: /dashboard     # restore the user where they left off

# §4 Notification preferences (per-user; the SHARED channels stay in app config.yaml)
notifications:
  desktop: true
  email: true
  minSeverity: warning       # info | warning | critical
  mode: immediate            # immediate | daily

# Remembered per-table sort/filter, keyed by table id (e.g. the domains table)
tables:
  views:
    domains:
      sort: { col: domain, dir: asc }
      filters:
        scheduleEnabled: { op: eq, value: true }
```

> **Relationship to [Settings](./settings.mdx).** Settings §4 and §8 describe a per-user block; on
> disk that block is realized as these `users/<email>/config.yaml` files — one per user — rather than
> a `users.<sub>` map inside the app config. The Settings REST contract is unchanged: `GET /api/settings`
> merges `defaults` (app config) with the caller's own `users/<email>/config.yaml`; `PUT /api/settings`
> writes only the caller's per-user file for all-users/per-user fields, while `PUT /api/settings/admin`
> writes the admin-only blocks in the app `config.yaml`.

### Email → directory policy

* **Lowercase** the email.
* Store it **verbatim as the folder name** (`bryan@whitehatengineering.com`) so the tree is
  human-browsable.
* **Sanitize defensively:** reject/strip any `/`, `\`, `..`, or NUL before using the value as a path
  segment. It comes from a trusted JWT, but we never build a path from an unsanitized string.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/config-store.ts` — `readUserConfig()` / `updateUserConfig()` (lazy per-user file) and `sanitizeUserKey()` (lowercase, strip `/`,`\`,`..`,NUL, `default` fallback); the folder is created by `stateSubdir("users", key)`.

## 5. `domains.yaml` — the domain list + each domain's sync on/off flag

The monitored-domains list is the source of truth for the [Domains](./domains.mdx) CRUD and persists
to **`~/.email_delivery_hero/domains.yaml`** (implemented in
`code/packages/backend/src/modules/domains/domains.service.ts` via the `yaml-store` helper). It is a
**YAML sequence** — deliberately YAML, not JSON, so an operator can read and hand-edit the domain list
on localhost. **Each record carries its own `scheduleEnabled` flag — this is "what was turned on or
off on sync"** (whether the domain rides the recurring scheduled checks). The latest per-category
status shown in the UI is not duplicated here; it is derived at read time from `audits.json` (§6).
A legacy `domains.json` from an earlier install is migrated to `domains.yaml` on first read.

```yaml
- id: 3f1c… (uuid)
  name: whitehatengineering.com
  label: Primary marketing domain
  dkimSelectors:
    - google
    - selector1
  sendingIps:
    - 203.0.113.10
    - 203.0.113.11
  scheduleEnabled: true
  addedBy: bryan@whitehatengineering.com
  createdAt: 2026-07-01T14:03:11.204Z
  updatedAt: 2026-07-01T14:03:11.204Z
- id: 7a20… (uuid)
  name: act3ai.com
  label: Product / transactional
  dkimSelectors:
    - google
  sendingIps:
    - 198.51.100.20
  scheduleEnabled: false
  addedBy: bryan@whitehatengineering.com
  createdAt: 2026-07-01T14:04:00.000Z
  updatedAt: 2026-07-01T14:04:00.000Z
```

* `scheduleEnabled` is toggled per domain in the Domains table/form ([Domains §6](./domains.mdx)); a
  domain is on the recurring schedule only when the **global** `schedule.enabled` (§3) **and** its own
  `scheduleEnabled` are both true.
* Adding/removing a domain, editing its IPs/selectors/label, and flipping its sync toggle all rewrite
  this file through the `yaml-store` helper (§9). Records written by an older version that lack
  `label` / `scheduleEnabled` are backfilled with defaults (empty label, schedule **on**) on read.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/domains/domains.service.ts` — class `DomainsService` (CRUD → `writeYaml(domains.yaml)`, one-time `domains.json` adoption, `onRemoved` listeners that purge audit history).

## 6. `audits.json` — the latest-result cache

The **latest** audit result per domain persists under the state root (already implemented in
`code/packages/backend/src/modules/audit/audit.service.ts`) as a single JSON map of **latest result
keyed by domain id** — the fast cache the Dashboard's Domain-health table reads. The full per-run
history is the `runs/` YAML tree (§7).

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/audit.service.ts` — class `AuditService`, method `persistResult()` (writes `audits.json` + the run file under the `writeChain` mutex) and `latest()` / `latestAll()` reads.

```json
{
  "whitehatengineering-com": {
    "id": "2026-07-01T14-03-11-a1b2",
    "domain": "whitehatengineering.com",
    "startedAt": "2026-07-01T14:03:10.900Z",
    "finishedAt": "2026-07-01T14:03:11.204Z",
    "findings": [
      { "checkId": "dmarc", "severity": "warning",
        "title": "DMARC policy is p=none",
        "detail": "_dmarc.whitehatengineering.com → v=DMARC1; p=none",
        "remediation": "Strengthen to p=quarantine once reports look clean." }
    ],
    "score": 85,
    "status": "warning"
  }
}
```

* **Role.** `audits.json` is a derived cache — it is always reconstructible from the newest run file
  per domain in `runs/` (§7). It exists so the Dashboard's top table renders without scanning the
  run tree.
* **Regression detection** ([Engineering §8](./engineering.mdx)) diffs each new run against the
  previous one (the two newest files in `runs/<domain>/`) to surface newly-introduced/worsened
  problems on the [Dashboard](./dashboard.mdx).

## 7. `runs/` — the run history (one YAML file per run)

Every health-check **run** — one execution of all checks for **one domain**, per the locked
**Run → Test → Sub-test** hierarchy in [Dashboard §1](./dashboard.mdx) — persists as **one YAML
file** under the state root:

```
~/.email_delivery_hero/
└── runs/
    ├── whitehatengineering.com/
    │   ├── 2026_06_28_06_00PM.yaml
    │   ├── 2026_06_30_03_00AM.yaml
    │   └── 2026_07_01_09_14AM.yaml      ← the newest run for this domain
    └── act3ai.com/
        └── 2026_07_01_09_14AM.yaml
```

A domain accumulates **many runs** over time; each run is one file, and the files never change once
written.

### 7.1 Directory rule

* `runs/` sits directly under the state root.
* One subdirectory **per domain**, named the **domain name, lowercased, verbatim**
  (`whitehatengineering.com`) — so different domains' runs live in different directories and the
  tree is human-browsable. The name is defensively sanitized like the email folder key (§4 policy:
  reject/strip `/`, `\`, `..`, NUL) before use as a path segment.
* The domain directory is created lazily on that domain's first run.

### 7.2 Filename rule — the run timestamp (LOCKED)

The filename is the run's **start time**, in this exact order — **year, month, day, hour, minutes,
AM/PM, `.yaml`**:

```
YYYY_MM_DD_hh_mm{AM|PM}.yaml       e.g.  2026_07_01_09_14AM.yaml
```

* `YYYY` 4-digit year · `MM` 2-digit month · `DD` 2-digit day · `hh` 2-digit **12-hour-clock**
  hour · `mm` 2-digit minutes — all zero-padded, joined by underscores — then `AM` or `PM`
  (uppercase, no separator before it), then `.yaml`.
* 12-hour-clock edge cases: midnight 00:xx → `12_xxAM`; noon 12:xx → `12_xxPM`.
* The time is rendered in the app's configured **local timezone** (`schedule.timezone` in §3,
  defaulting to the system tz) — the filename is for the human browsing the tree; the file body
  keeps the precise ISO-8601 UTC instants.
* **Collision rule:** if a second run for the same domain starts in the same minute, append `_2`
  (`_3`, …) before `.yaml`.
* Lexicographic sort of these names **is** chronological sort within a day segment-by-segment
  (the AM/PM hour is the one non-lexicographic wrinkle — listings must parse the timestamp, not
  string-sort blindly across noon/midnight).

### 7.3 File shape — the run YAML specification

One document per run: a `run:` metadata block, then **one top-level key per test category** — the
six locked categories, keys `spf`, `dkim`, `dmarc`, `blacklists`, `dns_infra`, `spam_content`:

```yaml
# ~/.email_delivery_hero/runs/whitehatengineering.com/2026_07_01_09_14AM.yaml
run:
  run_id: 2026-07-01T16-14-02-a1b2      # unique id — what /runs/:runId resolves
  domain: whitehatengineering.com
  started_at: 2026-07-01T16:14:02.000Z  # ISO-8601 UTC (09:14 AM PDT — the filename time)
  finished_at: 2026-07-01T16:14:05.310Z
  score: 88
  status: warning                        # healthy | warning | critical

spf:          { ... }   # each category section owns: status (worst severity),
dkim:         { ... }   # the structured observation for that category,
dmarc:        { ... }   # tool_runs: [] — one record per external tool invocation
blacklists:   { ... }   #   {tool, command, started_at, duration_ms, exit_code,
dns_infra:    { ... }   #    output_format: json|text, parsed: <captured output>, error}
spam_content: { ... }   # tests: [] — one row per sub-test {id, title,
                        #   result: pass|fail|warn|info, evidence, fix}
                        # problem_states: [] — matched problem-state ids
```

The **exact shape of each category section is owned by that category's spec** — see the
"Test results (YAML)" section of each:
[SPF](./checks/spf.mdx), [DKIM](./checks/dkim.mdx), [DMARC](./checks/dmarc.mdx),
[Blacklists](./checks/blacklists.mdx), [DNS & Infrastructure](./checks/dns.mdx),
[Spam & Content](./spam_checks.mdx). This document owns only the envelope: the path, the
filename, the `run:` block, and the six category keys.

### 7.4 Write / read / retention rules

* **Written once, atomically** (write-temp-then-rename via the store layer, §9) when the run
  finishes; a run file is immutable history and is never edited afterward. The same write also
  refreshes the domain's entry in `audits.json` (§6).
* **Read by** `GET /api/audit/runs` (listing — newest `started_at` first, optional `?domainId=`)
  and `GET /api/audit/runs/:runId` (one run — the run report and every "view one category run"
  page render from this file).
* **Retention:** files older than `storage.retentionDays` (§3) are pruned, and at most the newest
  **50 runs per domain** are kept ([Dashboard §1](./dashboard.mdx)); pruning deletes whole files,
  never rewrites them. Deleting a domain removes its `runs/<domain>/` directory.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/runs-store.ts` — `saveRun()` (one immutable YAML/run, filename from `runFileTimestamp()` + `_2` collision rule), `listRuns()` / `getRun()` (decode + sort on ISO `started_at`), `deleteRun()`, `deleteDomainRuns()`, `pruneRuns()`, and `migrateLegacyRunsJson()`.

### 7.5 Current implementation & migration (code today)

* **Code today persists history as `runs.json`** — a flat JSON array of `AuditResult` records,
  newest first, capped at the newest **50 runs per domain** (`RUNS_KEPT_PER_DOMAIN` in
  `modules/audit/audit.service.ts`). Every finished run goes through `persistResult()`, which
  writes **both** files in one serialized step — it overwrites the domain's entry in `audits.json`
  (§6) **and** prepends to `runs.json` — under the `writeChain` per-file mutex, so the parallel
  scan fan-out (concurrency 4, [progress_ui](./progress_ui.mdx)) never clobbers a concurrent
  write. (Older spec wording that `persistResult` targets only one of the two files is drift
  **D3**, §16.)
* The built REST surface is already the §7.4 contract: `GET /api/audit/runs?domainId=`,
  `GET /api/audit/runs/:runId`, `DELETE /api/audit/runs/:runId`, plus the triggers
  `POST /api/audit/run/:domainId` and a batch `POST /api/audit/run` the UI does not call

## 7A. `blacklists/<domain-name>/` — the per-check deep store (built) + zone registry

The Blacklists check is the first **grown** check with its own per-domain store — the pattern any
future deep-results check follows. Implemented in
`code/packages/backend/src/modules/audit/checks/blacklist/store.ts` via `stateSubdir()` + the
`yaml-store` helper; full data model in [checks/blacklists.mdx](./checks/blacklists.mdx) §12.

```
blacklists/<sanitized-domain-name>/
├── latest.yaml            # the newest full BlacklistRunResults
├── runs/<audit_id>.yaml   # one file per run, pruned to the newest 50 (lexical = chronological)
└── portals.yaml           # Record<provider, "unverified"|"verified_clean"|"problem_reported">
```

* **Keyed by domain NAME**, not UUID — `sanitize()` maps any char outside `[A-Za-z0-9._-]` to `_`.
  Deliberately operator-facing (a human finds `blacklists/act3ai.com/` instantly) and consistent
  with the `runs/<domain>/` rule (§7.1), but it splits keying from `audits.json` (**D11**, §16):
  delete-and-re-add a domain (new UUID) and the name-keyed blacklist history survives while the
  UUID-keyed audit history starts over. Renames are impossible (`name` is locked after create, §5),
  which contains the damage.
* **`BlacklistRunResults` is all snake_case** — `schema_version: 1`, `technology`, `domain`,
  `audit_id` (timestamp-prefixed), `ran_at`, `duration_ms`, `resolver{}`,
  `targets{ips[],domains[]}`, `zone_health[]`, `results[]`, `positive_reputation`,
  `provider_portals[]`, `summary`, `diff`.
* **Written when:** at the end of the blacklist checker's `run()` on every audit of the domain. A
  store-write failure is swallowed (logged, run continues) because the same payload also rides in
  the run record at `results.blacklist` — the run is stored **twice**, with two unrelated ids
  (`runId` vs `audit_id`) until the §7.5 migration aligns them (**D12**, §16).
* **REST:** `GET /api/blacklists/results` (latest per domain) ·
  `GET /api/blacklists/results/:domain` (full latest run) ·
  `GET /api/blacklists/results/:domain/history` (compact rows from `runs/*.yaml`) ·
  `PATCH /api/blacklists/:domain/portals/:provider` (body `{ state }`).
* **UI read:** BlacklistsPage summary cards; BlacklistDomainPage (verdict, per-zone results,
  history sparkline); BlacklistStatePage (one problem state).
* **User can edit:** the **provider-portal checklist** — each provider pill cycles
  Unverified → Clean → Problem and PATCHes `portals.yaml`. Real user-editable persisted state.

## 9. The store layer (how code touches disk)

All filesystem access for state goes through a thin **`store` module** so callers never deal with
paths, parsing, atomic writes, or the local-vs-server difference ([Engineering §7](./engineering.mdx)).

* **Atomic writes.** Serialize → write to a **unique** `*.tmp` (per-write, so concurrent writers never
  share a temp path) → `rename` over the real file, so a crash mid-write never leaves a half-written
  `config.yaml`/`domains.yaml`; a failed write cleans up the temp file, logs, and rethrows.
* **Per-file mutex.** Read-modify-write is serialized per file to avoid concurrent-write corruption
  (matters in server mode); every write stamps `updated_at`.
* **Defaults on absence.** Reading a missing file returns the typed defaults and logs nothing —
  expected first-run absence is **not** an error and must not spam `error.log`.
* **Schema validation on read.** Files are validated against the record/config schema on load;
  malformed data is rejected loudly rather than silently trusted.
* **Two scopes.** App-scoped (`config.yaml`, `domains.yaml`, `audits.json`) and user-scoped
  (`users/<email>/config.yaml`, created lazily) — the user-scoped opener is what makes storage
  per-logged-in-user (§4).

The store is isolated behind an interface so a later move to a real database is a single-module
change; services never touch the filesystem directly.

### As built today — two helpers, not yet converged (D16, §16)

* **`shared/yaml-store.ts`** (domains + blacklists store + zone overrides): `readYaml` returns the
  fallback on a missing, empty, or even **malformed** file (logs WARN+ERROR, never throws);
  `writeYaml` mkdirs the parent, writes a **unique** temp (`<path>.<pid>.<seq>.tmp`), renames over
  the target, and unlinks the temp on failure (then rethrows).
* **`shared/json-store.ts`** (audits.json + runs.json): same read semantics; `writeJson` uses a
  **fixed** `<path>.tmp` and on failure renames it to `<path>.tmp.failed` (leaves an artifact) —
  no unique temp, no parent mkdir.
* The only per-file mutex in code is `AuditService.writeChain` (audits.json + runs.json);
  `portals.yaml` read-modify-write is unguarded.

**Convergence contract:** unique-temp + unlink-on-fail + parent-mkdir everywhere; a serialization
guard for every read-modify-write; `schema_version` validation on read (loud on *corrupt*, silent
on *absent*) — silently falling back on a corrupt file hides data loss.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/yaml-store.ts` — `readYaml()` / `writeYaml()` (unique-temp + rename + unlink-on-fail); JSON equivalents in `shared/json-store.ts` (`readJson()`/`writeJson()`); the per-file mutex is `withFileLock()` in `shared/config-store.ts`.

## 10. Browser `localStorage` conventions (frontend)

The React frontend uses browser `localStorage` **only** as a fast, pre-network cache — the **server**
(`users/<email>/config.yaml`) is always the source of truth. This mirrors the reference app's
`theme.ts`, and it is bounded by two hard rules:

* **Never store secrets or long-lived tokens** in `localStorage`
  ([Authentication](./authentication.mdx) §security). Auth is the `HttpOnly` `oaf_edh` cookie plus a
  server-side session (§8); the JWT is held in memory by `@auth/react`, not persisted to
  `localStorage` by app code.
* **`localStorage` is a mirror, not the record.** On sign-in the app hydrates from
  `GET /api/settings`; on change it writes through to the server and updates the mirror. Clearing
  `localStorage` loses nothing but the pre-boot hint.

| Key | Value | Purpose |
| --- | --- | --- |
| `edh.theme` | `system` \| `light` \| `dark` | Apply the theme instantly at boot with no flash, before the session/network is ready. Reconciled with `users/<email>/config.yaml → ui.theme` after load. |
| `oaf_edh…` (library-owned) | OpenAuthFederated session snapshot | Written and owned by `@auth/react`, namespaced with the `oaf_edh` prefix so it never collides with other OAF apps on localhost. App code does not read/write these directly. |

Because per-user truth is server-side, a user gets the **same** settings on any browser they sign in
from — `localStorage` only accelerates the first paint.

> **Built today:** the only app-owned key in code is **`edh.scheduled`** (`"on"` / `"off"`) — the
> Dashboard "Scheduled" master toggle, written and read only by DashboardPage. It is **client-only**
> (never sent to the backend; real scheduling today is the `EDH_PERIODIC_AUDIT_MINUTES` env switch,
> §14 · scheduled_checks row), so the toggle *looks* functional but persists nothing server-side —
> drift **D17** (§16). When the `schedule.enabled` setting (§3) lands, the flag moves server-side
> and this key becomes its mirror. `edh.theme` is **planned** — no theme system exists in the
> frontend yet (colors are fixed CSS variables), so nothing writes it today (**D18**). No
> sessionStorage, no app-written cookies; TanStack Query caches API reads in memory only, and the
> scan-progress dock is in-memory React state lost on reload — by design, nothing to persist.

## 11. Logging

Logs are two append-only flat files **directly at the state root** (no `logs/` subdirectory),
implemented in `code/packages/backend/src/shared/logging.ts` +
`code/packages/backend/src/shared/rolling-file-writer.ts`:

```
~/.email_delivery_hero/log.log     # INFO/DEBUG/VERBOSE — the normal event trail
~/.email_delivery_hero/error.err   # WARN/ERROR/FATAL — the fault trail
```

* One line per event: `[ISO-8601 UTC] [LEVEL] [Context] message`. WARN/ERROR/FATAL are written to
  **both** files (`error.err` is the complete fault list; `log.log` keeps them inline for context).
* The log dir defaults to the state root; override with **`EDH_LOG_DIR`** to point logs elsewhere
  without moving the rest of the state.
* **Never let logging throw** — a failed write falls back to `stderr` and the request continues.
* Rotation is **on**: each file is capped (5 MiB) with 5 rotated backups (`log.log.1`…, `error.err.1`…),
  so neither grows unbounded; expected-absence and 4xx are logged as INFO/WARN, not errors.
* The full behavior — the global exception filter, process handlers, per-checker capture, and the
  frontend-error ingest that make *anything that can go wrong* land in `error.err` — is specified in
  **[Error handling & logging](./errors.mdx)**. This section is only the on-disk (storage) view.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/logging.ts` — class `AppLogger` (`logInfo`/`logWarn`/`logError`; WARN/ERROR/FATAL to both files) backed by `RollingFileWriter` in `shared/rolling-file-writer.ts` (5 MiB cap × 5 backups, synchronous `appendFileSync`).

## 13. Acceptance criteria

Storage is correct when **all** of the following hold:

1. All persisted state lives under a single **state root** (`~/.email_delivery_hero/`, override
   `EDH_STATE_DIR`), created on first use, never committed to the repo.
2. App-level settings live in a **hierarchical** `config.yaml` (one nested block per concern:
   `server`, `checks`, `schedule`, `notifications`, `storage`, `tools`, `access`, `defaults`), which
   is **default-everything** — a fresh install runs with no file present.
3. Per-logged-in-user settings live in `users/<email>/config.yaml`, keyed by the **verified** email
   from the OpenAuthFederated JWT, created **lazily** on first write; reading an unknown user returns
   the app `defaults` block with no error and no write.
4. `GET /api/settings` merges `defaults` with the caller's own `users/<email>/config.yaml`;
   per-user/all-users writes go only to that user's file; admin-only writes go to the app `config.yaml`
   (`PUT /api/settings/admin`, `403` without `role:admin`).
5. The **domain list** and **each domain's `scheduleEnabled` (sync on/off)** persist in `domains.yaml`
   (human-readable YAML; a legacy `domains.json` is migrated on first read); a domain is on the
   recurring schedule only when global `schedule.enabled` **and** the domain's `scheduleEnabled` are
   both true.
6. The latest audit result per domain persists in `audits.json` (a derived cache, reconstructible
   from `runs/`).
7. **Every run persists as one YAML file at `runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml`** — the
   domain subdirectory is the lowercased domain name; the filename is the run's start time in the
   configured local timezone (year, month, day, 12-hour-clock hour, minutes, AM/PM, `.yaml`,
   zero-padded, `_2` suffix on same-minute collision); the file holds the `run:` metadata block plus
   the six category keys (`spf`, `dkim`, `dmarc`, `blacklists`, `dns_infra`, `spam_content`), each
   with its structured observation, `tool_runs`, `tests`, and `problem_states` per its category
   spec. Run files are written once, atomically, and never edited; pruning (per
   `storage.retentionDays` and the 50-runs-per-domain cap) deletes whole files.
8. Auth session records (`sessions/`) and the signing secret (`.auth_session_secret`) live under the
   state root, separate from settings; **no** provider secret is stored there or in `config.yaml`.
9. Every read/write goes through the `store` layer: atomic write-temp-then-rename, per-file mutex,
   defaults on absence, schema validation on read; services never touch the filesystem directly.
10. The frontend uses `localStorage` only as a **mirror** (`edh.theme`) with the server as the source
    of truth; **no secret or long-lived token** is ever placed in `localStorage`.
11. Logs are appended to `log.log` (INFO/DEBUG/VERBOSE) and `error.err` (WARN/ERROR/FATAL) at the
    state root, size-capped with rotation; logging never throws. Full behavior in [errors.mdx](./errors.mdx).
12. The layout, formats, and code are **identical** in local and server mode — only the resolved
    state root and the logged-in user differ; a state dir copies between machines unchanged.
13. The Blacklists deep store follows §7A — name-keyed per-domain dirs, `schema_version: 1`
    snake_case YAML, `latest.yaml` + 50-run pruned history, user-editable `portals.yaml` via
    `PATCH /api/blacklists/:domain/portals/:provider`; any future per-check deep store follows the
    same pattern and gets its §12 inventory row **before** it exists on disk.
14. Check output lands per the §15 conventions — container keys `results.<check_id>` /
    `results["infra.<x>"]` only, camelCase envelope, snake_case payloads, stable prefix-matched
    finding ids.
15. Every persisted file/key appears in the §12 inventory, every PM spec's storage is mapped in
    §14 (file → keys → backend writer → UI read path → user-editable fields), and every known
    code-vs-spec divergence is a numbered §16 drift-ledger item with a resolution — never a
    silent fact.

## 15. Result-shape conventions & code↔store map

### Conventions (the contract for every check)

1. **Container key** — a check's structured snapshot lives at **`results.<check_id>`**; the infra
   family uses **`results["infra.<x>"]`** (`infra.mx_routing`, `infra.reverse_dns`,
   `infra.dns_health`, `infra.dnssec`). These are the only two forms. The nine legacy families in
   older check specs (`checks.<x>`, `checks.infra_<x>`, `checkResults.<x>`, `checkData.<x>`,
   `content.<x>`, bare top-level keys, …) are drift, normalized as each spec is grown (**D10**).
2. **Case** — the run-record **envelope is camelCase** (`runId`, `domainId`, `startedAt`); every
   **`results.<check>` payload is snake_case** (`lookup_count`, `working_selectors`, `audit_id`).
   The grown checks (spf, dmarc, dkim, blacklists, infra) already comply. The §7.3 run-YAML
   category sections are snake_case throughout.
3. **Findings** — every finding carries a stable `id` (prefix-matched to problem states and
   routes, e.g. per-selector `dkim.<selector>.<problem>`), a `checkId`, and a severity from
   `ok | info | warning | critical`.
4. **Deep stores** — follow the §7A pattern (`schema_version: 1`, snake_case YAML, latest +
   pruned history), inventoried in §12 before they exist.
5. **Per-domain check config** — goes on the `domains.yaml` record (nested `checks:` sub-map when
   it lands) or app `config.yaml` if install-wide — never the legacy `domains.json` (**D6**).

### Code↔store map (ground truth, Built today)

| Code | Owns |
| --- | --- |
| `shared/state-dir.ts` | state root, `EDH_STATE_DIR` / `EDH_LOG_DIR`, `stateSubdir()` |
| `shared/yaml-store.ts` / `shared/json-store.ts` | the two write helpers (§9 as-built) |
| `modules/domains/domains.service.ts` + `domain.types.ts` | `domains.yaml` (+ one-time `domains.json` adoption) |
| `modules/audit/audit.service.ts` | `audits.json` + `runs.json` (`persistResult`, `writeChain`, `RUNS_KEPT_PER_DOMAIN`), run REST |
| `modules/audit/checks/blacklist/store.ts` + `zones.ts` | `blacklists/<name>/…`, `blacklist_zones.yaml`, bundled registry |
| `modules/auth/auth-frontend.ts` (+ `@auth/backend`) | `.auth_session_secret`, `users/<email>/sessions/` |
| `shared/logging.ts` + `rolling-file-writer.ts` | `log.log`, `error.err`, rotation |
| `modules/audit/audit-scheduler.service.ts` + `code/deploy/launchd/trigger-scheduler.mjs` | scheduling (nothing persisted — §14 · scheduled_checks) |
| frontend `src/api/{domains,audit,blacklists}.ts` | every UI read/write path (TanStack Query hooks) |
| frontend `src/config/left_bar.ts` | raw import of `pm/left_bar.yaml` |
