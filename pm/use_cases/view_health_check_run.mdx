# Use case: view one health-check run

This walkthrough shows what happens when the user **clicks a row on the Dashboard**: they land on a
detail page for **one health-check run** (one domain, one day) and read it in full — the run summary,
every sub-check finding grouped by the six categories, and the exact fix for each problem. It builds
on the sibling specs and does not repeat them:

* [Overview](../overview.mdx) — the product charter and the list of checks performed.
* [Engineering](../engineering.mdx) — the audit engine, the `Finding` / `AuditResult` shape, the
  `GET /api/audits/:auditId` endpoint, and the periodic scheduler that diffs runs.
* [Dashboard](../dashboard.mdx) — the TanStack Table whose rows are runs; this page is where a row
  click leads.
* [Spam & content checks](../spam_checks.mdx) — the full catalog of sub-checks within the six
  categories shown here.
* [Audit a domain](./audit_a_domain.mdx) — the base add-domain / run-audit loop that produces a run.
* [Domain on a blacklist](./domain_on_blacklist.mdx) — the delisting flow linked from Blacklist
  findings on this page.

## 1. Persona and goal

| | |
| --- | --- |
| **Persona** | A deliverability owner (signed in as `bryan@whitehatengineering.com`). |
| **Goal** | Open one day's health-check run for `whitehatengineering.com` and read every finding, grouped by category, with the exact fix for each problem. |
| **Precondition** | The domain has at least one persisted run (`~/.email_delivery_hero/audits/<domainId>/<auditId>.json`) so a Dashboard row exists to click. |

---

## 2. Getting here

1. The user is on the **Dashboard** (`http://localhost:9311/` → Dashboard), a TanStack Table where
   each **row is one health-check run for one day**, with a **Date** column plus **6 category
   columns**: **SPF**, **DKIM**, **DMARC**, **Blacklists**, **DNS & Infrastructure**, and
   **Spam & Content**. Each cell is color-coded — dark green = healthy, dark red = critical,
   amber = some problems, light gray = never run (see [Dashboard](../dashboard.mdx)).
2. They **click a row** (a given day's run for a domain). The frontend navigates to the run detail
   route:

   ```
   /domains/:id/audits/:auditId
   ```

   (also reachable from the **Audits** left-bar view). The `:auditId` is the `id` of the persisted
   `AuditResult`; `:id` is the domain id.
3. The page loads the run by calling **`GET /api/audits/:auditId`** (TanStack Query). The backend
   reads `~/.email_delivery_hero/audits/<domainId>/<auditId>.json` and returns the full
   `AuditResult { id, domain, startedAt, finishedAt, findings, score, status }`. If the id is
   unknown the API returns `404` and the page shows a "run not found" state with a **Back to
   dashboard** link.

---

## 3. The run summary header

The top of the page is a summary card for this single run, mirroring the Dashboard cell colors so
the detail matches the row the user just clicked:

* **Date / timestamp** — the run's `startedAt` (and `finishedAt` / duration), e.g.
  "Tue 2026-06-30, 04:00 PST · completed in 3.1s".
* **Domain(s)** — the `domain` this run covered (`whitehatengineering.com`), with its registered
  sending IPs and DKIM selectors shown as context.
* **Overall score + status badge** — the numeric `score` and the `status` badge (the worst severity
  present: `ok` / `warning` / `critical`).
* **Six category chips** — one chip each for **SPF**, **DKIM**, **DMARC**, **Blacklists**,
  **DNS & Infrastructure**, **Spam & Content**, colored to match the Dashboard cell for that
  category on this run (dark green / dark red / amber / light gray). Clicking a chip scrolls to that
  category's section below.
* **Open-problem count** — the number of non-`ok` findings still present in this run (e.g.
  "2 open problems"), so the user knows the size of the work at a glance.

---

## 4. Findings grouped by category

Below the header, the run's `findings[]` are grouped into the **six categories**. Each category is a
collapsible section with its own status chip and a table of **every sub-check** that ran (see the
full catalog in [Spam & content checks](../spam_checks.mdx)). Each row shows:

**sub-check | severity | detail (what was observed) | remediation (the concrete fix)**

Healthy sub-checks render as green **ok** rows with no fix needed; problems render **amber**
(`warning`) or **red** (`critical`) with the concrete remediation string shown prominently.

An example run for `whitehatengineering.com`:

### SPF

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| SPF record present | **ok** | One record: `v=spf1 include:_spf.google.com ~all` | — |
| Single SPF record | **ok** | Exactly one `v=spf1` TXT at the apex | — |
| DNS-lookup limit (≤10) | **ok** | 4 mechanism lookups resolved | — |
| `all` qualifier | **warning** | Uses `~all` (softfail) | Consider `-all` (hardfail) once all senders are enumerated to stop spoofers cleanly. |

### DKIM

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| Selector `google` | **critical** | No TXT at `google._domainkey.whitehatengineering.com` | Enable DKIM in Google Admin and publish the provided `google._domainkey` TXT public key. |
| Key parses (`v=DKIM1; k=rsa; p=…`) | **critical** | No key to parse — record missing | Publish the DKIM public key TXT; re-check once DNS propagates. |

### DMARC

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| DMARC record present | **ok** | `v=DMARC1; p=none; rua=mailto:dmarc@whitehatengineering.com` | — |
| Policy strength | **warning** | `p=none` (monitor only — no enforcement) | Strengthen to `v=DMARC1; p=quarantine; rua=mailto:dmarc@whitehatengineering.com`. |
| Aggregate reporting (`rua`) | **ok** | `rua` mailbox present | — |

### Blacklists

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| Spamhaus ZEN | **ok** | `24.113.0.203.zen.spamhaus.org` did not resolve — not listed | — |
| Barracuda | **ok** | Not listed | — |
| SpamCop | **ok** | Not listed | — |

### DNS & Infrastructure

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| MX records | **ok** | `1 aspmx.l.google.com`, `5 alt1.aspmx.l.google.com` — reachable | — |
| Reverse DNS / PTR | **warning** | Sending IP `203.0.113.24` has PTR `mail.whitehatengineering.com` but forward A does not confirm | Add/repair forward A so PTR and A agree (FCrDNS). |
| Nameserver reachability | **ok** | 2 NS resolvable and authoritative | — |

### Spam & Content

| Sub-check | Severity | Detail (observed) | Remediation (how to fix) |
| --- | --- | --- | --- |
| Content spam score | **ok** | Sample message scored 0.8 (well under threshold) | — |
| List-Unsubscribe header | **warning** | Missing `List-Unsubscribe` / one-click header | Add `List-Unsubscribe` and `List-Unsubscribe-Post: List-Unsubscribe=One-Click` to bulk mail. |
| TLS / MTA-STS | **ok** | STARTTLS offered; MTA-STS policy present | — |

The user works top to bottom, reading each non-`ok` row's **remediation** — that "how to fix" line
is the whole point of the page.

## 5. Actions

The page exposes these actions:

* **Run checks again** — re-runs this domain's audit (`POST /api/domains/:id/audit`), producing a
  **new run** (a new `AuditResult` / a new Dashboard row for today). When it completes the UI
  navigates to the fresh run so the user can confirm fixes flipped to `ok`. See
  [Audit a domain](./audit_a_domain.mdx).
* **Back to dashboard** — returns to the Dashboard table.
* **Delisting links** — any **Blacklists** finding that is `critical` (an IP listed on a DNSBL)
  links to the delisting flow with the blocklist-specific removal steps — see
  [Domain on a blacklist](./domain_on_blacklist.mdx).
* **Copy the fix** — every non-`ok` finding's `remediation` has a **copy-to-clipboard** button; for
  record-level fixes (SPF/DKIM/DMARC TXT) it copies the **exact TXT record string** so the user can
  paste it straight into their DNS provider without retyping.

---

## 6. New vs resolved (regression view)

Because the **periodic scheduler diffs each run against the prior run** (see
[Engineering §8](../engineering.mdx)), this page annotates each finding relative to the **previous
day's run** for the same domain:

* **NEW** — a problem present in this run but absent from the prior run (e.g. the DKIM record was
  deleted since yesterday). Badged so regressions stand out.
* **STILL PRESENT** — a problem carried over from the prior run (unchanged, still open).
* **RESOLVED** — a finding that was a problem in the prior run and is now `ok` in this run (shown as
  a cleared/struck item so the user sees progress).

A small "**vs previous run (2026-06-29)**" toggle at the top lets the user filter to only NEW,
only RESOLVED, or all findings. If this is the first run for the domain (no prior run to diff),
the regression annotations are omitted and every finding is simply shown at its current severity.

---

## 7. Acceptance criteria

1. Clicking a Dashboard row navigates to `/domains/:id/audits/:auditId` and the page loads that run
   via `GET /api/audits/:auditId`, reading the persisted `AuditResult` JSON; an unknown id yields a
   404 "run not found" state.
2. The summary header shows the run's date/timestamp, domain(s), overall score + status badge, six
   category chips colored to match the Dashboard cells, and the open-problem count.
3. Findings are grouped into the six categories (SPF, DKIM, DMARC, Blacklists, DNS & Infrastructure,
   Spam & Content); each sub-check is a row with severity, observed detail, and — for every non-`ok`
   finding — a concrete remediation string.
4. Every non-`ok` finding shows its fix prominently with a copy-to-clipboard control that copies the
   exact TXT record for record-level fixes.
5. The page offers **Run checks again** (creating a new run), **Back to dashboard**, and delisting
   links for critical Blacklist findings.
6. Each finding is annotated NEW / STILL PRESENT / RESOLVED relative to the previous run for the same
   domain, and this annotation is omitted only when there is no prior run to diff.
