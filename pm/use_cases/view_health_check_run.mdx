# Use case: view one health-check run

This walkthrough shows what happens when the user **clicks a row on the Dashboard**: they land on
the report for **one health-check run** — one domain, one execution of all checks, stamped with a
`startedAt` and `finishedAt` date-time. A domain accumulates **many runs** over time, and this page
lets the user read any one of them in full and hop between them. It builds on the sibling specs and
does not repeat them:

* [Overview](../overview.mdx) — the product charter and the list of checks performed.
* [Engineering](../engineering.mdx) — the audit engine, the finding shape, and the periodic
  scheduler that diffs runs.
* [Storage](../storage.mdx) — the run YAML file this page renders
  (`~/.email_delivery_hero/runs/<domain>/<timestamp>.yaml`); this spec references it and does not
  redefine it.
* [Dashboard](../dashboard.mdx) — the two TanStack Tables whose row clicks lead here, and the
  locked **Run → Test → Sub-test** vocabulary.
* [Spam & content checks](../spam_checks.mdx) — the full catalog of sub-tests within the six
  categories shown here.
* [View one category run](./view_one_category_run.mdx) — the page each category section's chevron
  opens: one category's sub-tests within this run, in full detail.
* [Audit a domain](./audit_a_domain.mdx) — the base add-domain / run-checks loop that produces a run.
* [Domain on a blacklist](./domain_on_blacklist.mdx) — the delisting flow linked from Blacklist
  findings on this page.

## 1. Persona and goal

| | |
| --- | --- |
| **Persona** | A deliverability owner (signed in as `bryan@whitehatengineering.com`). |
| **Goal** | Open one health-check run for `whitehatengineering.com`, read every test category and its sub-tests, get the exact fix for each problem, and see how the run compares to the previous one. |
| **Precondition** | The domain has at least one persisted run (`~/.email_delivery_hero/runs/whitehatengineering.com/2026_07_01_09_14AM.yaml`) so a Dashboard row exists to click. |

---

## 2. Getting here

1. The user is on the **Dashboard** (`http://localhost:4444/` → Dashboard), which shows **two
   tables** (see [Dashboard](../dashboard.mdx)): **Table 1 — Domain health** (one row per monitored
   domain, cells from its newest run) and **Table 2 — Runs** (one row per run, newest first). Both
   share the six locked test columns — **SPF**, **DKIM**, **DMARC**, **Blacklists**,
   **DNS & Infrastructure**, **Spam & Content** — color-coded dark green = healthy, dark red =
   critical, amber = some problems, light gray = never run.
2. They **click anywhere on a row**:
   * **Table 1 (Domain health)** row click → the domain's **newest** run report, via the
     newest-run alias route **`/domains/:domainId`**.
   * **Table 2 (Runs)** row click (or its **›** chevron) → **that specific run's** report:

     ```
     /domains/:domainId/runs/:runId
     ```

   * Clicking a **category cell** instead opens the
     [view-one-category-run page](./view_one_category_run.mdx) for that category — not this page.
     Row controls (play, kebab, chevron) capture their own clicks.
3. The page loads the run by calling **`GET /api/audit/runs/:runId`** (TanStack Query). The backend
   reads the run's **YAML file** — one file per run at
   `~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml`, e.g.
   `~/.email_delivery_hero/runs/whitehatengineering.com/2026_07_01_09_14AM.yaml` (see
   [Storage](../storage.mdx)) — and returns the `run:` metadata block (`run_id`, `domain`,
   `started_at`, `finished_at`, `score`, `status`) plus the six top-level category keys (`spf`,
   `dkim`, `dmarc`, `blacklists`, `dns_infra`, `spam_content`), each carrying its structured
   observation, `tool_runs:`, and `tests:` (sub-test rows). If the id is unknown the API returns
   `404` and the page shows a "run not found" state with a **Back to dashboard** link. The
   newest-run alias `/domains/:domainId` resolves the domain's most recent run
   (`GET /api/audit/runs?domainId=` → first item) and renders the same page.

## 4. The run summary header

The top of the page is a summary card for this single run, mirroring the Dashboard cell colors so
the detail matches the row the user just clicked:

* **Domain** — the `domain` this run covered (`whitehatengineering.com`), with its registered
  sending IPs and DKIM selectors shown as context.
* **Run timestamp** — the run's `started_at` and `finished_at` with the computed duration, e.g.
  "Started Tue 2026-07-01 09:14 AM · finished 09:14 AM · completed in 3.1s". This is the same
  moment encoded in the run's YAML filename (`2026_07_01_09_14AM.yaml`).
* **Overall score + status badge** — the numeric `score` and the `status` badge (the worst severity
  present: healthy / warning / critical).
* **Six category chips** — one chip each for **SPF**, **DKIM**, **DMARC**, **Blacklists**,
  **DNS & Infrastructure**, **Spam & Content**, colored to match the Dashboard cell for that
  category on this run (dark green / dark red / amber / light gray). Clicking a chip scrolls to
  that category's section below (§5); the chip's small **›** chevron opens the category's full
  page instead (same target as the section's **Open ›**).
* **Open-problem count** — the number of non-pass sub-tests still present in this run (e.g.
  "3 open problems"), so the user knows the size of the work at a glance.

### 4.1 The run switcher

Because a domain has **many runs**, the header carries a run switcher:

* **‹ prev / next ›** — steps to the chronologically previous / next run **for this domain**
  (URL changes to that run's `/domains/:domainId/runs/:runId`). "next ›" is disabled on the newest
  run; "‹ prev" is disabled on the oldest kept run.
* **Runs dropdown** — the current run's timestamp rendered as a dropdown listing this domain's
  runs newest-first (fed by `GET /api/audit/runs?domainId=`), each entry showing its
  `started_at` and status color dot; picking one navigates to it.
* **NEWEST badge** — shown when the run being viewed is the domain's most recent run (always the
  case when arriving via the `/domains/:domainId` alias).

## 5. The body — the six test categories

The body of the page is **the list of the six test categories** (the locked Level-2 "tests" of the
Run → Test → Sub-test hierarchy). Each category renders as a **section card**, in the locked order
SPF → DKIM → DMARC → Blacklists → DNS & Infrastructure → Spam & Content, built from that
category's top-level key in the run YAML (`spf`, `dkim`, `dmarc`, `blacklists`, `dns_infra`,
`spam_content`). Each card shows:

* **Category name + status chip** — colored per the locked mapping (dark green / dark red / amber /
  light gray), identical to the header chip and the Dashboard cell.
* **Sub-test pass/fail counts** — e.g. "2 of 3 pass" (from the category's `tests:` rows).
* **The top problem** — the single worst open sub-test's title (e.g. "selector `google`: no TXT
  published"), so a scan down the page reads like a triage list. Omitted when all pass.
* **Open › chevron** — opens the **view-one-category-run page** for this category of **this run**:

  ```
  /domains/:domainId/runs/:runId/<slug>
  ```

  with the locked slugs `spf`, `dkim`, `dmarc`, `blacklists`, `dns`, `spam-content` (newest-run
  alias: `/domains/:domainId/<slug>`). That page —
  [view one category run](./view_one_category_run.mdx) — carries the **full** sub-test detail,
  the category-specific panels (record panels, include trees, per-source-IP tables …), and the
  tool-run transparency strip.
* **Sub-test preview rows** — the category's `tests:` rows rendered compactly, one line per
  sub-test:

  **status icon | sub-test id + title | observed detail | remediation (the concrete fix)**

  Passing sub-tests render as green **✓** rows with no fix needed; problems render **amber ⚠**
  (`warn`) or **red ✗** (`fail`) with the concrete remediation string shown prominently and a
  **copy-the-fix** button (§6). This is a **preview** — expanded evidence, category panels, and
  "How we measured" live on the category page behind **Open ›**.

An example run for `whitehatengineering.com` (previews abridged):

The user works top to bottom, reading each non-passing row's **remediation** — that "how to fix"
line is the whole point of the page — and presses **Open ›** wherever they need the full evidence.

**Findings sourced from ingested reports.** Some sub-tests come not from a live DNS lookup but from
the **DMARC aggregate / TLS-RPT report emails** the domain received (see
[Report-email ingestion](../emails.mdx)). These render in-line in the same categories —
`dmarc.report_*` and `dmarc.real_pass_rate` under **DMARC**, `infra.tls_rpt_reports_ingested`
under **DNS & Infrastructure** — each tagged with a small **"from reports"** chip; the full
per-source-IP table (source IP, count, disposition, SPF/DKIM eval + alignment, header-from,
envelope) lives on the category page behind **Open ›**. They obey the same six-category roll-up
and copy-the-fix rules as every other sub-test; the dedicated **Reports** view is where the full
per-domain report picture lives.

## 6. Actions

The page exposes these actions:

* **Run checks again** — re-runs this domain's checks (`POST /api/audit/run/:domainId`), producing
  a **new run** — a new YAML file under `~/.email_delivery_hero/runs/<domain>/` and a new top row
  in the Dashboard's Runs table. When it completes the UI navigates to the fresh run so the user
  can confirm fixes flipped to pass. See [Audit a domain](./audit_a_domain.mdx).
* **Back to dashboard** — returns to the Dashboard.
* **Run switcher** — prev/next and the runs dropdown (§4.1) move between this domain's runs.
* **Delisting links** — any **Blacklists** sub-test that fails (an IP listed on a DNSBL) links to
  the delisting flow with the blocklist-specific removal steps — see
  [Domain on a blacklist](./domain_on_blacklist.mdx).
* **Copy the fix** — every non-passing sub-test's remediation has a **copy-to-clipboard** button;
  for record-level fixes (SPF/DKIM/DMARC TXT) it copies the **exact TXT record string** so the user
  can paste it straight into their DNS provider without retyping.
* **Open ›** — per category section, opens that category's
  [view-one-category-run page](./view_one_category_run.mdx) for this run.

## 7. New vs resolved (regression view)

Because the **periodic scheduler diffs each run against the prior run** (see
[Engineering §8](../engineering.mdx)), this page annotates each sub-test relative to the
**previous run** for the same domain (the preceding YAML file in the domain's runs directory):

* **NEW** — a problem present in this run but absent from the prior run (e.g. the DKIM record was
  deleted since the last run). Badged so regressions stand out.
* **STILL PRESENT** — a problem carried over from the prior run (unchanged, still open).
* **RESOLVED** — a sub-test that was a problem in the prior run and now passes in this run (shown
  as a cleared/struck item so the user sees progress).

A small "**vs previous run (2026-06-30 03:00 AM)**" toggle at the top lets the user filter to only
NEW, only RESOLVED, or all sub-tests. If this is the first run for the domain (no prior run to
diff), the regression annotations are omitted and every sub-test is simply shown at its current
severity.

## 8. Acceptance criteria

1. Clicking a Dashboard **Runs** row (or its chevron) navigates to `/domains/:domainId/runs/:runId`;
   clicking a **Domain health** row navigates to the newest-run alias `/domains/:domainId`. The
   page loads the run via `GET /api/audit/runs/:runId`, whose data comes from the run's YAML file
   at `~/.email_delivery_hero/runs/<domain>/<timestamp>.yaml`; an unknown id yields a 404 "run not
   found" state.
2. The summary header shows the domain, the run's `started_at` / `finished_at` / duration, the
   overall score + status badge, six category chips colored to match the Dashboard cells, and the
   open-problem count.
3. The header's **run switcher** offers ‹ prev / next ›, a newest-first runs dropdown for this
   domain, and a NEWEST badge on the most recent run; navigation swaps the `:runId` in the URL.
4. The body lists the **six categories** in locked order; each section card shows the category's
   status chip, its sub-test pass/fail counts, its top open problem, preview sub-test rows
   (result icon, observed detail, remediation), and an **Open ›** chevron that navigates to
   `/domains/:domainId/runs/:runId/<slug>` (slugs: `spf`, `dkim`, `dmarc`, `blacklists`, `dns`,
   `spam-content`).
5. Every non-passing sub-test shows its fix prominently with a copy-to-clipboard control that
   copies the exact TXT record for record-level fixes; report-sourced sub-tests carry the
   "from reports" chip.
6. The page offers **Run checks again** (creating a new run and navigating to it), **Back to
   dashboard**, and delisting links for failing Blacklist sub-tests.
7. Each sub-test is annotated NEW / STILL PRESENT / RESOLVED relative to the previous run for the
   same domain, and this annotation is omitted only when there is no prior run to diff.
