# Use case: view one category run

This is the **generic spec** for the page that shows **one test category within one run** — the
sub-test results for that category (Level 2 of the locked **Run → Test → Sub-test** hierarchy) as
observed by that single `startedAt`/`finishedAt`-stamped execution. There are six instances of the
page, one per locked category — **SPF, DKIM, DMARC, Blacklists, DNS & Infrastructure,
Spam & Content** — all sharing the frame defined here; each category's own spec defines its extra
panels. It builds on the sibling specs and does not repeat them:

* [Dashboard](../dashboard.mdx) — the two tables whose **category-cell clicks** lead here, and the
  locked vocabulary, colors, and routes.
* [View one health check run](./view_health_check_run.mdx) — the run report whose category-section
  **Open ›** chevron leads here; the frame this page slots under.
* [Storage](../storage.mdx) — the run YAML file whose category key this page renders.
* [Engineering](../engineering.mdx) — the audit engine, finding shape, and scheduler.
* The six category specs, which own the sub-test catalogs and the category-specific panels this
  page hosts: [SPF](../checks/spf.mdx) · [DKIM](../checks/dkim.mdx) ·
  [DMARC](../checks/dmarc.mdx) · [Blacklists](../checks/blacklists.mdx) ·
  [DNS & Infrastructure](../checks/dns.mdx) · [Spam & Content](../checks/spam_content.mdx).

## 2. Getting here

Four ways in, all landing on the same page:

1. **Dashboard Table 1 (Domain health) — category cell click.** Clicking a category **cell** on a
   domain's row (rather than the row itself) opens this page for that category on the domain's
   **newest run**, via the newest-run alias route **`/domains/:domainId/<slug>`**.
2. **Dashboard Table 2 (Runs) — category cell click.** Clicking a category cell on a run's row
   opens this page for that category on **that specific run**:

   ```
   /domains/:domainId/runs/:runId/<slug>
   ```

3. **Run report — category section chevron.** On the
   [run report](./view_health_check_run.mdx), each of the six category sections carries an
   **Open ›** chevron (and each header category chip a small **›**) that opens this page for that
   category of the run being viewed.
4. **Direct URL.** The routes above are bookmarkable/shareable. The newest-run alias re-resolves to
   the domain's most recent run on every visit.

The **slugs are locked**: `spf`, `dkim`, `dmarc`, `blacklists`, `dns`, `spam-content` — mapping
1:1 to the six categories and to the YAML category keys `spf`, `dkim`, `dmarc`, `blacklists`,
`dns_infra`, `spam_content`.

## 4. The header

* **Domain + category name** — e.g. "whitehatengineering.com · DKIM".
* **Category status chip** — the category's roll-up status for this run, in the locked colors
  (dark green healthy / dark red critical / amber warning / light gray never-run), identical to
  the Dashboard cell the user clicked.
* **Run timestamp** — this run's `started_at` / `finished_at` / duration, so the user always knows
  **which** run they are reading (a domain has many).
* **Run switcher — category-sticky.** ‹ prev / next › steps plus a newest-first runs dropdown for
  this domain (fed by `GET /api/audit/runs?domainId=`), exactly as on the run report — but
  navigation **stays on this category across runs**: pressing ‹ prev on
  `/domains/:d/runs/:r2/dkim` goes to `/domains/:d/runs/:r1/dkim`. This is how a user watches one
  category evolve run over run. A **NEWEST** badge marks the most recent run.
* **Re-run button** — runs this domain's checks again (`POST /api/audit/run/:domainId`, a full
  run — runs are atomic per domain); on completion the page navigates to the new run's page **for
  this same category**, so the user immediately sees whether their fix flipped the sub-tests.
* **Back link** — "‹ Back to run report" returns to this run's
  [run report](./view_health_check_run.mdx) (`/domains/:domainId/runs/:runId`), scrolled to this
  category's section.

---

## 5. The sub-test results table

The heart of the page: **one row per sub-test** of this category in this run, rendered from the
category's `tests:` rows in the run YAML. Sort order is fixed fail → warn → info → pass (never
alphabetical). Each row:

* **Result icon** — ✗ red (fail) / ⚠ amber (warn) / ⓘ slate (info) / ✓ green (pass).
* **Sub-test id + title** — the id in small caps (e.g. `dkim.selector_google`) with the
  plain-language title.
* **Observed value / evidence** — what the check actually saw, monospace for DNS names, records,
  and IPs (e.g. the raw TXT answer, the DNSBL response code, the header value).
* **Expandable accordion** (`[v]`) — opens the full detail: *observed* (the exact evidence), *why
  it matters* (one sentence), and the **Fix** block — the concrete remediation with a
  **copy-to-clipboard** button. Copy buttons always copy the **publishable value** (the exact TXT
  record / header line / DNS entry), never prose. NEW / STILL PRESENT / RESOLVED regression badges
  and "from reports" chips carry over from the run report unchanged.

A counts strip above the table ("0 passed · 2 failed") matches the pass/fail counts shown on the
run report's category section and the Dashboard cell metric.

## 7. Tool-run transparency — "How we measured"

An expandable strip (collapsed by default) listing the category's **`tool_runs:`** from the run
YAML — every external tool invocation the checker made for this category in this run. One row per
tool run:

**`$ command` (monospace, copyable) | duration | exit code**

e.g. `$ dig +short TXT google._domainkey.whitehatengineering.com · 0.09s · exit 0`. A non-zero
exit code renders the row amber. This lets a skeptical operator reproduce every measurement by
hand in a terminal.

---

## 8. Empty, never-run, loading, and 404 states

* **Never run** — the domain exists but has no runs (arrival via the newest-run alias): gray
  header chip, "No runs yet — run checks to see [category] results" with a **Run checks** CTA.
* **Category absent from run** — the run YAML predates a category or the checker was skipped: the
  sub-test table is replaced by "This run did not measure [category]" with the Re-run CTA.
* **A failing result is NOT empty** — e.g. "no DKIM record found" renders the fail rows with their
  concrete publish-this remediations; emptiness refers only to absent data, never to bad results.
* **Loading** — skeleton header + skeleton table rows sized to the table columns; no layout shift.
* **404** — unknown `:runId`, unknown `:domainId`, or an unrecognized slug: "not found" state with
  a **Back to dashboard** link.

## 9. Data source

Everything on this page comes from **one category key inside one run's YAML file** (see
[Storage](../storage.mdx)): the backend serves the run via **`GET /api/audit/runs/:runId`**, the
frontend selects the slug's category key — `spf`, `dkim`, `dmarc`, `blacklists`, `dns_infra`, or
`spam_content` — and renders its **structured observation** (feeding the category-specific
panels), its **`tool_runs:`** (feeding §7), and its **`tests:`** rows (feeding §5). The run
switcher and newest-run alias resolve through `GET /api/audit/runs?domainId=`. No
category-specific endpoint exists: the run file is the single source of truth, which is what makes
prev/next comparison across runs cheap and exact.

Implemented in code:
The per-category pages live under `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/pages/domains/` — e.g. `SpfPage.tsx` (`SpfPage`), `DkimPage.tsx` (`DkimPage`), `DmarcPage.tsx`, `DnsPage.tsx` — plus `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/pages/blacklists/BlacklistsPage.tsx`; they share the sub-test table `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/components/TestResultsTable.tsx` — `TestResultsTable`. The slug↔category routing is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/lib/categories.ts` — function `techPageRoute()`. Data comes from `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/audit.controller.ts` — class `AuditController`, method `getRun()` (`GET /api/audit/runs/:runId`).

## 10. Acceptance criteria

1. Clicking a **category cell** on Dashboard Table 1 opens `/domains/:domainId/<slug>` (this
   category on the domain's newest run); on Table 2 it opens
   `/domains/:domainId/runs/:runId/<slug>` (this category on that run). The run report's category
   **Open ›** chevron opens the same route for the run being viewed. Slugs are exactly `spf`,
   `dkim`, `dmarc`, `blacklists`, `dns`, `spam-content`.
2. The header shows the domain, category name, the category's status chip in the locked colors,
   and the run's `started_at` / `finished_at` / duration.
3. The run switcher (‹ prev / next ›, runs dropdown, NEWEST badge) navigates between this
   domain's runs while **staying on this category**; Re-run creates a fresh run and lands on this
   category of the new run; the back link returns to this run's report.
4. The sub-test table renders one row per sub-test from the category's `tests:` rows — result
   icon, sub-test id + title, observed evidence — sorted fail → warn → info → pass, each row
   expandable to the full detail and a concrete **Fix** with copy-to-clipboard that copies the
   exact publishable value.
5. Category-specific panels render per that category's own spec (§6 table); this page provides
   only the shared frame.
6. The "How we measured" strip lists every `tool_runs:` entry for this category in this run —
   copyable command, duration, exit code.
7. Empty / never-run / category-absent / loading / 404 states render per §8; a failing check
   result always renders as results with fixes, never as an empty state.
8. All data comes from the category key inside the run YAML via `GET /api/audit/runs/:runId`
   (list/alias resolution via `GET /api/audit/runs?domainId=`); no category-specific endpoint.
