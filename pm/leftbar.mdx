# EmailDeliveryHero — Left Bar

The **left bar** is a **narrow** (`w-44`, 176px) fixed navigation column on every screen — kept as
narrow as the nav labels allow so the Dashboard (the widest screen) gets the space. There are
**two** left bars —
the **app bar** (main navigation) and the **settings bar** (under `/settings/*`) — both data-driven
from **`pm/left_bar.yaml`**, which the frontend imports raw (`src/config/left_bar.ts`, via Vite
`?raw`). There is no second code copy of the nav content.

Division of labor:

* **`pm/left_bar.yaml`** — the **machine source of truth** for the *content*: items, labels, icons,
  routes, order, gates, the header label, the account-menu items.
* **This file** — the **human spec** for everything around that content: the wordmark treatment,
  the color rules, states, layout, and acceptance criteria. Where prose here and the yaml disagree
  on content, the yaml wins; on styling, this file wins.

Siblings: [Overview](./overview.mdx) (product charter), [UI](./ui.mdx) (app frame),
[Dashboard](./dashboard.mdx) (the screen to the right of this bar),
[Security](./security.mdx) (optional login / the `default` user).

---

## 1. No wordmark (top of the bar)

The app bar has **no wordmark and no header band** — the nav list starts at the top of the column
(small top padding only). The "Email Delivery Hero" wordmark was **removed** from the left bar so
the bar can be as narrow as possible and hand the width back to the Dashboard.

* The app bar renders **no** product-name header, no icon mark, and no top border band.
* The product name still appears as the **`BrandHeader`** on the Dashboard screen itself (see
  [UI §3](./ui.mdx)) — that is a separate surface and is **unchanged**; only the left-bar copy is gone.
* `pm/left_bar.yaml` still carries `header.label` for the **settings** bar's back-header ("Settings")
  and for the (now unused) app header label; the frontend simply no longer renders an app-bar header.

---

## 2. Color rules (LOCKED)

The left bar's text is **black**. The complete rule set:

| What | Was | Now |
| --- | --- | --- |
| Nav item labels + icons | medium grey (`text-slate-700`) | **black** (`text-black`) |
| Settings group headers (e.g. "GENERAL") | medium grey (`--edh-muted`) | **black** (`text-black`) |
| Account slot name / email lines | name dark, email medium grey (`--edh-muted`) | **both black** (`text-black`) |
| Back-to-app link (settings bar) | medium grey | **black** |
| Active nav item | green text on green-tint background | **stays green** |
| "Sign in" accent in the account slot | green | **stays green** |
| Hover background (`hover:bg-slate-100`) | light grey fill | unchanged (it is a fill, not text) |

The principle, verbatim: **everything green stays green; everything that was medium grey becomes
black.** Icons inherit their item's text color, so they follow the same rule automatically. Black
on white passes WCAG AAA; the green active state keeps its tint background for shape.

## 3. The app bar (main navigation)

A flat, ordered list starting at the top of the bar; the account slot pinned to the bottom. Content from
`pm/left_bar.yaml` (`Location: app`):

| Order | Item | Icon (lucide) | Route | Purpose |
| --- | --- | --- | --- | --- |
| 1 | **Dashboard** | `LayoutDashboard` | `/` | The two-table health view — domain health grid + run history ([Dashboard](./dashboard.mdx)). |
| 2 | **Domains** | `Globe` | `/domains` | Add and manage the email-sending domains you monitor. |
| 3 | **Audits** | `ShieldCheck` | `/audits` | Deliverability audit results — SPF, DKIM, DMARC, MX, and blacklist findings. |
| 4 | **Blacklists** | `Ban` | `/blacklists` | DNSBL status for your sending IPs, with delisting steps. |
| 5 | **Reports** | `Mailbox` | `/reports` | The report library — the newest health-check reports (one per run, from the run history); planned second section: ingested DMARC (`rua`) + TLS-RPT report emails ([emails.mdx](./emails.mdx)). |
| 6 | **Settings** | `Settings` | `/settings` | Account, scheduling, and app configuration. |

States:

* **Active** — the item whose route matches the current path (`/` exact; others by prefix):
  green text + `--edh-primary`/10 tint background, `font-medium`.
* **Inactive** — black text (§2), transparent background.
* **Hover** — `bg-slate-100` fill; text color unchanged.
* Each item shows its yaml `description` as the hover `title` tooltip.

`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/components/layout/Sidebar.tsx` — component `Sidebar` (renders both bar variants);
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/config/left_bar.ts` — `appNavItems` / `settingsGroups` (parse `pm/left_bar.yaml`);
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/frontend/src/components/layout/NavIcon.tsx` — component `NavIcon`.

## 4. The settings bar

Rendered instead of the app bar under `/settings/*`. Content from `pm/left_bar.yaml`
(`Location: settings`):

* **Header** — a back affordance: `ArrowLeft` icon + **"Settings"**, routing back to `/`. Black
  text (§2).
* **Groups** (group headers uppercase, black):
  * **General** → **Account** (`/settings`, the default section) · **Scheduling**
    (`/settings/scheduling` — how often periodic re-audits run).
  * **Administration** → **Admin** (`/settings/admin`, gated `role:admin`).
* A group whose items are all gated out does not render its header.

---

## 5. The account slot (bottom of the bar)

Pinned to the bottom of both bars, always visible ([UI §1.5](./ui.mdx),
[Security §3.2](./security.mdx)):

* **Logged out** → a single **"Sign in"** button (green accent — stays green), no subtitle. The
  former "Using default settings" caption was **removed** to keep the slot compact. No screen ever
  forces sign-in.
* **Signed in** → avatar + **name** + **email**, both text lines **black** (§2). Clicking opens an
  **upward-expanding** popup menu (items from the yaml `footer`): **Settings**, **Admin** (only
  with `role:admin`), **Log out**. Log out returns to the `default` user — it never ejects the
  person from the app.

---

## 6. Layout & behavior

* Narrow fixed width **176px** (`w-44`) — as narrow as the nav labels allow — full height, white
  background, right border `--edh-border`; the nav list scrolls independently if it overflows.
* The bar **variant is chosen by route**: settings bar under `/settings/*`, app bar everywhere
  else.
* Permission gates come from OpenAuthFederated claims (e.g. `role:admin`); a gated item is
  **hidden**, not disabled, when the claim is absent. The bar renders for everyone — signed in or
  the `default` user.
* Icons resolve by lucide name in `NavIcon.tsx`; an unknown icon name renders no icon rather than
  crashing.

---

## 7. Acceptance criteria

1. The app bar has **no wordmark and no header band** — the nav list starts at the top. The bar is
   `w-44` (176px) wide.
2. All left-bar text is **black**: nav labels, icons, settings group headers, the back link, and
   the account-slot text. Nothing in the bar renders in medium grey.
3. Green elements keep their green: the active nav item (text + tint background) and the Sign-in
   accent.
4. Both bars render their content **from `pm/left_bar.yaml`** — items, order, icons, routes,
   labels, and gates; no hard-coded copies in components.
5. The app bar lists exactly: Dashboard, Domains, Audits, Blacklists, Reports, Settings — in that
   order; the settings bar shows the back header and the General/Administration groups.
6. Active state matches by route (exact `/`, prefix otherwise); gated items are hidden without the
   claim; hover shows the light-grey fill without changing text color.
7. The account slot sits at the bottom, behaves per §5, and its popup expands upward.
