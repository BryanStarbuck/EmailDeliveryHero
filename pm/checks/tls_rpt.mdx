# TLS-RPT (SMTP TLS Reporting)

TLS-RPT (SMTP TLS Reporting, RFC 8460) is a DNS-published opt-in that lets a domain owner receive
aggregate reports about the success and failure of TLS negotiation on inbound SMTP. A remote sending
MTA that supports TLS reporting reads a TXT record at `_smtp._tls.<domain>`, and when it later
encounters a STARTTLS failure, an MTA-STS policy violation, or a DANE/TLSA validation error while
delivering to your domain, it batches those events into a daily JSON report and sends it to the
address you named. Without TLS-RPT you are blind: a receiver silently downgrades to cleartext or
bounces mail on a broken policy and you never learn why deliverability dropped. This check verifies
that the record exists, is syntactically valid per RFC 8460, and names a well-formed reporting
endpoint — the DNS-observable half of the story. Actually *ingesting* the reports (standing up a
mailbox or HTTPS endpoint and parsing the JSON) is a separate, future capability and is called out
explicitly below.

## 1. What it checks

TLS-RPT is defined by **RFC 8460 (SMTP TLS Reporting)**. The domain publishes a single TXT record at
the well-known name `_smtp._tls.<domain>`. Its two mandatory-ish tags are:

- `v=TLSRPTv1` — version, MUST be the first tag, value fixed.
- `rua=` — one or more reporting URIs, comma-separated, each either `mailto:<addr>` or
  `https://<url>`. This is the destination for the daily aggregate reports.

Example valid record:

```
_smtp._tls.example.com.  IN TXT  "v=TLSRPTv1; rua=mailto:tls-reports@example.com"
```

Receivers care because TLS-RPT is the feedback loop for the entire outbound-encryption stack. When a
big receiver (Google, Microsoft, etc.) supports TLS reporting and your domain publishes MTA-STS or
DANE, that receiver enforces your policy — and if enforcement *fails* (expired MTA-STS policy, a
certificate name mismatch, a stale TLSA record after a key roll, a receiver that only offers
cleartext), the receiver records it and mails you a report. TLS-RPT itself does not enforce anything
and does not directly change whether a given message is accepted; it is diagnostic. But its absence
is a real operational gap: the common real-world failure mode is a silent MTA-STS or DANE
misconfiguration that quietly downgrades or bounces a slice of your mail for days before anyone
notices. A degraded-but-not-dead TLS posture erodes inbox placement (some receivers weight transport
security) and, worse, can cause hard delivery failures under `enforce` policies that you cannot
diagnose without these reports. Publishing TLS-RPT costs one DNS record and turns those silent
failures into a dated, structured feed.

## 2. Sub-checks

| Sub-check (checkId) | What it verifies | How measured | Typical critical / warning finding | Concrete remediation |
|---|---|---|---|---|
| `infra.tls_rpt_present` | A TXT record exists at `_smtp._tls.<domain>` | `resolveTxt('_smtp._tls.' + domain)` returns ≥1 record containing `v=TLSRPTv1` | warning: no TLS-RPT record found (you are blind to TLS failures) | Publish `_smtp._tls.example.com. TXT "v=TLSRPTv1; rua=mailto:tls-reports@example.com"` |
| `infra.tls_rpt_syntax` | Record is well-formed: starts `v=TLSRPTv1`, valid `tag=value` pairs, `;`-separated | Parse the joined string; first tag must be `v=TLSRPTv1`; every tag matches `^[a-z]+=` | critical: record present but `v=` missing/not first → whole record ignored by receivers | Rewrite record so it begins exactly `v=TLSRPTv1;` then `rua=...` |
| `infra.tls_rpt_version` | Version token is exactly `TLSRPTv1` | Exact-match the `v=` value | critical: `v=TLSRPTv0` / typo → record invalid | Set `v=TLSRPTv1` |
| `infra.tls_rpt_rua_present` | At least one `rua=` endpoint is declared | Presence of a `rua` tag with non-empty value | critical: record has no `rua` → receivers have nowhere to send reports; record is useless | Add `rua=mailto:tls-reports@example.com` |
| `infra.tls_rpt_rua` | Every `rua` URI is a well-formed `mailto:` or `https:` URI | Split `rua` on `,`; validate each against mailto/https URI grammar | warning: malformed URI (e.g. `mail:` typo, bare address, `http:` not `https:`) | Use `rua=mailto:addr@example.com` or `rua=https://reports.example.com/tlsrpt` (HTTPS, not HTTP) |
| `infra.tls_rpt_rua_scheme` | Only `mailto:`/`https:` schemes are used (no `http:`, `ftp:`) | Scheme prefix check per URI | warning: unsupported scheme; receivers skip it | Replace with `mailto:` or `https:` |
| `infra.tls_rpt_single` | Exactly one TLS-RPT TXT record is published | Count TXT strings at the name containing `v=TLSRPTv1` | critical: multiple TLSRPTv1 records → RFC 8460 says treat as no policy | Delete the duplicate; keep exactly one `v=TLSRPTv1` TXT |
| `infra.tls_rpt_no_extra_txt` | Non-TLSRPT TXT junk isn't polluting the name | Warn if TXT strings exist at `_smtp._tls` that are not TLSRPTv1 | info: stray TXT at reporting name | Remove unrelated TXT records from `_smtp._tls.<domain>` |
| `infra.tls_rpt_unknown_tags` | No unknown/typo'd tags (only `v`, `rua` defined by RFC 8460) | Flag tags outside {`v`,`rua`} | info: unknown tag ignored by receivers (possible typo, e.g. `ruf=`) | Remove non-standard tags; TLS-RPT has no `ruf`/`pct` |
| `infra.tls_rpt_endpoint_reachable` | (Advisory) mailto domain resolves + has MX, or https host resolves | For `mailto:` extract domain → `resolveMx`; for `https:` → `resolve4/6` of host | warning: `rua` mailbox domain has no MX → reports will bounce | Point `rua` at a domain with valid MX, or fix the reporting mailbox's DNS |
| `infra.tls_rpt_rua_external_authorized` | (Advisory) cross-org `rua` (3rd-party report processor) is intended | Compare `rua` domain to audited domain; note when external | info: reports go to a third party (`rua=mailto:...@report-vendor.com`) | Confirm the external processor is intended; RFC 8460 allows external rua without an authorization record |
| `infra.tls_rpt_ttl_sane` | (Advisory) record TTL isn't pathologically low/high | Read TTL (future: via `dig`, not exposed by node:dns) | info: TTL 60s causes excess lookups; 7d slows fixes | Set TTL 3600–86400 |
| `infra.tls_rpt_reports_ingested` | **(SHIPPED — fed by the [../emails.mdx](../emails.mdx) ingestion pipeline)** Actual JSON reports are being received/parsed; failure volume + trend | `deriveTlsRptFindings` aggregates the ingested reports for the configured window (`aggregateTlsRpt`): total ok/failed sessions, reporters, policy types, `failure-details[]` result types | warning: any `failure-details` volume (`starttls-not-supported`/`certificate-host-mismatch`/`validation-failure`/…); info at zero failures (healthy baseline) or when nothing is ingested yet / ingestion is disabled | Fix the reported layer (the §9.5 result-type → fix table): MX cert, STARTTLS, MTA-STS policy, or TLSA record |
| `infra.tls_rpt_no_policy_found` | **(SHIPPED, report-derived)** A reporter saw `policy-type: no-policy-found` while another reporter saw your `sts`/`tlsa` policy in the same window — the policy is not consistently visible | `deriveTlsRptFindings` compares `policy-type` values across the window's ingested report rows | warning: policy invisible to some reporters (DNS propagation, split answers, broken policy fetch) | Verify the `_mta-sts.<domain>` TXT record and the HTTPS policy file (or the TLSA record) resolve from everywhere |

## 3. How it is measured

**First-round detection (pure `node:dns/promises`, deterministic):**

1. **Lookup.** Call `resolveTxt('_smtp._tls.' + ctx.domain)`. `resolveTxt` returns `string[][]`
   (each record is an array of character-strings that must be concatenated with no separator — a long
   record split across 255-byte chunks). Join each inner array, then trim.
2. **Filter to TLS-RPT.** Keep only joined strings whose (case-insensitive on tag, but value
   `TLSRPTv1` is case-sensitive per spec) first tag is `v=TLSRPTv1`. Ignore SPF/other TXT that could
   never legitimately appear here.
3. **Presence & singleton.** Zero TLSRPTv1 strings → `infra.tls_rpt_present` = warning, stop. Two or
   more → `infra.tls_rpt_single` = critical (RFC 8460 §3.1: if more than one record exists, treat as
   though no record is published).
4. **Tokenize.** Split the single record on `;`, trim, split each on the first `=` into tag/value.
   Enforce: first non-empty tag is `v`, value exactly `TLSRPTv1` (`infra.tls_rpt_version`,
   `infra.tls_rpt_syntax`). Collect all tags; anything not in {`v`,`rua`} → `infra.tls_rpt_unknown_tags`.
5. **rua parse.** If no `rua` → critical (`infra.tls_rpt_rua_present`). Else split the `rua` value on
   `,`, trim each URI, and validate:
   - `mailto:` — must have a non-empty address parseable as `local@domain`.
   - `https:` — must parse as a URL with `https:` scheme and a host.
   - `http:`, `mailto` without colon, bare addresses → `infra.tls_rpt_rua`/`_scheme` warning.
   RFC 8460 also permits a size limit suffix on `mailto`/`https` URIs (`!10m`); accept and record it.
6. **Endpoint advisory (still DNS-only).** For each `mailto:` URI, extract the domain and
   `resolveMx` it; empty MX → `infra.tls_rpt_endpoint_reachable` warning. For each `https:` URI,
   `resolve4`/`resolve6` the host; NXDOMAIN → warning. Flag external-org rua as info
   (`infra.tls_rpt_rua_external_authorized`).

**Edge cases:** DNS `ENODATA`/`ENOTFOUND` on the label = "not present" (warning, not an error);
`SERVFAIL`/timeout = surface as an `info` "could not resolve, retry" rather than asserting absence.
Multi-chunk TXT must be reassembled before parsing. Case: tags are case-insensitive but the
`TLSRPTv1` version value is case-sensitive. A record present but with only `v=TLSRPTv1;` and no `rua`
is *syntactically parseable but operationally useless* → critical, because receivers have nowhere to
report.

**Caching/rate-limits:** one TXT + at most a few MX/A lookups per domain per run; negligible. Respect
DNS TTL via the resolver; no external rate limits in first round. The `_endpoint_reachable` MX/A
probes are still DNS and stay first-round; anything requiring SMTP connection or HTTPS `POST`
(delivering a test report) is future.

**Severity mapping:** missing record → warning (a gap, not an active break). Malformed/duplicate/
missing-rua record → critical (present-but-broken is worse than absent: it signals intent that
receivers silently discard). Advisory reachability and stylistic issues → warning/info.
`infra.tls_rpt_reports_ingested` severities derive from parsed report content and are future.

**Future ingestion detection:** stand up a mailbox (e.g. `tls-reports@`) or an HTTPS collector,
receive the daily `application/tlsrpt+gzip` reports, gunzip, parse the JSON (`organization-name`,
`date-range`, `policies[]` with `policy-type` of `sts`/`tlsa`/`no-policy-found`, and
`summary.total-successful-session-count`/`total-failure-session-count`, plus `failure-details[]`
with `result-type` like `starttls-not-supported`, `certificate-host-mismatch`,
`validation-failure`). Store per-day counts, compute a trend, and correlate spikes to the MTA-STS /
DANE / TLS-transport checks.

## 4. User interface

TLS-RPT contributes to the **DNS & Infrastructure** dashboard column alongside MTA-STS, DANE, reverse
DNS, and DNSSEC. Its worst sub-check severity feeds the category cell roll-up (a missing record makes
it at most amber; a broken/duplicate record can push it red per the roll-up rule that any critical =
dark red). `info`-only findings (unknown tag, external rua) never turn the cell amber.

**Audit-detail view.** Findings render as grouped rows under a "TLS-RPT (SMTP TLS Reporting)"
sub-header inside the DNS & Infrastructure section, using the shared React 19 + shadcn/ui finding
row: a severity `Badge` (`ok`/`info`/`warning`/`critical`), the sub-check title, an observed-detail
line (the raw record + parsed rua endpoints), the remediation string, and a copy-to-clipboard button
(lucide-react `Copy`) that yields the exact DNS record to publish.

```
DNS & Infrastructure ▸ TLS-RPT                                    [ ⚠ warning ]
┌────────────────────────────────────────────────────────────────────────────┐
│ ⚠  No TLS-RPT record                _smtp._tls.example.com → NXDOMAIN        │
│    Fix: publish TXT "v=TLSRPTv1; rua=mailto:tls-reports@example.com"  [Copy] │
│ ✔  Record is a singleton            1 TLSRPTv1 record found                  │
│ ⓘ  Reports go to external org       rua=mailto:tls@vendor.com                │
│ ⋯  Report ingestion (future)        not yet collecting JSON reports         │
└────────────────────────────────────────────────────────────────────────────┘
```

**Per-domain config inputs.** None required for the first-round DNS checks (the record name is
derived from the domain). A future admin-only panel captures the ingestion mailbox/HTTPS endpoint and
optional third-party report-processor credentials. The generated record string is shown read-only
with a copy button so the user can paste it into DNS.

**States.** Empty/never-run → light-gray cell, "Not yet audited". Loading → skeleton row while the
TXT lookup runs. `infra.tls_rpt_reports_ingested` is now a **live** row with four states: muted
"ingestion disabled" (admin master switch off — see [../emails.mdx](../emails.mdx) §8), info "no
TLS reports ingested yet" with a CTA to the Reports page's **Ingest now**, info "Inbound TLS
healthy — N ok / 0 failed" (baseline), or warning "Inbound TLS failures reported (N session(s))"
with the failing result types. Admin-only settings (report endpoint config) live behind the
OpenAuthFederated-gated settings route.

## 5. Information schema (database schema)

Reference the shared core tables (owned by engineering, do not redefine): `domains`, `audit_runs`,
and `findings` (each parsed sub-check emits a row into `findings` with `category='infra'` and
`check_id='infra.tls_rpt_*'`). This check adds one structured result table and one future table.

```sql
-- Structured observation from one TLS-RPT run (first-round).
CREATE TABLE tls_rpt_check_results (
    id              BIGSERIAL   PRIMARY KEY,
    audit_run_id    BIGINT      NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id       BIGINT      NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    present         BOOLEAN     NOT NULL,               -- record exists at _smtp._tls.<domain>
    record_count    SMALLINT    NOT NULL DEFAULT 0,     -- # of TLSRPTv1 TXT strings (singleton check)
    raw_record      TEXT,                               -- reassembled TXT string as published
    version_ok      BOOLEAN     NOT NULL DEFAULT FALSE, -- starts v=TLSRPTv1
    rua_endpoints   JSONB       NOT NULL DEFAULT '[]',  -- [{ "uri":"mailto:...", "scheme":"mailto", "domain":"example.com", "size_limit":null, "reachable":true, "external":false }]
    valid           BOOLEAN     NOT NULL DEFAULT FALSE, -- syntactically valid AND has ≥1 well-formed rua
    unknown_tags    JSONB       NOT NULL DEFAULT '[]',  -- ["ruf", ...]
    checked_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_tls_rpt_results_run    ON tls_rpt_check_results(audit_run_id);
CREATE INDEX idx_tls_rpt_results_domain ON tls_rpt_check_results(domain_id, checked_at DESC);
CREATE UNIQUE INDEX uq_tls_rpt_results_run_domain ON tls_rpt_check_results(audit_run_id, domain_id);

-- FUTURE: parsed daily aggregate reports (needs a mailbox/HTTPS endpoint + parser).
CREATE TABLE tls_rpt_reports (
    id                BIGSERIAL   PRIMARY KEY,
    domain_id         BIGINT      NOT NULL REFERENCES domains(id) ON DELETE CASCADE,
    report_date       DATE        NOT NULL,             -- date-range start from the JSON report
    reporter_org      TEXT,                             -- organization-name (e.g. "google.com")
    policy_type       TEXT        NOT NULL,             -- 'sts' | 'tlsa' | 'no-policy-found'
    success_count     BIGINT      NOT NULL DEFAULT 0,   -- total-successful-session-count
    failure_count     BIGINT      NOT NULL DEFAULT 0,   -- total-failure-session-count
    failure_details   JSONB       NOT NULL DEFAULT '[]',-- [{ "result-type":"certificate-host-mismatch", "count":12, ... }]
    ingested_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_tls_rpt_reports_domain_date ON tls_rpt_reports(domain_id, report_date DESC);
CREATE INDEX idx_tls_rpt_reports_policy      ON tls_rpt_reports(policy_type);
```

**Mapping to today's JSON file store.** There is no database yet; storage is the file store under
`~/.email_delivery_hero/`. A `tls_rpt_check_results` row maps to the TLS-RPT slice of the persisted
audit document `audits/<domainId>/<auditId>.json` — specifically a `checks.tls_rpt` object holding
`{ present, recordCount, rawRecord, versionOk, ruaEndpoints[], valid, unknownTags[], checkedAt }`,
plus the emitted `findings[]` entries for each `infra.tls_rpt_*` sub-check (already part of the shared
audit JSON). The future `tls_rpt_reports` table has no file-store analog today; when ingestion ships
it will land as `audits/<domainId>/tls_rpt_reports/<date>.json` (or directly in the DB if the
DB migration happens first). The store module abstracts both, so the file-to-DB reconciliation is a
single-module change: `tls_rpt_check_results` ← the `checks.tls_rpt` object, keyed by `(auditId,
domainId)`.

## 6. When we run the check

TLS-RPT runs as part of every audit — both **on-demand** (`POST /api/domains/:id/audit`, `POST
/api/scheduler/run`) and on the **periodic scheduler** (in-process interval job, cadence from
`config.yaml`, e.g. every 6h). The first-round sub-checks are pure DNS: one `resolveTxt` plus a
handful of `resolveMx`/`resolve4` for the endpoint-reachability advisory — cheap, deterministic, no
external cost or rate limit, so there is no reason to gate them behind a slower cadence. Honor DNS TTL
via the resolver; add light exponential back-off on `SERVFAIL`/timeout and treat transient failures
as `info` rather than flipping `present` to false.

**Regression detection.** The scheduler diffs each run against the previous stored
`tls_rpt_check_results`/`checks.tls_rpt` snapshot. Newly-appearing or worsened findings are flagged as
"new problems": e.g. a record that disappeared (`present` true→false), a rua that became malformed, a
second record appearing (singleton→duplicate = new critical), or an endpoint whose MX/A stopped
resolving. These are the exact silent breakages TLS-RPT exists to prevent, so surfacing them on the
dashboard closes the loop even before report ingestion exists.

`infra.tls_rpt_reports_ingested` **has shipped**: the checker calls `deriveTlsRptFindings` on every
run, which re-derives the finding from the reports already ingested by the
[../emails.mdx](../emails.mdx) pipeline (mailbox/drop-folder → gunzip → parse → store). Running the
audit does **not** fetch new report emails — ingestion has its own trigger (the Reports page's
**Ingest now** and its own cadence, since reports arrive daily) independent of the DNS audit
interval. With the admin master switch off it degrades to a single muted `info` "ingestion
disabled" finding that contributes nothing to the score. Only `infra.tls_rpt_ttl_sane` remains a
registered non-scoring future stub (node:dns exposes no TTL).

## 8. Acceptance criteria

1. For a domain with a valid `v=TLSRPTv1; rua=mailto:...` record, all first-round sub-checks return
   `ok` and `tls_rpt_check_results.valid = true`, `present = true`, `record_count = 1`.
2. A domain with no `_smtp._tls` TXT yields exactly one `infra.tls_rpt_present` finding at `warning`
   whose remediation is the concrete record `v=TLSRPTv1; rua=mailto:tls-reports@<domain>`.
3. A record missing the `v=TLSRPTv1` prefix (or with `v` not first) yields `infra.tls_rpt_syntax`
   `critical`; the record is treated as invalid.
4. Two or more TLSRPTv1 records yield `infra.tls_rpt_single` `critical` and the checker treats the
   domain as having no policy.
5. A record present with no `rua` yields `infra.tls_rpt_rua_present` `critical`.
6. A malformed rua (`http:`, bare address, `mail:` typo) yields `infra.tls_rpt_rua`/`_scheme`
   `warning`, and well-formed URIs are captured in `rua_endpoints` JSONB with scheme + domain.
7. A rua `mailto:` whose domain has no MX yields `infra.tls_rpt_endpoint_reachable` `warning`; a
   third-party rua domain yields `infra.tls_rpt_rua_external_authorized` `info`.
8. Multi-chunk (255-byte-split) TXT records are reassembled before parsing and parse identically to a
   single-string record.
9. Transient DNS errors (`SERVFAIL`/timeout) do NOT set `present = false`; they emit an `info`
   retry-advisory instead.
10. Every non-`ok` finding carries a concrete `remediation` string (the exact record or step), never a
    generic message.
11. `infra.tls_rpt_reports_ingested` is live: with ingestion disabled it returns the muted `info`
    "ingestion disabled" finding (zero score impact); with zero reports it returns the `info`
    "no TLS reports ingested yet" CTA; with ingested reports it is `info` at zero failure sessions
    (healthy baseline, reporters + window named) and `warning` on any `failure-details` volume,
    naming each result type and its layer-specific fix. `infra.tls_rpt_no_policy_found` fires
    `warning` when one reporter saw `no-policy-found` while another saw `sts`/`tlsa` in the window.
12. The scheduler diff flags record disappearance, new duplicates, and rua breakage as new problems
    against the prior `checks.tls_rpt` snapshot.
13. The explainer page `/domains/:domainId/runs/:runId/dns/check/tls-rpt` (newest alias
    `/domains/:domainId/dns/check/tls-rpt`) renders the five locked blocks of
    [dns.mdx §14.2](./dns.mdx) in order, with this spec's interior copy (§9), and the alias
    rewrites itself to the canonical run URL.
14. Block 1 contains the anchored sub-headings `#concept-tls-rpt`, `#concept-rua`, and
    `#concept-reporting-mailbox`; every underlined "TLS-RPT" / "rua (TLS)" / "reporting mailbox"
    term anywhere in the app deep-links to them run-scoped (§10.2).
15. Block 2 shows every `infra.tls_rpt_*` sub-test row from the viewed run (fail rows
    pre-expanded), the raw pane with the reassembled TXT string (or `_smtp._tls.<domain> →
    NXDOMAIN` when absent) with a copy button and `tool_runs[]` traceability, and the §9.3 parsed
    field-by-field table with a per-field ✓/✗/⚠/ⓘ verdict chip; uncaptured fields render "not
    captured this run", never invented data.
16. Block 2's ingested-reports panel renders one row per reporter/day from `aggregateTlsRpt`
    (reporter, date, policy-type, ok/failed sessions, failure result types) with the window and
    totals chips — for the real act3ai.com corpus that is the Microsoft Corporation daily `sts`
    rows with 2-3 ok / 0 failed sessions — and never counts a DMARC attachment (`application/zip`
    from google.com) as a TLS report.
17. Blocks 3 and 4 are conditional on the observed state per the §9.4 copy table and the §9.5 fix
    recipes; every fix step carries a copy button with the exact record/text, never prose-only.
18. Block 5's button posts `POST /api/audit/run/:domainId/check/dns/tls-rpt`, which writes a NEW
    run file tagged `run.scope: dns.tls_rpt`, follows the idle→running→done-with-diff-flash→error
    state machine of [dns.mdx §15.2](./dns.mdx), and is labeled with its exact scope and ~2 s

### 9.1 Block 1 — What this is (concept copy and anchors)

Three short paragraphs, plain language first, standards second. Static content — renders
instantly, independent of run data:

1. **The feedback loop** (`#concept-tls-rpt`). TLS-RPT is how you find out when other mail
   servers *couldn't* speak TLS to yours. One TXT record at `_smtp._tls.<domain>` names an
   address; participating receivers (Google, Microsoft) then mail you one report per day
   describing every TLS session they attempted against your MX — how many succeeded, how many
   failed, and why. It enforces nothing itself; it is the smoke detector for MTA-STS, DANE, and
   STARTTLS (each term deep-links per §10.2).
2. **The reporting address** (`#concept-rua`). The `rua=` tag lists where reports go: one or
   more comma-separated `mailto:` or `https:` URIs, optionally with a `!10m`-style size cap
   (`#concept-size-limit`). Explicitly contrast with DMARC's `rua` — same tag name, different
   record, and TLS-RPT has **no** `ruf`/`pct`/`sp` and needs no external-authorization record.
3. **The reporting mailbox and the daily report** (`#concept-reporting-mailbox`,
   `#concept-report-json`). What actually arrives: a gzip'd JSON attachment
   (`application/tlsrpt+gzip`) with `organization-name`, a one-day `date-range`, and per-policy
   `summary` counts plus `failure-details[]`. Show the real-world shape using the act3ai.com
   corpus: Microsoft's daily report echoing back the domain's own MTA-STS policy
   (`policy-type: sts`, `mode: enforce`, the five Google MX lines) with
   `total-successful-session-count`. This app can *be* that mailbox — the
   [../emails.mdx](../emails.mdx) ingestion pipeline parses these reports into the panel in §9.3.

Anchor contract (fixed by the [dns.mdx §14.6](./dns.mdx) concept map — the first three are the
category-wide destinations; the last two are page-local extras):

| Anchor | Term(s) that link here from anywhere in the app |
|---|---|
| `#concept-tls-rpt` | TLS-RPT, SMTP TLS Reporting, `_smtp._tls` |
| `#concept-rua` | `rua` (TLS), reporting URI, reporting endpoint |

### 9.4 Block 3 — What it means (conditional copy)

| Observed state | Copy direction (plain language) |
|---|---|
| Record missing (`_present` warning) | You are flying blind: if STARTTLS, MTA-STS, or DANE breaks, mail silently downgrades to cleartext or bounces for days before you notice via complaints. TLS-RPT never blocks mail itself — this is a *visibility* gap, not a delivery gate. |
| Record present but broken (`_single`/`_syntax`/`_version`/`_rua_present` critical) | Worse than absent: RFC 8460 makes receivers treat a duplicate/malformed record as *no policy*, so you believe you have a smoke detector and you don't. |
| rua unreachable (`_endpoint_reachable` warning) | Receivers generate the reports and they bounce — same blindness, one hop later. |
| Healthy record, reports arriving, 0 failures | The full loop works: your record, the receiver's reporter, and your mailbox. This baseline is what makes a later regression detectable (§11). |
| Failures reported (`_reports_ingested` warning) | Real senders failed TLS to your MX in the window. Under `mode: enforce` (MTA-STS) or DANE those sessions are *bounced*, not downgraded — this is active mail loss, and the result types name the broken layer. |
| `_no_policy_found` warning | Part of the internet can't see your TLS policy — propagation, split answers, or a broken policy fetch; enforcement is inconsistent. |

Enforcement framing per the category ladder: TLS-RPT is *hardening/visibility* (nobody rejects
mail for lacking it), but the failures it reports are the *rejected now* class.

### 9.5 Block 4 — What you can do (fix recipes)

Numbered steps, every step with a copy button carrying the exact text — never prose-only.
Conditional on state:

1. **No record** → publish (copy button):
   `_smtp._tls.<domain>. 3600 IN TXT "v=TLSRPTv1; rua=mailto:tls-reports@<domain>"`
   plus the provider path (registrar/DNS console → TXT → host `_smtp._tls`).
2. **Duplicate records** → delete all but one; the kept record is offered on the copy button.
3. **Malformed / wrong version / missing rua** → the full corrected record string (the checker's
   `remediation` already carries it verbatim — render it, don't re-generate).
4. **Bad scheme / bare address** → the per-URI rewrite (`mailto:` prefix or `https://` URL).
5. **Mailbox undeliverable** → point `rua` at a domain with MX, or fix the mailbox domain's MX.
6. **Reports not being read** → make this app the reader: Settings → Admin (mailbox/drop-folder,
   [../emails.mdx](../emails.mdx) §8), then Reports → **Ingest now**.
7. **Failure result types** (the code-true fix table — one row per reported `result-type`):

| `result-type` in `failure-details[]` | Fix (copy-button text) |
|---|---|
| `starttls-not-supported` | Ensure every MX offers STARTTLS |
| `certificate-host-mismatch` | Replace the MX certificate so its name matches the MX host |
| `certificate-expired` | Renew the expired MX certificate |
| `validation-failure` | Install a certificate from a trusted CA on the MX |
| `tlsa-invalid` | Repair the TLSA record after the key roll |
| `dnssec-invalid` | Fix the DNSSEC chain for the TLSA record |
| `sts-policy-fetch-error` | Make `https://mta-sts.<domain>/.well-known/mta-sts.txt` reachable |
| `sts-policy-invalid` | Correct the MTA-STS policy file syntax |
| `sts-webpki-invalid` | Fix the certificate on the mta-sts policy host |
| *(unknown type)* | "investigate `<type>`" + link to RFC 8460 §6.6 |

Ladder cross-links lead the block when relevant: MX routing broken → "fix MX first — nobody can
even connect"; failures naming `sts-*` → the [MTA-STS explainer](./mta_sts.mdx); `tlsa-*`/
`dnssec-*` → the [DANE](./dane_tlsa.mdx) / DNSSEC explainers; `starttls-*`/`certificate-*` →
the [STARTTLS & MX certificates](./tls_transport.mdx) explainer.
