# EmailDeliveryHero — DMARC Check

**DMARC** (Domain-based Message Authentication, Reporting & Conformance) is the policy layer that
sits on top of SPF and DKIM. A DMARC record published at `_dmarc.<domain>` tells receivers (a) what
to do with mail that fails **both** SPF-and-DKIM **alignment** (`p=` / `sp=` / `np=`), and (b) where
to send **aggregate** (`rua`) and **failure** (`ruf`) reports so the domain owner gets visibility
into who is sending as them. Without DMARC at an enforcing policy, anyone can spoof your exact
domain in the `From:` header and receivers have no owner-published instruction to reject it — the
single largest lever against direct-domain phishing, and, since the **2024 Gmail/Yahoo bulk-sender
rules**, an effectively mandatory record for any sender pushing volume.

This check belongs to the **DMARC** dashboard category; all sub-check ids use the **`dmarc`**
checkId prefix. In the locked **Run → Test → Sub-test** hierarchy
([../dashboard.mdx](../dashboard.mdx) §1), a **run** is one execution of all checks for one domain
(with `startedAt`/`finishedAt`; a domain accumulates many runs), **DMARC** is one of the six locked
tests (categories) inside every run, and the §2 sub-checks are its **sub-tests**.

## 1. What it checks

**The standard.** DMARC was defined in **RFC 7489** (informational, 2015) and was **obsoleted in
May 2026** by the **DMARCbis** set: **RFC 9989** (core), **RFC 9990** (aggregate reporting), and
**RFC 9991** (failure reporting). `v=DMARC1` records are unchanged on the wire, but the rules this
checker must honor changed:

- **`pct=`, `rf=`, and `ri=` are removed** — receivers may ignore them; owners should delete them.
  Percentage-based staged rollout is no longer part of the model.
- **`np=`** (non-existent-subdomain policy), **`psd=`** (public-suffix-domain flag), and **`t=`**
  (testing flag; `t=y` ≈ the old `pct=0`) are first-class tags.
- The **organizational-domain lookup is now a DNS Tree Walk** (walking up labels looking for
  `_dmarc` records) instead of the Public Suffix List. A subdomain with no record of its own is
  governed by the closest parent record's `sp=`.

**Why receivers care.** SPF and DKIM each answer a narrow question ("is this IP allowed?" / "is this
signature intact?") but neither ties the authenticated identity to the **`From:` header the human
sees**. DMARC closes that gap: it passes only if SPF **or** DKIM passes **and** the passing identity
is **aligned** with the `From:` domain. A published, enforcing DMARC policy is what lets Gmail,
Yahoo, Microsoft, and others reject or quarantine look-alike/spoofed mail on the owner's behalf.

**Common real-world failure modes.** (1) The domain publishes `p=none` and never moves off it —
monitor-only forever, zero protection. (2) A strong `p=reject` at the org domain but `sp=none`,
leaving every subdomain wide open. (3) `rua` pointing at a third-party analytics provider
**without** the required `<domain>._report._dmarc.<provider>` authorization record, so the provider
silently receives nothing. (4) A wildcard TXT record answering the `_dmarc` query with junk, which
some receivers treat as an invalid record.

**How a failure degrades inbox placement.** No DMARC (or `p=none`) means receivers apply only their
own heuristics; spoofed and even legitimate mail rides on weaker signals and is more likely to land
in spam. Under the 2024 Gmail/Yahoo rules, bulk senders **without** any DMARC record can be
rate-limited or rejected outright. Conversely, a broken/malformed record is treated as **no
policy** — a silent regression that removes protection the owner believes they still have.

## 2. Tests we perform (sub-checks)

The comprehensive first-round set for the `dmarc` category. Every test emits a finding — **pass
(`ok`) findings included**, so the DMARC page can render an explicit passed/failed row per test.
Each remediation is the exact record/tag/step to apply.

| Sub-check (finding id) | What it verifies | How measured | Typical critical / warning finding | Concrete remediation |
| --- | --- | --- | --- | --- |
| `dmarc.present` / `dmarc.missing` | A TXT record exists at `_dmarc.<domain>` and begins with `v=DMARC1`; when absent, the **tree walk** checks whether a parent record covers the domain. | `resolveTxt('_dmarc.<domain>')`; join multi-string chunks; test prefix; on miss, walk up parent labels. | **critical:** no record anywhere (`dmarc.missing`); **info:** covered only by a parent record via `sp=`. | Publish TXT at `_dmarc.<domain>`: `v=DMARC1; p=none; rua=mailto:dmarc@<domain>` to start, then tighten. |
| `dmarc.multiple` | Exactly one `v=DMARC1` record (≥2 = the whole domain treated as no-policy by receivers). | Count TXT strings whose joined value starts with `v=DMARC1`. | **critical:** two or more DMARC records. | Delete the extra TXT record(s) so exactly one `v=DMARC1` string remains. |
| `dmarc.syntax` / `dmarc.syntax_p_position` | Record parses: `v=DMARC1` first; `p=` immediately second. | Tokenize on `;`, split `tag=value`, validate order. | **critical:** first tag not `v=DMARC1` → record ignored; **warning:** `p=` not second. | Rewrite so the record begins `v=DMARC1; p=…`. |
| `dmarc.policy` / `dmarc.no_policy` / `dmarc.p_none` / `dmarc.policy_ok` | A valid `p=` is present and **enforcing** (`quarantine` or `reject`). | Read `p=`; classify `none` / `quarantine` / `reject`. | **critical:** missing/invalid `p=`; **warning:** `p=none` (monitor-only). | Move to `p=quarantine`, then `p=reject` once `rua` confirms legit mail passes aligned. |
| `dmarc.testing` | The RFC 9989 `t=` flag is not silently disabling enforcement. | Read `t=`. | **warning:** `t=y` — policy is advisory only, effectively `p=none`. | Remove `t=y` (or set `t=n`) when ready to enforce. |
| `dmarc.subdomain` / `dmarc.subdomain_ok` | Effective `sp=` is at least as strong as `p=`. | Read `sp=` (defaults to `p=`); compare ranks. | **warning:** enforcing `p=` but weaker `sp=` leaves subdomains spoofable; **warning:** invalid `sp=` value. | Set `sp=reject` (or remove `sp=` so subdomains inherit `p=`). |
| `dmarc.np` | `np=` (non-existent-subdomain policy) set on enforcing records. | Read `np=`. | **info:** enforcing `p=` but no `np=` → spoofed made-up subdomains not explicitly rejected. | Add `np=reject` if you never send from non-existent subdomains. |
| `dmarc.alignment` / `dmarc.adkim_strict` / `dmarc.aspf_strict` | Alignment modes valid; strict (`s`) flagged because it breaks subdomain/ESP senders. | Read `adkim`/`aspf`; default `r`. | **warning:** `adkim=s` or `aspf=s` with subdomain/ESP senders → legit mail fails alignment; **warning:** invalid value. | Use relaxed `adkim=r; aspf=r` unless exact-domain matching is a hard requirement. |
| `dmarc.pct` | `pct=` handling — **removed in RFC 9989**. | Read `pct=`; range-check. | **warning:** `pct<100` (partial, non-deterministic enforcement); **info:** `pct=100` present at all. | Remove the `pct` tag; the policy then applies to 100 % of mail. |
| `dmarc.rua` / `dmarc.rua_invalid` / `dmarc.rua_ok` | Aggregate reporting enabled with valid `mailto:` URIs. | Parse `rua=` into a URI list; validate scheme/address. | **warning:** no `rua` → flying blind; **warning:** non-`mailto:` / malformed URI. | Add `rua=mailto:dmarc@<domain>` (or your analytics mailbox). |
| `dmarc.ruf` | Failure/forensic reporting considered (optional, privacy-sensitive). | Parse `ruf=`. | **info:** no `ruf`. | Optionally add `ruf=mailto:…; fo=1`. |
| `dmarc.fo` | `fo=` tokens valid (`0`/`1`/`d`/`s`, colon-combinable); `fo=1` advised when `ruf` set. | Read `fo=`; validate token set. | **info:** invalid tokens, or `ruf` set with default `fo=0` (only both-fail reports). | Set `fo=1` to report when **either** SPF or DKIM fails alignment. |
| `dmarc.ri` / `dmarc.rf` | Legacy `ri=`/`rf=` sane if present (both removed in RFC 9989). | Read + range/value-check. | **info:** non-numeric `ri`, `ri<3600`, or `rf≠afrf`. | Remove the tags (receivers report daily in `afrf` regardless). |
| `dmarc.report_uri_size` | Optional per-URI `!<size>` suffixes (e.g. `mailto:x@y!10m`) well-formed. | Parse the `!size` suffix on each URI. | **info:** malformed size suffix. | Fix or drop the suffix; plain `mailto:` if unsure. |
| `dmarc.external_report_auth` / `dmarc.external_report_auth_ok` | Every `rua`/`ruf` destination on a **foreign** org domain publishes the authorization record. | For each external report domain `R` while auditing `D`: `resolveTxt('D._report._dmarc.R')`, require `v=DMARC1` (wildcard publishers pass automatically). | **critical:** external destination with **no** authorization → reports are silently discarded. | Have `R` publish TXT `<D>._report._dmarc.<R>` = `v=DMARC1` (report providers document this), or use an in-domain mailbox. |
| `dmarc.deprecated_tags` | No obsolete (`pct`/`rf`/`ri`) or unknown/typo'd tags lingering. | Diff observed tags against the RFC 9989 tag set. | **info:** obsolete or unknown tags present. | Remove them at the next DNS edit; keep the record minimal. |
| `dmarc.lookup_failed` | DNS answered at all (SERVFAIL/timeout is transient, not "no record"). | Distinguish NXDOMAIN/NODATA from resolver errors. | **warning:** could not resolve — re-run. | Retry; verify the domain's nameservers respond for `_dmarc`. |
| `dmarc.real_pass_rate` *(future — [../emails.mdx](../emails.mdx))* | Actual SPF/DKIM/DMARC pass rates measured from ingested `rua` XML. | Parse per-source-IP dispositions/alignment from aggregate reports. | **warning:** high volume from unaligned sources before enforcing. | Fix or authorize the failing sources **before** raising policy. |

### How a test run executes the tools

First-round detection itself runs in-process on `node:dns/promises`; those lookups populate
`record:` and `external_report_auth[]` directly and are **not** tool runs. Every **shell-out** to
an external binary during a run, however, is recorded as one entry in **`dmarc.tool_runs[]`** in
the run YAML (§5, [../storage.mdx](../storage.mdx)):
`{tool, command, started_at, duration_ms, exit_code, output_format, parsed, error}`.

Execution rules:

- **Exact argv.** `command` holds the exact command line with **every** input argument substituted
  (the domain, the report file, the resolver IP…) — copy-paste-reproducible from the UI.
- **Prefer JSON.** Every tool that can emit JSON is invoked with its JSON flag
  (`output_format: json`); only kdig is `text`. Stdout is captured whole, `JSON.parse`d when
  `json`, and stored under `parsed` (large outputs pruned to the fields we consume, noted per row
  below). Stderr is captured separately and lands in `error` on failure.
- **One entry per invocation**, retries included. Non-zero exit or timeout keeps the entry with
  `exit_code` set (or `null` on timeout/kill), `parsed: null`, and `error` holding stderr or
  `"timeout after <n>ms"`. `duration_ms` is wall-clock around the spawn.
- **Missing binary** (preflight `command -v <tool>`) → the invocation is skipped entirely — no
  fabricated entry — and surfaced once per run as an `info` finding naming the `brew install` line.
- `started_at` is ISO-UTC; entries appear in invocation order.

The tools a DMARC run shells out to, in order:

| # | Tool | Exact command line (argv) | Output | Timeout | Captured → feeds |
| --- | --- | --- | --- | --- | --- |
| 1 | **doggo** | `doggo _dmarc.<domain> TXT --json` — e.g. `doggo _dmarc.example.com TXT --json` | json | 10 s | Full answer array → `parsed.answers`; the raw TXT evidence shown on every test row; cross-checks the in-process `record.raw_record` / `found_at` (a mismatch = resolver inconsistency, noted in `tests[].detail`). |
| 2 | **doggo** (per external report destination, deduped) | `doggo <domain>._report._dmarc.<report-domain> TXT --json` — e.g. `doggo example.com._report._dmarc.thirdparty.net TXT --json` | json | 10 s | Empty `answers` = unauthorized → `external_report_auth[].authorized` and the `dmarc.external_report_auth` evidence. One invocation (one entry) per distinct external report domain. |
| 3 | **checkdmarc** | `checkdmarc <domain> -f json` — e.g. `checkdmarc example.com -f json` | json | 60 s (it also probes MX/DNSSEC) | The `dmarc` object of its JSON (pruned to `record`, `valid`, `location`, `tags`, `warnings`) → `parsed`; the conformance oracle cross-validating `record.parsed`, `policy`, `sp` effectiveness, obsolete tags, and external-destination authorization. Any disagreement with our checker is appended to the affected `tests[].detail`. |
| 4 | **kdig** (conditional) | `kdig @8.8.8.8 +short TXT _dmarc.<domain>` | text | 10 s | Raw stdout lines → `parsed.lines`. Only invoked when the in-process lookup and doggo disagree, or on `dmarc.lookup_failed` — "does a public resolver see the same record?" evidence. |
| 5 | **mailauth** (optional — only when the user supplies a message file) | `npx mailauth report <path-to-message.eml>` | json | 30 s | `dmarc`/`spf`/`dkim` alignment verdicts (pruned) → `parsed`; message-level evidence for PS-08/PS-13 drill-downs. |
| 6 | **parsedmarc** *(future — rua ingestion)* | `parsedmarc <report.xml.gz>` | json | 120 s | Normalized report JSON → the [../emails.mdx](../emails.mdx) pipeline and `dmarc_aggregate_reports`. Runs inside the ingestion job, whose own `tool_runs` live on that job — **not** in a domain run. |

Rows 1–3 run on every DMARC run; row 4 is conditional; rows 5–6 are optional/future. Each entry's
`parsed` payload is diagnostic **provenance**: the structured truth the checker asserts stays in
`record:` / `external_report_auth[]` / `tests[]`, and `tool_runs[]` proves where it came from.

## 4. How it is measured

**First-round detection (pure `node:dns/promises`).**

1. **Lookup.** `resolveTxt('_dmarc.<domain>')`. `resolveTxt` returns `string[][]` (an array of
   records, each an array of character-strings that must be **concatenated without a separator**
   before parsing). Filter to records whose joined value starts with `v=DMARC1`.
2. **Tree walk.** If the domain has no record of its own, walk up parent labels
   (`_dmarc.<parent>`, stopping at the 2-label domain) per RFC 9989. A parent hit means the domain
   is **covered** by that record's `sp=` — an `info`, not a `critical`; the parent record is then
   evaluated with the effective policy = its subdomain policy.
3. **Single-record guard.** Zero `v=DMARC1` records anywhere → `dmarc.missing` critical. ≥2 →
   `dmarc.multiple` critical (per spec, receivers discard **all** of them).
4. **Tokenize.** Split the joined record on `;`, trim, split each token on the first `=` into
   `tag=value`. Enforce ordering: `v` first and equal to `DMARC1` (critical), `p` second (warning).
5. **Classify policy.** `p ∈ {none, quarantine, reject}`; anything else → critical. `none` →
   warning. `sp` defaults to `p`; `np` defaults to `sp`. `t=y` → warning (testing mode).
6. **Alignment.** `adkim`/`aspf` default to `r`; value `s` flagged as a warning (breaks
   subdomain/ESP senders); anything else invalid.
7. **Reporting tags.** Parse `rua`/`ruf` into URI lists (comma-separated `mailto:`, optional
   `!<size>` suffix). Validate `fo` ∈ {`0`,`1`,`d`,`s`}, `ri` numeric, `rf` = `afrf`, and flag
   `pct`/`rf`/`ri` as obsolete.
8. **External report authorization.** For every `rua`/`ruf` URI whose mail domain is not the
   audited domain (or a subdomain of it), issue
   `resolveTxt('<audited-domain>._report._dmarc.<report-domain>')` and require a `v=DMARC1` value.
   Missing → **critical** (reports silently discarded); resolver error → info "re-run to confirm".

**Edge cases.**
- **CNAME'd `_dmarc`.** Resolvers follow the CNAME transparently; record it in `detail` for clarity.
- **Multi-string TXT (>255 bytes).** Always join character-strings before prefix-matching.
- **Case-insensitivity.** Tag names and policy values are case-insensitive; lower-case before
  comparing, preserve original case in `raw_record`.
- **Whitespace / trailing semicolons.** Tolerate; don't fail syntax on cosmetic spacing.
- **`NXDOMAIN` vs `NODATA`.** Both mean "no DMARC" → `dmarc.missing`; distinguish in `detail`.
- **Wildcard junk.** TXT strings at `_dmarc` that don't start `v=DMARC1` are ignored (receivers do
  the same), but a valid-looking DMARC string at the apex or `dmarc.<domain>` is the misplaced-
  record problem state (PS-03).

**Caching / rate limits.** Pure DNS; negligible cost. `_report._dmarc` probes are one lookup per
distinct external report domain, deduped. No third-party quotas in the first round.

**Severity mapping.** Missing/duplicate/malformed record and missing external-report auth →
**critical**. `p=none`, `t=y`, weak `sp`, strict alignment, `pct<100`, missing/invalid `rua` →
**warning**. Obsolete/cosmetic/advisory items → **info**. `info` alone never turns the DMARC cell
amber.

## 5. Test results (YAML)

The information we extract from the tests. Every run persists as **one YAML file** — the full
run-file spec (the `run:` block, category ordering, retention) is owned by
[../storage.mdx](../storage.mdx), not redefined here — at:

```
~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml
```

e.g. `~/.email_delivery_hero/runs/whitehatengineering.com/2026_07_01_09_14AM.yaml` — `<domain>`
lowercased; the filename is the run's **start** time in the local timezone, 12-hour clock,
zero-padded, with an `AM`/`PM` suffix. The file opens with the `run:` block (`run_id`, `domain`,
`started_at`/`finished_at` in ISO-UTC, `score`, `status`) followed by one top-level key per locked
category: `spf, dkim, dmarc, blacklists, dns_infra, spam_content`. **This spec owns the `dmarc`
key.** Its section holds the worst-severity `status`, the structured observation (`record:`,
produced by the checker as `results.dmarc` — see §10), the shell-out provenance (`tool_runs:`,
§3), the per-sub-test rows (`tests:` — the findings), and the derived `problem_states`. This is
the canonical shape every DMARC UI surface and the regression differ consume:

```yaml
# ~/.email_delivery_hero/runs/example.com/2026_07_01_09_14AM.yaml — ONE file per run
run:                           # owned by ../storage.mdx — shown for context only
  run_id: 9f1c2e7a
  domain: example.com
  started_at: 2026-07-01T16:14:03Z     # ISO-UTC (the filename is the same instant, local tz)
  finished_at: 2026-07-01T16:14:41Z
  score: 68
  status: warning
# spf: … dkim: …               # the five sibling category keys, owned by their own specs
dmarc:
  status: warning              # worst severity across the tests below
  # ---- the parsed observation (results.dmarc) ----
  record:
    query_name: _dmarc.example.com
    record_found: true
    record_count: 1
    found_at: _dmarc.example.com     # a parent name here means tree-walk coverage
    raw_record: "v=DMARC1; p=none; rua=mailto:reports@thirdparty.net; adkim=r; aspf=r"
    parsed:                          # full tag map exactly as published
      v: DMARC1
      p: none
      rua: "mailto:reports@thirdparty.net"
      adkim: r
      aspf: r
    policy: none                     # none | quarantine | reject | null
    subdomain_policy: none           # effective sp (defaults to p)
    np_policy: null
    pct: null                        # legacy tag when present
    adkim: r
    aspf: r
    rua_uris: ["mailto:reports@thirdparty.net"]
    ruf_uris: []
    fo: null
    ri: null
    is_enforcing: false              # policy in (quarantine, reject) and not t=y
    external_reports_authorized: false
    external_report_auth:            # one row per external report destination
      - report_kind: rua
        report_uri: "mailto:reports@thirdparty.net"
        report_domain: thirdparty.net
        auth_name: example.com._report._dmarc.thirdparty.net
        authorized: false
  # ---- the shell-out provenance (one entry per external tool invocation; §3) ----
  tool_runs:
    - tool: doggo
      command: "doggo _dmarc.example.com TXT --json"
      started_at: 2026-07-01T16:14:04Z
      duration_ms: 142
      exit_code: 0
      output_format: json
      parsed:
        answers:
          - name: _dmarc.example.com.
            type: TXT
            answer: '"v=DMARC1; p=none; rua=mailto:reports@thirdparty.net; adkim=r; aspf=r"'
      error: null
    - tool: doggo
      command: "doggo example.com._report._dmarc.thirdparty.net TXT --json"
      started_at: 2026-07-01T16:14:05Z
      duration_ms: 133
      exit_code: 0
      output_format: json
      parsed:
        answers: []              # NXDOMAIN — feeds external_report_auth[].authorized=false
      error: null
    - tool: checkdmarc
      command: "checkdmarc example.com -f json"
      started_at: 2026-07-01T16:14:05Z
      duration_ms: 5210
      exit_code: 0
      output_format: json
      parsed:                    # pruned to the dmarc object we consume
        dmarc:
          valid: true
          location: _dmarc.example.com
          record: "v=DMARC1; p=none; rua=mailto:reports@thirdparty.net; adkim=r; aspf=r"
          warnings:
            - "Unverified external destination: thirdparty.net"
      error: null
  # ---- the per-test rows (the findings, pass and fail alike) ----
  tests:
    - id: dmarc.present
      title: DMARC record found
      result: pass                   # pass | fail | warn | info
      evidence: "v=DMARC1; p=none; rua=mailto:reports@thirdparty.net; adkim=r; aspf=r"
    - id: dmarc.p_none
      title: DMARC policy is monitor-only (p=none)
      result: warn
      detail: p=none collects reports but delivers spoofed mail normally.
      fix: Raise to p=quarantine, then p=reject once rua reports run clean.
    - id: dmarc.external_report_auth
      title: Report destination thirdparty.net is not authorized
      result: fail
      detail: example.com._report._dmarc.thirdparty.net has no v=DMARC1 TXT record.
      dns_value_expected: 'example.com._report._dmarc.thirdparty.net TXT "v=DMARC1"'
      fix: Have thirdparty.net publish the authorization record, or use an in-domain mailbox.
    - id: dmarc.subdomain_ok
      title: Subdomains covered (effective sp=none)
      result: info
  # ---- matched problem states (see §9) ----
  problem_states: [PS-05, PS-10]
```

Field-name mapping: `record.*` keys are 1:1 with the `dmarc_check_results` columns in §10
(`raw_record`, `parsed`, `policy`, `subdomain_policy`, `pct`, `adkim`, `aspf`, `rua_uris`,
`ruf_uris`, `is_enforcing`, `external_reports_authorized`); `tests[]` rows are the `findings`
(`result: pass|fail|warn|info` ⇔ severity `ok|critical|warning|info`); `problem_states` is derived
(§9 mapping table) — never stored separately. `tool_runs[]` is **provenance** and lives only in
the run YAML (no §10 table); each entry's `parsed` payload feeds the `record.*` /
`external_report_auth[]` / `tests[].evidence` fields named in §3's execution table. The domain
name and run timestamps are **not** repeated inside `dmarc:` — the file's `run:` block owns them.

## 6. User interface — the summary surface and the full DMARC page

Two surfaces, following the app-wide pattern in [../ui.mdx](../ui.mdx): a **summary cell** on the
main view, and a **chevron** that opens a **full page dedicated to DMARC** for that domain.

### 6.1 The summary surface (main view)

DMARC is one of the six category columns on the Dashboard. The DMARC cell for a domain is the
**worst severity** among all `dmarc.*` findings in the latest run, and shows the single most
important fact about this technology at a glance:

- **Color** — dark green (enforced & healthy), amber (worst = warning, e.g. `p=none`), dark red
  (any critical, e.g. no record / duplicate / unauthorized report destination), light gray (never
  run). `info`-only findings keep the cell green.
- **Label** — the one-line summary of the most decision-relevant metric: the **policy level**.
  `p=reject` / `p=quarantine` / `p=none` / `No record` / `K of M fail`.
- **Hover** — tooltip with the top open problem's title.
- **Cell click** — clicking the DMARC **cell itself** opens the "view one category run" page (§6.2):
  on **Table 1 (Domain health)** it opens the domain's **newest** run's DMARC page
  (`/domains/:domainId/dmarc`, the newest-run alias); on a **Table 2 (Runs)** row it opens **that
  run's** DMARC page (`/domains/:domainId/runs/:runId/dmarc`). A chevron (›) on the cell's right
  edge (visible on hover, and on the run report's DMARC category chip) routes to the same place.
- **Row click (everywhere else)** — clicking anywhere else in the row opens the **whole run
  report**: `/domains/:domainId/runs/:runId` (Table 2), or its newest alias `/domains/:domainId`
  (Table 1), per [../dashboard.mdx](../dashboard.mdx) §6.

### 6.2 The full DMARC page — one category of one run (`/domains/:domainId/runs/:runId/dmarc`)

One page, purely about the DMARC sub-test results **of one specific run** for one domain — a
"view one category run" page. The canonical route is **run-scoped** —
`/domains/:domainId/runs/:runId/dmarc` — with the **newest-run alias** `/domains/:domainId/dmarc`
(the same page, resolved to the domain's latest run and showing the "newest" indicator). It is
reached two ways:

- **Dashboard DMARC cell click** (§6.1): a Domain-health cell opens the newest run's page (the
  alias route); a Runs-row cell opens that specific run's page.
- **Run report chevron**: the DMARC section/category chip on the run report
  (`/domains/:domainId/runs/:runId`) carries a chevron (›) into this page for the same run.

It must answer, top to bottom: *Which run am I looking at? Am I protected? What exactly did we
observe (the DNS values)? Which tests passed/failed? How do I fix each failure?* Every failed test
shows the relevant debugging values — the DNS name queried, the raw TXT answer, the expected
record, and (when report data exists) source IP addresses. An older run is a **historical
snapshot** — nothing on the page silently swaps to newer data; the run context strip makes the
vintage unmistakable.

Contents, in order (exact layout in §7):

1. **Header** — domain name, "DMARC" title, the category StatusCell (this run's roll-up), a
   **Re-run** button (starts a *new* run for just this domain, then navigates to the new run's
   DMARC page), and a back link to this run's report. Directly beneath: the **run context strip**
   — the run's timestamp (`startedAt`, `YYYY-MM-DD HH:mm`), a **‹ prev / next ›** run pager
   stepping through this domain's runs in `startedAt` order (each step swaps `:runId` in the
   route; ends disabled), and a **"newest"** badge when this run is the domain's latest (always
   shown when the page was reached via the alias route).
2. **Policy ladder** — a three-step `none → quarantine → reject` progress visual with the current
   step highlighted and the recommended next step (from §8) as a CTA.
3. **Parsed record panel** — the raw TXT string (copyable, monospace) above a two-column tag table:
   every published tag (`v`, `p`, `sp`, `np`, `adkim`, `aspf`, `pct`, `rua`, `ruf`, `fo`, `ri`,
   `rf`, `t`) with its value, its effective default when absent (grayed, e.g. `sp → none
   (inherits p)`), and a one-line meaning. Obsolete tags get a strikethrough badge.
4. **Test results table** — one row per §2 test: pass/fail/warn/info icon, test name, the observed
   value (DNS value / URI / policy), and — expanded via row accordion — the full detail, the
   evidence (exact DNS name + TXT answer), the expected value, and the **Fix** block with a
   copy-to-clipboard button carrying the exact record to publish.
5. **Report destinations panel** — one card per `rua`/`ruf` URI: destination mailbox, report
   domain, external? yes/no, authorization DNS name and its observed answer, authorized ✓/✗.
6. **Problem states** — cards for each matched §9 problem state, each with a one-paragraph
   explanation and a **"Learn more ›"** chevron that opens the per-problem drill-down page
   (`/domains/:domainId/runs/:runId/dmarc/:problemId`, newest alias
   `/domains/:domainId/dmarc/:problemId`) — concept, diagnosis, and the step-by-step path forward.
7. **Tool runs** — a collapsed provenance footer listing this run's `dmarc.tool_runs[]` entries
   (§3/§5): tool, the exact command line (copyable — the user can reproduce it in a terminal),
   duration, exit code, and an expandable view of the captured `parsed` output. Failed
   invocations render their `error` in red.
8. **Sources (future)** — once `rua` ingestion lands: a per-source-IP table (IP, PTR, message
   count, SPF/DKIM aligned pass, disposition) feeding PS-11/PS-12.

**Empty / never-run / loading states.** Never-run → gray header cell + "No audit yet — run one"
CTA. Loading → skeleton rows. A no-record result is **not** empty — it renders the `dmarc.missing`
critical with the concrete publish-this remediation and the policy ladder at step zero.

**Per-domain config inputs.** DMARC needs no extra per-domain input (unlike DKIM selectors). The
page offers an optional **"Suggested record" builder**: a small form (desired `p`, rua mailbox,
cover subdomains?) that emits a ready-to-publish TXT string — advisory only, never auto-published.

Written as the UX designer would hand it to engineering. Desktop-first (localhost tool), 1200 px
content max-width, the app-wide left nav (256 px) stays. Vertical order = triage order: verdict
first, evidence second, work items third, education last.

└────────────────────────────────────────────────────────────────────────────┘

Placement rules and rationale:

- **Highest: the verdict — anchored to one run.** The header row carries identity + status; the
  **run context strip** directly beneath it pins the page to one run — the run timestamp, the
  **‹ prev / next ›** pager (same domain, `startedAt` order, keyboard `←`/`→`; ends disabled),
  and the **★ newest** badge on the domain's latest run. Paging swaps `:runId` in the route and
  re-renders every band below from that run's YAML file — the strip is the only element that
  changes meaning between runs, so it sits above everything. The **policy ladder hero** under it
  is the one visual a non-expert needs — where am I on none→quarantine→reject and what's my next
  move. One accent CTA maximum on the page (the ladder's "next step").
- **Second band, two-up:** left = **Published record** (the ground truth users paste elsewhere;
  monospace, copy button top-right, tag table beneath with grayed inherited defaults); right =
  **Report destinations** (the most-missed failure — external authorization — gets its own card
  rather than drowning in the table). On narrow widths the right card stacks below the left.
- **Main table: test results.** A table, not cards — users scan many rows; sort order is fixed
  fail → warn → info → pass (never alphabetical). Each row: status icon (✗ red / ⚠ amber / ✓
  green / ⓘ slate), finding id in small caps, plain-language title, right-aligned chevron `[v]`
  accordion. Expanded row shows *observed* (exact DNS name + raw answer, monospace), *expected*,
  *why it matters* (one sentence), and the **Fix** strip: the exact TXT record/tag edit with
  `[copy fix]`. Copy buttons always copy the publishable value, never prose.
- **Lowest: problem-state cards, then the tool-runs footer.** Problem-state cards are education
  and deep dives; two per row, title + one-line hook + chevron to
  `/domains/:domainId/runs/:runId/dmarc/:problemId`. They repeat information the table already
  flagged on purpose: the table is for operators, the cards for learners. Beneath them, the
  **tool-runs footer** ships collapsed (provenance, not triage): one monospace row per
  `tool_runs[]` entry — exact command, duration, exit code, `[copy]` for the command line, `[v]`
  accordion for the captured `parsed` output/`error`. It is last because it answers "prove it",
  never "what do I do".
- **Type & color:** shadcn/ui primitives (`Card`, `Table`, `Accordion`, `Badge`), lucide icons
  (`ShieldCheck`/`ShieldAlert`/`Info`/`ChevronRight`/`Copy`/`Wrench`). Severity colors identical
  to the Dashboard cells. Monospace (`font-mono text-xs`) for every DNS name, TXT value, and IP.
- **Drill-down pages** (`/domains/:domainId/runs/:runId/dmarc/:problemId`; newest alias
  `/domains/:domainId/dmarc/:problemId`) reuse the same header row **and run context strip** (so
  the user never loses context — the data shown is this run's, not the latest), then: **Concept**
  (what this state means, 2–3 paragraphs), **Your data** (the §9 data fields rendered from this
  run's `dmarc` section), **Diagnose it yourself** (copyable terminal commands from §3), **Path
  forward** (numbered steps with copyable records), and a further-reading footer. Back chevron
  returns to the run's DMARC page anchor.

## 8. Recommendations — how we tell the user to fix DMARC

Every failing test carries its concrete one-line remediation (§2). Above the per-test fixes, the
page computes one **primary recommendation** from the rollout state machine — the ladder CTA:

| Current state | Gate | Primary recommendation |
| --- | --- | --- |
| No record (PS-01) | — | Publish `v=DMARC1; p=none; rua=mailto:dmarc@<domain>` today. Zero risk — it only turns on monitoring. |
| Broken record (PS-02/03/04) | — | Fix the record first; a malformed/duplicate record = no protection at all. |
| `p=none` | `rua` present, no failing tests, (future) aligned pass-rate ≥ ~99 % over 30 days | Raise to `p=quarantine`. Until report data exists, gate on: record healthy + SPF & DKIM categories green + user-confirmed 30-day monitoring. |
| `p=none`, no `rua` | — | Add `rua=` first — never enforce blind. |
| `p=quarantine` | clean for ≥ 30 days at 100 % | Raise to `p=reject`. |
| `p=reject` | — | Tighten the perimeter: explicit `sp=reject`, add `np=reject`, delete obsolete tags, keep `rua` forever. |
| Any level, legit mail bouncing | — | **Rollback rule:** step back exactly one notch, fix alignment (see PS-11), re-monitor ≥ 2 weeks, resume. |

Additional standing recommendations surfaced as `info` when applicable: keep relaxed alignment
unless strict is a hard requirement; ≤ 2 report URIs (the spec only guarantees two); put
`p=reject` DMARC (plus `v=spf1 -all`) on **parked/non-sending domains**; prefer aligned **DKIM**
on every stream (it survives forwarding; SPF does not) before going to `reject`.

## 9. Problem states

The health situations a domain can be in for DMARC. Each maps from finding ids (detection is
automatic), and each has a deep-dive below — and a drill-down UI page (§7). PS-00 is the healthy
goal state; the rest are unhealthy in descending typical severity.

| Id | Problem state | Detected by | Severity |
| --- | --- | --- | --- |
| PS-00 | Enforced and healthy | all tests pass, `policy ∈ {quarantine, reject}` | ok |
| PS-01 | No DMARC record at all | `dmarc.missing` | critical |
| PS-02 | Multiple DMARC records | `dmarc.multiple` | critical |
| PS-03 | Record invisible: wrong host / wildcard junk / CNAME breakage | `dmarc.missing` + heuristics (valid-looking record at apex or `dmarc.<domain>`) | critical |
| PS-04 | Syntax-invalid record | `dmarc.syntax`, `dmarc.no_policy`, `dmarc.policy` | critical |
| PS-05 | Stuck in monitoring (`p=none` forever, or `t=y`) | `dmarc.p_none`, `dmarc.testing` | warning |
| PS-06 | Partial enforcement (legacy `pct<100`) | `dmarc.pct` | warning |
| PS-07 | Subdomain gap (`sp` weaker than `p`; no `np`) | `dmarc.subdomain`, `dmarc.np` | warning |
| PS-08 | Strict alignment breaking legitimate mail | `dmarc.adkim_strict`, `dmarc.aspf_strict` | warning |
| PS-09 | Flying blind (no/dead aggregate reporting) | `dmarc.rua`, `dmarc.rua_invalid` | warning |
| PS-10 | Reports going nowhere (unauthorized external destination) | `dmarc.external_report_auth` | critical |
| PS-11 | Legitimate stream failing alignment *(future: rua data)* | `dmarc.real_pass_rate` | warning |
| PS-12 | Active spoofing observed *(future: rua data)* | `dmarc.report_*` unaligned unknown sources | critical |
| PS-13 | Forwarding fragility (SPF-only authentication) | cross-check: DKIM category unhealthy while DMARC relies on `aspf` | warning |

### PS-00 — Enforced and healthy

The goal state: one valid record, `p=reject` (or `quarantine` en route), subdomains covered,
reports flowing to an authorized destination.
- **Data fields:** `policy`, `subdomain_policy`, `np_policy`, `is_enforcing=true`,
  `external_reports_authorized=true`, `tests[]` all pass.
- **Diagnose/debug:** nothing to fix — `checkdmarc <domain> -f json` should agree (`valid: true`,
  empty warnings).
- **Tools:** `dig +short TXT _dmarc.<domain>` for a sanity glance.
- **More health metrics:** watch the scheduled-run regression diff (policy downgrades, lost
  authorization); future rua ingestion adds the real pass-rate trend.
- **UI:** ladder at step 3, all-green table; drill-down page explains *keeping* it healthy —
  monitoring is permanent because new streams/vendors/key rotations keep happening.

### PS-01 — No DMARC record at all

- **Concept:** anyone can spoof the exact domain; bulk mail is penalized by Gmail/Yahoo rules.
- **Data fields:** `record_found=false`, `record_count=0`, `query_name`, `found_at=null`.
- **Diagnose/debug:** `dig +short TXT _dmarc.<domain>` (empty) → confirm against a public resolver
  `kdig @8.8.8.8 +short TXT _dmarc.<domain>` (rules out local-resolver weirdness) → check the
  parent labels the same way (tree-walk coverage).
- **Tools:** dig/kdig/doggo; `checkdmarc` reports `"error": "A DMARC record does not exist…"`.
- **More health metrics:** none until the record exists.
- **UI:** ladder at step 0; single critical row with the copyable starter record
  `v=DMARC1; p=none; rua=mailto:dmarc@<domain>`; drill-down explains the none→reject journey.

### PS-02 — Multiple DMARC records

- **Concept:** receivers finding ≥2 `v=DMARC1` strings MUST treat DMARC as absent — all protection
  evaporates. Classic after switching DMARC vendors.
- **Data fields:** `record_count≥2`, `raw_record` (all records joined with ` | `).
- **Diagnose/debug:** `dig TXT _dmarc.<domain> +multiline` and count the answers; find which DNS
  provider/console entry created each (often one from the registrar template, one from a vendor).
- **Tools:** dig; `doggo --json` makes the answer count explicit for scripts.
- **More health metrics:** regression diff — flag if the count ever rises above 1 again.
- **UI:** critical row listing every observed string; fix = "delete all but this one", with the
  survivor pre-selected (the one matching the user's report destination, when detectable).

### PS-03 — Record invisible: wrong host / wildcard junk / CNAME breakage

- **Concept:** a valid-looking `v=DMARC1…` string exists — at the apex, at `dmarc.<domain>`
  (missing underscore), at `_dmarc._dmarc.<domain>` (console auto-append) — or a wildcard TXT
  answers `_dmarc` with non-DMARC junk, or `_dmarc` is a CNAME to a dead target. Receivers never
  see a policy.
- **Data fields:** `record_found=false` at `query_name` plus heuristic probes: TXT at apex and
  `dmarc.<domain>` matched against `^v=DMARC1`; CNAME chain of `_dmarc`.
- **Diagnose/debug:** `dig +short TXT <domain> | grep -i dmarc`; `dig +short TXT dmarc.<domain>`;
  `dig +short CNAME _dmarc.<domain>` then resolve the target; query a random label
  (`dig +short TXT zz-random.<domain>`) to expose wildcards.
- **Tools:** dig/doggo; `dnsx` for sweeping many candidate names at once.
- **More health metrics:** once fixed, PS-01 metrics apply.
- **UI:** the drill-down shows a "where we looked" table (name → answer) so the misplacement is
  visually obvious; fix = publish at exactly `_dmarc.<domain>` and delete the misplaced string.

### PS-04 — Syntax-invalid record

- **Concept:** `v=dmarc1` case errors, missing `v=`, `p=monitor`/misspellings, commas or colons for
  separators, smart quotes, duplicated tags — receivers discard the record; the domain silently has
  no policy while the owner believes it does.
- **Data fields:** `raw_record`, `parsed` (tag map — shows what survived parsing), failing test ids.
- **Diagnose/debug:** read `raw_record` character-by-character in the UI's monospace panel (smart
  quotes and stray spaces are visible there); `checkdmarc` names the offending tag.
- **Tools:** `checkdmarc <domain>`; learndmarc.com for an interactive second opinion.
- **More health metrics:** regression diff on `raw_record` — any change to the string is surfaced.
- **UI:** the parsed-record table renders unknown/invalid values with a red badge in place; fix
  block shows a corrected full record assembled from the valid parts.

### PS-05 — Stuck in monitoring (`p=none` forever, or `t=y`)

- **Concept:** monitor-only is the right *first* step and the wrong *permanent* state — reports
  flow but spoofed mail is delivered normally. `t=y` (RFC 9989) is the same trap wearing a new tag.
- **Data fields:** `policy=none` (or `parsed.t=y`), `is_enforcing=false`, `rua_uris`, plus (future)
  time-in-state from run history and the aligned pass-rate.
- **Diagnose/debug:** is anything blocking promotion? Check the SPF and DKIM categories are green;
  check `rua` reports are actually arriving (parsedmarc); confirm every legitimate sender passes
  aligned.
- **Tools:** `checkdmarc` (policy read-back); `parsedmarc` on the report mailbox; `mailauth report
  msg.eml` on a message from each sending system to verify alignment before enforcing.
- **More health metrics:** days at `p=none` (from run history); % aligned pass from rua (future) —
  the promotion gate in §8.
- **UI:** ladder highlights step 1 with the CTA "Raise to quarantine"; drill-down walks the §8
  state machine and its gates, with the exact edited record for the next step.

### PS-06 — Partial enforcement (legacy `pct<100`)

- **Concept:** `pct=25` enforces on a 25 % sample — non-deterministic protection, and RFC 9989
  removed the tag so modern receivers may ignore it entirely. `pct=0` is "stealth none".
- **Data fields:** `pct`, `policy`, `raw_record`.
- **Diagnose/debug:** confirm the value; ask *why* it's partial — usually an abandoned ramp.
- **Tools:** dig; `checkdmarc` flags obsolete `pct` explicitly.
- **More health metrics:** same promotion gates as PS-05; when clean, jump straight to 100 %.
- **UI:** warning row "policy applies to only N % of mail"; fix = remove the tag (copy button holds
  the record minus `pct`).

### PS-07 — Subdomain gap

- **Concept:** `p=reject; sp=none` locks the apex while `anything@foo.<domain>` stays spoofable;
  missing `np=` leaves *non-existent* subdomains (which can never have SPF/DKIM) to the weaker
  `sp`. Attackers actively probe for exactly this.
- **Data fields:** `policy`, `subdomain_policy`, `np_policy`, `found_at` (a subdomain audited via
  tree walk shows which parent record governs it).
- **Diagnose/debug:** `dig +short TXT _dmarc.<domain>` and read `sp=`/`np=`; enumerate known
  sending subdomains and check each for its own record (`dnsx` sweep).
- **Tools:** dig, dnsx; `checkdmarc` warns on ineffective `sp`.
- **More health metrics:** count of subdomains with their own records vs inherited; regression diff
  on `sp`/`np` changes.
- **UI:** warning row with an inheritance diagram (org record → subdomain effective policy);
  fix = `sp=reject` (and `np=reject`) appended to the existing record.

### PS-08 — Strict alignment breaking legitimate mail

- **Concept:** `adkim=s`/`aspf=s` require the DKIM `d=`/Return-Path domain to *exactly* equal the
  `From:` domain — mail signed as `mail.<domain>` or sent by an ESP stops passing. Almost never
  required; the failure only shows up when policy is enforced, i.e. at the worst time.
- **Data fields:** `adkim`, `aspf`, plus the DKIM category's selector domains for cross-reference.
- **Diagnose/debug:** send a message from each system through `mailauth report msg.eml` and read
  the alignment verdicts; check rua rows (future) for org-domain-only matches failing.
- **Tools:** mailauth; swaks to generate live test sends; learndmarc.com interactively.
- **More health metrics:** aligned pass-rate split by mechanism (future rua data).
- **UI:** warning rows per strict tag; fix = `adkim=r; aspf=r` unless the drill-down's "do you
  really need strict?" checklist says otherwise.

### PS-09 — Flying blind (no/dead aggregate reporting)

- **Concept:** without working `rua` you can't see spoofers, can't see which legitimate streams
  fail, and can't safely progress policy. Dead variants: no tag, malformed URI, mailbox domain
  with no MX, vendor mailbox you stopped paying for.
- **Data fields:** `rua_uris` (empty or malformed), (future) MX/A probe of each rua domain, days
  since last ingested report (once [../emails.mdx](../emails.mdx) ingestion is live).
- **Diagnose/debug:** validate the URI shape; `dig +short MX <rua-domain>`; send a probe message
  and watch for a report within 48 h.
- **Tools:** dig; parsedmarc watching the mailbox confirms end-to-end arrival.
- **More health metrics:** reports-received-per-week trend; "rua set but zero reports in N days"
  becomes its own future finding.
- **UI:** warning row; fix = add `rua=mailto:dmarc@<domain>`; drill-down explains mailbox vs
  analytics-provider trade-off.

### PS-10 — Reports going nowhere (unauthorized external destination)

- **Concept:** `rua` points at a different org domain, and that domain does not publish
  `<your-domain>._report._dmarc.<their-domain>` = `v=DMARC1` — compliant report generators
  **silently skip you**. The classic "we set up DMARC months ago and never got a single report".
- **Data fields:** `external_report_auth[]` — per destination: `report_kind`, `report_uri`,
  `report_domain`, `auth_name`, `authorized`.
- **Diagnose/debug:** `dig +short TXT <domain>._report._dmarc.<report-domain>` — expect
  `"v=DMARC1"`; also try the wildcard the big providers publish:
  `dig +short TXT '*._report._dmarc.<report-domain>'`.
- **Tools:** dig/doggo; `checkdmarc` verifies this natively (its `unverified external destination`
  warning).
- **More health metrics:** re-probe each scheduled run; a previously authorized destination losing
  its record is a regression finding.
- **UI:** critical row per destination with the exact `auth_name` queried and its observed answer
  (NXDOMAIN); fix = the exact TXT record the *destination* must publish (copy button), plus the
  fallback "or use mailto:dmarc@<your-domain>".

### PS-11 — Legitimate stream failing alignment *(future — rua data)*

- **Concept:** an ESP (SendGrid, Mailchimp, …) passes raw SPF/DKIM for *its* domain but not
  aligned to yours — fine at `p=none`, spam-foldered at `quarantine`, bounced at `reject`. The
  telltale in reports: `auth_results` pass + `policy_evaluated` fail.
- **Data fields:** per-source rows from `dmarc_aggregate_reports`: `source_ip`, PTR, volume,
  `spf_result` vs `spf_aligned`, `dkim_result` vs `dkim_aligned`, `disposition`.
- **Diagnose/debug:** identify the source (PTR + volume pattern), then per-vendor: enable custom
  Return-Path (SPF alignment) and custom DKIM signing (`d=`your domain) via the vendor's CNAMEs.
- **Tools:** parsedmarc (normalize reports); `mailauth report` on a message from that stream;
  vendor dashboards.
- **More health metrics:** aligned pass-rate per source over time; volume-weighted readiness score
  gating the §8 promotion.
- **UI:** sources table row expands to a per-vendor fix card (recognized PTRs get vendor-specific
  instructions); drill-down explains alignment with the raw-vs-aligned distinction.

### PS-12 — Active spoofing observed *(future — rua data)*

- **Concept:** report rows from IPs that are not yours failing both mechanisms = someone is
  sending as your domain right now. At `p=none` that mail is being delivered.
- **Data fields:** unknown-source rows: `source_ip`, PTR, geo, volume, `disposition` (what
  receivers actually did).
- **Diagnose/debug:** confirm the IP isn't a forgotten legitimate system (PS-11 first!); then this
  is the strongest argument to finish the §8 ladder.
- **Tools:** parsedmarc; whois/PTR on the source IP; dmarc-cat for eyeballing a single report.
- **More health metrics:** spoof volume trend; % of spoofed mail receivers rejected (rises as
  policy tightens — the payoff chart).
- **UI:** red sources-table rows; drill-down shows the payoff chart and drives the enforcement CTA.

### PS-13 — Forwarding fragility (SPF-only authentication)

- **Concept:** forwarders change the connecting IP (SPF fails downstream); mailing lists rewrite
  content (DKIM fails). A domain whose DMARC pass depends **only on SPF** will have legitimate
  forwarded mail bounce at `p=reject`. Aligned DKIM on every stream is the fix — it survives
  forwarding.
- **Data fields:** cross-category: DKIM category health + (future) rua rows where SPF fails but
  DKIM passes aligned (benign forwarders) vs both-fail from forwarder IPs.
- **Diagnose/debug:** confirm every sending system DKIM-signs with an aligned `d=`; test by
  forwarding a message through a third-party mailbox and running `mailauth report` on the result.
- **Tools:** mailauth; swaks; the DKIM check ([./dkim.mdx](./dkim.mdx)).
- **More health metrics:** share of DMARC passes carried by DKIM vs SPF (future rua data) — DKIM
  share should approach 100 % before `p=reject`.
- **UI:** advisory card on the DMARC page when DKIM is unhealthy but DMARC is enforcing; drill-down
  explains forwarding mechanics with a diagram.

## 10. Information schema (database schema)

Reference the shared core tables (owned by engineering, **not** redefined here): `domains`,
`sending_ips`, `dkim_selectors`, `audit_runs`, `findings`. Human-readable DMARC findings land in the
shared `findings` table with `category='dmarc'` and `check_id` = the sub-check id. The structured,
machine-parsed observation for each run lives in the check-specific tables below.

```sql
-- One row per audit run per domain: the parsed DMARC record and derived flags.
CREATE TABLE dmarc_check_results (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    audit_run_id     BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id        BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    record_found     BOOLEAN NOT NULL,
    record_count     INT     NOT NULL DEFAULT 0,   -- >1 => dmarc.multiple critical
    found_at         TEXT,                          -- _dmarc.<domain> or the covering parent name
    raw_record       TEXT,                          -- exact joined TXT string as observed
    parsed           JSONB,                         -- full tag map { v,p,sp,np,psd,t,adkim,aspf,pct,rua,ruf,fo,ri,rf }
    policy           TEXT,                          -- p=  : none | quarantine | reject | null
    subdomain_policy TEXT,                          -- sp= (effective; defaults to policy)
    np_policy        TEXT,                          -- np= (RFC 9989)
    pct              INT,                            -- pct= (obsolete) or null
    adkim            CHAR(1),                        -- 'r' | 's'
    aspf             CHAR(1),                        -- 'r' | 's'
    rua_uris         JSONB,                          -- ["mailto:dmarc@example.com", ...]
    ruf_uris         JSONB,                          -- ["mailto:forensics@example.com", ...]
    fo               TEXT,                           -- fo= tokens, e.g. '1'
    ri               INT,                            -- report interval seconds (obsolete tag)
    is_enforcing     BOOLEAN NOT NULL DEFAULT FALSE, -- policy in (quarantine, reject) and not t=y
    external_reports_authorized BOOLEAN,             -- all external rua/ruf have _report._dmarc auth
    checked_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX uq_dmarc_results_run_domain ON dmarc_check_results(audit_run_id, domain_id);
CREATE INDEX ix_dmarc_results_domain            ON dmarc_check_results(domain_id);
CREATE INDEX ix_dmarc_results_enforcing         ON dmarc_check_results(is_enforcing);

-- One row per external report URI observed, tracking its _report._dmarc authorization state.
CREATE TABLE dmarc_external_report_auth (
    id             BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    result_id      BIGINT NOT NULL REFERENCES dmarc_check_results(id) ON DELETE CASCADE,
    report_kind    TEXT   NOT NULL,   -- 'rua' | 'ruf'
    report_uri     TEXT   NOT NULL,   -- e.g. mailto:d@thirdparty.example
    report_domain  TEXT   NOT NULL,   -- thirdparty.example
    auth_name      TEXT   NOT NULL,   -- <audited>._report._dmarc.thirdparty.example
    authorized     BOOLEAN NOT NULL,  -- v=DMARC1 present at auth_name?
    checked_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ix_dmarc_ext_auth_result ON dmarc_external_report_auth(result_id);

-- FUTURE (rua XML ingestion; not first round): parsed aggregate-report rows (RFC 9990 schema).
CREATE TABLE dmarc_aggregate_reports (
    id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    audit_run_id  BIGINT REFERENCES audit_runs(id) ON DELETE CASCADE, -- nullable: reports arrive async
    domain_id     BIGINT NOT NULL REFERENCES domains(id) ON DELETE CASCADE,
    report_org    TEXT,              -- reporting receiver (e.g. google.com)
    date_begin    TIMESTAMPTZ,
    date_end      TIMESTAMPTZ,
    source_ip     INET   NOT NULL,
    message_count INT    NOT NULL,
    disposition   TEXT,              -- none | quarantine | reject (applied)
    spf_result    TEXT,              -- raw auth_results value
    dkim_result   TEXT,              -- raw auth_results value
    spf_aligned   BOOLEAN,           -- policy_evaluated spf
    dkim_aligned  BOOLEAN,           -- policy_evaluated dkim
    ingested_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ix_dmarc_agg_domain_ip ON dmarc_aggregate_reports(domain_id, source_ip);
```

**Mapping onto today's file store.** There is **no database in the first round**. The audit
runner persists every run as **one YAML file** at
`~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml`
([../storage.mdx](../storage.mdx)); this spec owns that file's top-level **`dmarc:`** key — the §5
document. The checker's structured payload (`results.dmarc`) serializes as §5's `record:` block,
mapping 1:1 to `dmarc_check_results`; the per-URI list under `external_report_auth[]` maps to
`dmarc_external_report_auth`; the human-readable findings (`checkId:"dmarc"`) serialize as the
`tests[]` rows and map to the shared `findings` table. `tool_runs[]` is YAML-only provenance with
**no** table above — if it ever needs querying, it becomes a generic `tool_runs` table shared by
all six categories, keyed by `audit_run_id` + category. Future `rua` ingestion writes per-report
JSON that maps to `dmarc_aggregate_reports`. Because the store module fronts all of this, the
file→DB move is a single-module change.

## 11. When we run the check

- **On-demand.** Runs on every manual audit (`POST /api/audit/run/:domainId` and the run-all
  endpoint). DMARC is cheap (one TXT lookup + tree-walk fallbacks + deduped `_report._dmarc`
  probes), so it always runs inline with the rest of the engine.
- **Periodic.** The in-process scheduler re-audits each monitored domain on the configured cadence.
  Each run's `dmarc.*` findings are diffed against the previous run; any newly appearing or
  worsened finding is a **new problem** surfaced on the Dashboard. High-value regressions: a record
  silently dropped, a policy walked back from `reject`, a new duplicate record, `t=y` appearing, or
  an external `rua` losing its authorization record.
- **First-round vs future.** All §2 sub-checks except `dmarc.real_pass_rate` are first-round pure
  DNS. `dmarc.real_pass_rate` and PS-11/PS-12 are gated behind the `rua` ingestion pipeline
  ([../emails.mdx](../emails.mdx)) + the `dmarc_aggregate_reports` table.
- **Cost / rate-limit / back-off.** No paid feeds or provider APIs in the first round; DNS lookups
  only. Resolver failures get a short bounded retry and record as a **warning** "could not resolve"
  rather than a false critical.
- **Regression detection** applies exactly as in [../scheduled_checks.mdx](../scheduled_checks.mdx):
  compare the `dmarc` section of the domain's two newest run files
  (`~/.email_delivery_hero/runs/<domain>/`, [../storage.mdx](../storage.mdx)) — `record:` and
  `tests[]` — by finding id + severity. Every run appends a new file; nothing is overwritten, so
  the diff (and the §6.2 prev/next pager) always has both sides.

## 13. Acceptance criteria

1. Resolving `_dmarc.<domain>` correctly joins multi-string TXT chunks and identifies a `v=DMARC1`
   record; absence anywhere on the tree walk yields a `dmarc.missing` **critical** with a concrete
   publish-this remediation; a parent-only record yields a coverage **info** naming the parent.
2. Two or more `v=DMARC1` records yield a `dmarc.multiple` **critical**.
3. A record whose first tag is not `v=DMARC1` yields **critical**; `p=` not second yields
   **warning**; a missing or invalid `p=` yields **critical**.
4. `p=none` yields a **warning**; `t=y` yields a **warning**; `p=quarantine`/`p=reject` (without
   `t=y`) sets `is_enforcing=true` and yields an **ok** `dmarc.policy_ok` finding.
5. Enforcing `p=` with a weaker effective `sp=` yields a `dmarc.subdomain` **warning**; `sp`/`np`
   correctly default from `p`/`sp` when absent; a passing state emits `dmarc.subdomain_ok`.
6. `adkim=s`/`aspf=s` yield strict-alignment **warnings**; `pct` presence yields an obsolete-tag
   flag (`pct<100` a **warning**, `pct=100` an **info**); obsolete `ri`/`rf` yield **info**.
7. Missing `rua` yields a **warning**; malformed `rua` URIs yield a **warning**; an external
   `rua`/`ruf` domain with **no** `<domain>._report._dmarc.<report-domain>` `v=DMARC1` record
   yields `dmarc.external_report_auth` **critical**; a present one clears it with an **ok** finding.
8. Pass results are emitted as `ok` findings so the DMARC page renders an explicit passed/failed
   row per test, and the category cell label reflects the policy level.
9. Each run persists its DMARC results under the top-level **`dmarc`** key of the run's YAML file
   at `~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml` (e.g.
   `~/.email_delivery_hero/runs/whitehatengineering.com/2026_07_01_09_14AM.yaml`;
   [../storage.mdx](../storage.mdx)), with all §5 `record:` fields (snake_case, matching
   `dmarc_check_results` column names) including `external_report_auth[]`, the `tests[]` rows, and
   the derived `problem_states` — and never repeats the `run:` block's domain/timestamps.
10. Every non-`ok` finding carries a concrete `remediation` (the exact TXT record or tag edit),
    never a generic "check your DNS".
11. The Dashboard DMARC cell reflects the worst severity; `info`-only never turns it amber.
    Clicking the DMARC **cell** on the Domain-health table opens the domain's **newest** run's
    DMARC page (`/domains/:domainId/dmarc`); on a Runs-table row it opens **that run's** DMARC
    page (`/domains/:domainId/runs/:runId/dmarc`); clicking anywhere else in the row opens the
    whole run report (`/domains/:domainId/runs/:runId`, newest alias `/domains/:domainId`); the
    run report's DMARC category chip carries a chevron into the same run-scoped page.
12. The full DMARC page is **run-scoped**: it serves both `/domains/:domainId/runs/:runId/dmarc`
    and the newest alias, and renders the run context strip (run timestamp, ‹ prev / next › pager
    across the same domain's runs with ends disabled, "newest" badge on the latest run), the
    policy ladder, the raw + parsed record panel, the fail-first test-results table with
    observed/expected DNS values and copyable fixes, the report-destinations panel with
    per-destination authorization state, the matched problem-state cards linking to run-scoped
    drill-down pages, and the collapsed tool-runs provenance footer. Paging swaps `:runId` and
    re-renders entirely from that run's YAML — older runs never show newer data.
13. A second scheduled run that sees a dropped record, a policy downgrade, a new duplicate, `t=y`
    appearing, or a lost external-report authorization flags each as a **new problem** via the
    regression diff of the two newest run files.
14. `dmarc.real_pass_rate` (and PS-11/PS-12) are absent in the first round and clearly gated behind
    future `rua` ingestion.
15. Every external tool invocation during a run is captured as one `dmarc.tool_runs[]` entry —
    `tool`, the exact argv `command` (every input argument substituted, reproducible verbatim),
    `started_at`, `duration_ms`, `exit_code`, `output_format` (`json` wherever the tool supports
    it), the captured/parsed stdout under `parsed`, and `error: null` on success (stderr/timeout
    text on failure, with `parsed: null`). In-process `node:dns` lookups produce **no** tool-run
    entries; a binary missing from PATH is skipped with a single `info` finding, never a
    fabricated entry.
16. The §3 execution table is honored: doggo (record + per-external-destination authorization,
    `--json`, 10 s) and checkdmarc (`-f json`, 60 s) run on every DMARC run; kdig runs only on
    resolver disagreement or lookup failure; mailauth only with a user-supplied `.eml`; parsedmarc
    only inside the future ingestion job — and each entry's `parsed` output feeds exactly the
    structured fields that table names.
