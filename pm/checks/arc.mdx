# EmailDeliveryHero — ARC (Authenticated Received Chain)

ARC (**Authenticated Received Chain**, **RFC 8617**) is the mechanism that lets an intermediary — a
forwarder, a mailing list, or a security gateway — **preserve the SPF/DKIM/DMARC authentication
results it observed** and hand them, cryptographically sealed, to the next hop. It exists because
forwarding routinely **breaks DMARC**: the forwarder changes the envelope (breaking SPF alignment)
and often rewrites the body or `Subject:` (breaking the DKIM signature), so a message that passed
DMARC at the origin arrives at the final receiver failing both. ARC gives the final receiver a
trustworthy record of "this message *did* pass at the last honest hop," which the receiver **may**
choose to honor as a local override instead of rejecting on a `p=reject` policy.

The crucial architectural fact for this app: **ARC is not a published DNS record.** It is a set of
three header fields (`ARC-Seal`, `ARC-Message-Signature`, `ARC-Authentication-Results`) **added by
intermediaries to a message in transit**. You cannot audit a domain's ARC posture from DNS alone —
you need a **captured, forwarded message sample** to see whether a chain was applied and whether it
validates. Consequently **most of this check is FUTURE** (message-sample-dependent), and the parts
that ship first round are **advisory/informational**: they warn that a domain's sending patterns
*could* be broken by forwarding, and they check the one DNS-answerable piece (the intermediary's
signing selector). Be honest with the user: for a domain that never sends through forwarders, ARC is
simply **not applicable**, and even where it applies, ARC is an *advisory* signal — receivers are
free to ignore it.

## 1. What it checks

**Background & standard.** RFC 8617 defines ARC as three cooperating header fields, stamped as a
numbered set at each participating hop, identified by an **instance number** `i=` starting at `1`:

* **`ARC-Authentication-Results` (AAR)** — a snapshot of the `Authentication-Results` the
  intermediary observed *before* it altered the message (the SPF/DKIM/DMARC verdicts at that hop).
* **`ARC-Message-Signature` (AMS)** — a DKIM-style signature over the message headers and body *as
  the intermediary received them*, so a later validator can confirm the AAR describes this message.
* **`ARC-Seal` (AS)** — a signature over the **entire prior chain** (all earlier AS/AMS/AAR sets plus
  this hop's AAR and AMS). The seal is what makes the chain tamper-evident hop-to-hop, and carries the
  **chain validation status** `cv=` (`none` at `i=1`, then `pass` or `fail`).

**Why receivers care.** Large receivers (Google, Microsoft) evaluate ARC when a message fails DMARC:
if a valid ARC chain (`cv=pass`) shows the message passed authentication at a **trusted** forwarder,
the receiver may deliver it rather than quarantine/reject. Trust is per-receiver and reputation-based
— ARC does **not** compel delivery; it merely *supplies evidence* the receiver may weigh.

**Common real-world failure mode.** A domain publishes `p=reject`, then a subset of its mail is
forwarded (employee `.forward` rules, a `listserv`/Google Group, an "email to a friend" relay, a
security appliance that re-injects mail). At the final receiver SPF and DKIM both fail, DMARC fails,
and — because the forwarder does **not** apply ARC (or applies a broken chain) — the legitimate mail
is rejected. The symptom is "some recipients never get our mail, but only when it goes through
list X." ARC applied correctly by that forwarder fixes it.

**How a failure degrades inbox placement.** Without ARC on a forwarding path, DMARC-failing forwarded
mail is quarantined or rejected at strict receivers; with a broken ARC chain (`cv=fail`, out-of-order
instances, an unresolvable signing selector) the receiver discards the ARC evidence and falls back to
the raw DMARC failure — the same bad outcome. ARC never *hurts* a directly-sent message; its absence
only matters on hops that mutate the message.

## 2. Sub-checks

Expanded well beyond the seed list. Advisory sub-checks (marked *advisory*) are the ones that ship
first round from configuration/DNS; the message-sample sub-checks are FUTURE (need a captured
forwarded message — see §3).

| Sub-check (`checkId`) | What it verifies | How measured | Typical critical / warning finding | Concrete remediation |
| --- | --- | --- | --- | --- |
| `arc.applicable` *(advisory)* | Whether ARC is even relevant: does the domain enforce DMARC (`p=quarantine`/`reject`) **and** send through forwarders/lists? If not, ARC is N/A. | Read the DMARC policy (from the DMARC checker) + the domain's configured "uses forwarding" flag. | **info:** DMARC not enforcing, or no forwarding configured → ARC not applicable. | None needed; revisit if you adopt `p=reject` or start sending via lists/forwarders. |
| `arc.forwarding_risk` *(advisory)* | The domain sends through paths that break DMARC without ARC (mailing lists, `.forward`, gateways) while enforcing DMARC. | Config flag + heuristic over known list/forwarder indicators the operator enters per domain. | **warning:** enforcing DMARC + forwarding, but ARC status unverified (likely silent losses). | Confirm each forwarder/list applies ARC (`cv=pass`); for lists you run, enable ARC sealing on the MLM (e.g. Mailman 3, `mlmmj` + `OpenARC`). |
| `arc.selector_dns` | The ARC-signing intermediary's DKIM-style selector resolves in DNS: `<d>._domainkey.<s>` from a known/observed AMS+AS `d=`/`s=`. | `resolveTxt('<selector>._domainkey.<signing-domain>')` for each configured/observed ARC signer. | **critical:** signer's selector NXDOMAIN/empty `p=` → AMS/AS cannot be verified. **warning:** weak/legacy key. | Ask the forwarder to publish/repair its ARC signing key; if you operate the sealer, publish the 2048-bit key at `<selector>._domainkey.<domain>`. |
| `arc.chain_present` | An ARC header set (AS + AMS + AAR at `i=1…n`) is present on a **sampled forwarded message**. | Parse `ARC-*` headers from a captured sample (§3). | **warning:** forwarded sample carries **no** ARC headers (forwarder does not seal). | Enable ARC on the forwarder/MLM, or route mail through an intermediary that seals (many do by default). |
| `arc.seal_valid` | The ARC-Seal chain validates cryptographically and the newest seal is **`cv=pass`**. | Verify each `ARC-Seal` signature over the cumulative chain; read final `cv=`. | **critical:** `cv=fail` (broken/tampered chain) → receivers ignore the ARC evidence. | Fix the sealing intermediary (clock skew, header mutation after sealing, wrong key); re-test the sample until `cv=pass`. |
| `arc.ams_valid` | The most recent `ARC-Message-Signature` verifies against the message as sealed (body hash + header hash match). | DKIM-style AMS verification on the sample. | **critical:** AMS does not verify (message altered after sealing, or bad `bh=`). | Ensure nothing rewrites the message body/headers *after* the ARC signer; place the sealer last in the pipeline. |
| `arc.instance_ordering` | The `i=` instances are **contiguous and correctly ordered** (`1,2,3…` with no gaps/dupes), each set having exactly one AS, AMS, AAR. | Collect all `i=` values across the three header types; assert a complete, gapless sequence. | **critical:** missing/duplicate/out-of-order instance → chain invalid per RFC 8617 §5.2. | Fix the offending hop's sealer so it increments `i=` correctly and emits all three headers per hop. |
| `arc.oldest_pass` | The **oldest** hop's preserved authentication results (`i=1` AAR) show the message **passed** SPF/DKIM/DMARC at origin — the evidence downstream trust rests on. | Parse the `i=1` `ARC-Authentication-Results`; confirm `dmarc=pass` (or aligned dkim/spf) there. | **warning:** oldest AAR shows the message already failed auth at origin (nothing worth preserving). | Fix the origin authentication first (see [DKIM](./dkim.mdx)/[DMARC](./dmarc.mdx)); ARC cannot rescue mail that never passed. |
| `arc.cv_at_i1` | The first seal correctly reports `cv=none` (per spec, `i=1` has no prior chain to validate). | Read `cv=` on the `i=1` `ARC-Seal`. | **warning:** `i=1` seal with `cv=pass/fail` (malformed sealer). | Report the malformed sealer to the intermediary; `cv` must be `none` at the first hop. |
| `arc.signature_algorithm` | ARC signatures use a modern algorithm (`rsa-sha256` or `ed25519-sha256`; no SHA-1) with an adequate key length (mirrors DKIM RFC 8301). | Read `a=` on AMS/AS; inspect the resolved key modulus. | **warning:** weak/legacy algorithm or <1024-bit key on the signer. | Ask the signer to reissue with RSA-SHA256 / 2048-bit (or ed25519). |
| `arc.aar_completeness` *(advisory)* | The AAR at each hop actually records the SPF/DKIM/DMARC methods (not an empty/`none` stub). | Parse each AAR's `Authentication-Results` methods. | **info:** AAR present but records no usable auth methods. | Configure the sealer to snapshot full `Authentication-Results` before mutation. |
| `arc.receiver_honors` *(advisory)* | Whether downstream receivers (Gmail/Outlook) actually honored the chain and delivered DMARC-failing forwarded mail to the inbox. | Requires seed-list placement + DMARC `rua` evidence (fully future/paid). | **info:** cannot confirm receivers honored ARC (trust is per-receiver, reputation-based). | Build the forwarder's/sealer's sending reputation; ARC honoring is a receiver-side trust decision, not guaranteed. |

## 3. How it is measured

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/checks/arc/arc.check.ts` — the `arcCheck` checker, method `run()`

**Advisory / first-round path (deterministic).** Two things are answerable without a message sample:

1. **Applicability & forwarding risk** (`arc.applicable`, `arc.forwarding_risk`): read the DMARC
   policy the DMARC checker already parsed (enforcing vs `p=none`) and a per-domain
   **`usesForwarding`** flag / list of known forwarders the operator entered. If DMARC is not
   enforcing **or** the domain declares no forwarding, emit a single `info` "ARC not applicable" and
   stop. If DMARC is enforcing **and** forwarding is declared but no verified chain exists, emit the
   `warning`.
2. **Signer selector resolution** (`arc.selector_dns`, `arc.signature_algorithm`): for each ARC
   signing domain/selector — either configured for a known forwarder, or *observed* from a prior
   sample — run `resolveTxt('<selector>._domainkey.<signing-domain>')` via `node:dns/promises`, parse
   the DKIM key record, base64-decode `p=`, and read the modulus. This reuses the exact machinery of
   the [DKIM](./dkim.mdx) checker.

**Message-sample path (FUTURE).** Everything about an actual chain (`arc.chain_present`,
`arc.seal_valid`, `arc.ams_valid`, `arc.instance_ordering`, `arc.oldest_pass`, `arc.cv_at_i1`,
`arc.aar_completeness`) requires a **captured forwarded message**. Obtaining the sample:

* **Probe through a forwarder.** Use Homebrew **`swaks`** to send a known test message from the
  monitored domain **to an address that forwards** to a mailbox we control (e.g. a Google Group, a
  `listserv`, or a `.forward` alias set up for the audit). Example shape:
  `swaks --to list@example.org --from probe@<domain> --server <mx> --h-Subject "EDH-ARC-<auditId>"`.
* **Retrieve & parse the delivered copy.** Read the received message from the destination mailbox
  (IMAP fetch, or a maildrop the audit watches), extract the full header block, and parse the
  `ARC-Seal`, `ARC-Message-Signature`, `ARC-Authentication-Results`, and the receiver's own
  `Authentication-Results` (which frequently states `arc=pass (i=N ...)`).
* **Validate the chain.** Verify seals/signatures with an ARC validator: shell to **`OpenARC`**
  (`openarc`/`arcsign`/`arcverify`) or use a Node ARC/DKIM verification library. Read the final
  `cv=`, walk the `i=` sequence, and verify the `i=1` AAR's origin verdict.

**Edge cases & rate limits.** ARC selectors share DKIM's caching (respect TTL; a `dig` fallback is a
future cache-buster). Probe sends must be **throttled** — one probe per forwarder per audit at most —
to avoid tripping the forwarder's own spam controls or polluting reputation; store the last probe
timestamp per (domain, forwarder). A single "no ARC headers" result is inconclusive if the forwarder
sometimes seals and sometimes doesn't; treat `arc.chain_present` as `warning`, never `critical`.

**Mapping to severities.** Cryptographic/structural failures on a *present* chain (`arc.seal_valid`
`cv=fail`, `arc.ams_valid` fail, `arc.instance_ordering` gap, unresolved `arc.selector_dns`) are
**critical** — the chain is worthless. "Forwarder applies no ARC at all" and "oldest hop already
failed auth" are **warning**. Applicability, honoring, and completeness advisories are **info**
(they never turn the DMARC cell amber on their own, per the roll-up rules in
[Spam & Content catalog](../spam_checks.mdx)).

## 4. User interface

**Dashboard contribution.** ARC has **no cell of its own**; its findings feed the **DMARC** category
cell (column 3). Because ARC findings are advisory-heavy, an `info`-only ARC result does **not** turn
the DMARC cell amber; an `arc.seal_valid` `critical` on a captured sample *does* contribute red. When
ARC has never been sampled for a domain, it contributes nothing (the cell reflects only the DNS-based
DMARC sub-checks) and surfaces as a "Not yet sampled" advisory in detail.

**Audit-detail view (React 19 + shadcn/ui).** Inside the expanded DMARC group, ARC findings render as
their own labelled sub-group ("ARC — forwarding preservation (advisory)"). Each row uses the standard
finding layout shared with the other checks:

```
DMARC ▸ ARC (Authenticated Received Chain)                       [advisory]
────────────────────────────────────────────────────────────────────────────
● info      arc.applicable        DMARC=reject + 1 forwarder declared → ARC applies
▲ warning   arc.forwarding_risk   "acme-users" Google Group; ARC status unverified   [Copy fix]
✕ critical  arc.selector_dns      lists.acme.org selector "arc1" → NXDOMAIN            [Copy fix]
◻ n/a       arc.chain_present     No message sample captured yet          [Capture sample…]
```

* Severity badge (`ok`/`info`/`warning`/`critical`), the observed **detail** (parsed `cv=`, `i=`
  range, signer `d=`/`s=`, oldest-hop verdict), the **remediation** string, and a **Copy** button
  that copies the exact fix (e.g. the selector TXT record to publish, or the MLM setting to toggle).
* A **"Capture sample…"** action on FUTURE sub-checks opens a small panel to configure the
  swaks-through-forwarder probe (target list/alias, destination mailbox) and shows the last capture's
  parsed chain in a compact table: `i= | signer (d=) | AMS | AS cv= | AAR verdict`.

**Per-domain config inputs.** On the domain's settings, an **ARC / forwarding** section: a
`usesForwarding` toggle, a repeatable list of **forwarders/mailing lists** (label + address + optional
known ARC signing domain/selector), and the probe destination mailbox. These drive `arc.applicable`
and `arc.selector_dns` today and the capture probe later.

**States.** *Empty/never-run:* "ARC not yet evaluated." *Not applicable:* a muted "ARC N/A — no
enforcing DMARC or no forwarding declared" note. *Loading:* skeleton rows. *Sampled:* the parsed
chain table. **Admin-only:** enabling the live swaks probe (which sends real mail) is gated behind
`role:admin`; non-admins see advisory + DNS results only.

**Where every ARC row leads.** The rows above are no longer terminal: the ARC sub-group heading and
every `arc.*` finding row (on the run report's DMARC section and on the DMARC category page's
sub-tests band) deep-link into the **ARC explainer page** at
`/domains/:domainId/runs/:runId/dmarc/check/arc` (§9), and each finding also maps to one **ARC-nn
problem drill-down** at `/domains/:domainId/runs/:runId/dmarc/:problemId` (§10). The DMARC category
page currently filters its test-results table to `checkId === "dmarc"` (DmarcPage.tsx); the `arc`
band row specified in [dmarc.mdx §6.3](./dmarc.mdx) is the surface that reads `checkId === "arc"`
findings and carries the user here.

## 5. Information schema (database schema)

ARC reuses the **shared core tables** owned by [Engineering](../engineering.mdx) — `domains`,
`audit_runs`, and `findings` (each `arc.*` finding is one `findings` row with `category = 'dmarc'`
and `check_id` like `arc.seal_valid`). It also reuses `dkim_selectors` conceptually for the signer
selector. ARC adds **one result table** plus **one reference table**. Because first round is
advisory-only, **most columns are nullable** (they populate only once a message sample exists).

```sql
-- Per-audit structured ARC observation. Almost everything is nullable: first-round runs
-- record only applicability + selector-DNS results; the sample-derived columns fill in later.
CREATE TABLE arc_check_results (
    id                BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    audit_run_id      BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id         BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    message_sample_id TEXT,              -- id/hash of the captured forwarded message; NULL if none
    applicable        BOOLEAN,           -- DMARC enforcing AND forwarding declared
    chain_present     BOOLEAN,           -- ARC header set found on the sample
    chain_length      INT,               -- highest i= observed (hop count); NULL if no sample
    cv_result         TEXT CHECK (cv_result IN ('none','pass','fail')),  -- newest ARC-Seal cv=
    seal_valid        BOOLEAN,           -- entire seal chain verified
    ams_valid         BOOLEAN,           -- newest ARC-Message-Signature verified
    instances_ok      BOOLEAN,           -- i= contiguous, ordered, complete
    oldest_pass       BOOLEAN,           -- i=1 AAR shows origin dmarc/dkim/spf pass
    signer_domain     TEXT,              -- d= of the newest ARC signature
    signer_selector   TEXT,              -- s= of the newest ARC signature
    selector_resolves BOOLEAN,           -- <selector>._domainkey.<signer_domain> resolved
    instances         JSONB,             -- per-hop detail: [{i, d, s, ams_valid, as_cv, aar}]
    probe_sent_at     TIMESTAMPTZ,       -- when the swaks-through-forwarder probe last ran
    notes             TEXT,              -- freeform (e.g. "forwarder seals intermittently")
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (audit_run_id, domain_id)
);
CREATE INDEX arc_results_domain_idx     ON arc_check_results (domain_id, created_at DESC);
CREATE INDEX arc_results_cv_idx         ON arc_check_results (cv_result);
CREATE INDEX arc_results_instances_gin  ON arc_check_results USING GIN (instances);

-- Reference/config: the forwarders & mailing lists a domain sends through, and the ARC signer
-- each is expected to use. Drives arc.applicable / arc.forwarding_risk / arc.selector_dns and
-- the capture probe target. Operator-maintained (see §4 per-domain config).
CREATE TABLE arc_forwarders (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    domain_id        BIGINT NOT NULL REFERENCES domains(id) ON DELETE CASCADE,
    label            TEXT NOT NULL,           -- "acme-users Google Group"
    forward_address  TEXT NOT NULL,           -- probe target that forwards to us
    signer_domain    TEXT,                    -- expected ARC d= (nullable until observed)
    signer_selector  TEXT,                    -- expected ARC s=
    probe_mailbox    TEXT,                    -- where the forwarded copy lands for capture
    enabled          BOOLEAN NOT NULL DEFAULT true,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (domain_id, forward_address)
);
CREATE INDEX arc_forwarders_domain_idx ON arc_forwarders (domain_id);
```

**Mapping to today's JSON file store.** No database exists first round; this maps onto the file store
under `~/.email_delivery_hero/` ([Engineering §store](../engineering.mdx)):

* `arc_check_results` → an **`arc`** object inside the per-run audit file
  `audits/<domainId>/<auditId>.json` (alongside the run's `findings[]`), holding the same fields
  (`applicable`, `chainPresent`, `chainLength`, `cvResult`, `sealValid`, `instances[]`, `notes`,
  `messageSampleId`, `probeSentAt`). One audit file = one `arc_check_results` row. First round this
  object is mostly `null`/absent except `applicable`, `forwardingRisk`, and the selector-DNS result.
* `arc_forwarders` → the per-domain **`arc.forwarders`** array in `domains.json` (operator-entered
  config), plus the global **`arc`** block in `config.yaml` for probe throttle defaults.
* Captured raw messages (future) live as opaque blobs under `audits/<domainId>/samples/<sampleId>.eml`;
  `message_sample_id` is the filename stem. The store module hides this so the DB migration is a
  single-module change.

## 6. When we run the check

**On-demand.** A manual audit (`POST /api/domains/:id/audit`) always runs the **advisory + DNS**
ARC sub-checks (`arc.applicable`, `arc.forwarding_risk`, `arc.selector_dns`,
`arc.signature_algorithm`) — they are cheap and deterministic. The **live capture probe** (swaks
through a forwarder) is **not** part of the routine audit: it sends real mail and is triggered
explicitly from the "Capture sample…" action (admin-only), or on an opt-in slow cadence.

**Periodic scheduler.** The in-process scheduler ([Scheduled checks](../scheduled_checks.mdx)) re-runs
the advisory/DNS ARC sub-checks on the normal domain cadence (e.g. every 6h) — selector NXDOMAIN or a
newly-declared forwarder is caught here. Capture probes, if enabled, run on a **much slower** cadence
(e.g. weekly per forwarder) with a per-(domain, forwarder) back-off recorded in `probe_sent_at`, so we
never spam a mailing list or damage the domain's reputation.

**First-round vs gated.** Only the advisory + selector-DNS sub-checks are first round; everything
sample-derived is gated behind the swaks-probe + ARC-validator dependency (§7).

**Cost / rate-limit / back-off.** DNS selector lookups are free and cached (respect TTL). Probe sends
cost real mail volume and reputation, so they are strictly throttled and admin-gated. Never probe a
list the operator did not explicitly register.

**Regression detection.** The scheduler diffs each run against the previous: a signer selector that
**stops resolving** (`arc.selector_dns` ok→critical), a chain that **starts failing**
(`arc.seal_valid` pass→fail on the next captured sample), or a **newly-declared forwarder with no
verified chain** all register as a new problem and can turn the DMARC cell's advisory state worse. An
`info`-only ARC change (applicability flips because DMARC moved to enforcing) is logged but does not
by itself worsen the cell.

Rule of thumb: **anything answerable from DNS/config ships first (all advisory-only)**; anything that
needs to capture and cryptographically validate a **real forwarded message** is a later round. The
`Checker`/`Finding` contract is unchanged across rounds — only the checker's internals grow — so the
Dashboard, store, and scheduler never change as ARC matures.

## 8. Acceptance criteria

A developer can verify the ARC check is correct when **all** of the following hold:

1. The checker registers under **`id: 'arc'`**, emits `Finding`s with `category` rolled into the
   **DMARC** dashboard cell, and every non-`ok` finding carries a **concrete** remediation (a selector
   TXT to publish, or the exact MLM/forwarder setting to change) — never a generic "check your DNS."
2. For a domain with `p=none` **or** no declared forwarding, the check emits a single `info`
   `arc.applicable` "not applicable" and does **not** turn the DMARC cell amber.
3. For an enforcing-DMARC domain that declares a forwarder, `arc.forwarding_risk` emits a `warning`
   until a valid chain is verified, and `arc.selector_dns` resolves each declared/observed signer via
   `node:dns/promises`, returning `critical` on NXDOMAIN/empty `p=`.
4. Given a captured forwarded sample, the checker parses `ARC-Seal`, `ARC-Message-Signature`, and
   `ARC-Authentication-Results`, populates `chain_length`, `cv_result`, `seal_valid`, `ams_valid`,
   `instances_ok`, `oldest_pass`, and stores per-hop detail in `instances` JSONB.
5. A tampered/out-of-order/`cv=fail` chain yields a **critical** finding; a forwarded sample with **no**
   ARC headers yields a **warning** (`arc.chain_present`), never a false `critical`.
6. `arc.cv_at_i1` flags any first-hop seal whose `cv` is not `none`; `arc.oldest_pass` reads the
   `i=1` AAR and warns when origin authentication already failed.
7. The advisory/DNS sub-checks run in every manual and scheduled audit; the **live swaks capture
   probe** runs only on explicit admin-gated trigger (or a slow opt-in cadence) and records
   `probe_sent_at` with per-(domain, forwarder) back-off.
8. Results persist to the audit run's **`arc`** JSON object today and map cleanly onto
   `arc_check_results` + `arc_forwarders` (with the nullable columns above) when the store moves to
   PostgreSQL — a single store-module change, no checker change.
9. The scheduler's diff flags a selector that stops resolving or a chain that starts failing as a
   **new problem** between runs; `info`-only applicability flips are logged but do not worsen the cell.
10. The UI renders ARC as an advisory sub-group inside the DMARC audit-detail group, with severity
    badges, parsed chain detail, copy-to-clipboard remediation, a "Capture sample…" action, and clear
    empty / not-applicable / never-sampled states, built on React 19 + shadcn/ui.
11. The **ARC explainer page** (§9) renders at the canonical run-scoped route

    strip, the five locked blocks of [dmarc.mdx §6.4](./dmarc.mdx) in order, and a status chip

12. Block 2 renders **only this run's data**: the applicability panel (DMARC policy + enforcing
    verdict from the run's `dmarc` section, `usesForwarding`, forwarder count), the finding rows

    declared forwarder** (query name + raw TXT with `p=`/`k=` highlighted; "signer unknown"

13. Block 4's fixes are **keyed to the current state** (§9.5): each fix step's copy button carries

15. The **unit-scoped history strip** colors each of the last 10 runs by that run's worst `arc.*`

## 9. The ARC explainer page — `/dmarc/check/arc`

The DMARC category spec ([dmarc.mdx §6.3–§6.5](./dmarc.mdx)) locks a shared route pattern and a
five-block page contract for every sub-test unit in the category, and registers **`arc`** as a
sibling-owned `checkKey`. **This spec owns the content of that page.** The chrome (header row, run
context strip, back link, pager, skeletons, unknown-key panel) is the category's; everything inside
the blocks is defined here.

```
/domains/:domainId/runs/:runId/dmarc/check/arc     (canonical, run-scoped)
/domains/:domainId/dmarc/check/arc                 (newest-run alias)
```

### 9.1 Entry points & status-chip derivation

The page is reachable from every surface that mentions ARC:

| Entry point | Surface | Link |
| --- | --- | --- |
| Sub-tests band row "ARC chain" | DMARC category page ([dmarc.mdx §6.3](./dmarc.mdx)) | whole-row → run-scoped explainer |
| ARC sub-group heading + finding-row chevrons | Run report's DMARC section (RunDetailPage, [dmarc.mdx §6.6](./dmarc.mdx)) | `…/runs/:runId/dmarc/check/arc` |
| ARC mini-row in the per-unit mini-list | Run report's DMARC section | same |
| "ARC" mentions in DMARC explainer block 3 copy (e.g. the `policy` unit's forwarding caveat) | sibling explainers | newest alias |
| History-strip chips on this page | the explainer itself | same route, other `:runId` |

**Status chip (block 2 headline + band row + mini-row).** One rule, shared by all three surfaces:
the **worst severity among the viewed run's `checkId === "arc"` findings** — `critical` → ✗ red,
`warning` → ⚠ amber, `info` → ⓘ slate, all-`ok` → ✓ green; **no `arc.*` findings in the run** →
gray "not measured" (the checker was disabled, errored before emitting, or the run predates ARC).
Because first round is advisory-heavy, the common healthy-ish states are slate (`arc.applicable`
N/A) and amber (`arc.forwarding_risk` open) — the chip label spells it out: `ⓘ N/A`,
`⚠ unverified`, `✗ signer key`, `✓ verified` (future, requires a `cv=pass` sample).

### 9.2 Block 1 — What this is

Static concept copy (2–4 short paragraphs, per the category contract), independent of run data so
it renders even for "not measured":

1. **The mechanism.** ARC (Authenticated Received Chain, RFC 8617) lets a forwarder or mailing
   list record "this message passed SPF/DKIM/DMARC when it reached me," sign that record, and pass
   it downstream. Gloss the three headers in one line each: `ARC-Authentication-Results` (the
   snapshot), `ARC-Message-Signature` (a DKIM-style signature over the message as received),
   `ARC-Seal` (a signature over the whole chain so far, carrying `cv=`).
2. **Why it exists.** Forwarding breaks DMARC: the envelope changes (SPF alignment lost) and the
   body/subject often change (DKIM signature broken). Concrete example: mail from
   `billing@example.com` to a Google Group arrives at Gmail failing both — under `p=reject` it is
   rejected even though it is legitimate.
3. **The crucial caveat.** ARC is **not a DNS record you publish** and **not a guarantee** —
   intermediaries add it in transit, and receivers *may* honor a valid chain (`cv=pass`) from a
   forwarder they trust. That is why this unit is advisory: EmailDeliveryHero can verify the
   pieces that live in DNS/config today and the chain itself only from a captured forwarded
   message (§3).

### 9.3 Block 2 — Your current state

Everything renders from the **viewed run's** persisted data — `results.arc` (§5) plus the run's
`checkId === "arc"` findings — never from a live lookup; an older run stays a snapshot. Top to
bottom:

1. **Status chip** (large, per §9.1) with a one-line verdict sentence, e.g. *"ARC applies
   (p=reject + 2 forwarders declared) but no chain has been verified yet."*
2. **Applicability panel** — three labeled rows that explain how the verdict was reached, each with
   observed value and provenance:
   * *DMARC policy*: `p=<policy>` → enforcing yes/no, sourced from the run's `dmarc` section (the
     sibling result the checker consumed — run graph `arc ← dmarc`); the value chips link to the
     `policy` explainer (§9.9).
   * *Forwarding declared*: the `usesForwarding` flag + declared-forwarder count, sourced from the
     domain's ARC / forwarding settings at run time.
   * *Verdict*: `applicable` / `forwardingRisk` from `results.arc`.
3. **Finding rows** — the run's `arc.*` findings sorted fail → warn → info → pass (same row
   anatomy as the category's test-results table): status icon, finding id in small caps,
   plain-language title, chevron accordion expanding to *observed* (the `evidence` string,
   monospace), *why it matters* (one sentence), and the remediation with a `[copy fix]` button.
   Each expanded row ends with a **"Full drill-down ›"** link to its ARC-nn problem page (§10).
4. **Raw record slot — signer-key cards.** ARC has no DNS record of its own, so the category
   contract's "raw record" slot renders **one card per declared forwarder**:
   * Signer known (`d=`/`s=` recorded): the query name
     `<selector>._domainkey.<signer-domain>` as the card title, the **raw TXT answer** beneath it
     in monospace with the `p=` and `k=` tags visually highlighted (the unit's "owned tags"), a
     resolve badge (✓ resolves / ✗ NXDOMAIN / ✗ revoked `p=` / ⓘ lookup failed), and `[copy]` for
     the raw value. Rendered from `rawKeyRecord` (§9.11) with the finding `evidence` string as the
     fallback for runs that predate it.
   * Signer unknown: a muted placeholder card — *"ARC signer not yet known for <label> — record
     its d=/s= in settings or capture a forwarded sample"* — linking to the settings editor.
   * Future, when a sample exists: a final card shows the captured `ARC-Seal` /
     `ARC-Message-Signature` / `ARC-Authentication-Results` header block (monospace, `cv=` and
     `i=` highlighted) from `messageSampleId`.
5. **Scoped parsed-field table** — the fields this unit reads, exactly the `results.arc` shape
   (§5), one row each with published value, meaning, and grayed rendering when null:

   | field | value (example) | meaning |
   | --- | --- | --- |
   | `applicable` | `true` | enforcing DMARC + forwarding declared → ARC matters here |
   | `forwardingRisk` | `true` | a declared path could drop mail until a chain is verified |
   | `forwarders[]` | 2 rows | sub-table: label · forward address · signer `d=` · `s=` · resolves · key type · key bits |
   | `messageSampleId` | — not yet sampled | the captured forwarded message the chain checks need |
   | `chainPresent` / `chainLength` | — not yet sampled | ARC header set found / highest `i=` (hop count) |
   | `cvResult` / `sealValid` / `amsValid` | — not yet sampled | newest seal's `cv=` / seal chain verified / newest AMS verified |
   | `instancesOk` / `oldestPass` | — not yet sampled | `i=` contiguous & ordered / origin passed at `i=1` |
   | `probeSentAt` | — never | last swaks-through-forwarder probe |

   The `forwarders[]` sub-table's key-type/bits columns come from §9.11; grayed "— not yet
   sampled" rows are rendered (not hidden) so the user sees what a future capture will add.
6. **Per-hop instances table (future).** When `instances` is non-null: `i= | signer (d=) | s= |
   AMS | AS cv= | AAR verdict`, one row per hop, the failing hop's row tinted red — the compact
   chain table already specified in §4.

### 9.4 Block 3 — What it means

Impact copy keyed to the current state (one short paragraph shown per state, passing state first
when healthy):

* **N/A — DMARC not enforcing** (`arc.applicable` info, `p=none`/absent): nothing is being
  rejected, so there is nothing for ARC to rescue; the real work is on the DMARC `policy` ladder.
  Deliverability impact today: none from ARC.
* **N/A — no forwarding declared**: directly-sent mail never needs ARC. Impact: none — *unless*
  the domain actually does send through lists it hasn't declared, in which case losses are
  invisible; the copy says exactly that.
* **Unverified forwarding path** (`arc.forwarding_risk` warning): the classic silent failure —
  "some recipients never get our mail, but only via list X." Under `p=quarantine/reject`,
  forwarded legitimate mail is being spam-foldered or rejected at strict receivers until each
  forwarder demonstrably seals a valid chain. This is the §1 failure mode restated for this
  domain's declared paths, naming them.
* **Signer key broken** (`arc.selector_dns` critical): the forwarder may be sealing, but receivers
  cannot verify the seal — cryptographically equivalent to no ARC at all; the ARC evidence is
  discarded and mail falls back to the raw DMARC failure.
* **Weak/legacy signer key** (`arc.signature_algorithm` warning): verifiers increasingly refuse
  <1024-bit or non-rsa/ed25519 keys (RFC 8301), so the chain's protection erodes receiver by
  receiver.
* **Healthy / verified (future)**: a `cv=pass` chain was captured on every declared path — the
  paragraph explains what that protects (forwarded mail surviving `p=reject`) and the residual
  caveat (honoring is per-receiver trust, `arc.receiver_honors`).

Where a state maps to a §10 problem page, this block reuses that page's concept copy (single
source, same rule as the category's PS-nn reuse).

### 9.5 Block 4 — What you can do about it

Numbered, copy-pasteable steps, keyed to the same states. Copy buttons always carry the
publishable value / runnable command, never prose:

* **N/A (either reason)** — the healthy collapse line: *"Nothing to do for ARC right now."* plus
  the standing recommendation: revisit when moving to `p=quarantine`/`reject` or when mail starts
  flowing through a list/forwarder; if forwarding exists but was never declared, step 1 links to
  the domain's **ARC / forwarding settings** editor (label + forwarding address + signer `d=`/`s=`
  if known — the §4 config inputs).
* **Forwarding declared but no forwarders registered** (`arc.forwarding_risk` warning, empty
  list): one step — register each forwarder in settings; the panel enumerates the fields and links
  to the editor.
* **Unverified forwarding path (per forwarder)**:
  1. If you **run the list** (Mailman 3): enable ARC sealing — `[copy]` carries the config block:
     ```ini
     # /etc/mailman3/mailman.cfg
     [ARC]
     enabled: yes
     authserv_id: lists.example.org
     domain: lists.example.org
     selector: arc1
     privkey: /etc/mailman3/arc.private.pem
     ```
  2. Generate the signing key — `[copy]` carries:
     ```
     openssl genrsa -out arc.private.pem 2048
     openssl rsa -in arc.private.pem -pubout -outform der | openssl base64 -A
     ```
  3. Publish the public key — `[copy]` carries the exact TXT record
     `arc1._domainkey.lists.example.org TXT "v=DKIM1; k=rsa; p=<base64-from-step-2>"`.
  4. If it is a **third-party forwarder** (Google Groups, a listserv you don't run): ask them to
     enable ARC sealing and tell you the `d=`/`s=` they sign with; record those in settings so the
     key check runs here. (Google Groups and Microsoft 365 seal by default — the step says to
     simply record their observed signer once a sample is captured.)
  5. Verify with the capture probe once available (§9.6), or manually: forward a test message
     through the path and inspect its headers for `ARC-Seal: … cv=pass`.
* **Signer key broken** (`arc.selector_dns` critical): steps 2–3 above scoped to the failing
  `<selector>._domainkey.<signer-domain>` (the copy button carries that exact name), plus the
  verify command `[copy]`: `doggo <selector>._domainkey.<signer-domain> TXT --json` — and the
  propagation note (TTL wait) shared with the category pages. When the signer is third-party, step
  1 becomes "report the broken key to <label>" with the evidence line to paste to them.
* **Weak/legacy key** (`arc.signature_algorithm` warning): reissue at 2048-bit RSA or ed25519 —
  same key-generation + TXT snippets, prefixed with "ask <label> to…" for third-party signers.
* **Future chain failures** (`cv=fail`, AMS invalid, ordering, `cv≠none` at `i=1`, origin already
  failing): each carries the §2 remediation verbatim, with the failing hop identified from the
  per-hop table; "origin already failed" cross-links to the [DKIM](./dkim.mdx) and
  [DMARC](./dmarc.mdx) categories since ARC cannot rescue mail that never passed.

### 9.6 Block 5 — Run this check now (and the Capture sample probe)

Two affordances, deliberately distinct:

* **Run this check now** — the category-standard re-run ([dmarc.mdx §6.5](./dmarc.mdx)). **v1:**

    signer-key card's body; the checker already holds it as `rec`).
  * `keyType: string | null` — the parsed `k=` (defaulted `"rsa"` when the record resolves).
  * `keyBits: number | null` — the estimated RSA modulus bits (null for ed25519/unresolved).
* **`results.arc`** gains:
  * `dmarcPolicy: string | null` — the `p=` the applicability verdict used.
  * `policySource: "sibling" | "dns" | null` — whether it came from the sibling `dmarc` result or
    the checker's own fallback lookup (the applicability panel's provenance line).

Database mapping: three nullable columns on `arc_check_results`' `instances`-style JSON (the
`forwarders` observation array already lives inside the per-run `arc` object) plus
`dmarc_policy TEXT` / `policy_source TEXT` columns — all nullable, consistent with §5's
advisory-first posture. Older persisted runs simply render the fallbacks (finding `evidence` for
the card body, "—" for key type/bits, provenance omitted).

## 10. Problem states & drill-down pages (ARC-nn)

Each finding the checker emits maps to exactly **one** problem drill-down, rendered at the
category's problem route with the same chrome as the DMARC spec's PS-nn pages ([dmarc.mdx
§7](./dmarc.mdx) drill-down layout: **Concept → Your data → Diagnose it yourself → Path forward →
further reading**, with header row + run context strip):

```
/domains/:domainId/runs/:runId/dmarc/:problemId    (canonical, run-scoped)
/domains/:domainId/dmarc/:problemId                (newest-run alias)
```

**Id namespace.** ARC problem ids use the **`ARC-nn`** prefix — disjoint from the DMARC spec's
`PS-nn` ids (that spec's registry is untouched) and, like them, disjoint from the literal `check/`
segment. The frontend problem-state registry (today `lib/dmarc-problems.ts`) gains the ARC entries
below, with this spec as their content source; matching stays "which ids fired in this run."
Matched ARC-nn cards append to the category page's PROBLEM STATES band **after** the PS-nn cards,
same card anatomy (title + one-line hook + chevron).

### ARC-01 — Signer key unresolvable

* **Matches:** any `arc.selector_dns.<slug>` critical (NXDOMAIN/empty answer, or revoked/empty
  `p=`). **Concept:** the ARC seal is a DKIM-style signature; without the public key at
  `<selector>._domainkey.<signer>`, no receiver can verify it — the chain is cryptographically
  worthless, so the forwarded mail falls back to its raw DMARC failure. **Your data:** the failing
  forwarder's observation row (label, address, `d=`, `s=`, `selectorResolves=false`,
  `rawKeyRecord` when revoked-not-missing) plus the finding evidence. **Diagnose:**
  `doggo <selector>._domainkey.<signer> TXT --json`, `dig +short TXT <name>`,
  `kdig @8.8.8.8 +short TXT <name>` (copyable). **Path forward:** the §9.5 key-publication steps
  scoped to this signer (third-party: the report-to-forwarder variant). **Further reading:**
  RFC 8617 §4.1.2–4.1.3, RFC 6376 key records.

### ARC-02 — Weak or legacy signer key

* **Matches:** `arc.signature_algorithm.<slug>` warning (`k=` not rsa/ed25519, or RSA <1024-bit).
  **Concept:** RFC 8301 minimums; verifiers increasingly hard-fail weak keys, eroding the chain
  receiver-by-receiver. **Your data:** the observation's `keyType`/`keyBits` + raw record.
  **Diagnose:** the same TXT lookups plus decoding the modulus
  (`… | openssl base64 -d -A | openssl rsa -pubin -inform der -noout -text`). **Path forward:**
  reissue at 2048-bit / ed25519 (the §9.5 snippets). **Further reading:** RFC 8301.

### ARC-03 — Unverified forwarding path

* **Matches:** `arc.forwarding_risk` (declared-but-none-registered) and every
  `arc.forwarding_risk.<slug>`; also hosts the "signer not yet known" info. **Concept:** the §1
  failure mode — enforcing DMARC + forwarding, ARC status unknown → silent losses on that path.
  **Your data:** the declared-forwarder list with per-row verification status; the DMARC policy
  context. **Diagnose:** send a message through the path manually and inspect the delivered copy's
  headers for `ARC-Seal … cv=`; `swaks` command shape from §3 (copyable, with the "sends real
  mail" caution). **Path forward:** the §9.5 per-forwarder steps (Mailman config / third-party
  request / record the signer). **Further reading:** RFC 8617, M3AAWG forwarding paper, Mailman 3
  docs.

### ARC-04 — Not applicable / not yet determined

* **Matches:** `arc.applicable` info in its N/A and could-not-determine variants, and transient
  selector-lookup infos. **Concept:** when ARC does and does not matter (the two gates: enforcing
  policy, forwarding). **Your data:** the applicability panel rows. **Diagnose:**
  `doggo _dmarc.<domain> TXT --json`. **Path forward:** none today; the revisit triggers, plus
  "declare your forwarders" when the gate is forwarding. **Further reading:** RFC 9989 policy,
  RFC 8617 §1.

### ARC-05 — Chain not yet sampled / chain absent

* **Matches:** the first-round `arc.chain_present` info stub; future: the warning when a captured
  sample carries no ARC headers. **Concept:** why the chain checks need a real forwarded message
  and what the capture probe does (§3). **Your data:** `messageSampleId`/`probeSentAt` (null
  today), the declared probe targets. **Diagnose:** the manual forward-and-inspect procedure.
  **Path forward:** run the admin Capture sample probe when available; meanwhile ARC-03's
  confirmation steps. **Further reading:** RFC 8617 §5.

### ARC-06 — Broken or invalid chain *(future)*

* **Matches:** the sample-derived criticals/warnings — `arc.seal_valid` (`cv=fail`),
  `arc.ams_valid`, `arc.instance_ordering`, `arc.cv_at_i1`, `arc.oldest_pass`, plus the
  `arc.aar_completeness`/`arc.receiver_honors` advisories as context rows. **Concept:** what each
  structural failure means per RFC 8617 §5.2 and why receivers discard a broken chain. **Your
  data:** the per-hop instances table with the failing hop tinted; `cvResult`, `sealValid`,
  `amsValid`, `instancesOk`, `oldestPass`. **Diagnose:** re-verify the captured `.eml` with
  OpenARC (`arcverify`) / `npx mailauth report <sample.eml>` (copyable). **Path forward:** the §2
  remediations for the specific failing sub-check; origin failures route to the DKIM/DMARC
  categories first. **Further reading:** RFC 8617 §5.2, OpenARC.

**Drill-down ↔ explainer linking.** Every ARC-nn page's Concept block ends with *"Full ARC
sub-test explainer ›"* → `…/dmarc/check/arc` (run-scoped), and the explainer's finding rows link
back per §9.3 item 3 — the two page kinds answer different questions (one problem deeply vs. the
whole unit) and must cross-link both ways. Back chevron returns to the run's DMARC page anchor,
exactly like PS-nn.
