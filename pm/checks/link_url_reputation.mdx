# Link / URL Reputation (URI Blocklists on Links Inside the Message)

Receiving mail systems do not only judge *who* sent a message ŌĆö they judge *where its links point*. Every URL in the body is extracted, its host reduced to a registrable domain, and that domain is checked against URI blocklists (Spamhaus DBL, SURBL, Invaluement ivmURI) exactly the way an RHSBL checks a sending domain. A single bad linked domain ŌĆö a listed URL shortener, a compromised WordPress site you link to, a tracking domain that got swept into DBL ŌĆö can spam-file the whole message **even when your SPF, DKIM, and DMARC are perfect**, because content filters (SpamAssassin `URIBL_*`, Rspamd `SURBL_*`/`DBL`, and every commercial gateway) weight body-URL reputation heavily. This spec is the URL-reputation sub-family of the **Spam & Content** dashboard category; every sub-check id uses the **`content`** prefix (`content.url_*`).

## 1. What it checks

There is no RFC that defines the URI blocklists themselves, but the query mechanism is the RHSBL half of **RFC 5782 (DNS Blacklists and Whitelists)**: the *registrable* domain of a link is queried **directly, with no reversal** ŌĆö `<linkdomain>.<zone>`, `A`/`TXT` lookup, `127.0.0.x` return codes decode the reason. The registrable domain is derived with the **Public Suffix List** (PSL, publicsuffix.org) so that `foo.bar.co.uk` reduces to `bar.co.uk`, not `co.uk`; SURBL and DBL are queried on that reduced domain (SURBL additionally tests a couple of nested labels for wildcard abuse). URLs themselves are structured by **RFC 3986**; internationalized hosts by **IDNA2008 / RFC 5890-5891** (punycode `xn--`).

Why receivers care: botnet and snowshoe spam rotates sending IPs constantly but must always drive the recipient to a payload URL, so the *link* domain is the most stable spam fingerprint. URI blocklists like SURBL and Spamhaus DBL catch the overwhelming majority of spam/phish/malware by domain, and mailbox providers additionally consult Google Safe Browsing and Microsoft SmartScreen on click. The common real-world failure modes: (1) linking through a **URL shortener** (bit.ly, t.co, tinyurl) that hides ŌĆö and cannot vouch for ŌĆö the true destination, which filters penalize on principle; (2) linking a **compromised legitimate site** (a hacked blog, an "abused-legit" DBL entry) you have no control over; (3) a **tracking/click domain** from your ESP that got listed by neighbor abuse; (4) **http-not-https** links, **raw-IP** URLs, or **punycode/homograph** lookalike domains, all of which are classic phishing signals. A single high-trust URI listing (DBL spam/phish, SURBL phishing bit) can drop inbox placement to near zero regardless of authentication.

## 2. Sub-checks

Each row extracts and evaluates the links in a **sample message** (see ┬¦3 for how samples are supplied). "Link domain" = the PSL-registrable domain of the URL host; "final domain" = the link domain *after* following shorteners/redirects. RHSBL zone rows reuse `dnsbl_zones` from [./blacklists.mdx](./blacklists.mdx). Every remediation is concrete: a per-domain delisting portal, or the exact link-hygiene edit.

## 3. How it is measured

**Sample source.** Unlike auth/DNS checks, this check needs the *message body*. A sample is supplied one of three ways, stored in `content_sample_messages` (owned by [./content_scoring.mdx](./content_scoring.mdx)): (1) the operator pastes/uploads a raw `.eml`; (2) the operator sends a real campaign to a per-domain **ingest mailbox/seed address** the app monitors; (3) an HTML/text snippet is pasted. If no sample exists, every `content.url_*` sub-check is **`info` "no sample message ŌĆö add one to check link reputation"**, never a false pass. Each audit pins the `sample_id` used so results are reproducible.

**Extraction & normalization.** The HTML part is parsed (DOM), harvesting `a[href]`, `area[href]`, `img[src]` used as links, `[background]`, inline CSS `url(...)`, and AMP `href`s; the text part is scanned with an RFC 3986-aware URL matcher. Each URL is parsed, lowercased, IDNA-encoded, and reduced to its **registrable domain via the Public Suffix List**. `mailto:`/`tel:`/`data:` are excluded from reputation but `mailto:` domains may feed alignment. The result is a de-duplicated map of `{ url, link_domain }`, plus per-URL flags (`is_https`, `is_ip_literal`, `is_punycode`, `is_shortener`).

**Redirect / shortener expansion (probe, future-round).** For shortener or otherwise-suspicious links, a bounded HTTP client issues `HEAD` (falling back to a ranged `GET`), following `301/302/303/307/308`, `<meta http-equiv=refresh>`, and obvious `Location`/JS hops up to a hop cap (e.g. 5) with a timeout (e.g. 5s) and no cookies/JS execution. The hop chain and the **final_domain** are recorded; the final domain is then run through the URI zones. This is disabled in first round (no outbound HTTP); until then `final_domain` is null and `content.url_redirect_chain`/`content.url_reachable`/`content.url_safe_browsing` are `info`-gated.

**URI-zone resolution.** For each unique link (and final) domain, the checker queries the enabled RHSBL zones with `node:dns/promises` `resolve4()` (`A`) and `resolveTxt()` (`TXT`): `NXDOMAIN` = not listed; any `127.0.0.x`/`127.0.1.x` answer = listed, stored as the return code and decoded via the zone's `return_code_map` (reused from `dnsbl_zones`) to a sub-label + severity. SURBL/URIBL bitmask codes are decoded bit-by-bit. The **RFC 5782 test-point probe** (`2.0.0.127.<zone>`ŌåÆ`127.0.0.2`, `1.0.0.127.<zone>`ŌåÆ`NXDOMAIN`) runs once per zone per audit; on failure the zone's results are marked `inconclusive` (info), never false-listed. Results dedupe per domain across the message so linking the same bad domain twice is one finding.

**Resolver policy & rate limits.** SURBL, Spamhaus DBL public mirrors, and URIBL **block large open resolvers (8.8.8.8, 1.1.1.1) and high-volume clients**; Invaluement is paid; Safe Browsing needs an API key. The checker reuses `dnsbl_zones.requires_registration`/`is_paid` to skip or degrade a zone cleanly. Queries are capped at a small concurrency (~8 in-flight), cached for the audit run honoring DNS TTL, and use jittered back-off on `SERVFAIL`/timeout. Because a message can contain many links, the domain set is deduped *before* querying so query count = (unique link domains) ├Ś (enabled URI zones), not (links) ├Ś (zones).

**Severity mapping.** High-trust URI listings (DBL spam/phish/malware, SURBL phishing/malware bit, URIBL black) ŌåÆ **critical**; abused-legit/grey/cracked bits and shortener/http/redirect-chain/alignment heuristics ŌåÆ **warning**; raw-IP literal and brand-homograph ŌåÆ **critical** (unambiguous phishing shapes); count/reachability/alignment advisories ŌåÆ **info** unless thresholds cross. The category roll-up (`content.url_aggregate`) takes the worst *weighted* severity; advisory `info` alone does not turn the Spam & Content cell amber.

**Dedicated panel.** A "Sample Message" panel (shared with content scoring) lets the operator paste an `.eml`, upload, or copy the per-domain ingest address; it shows the parsed link table with domains, schemes, and expansion status. **Per-domain config inputs.** The domain's own/related/allowlisted link domains (for `content.url_domain_alignment`), the sample message, and any pinned known-collateral link listings. **States:** *Never-run/no sample:* gray, "Add a sample message to check link reputation." *Loading:* skeleton + progress ("extracting linksŌĆ” querying 5 URI zonesŌĆ”"). *Inconclusive:* amber-outline info row "zone unavailable / paid feed not configured / redirect probe disabled (first round)." *All clean:* green, "All 12 links clean across DBL/SURBL/URIBL." Zone weights, paid/registration keys, Safe Browsing key, and shortener list are **admin-only**.

## 5. Information schema (database schema)

Reference the shared core tables (owned by engineering, do **not** redefine): `domains`, `sending_ips`, `dkim_selectors`, `audit_runs`, `findings`. This check **reuses** `dnsbl_zones` (from [./blacklists.mdx](./blacklists.mdx), filtered to `kind='domain'` for URI zones) and **references** `content_sample_messages` (owned by [./content_scoring.mdx](./content_scoring.mdx)). It adds one **per-URL** table and one **per-run summary** result table.

```sql
-- Per-extracted-URL detail for one audit's sample message.
CREATE TABLE message_urls (
    id            BIGSERIAL PRIMARY KEY,
    audit_run_id  BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id     BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    sample_id     BIGINT REFERENCES content_sample_messages(id) ON DELETE SET NULL, -- nullable: snippet-only samples
    url           TEXT   NOT NULL,          -- the raw URL as it appeared in the body
    link_domain   TEXT   NOT NULL,          -- PSL-registrable domain of the URL host
    final_domain  TEXT,                     -- registrable domain after redirect/shortener expansion (null in first round)
    is_shortener  BOOLEAN NOT NULL DEFAULT FALSE,
    is_https      BOOLEAN NOT NULL DEFAULT FALSE,
    is_ip_literal BOOLEAN NOT NULL DEFAULT FALSE,
    is_punycode   BOOLEAN NOT NULL DEFAULT FALSE,
    redirect_hops INTEGER,                  -- number of hops followed (null when probe disabled)
    listings      JSONB  NOT NULL DEFAULT '[]', -- [{ "zone":"dbl.spamhaus.org", "listed":true, "code":"127.0.1.4", "bit":"phish" }, ...]
    aligned       BOOLEAN,                  -- link_domain matches sender/org/allowlist (null = not evaluated)
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_message_urls_domain      ON message_urls (domain_id);
CREATE INDEX idx_message_urls_link_domain ON message_urls (link_domain);
CREATE INDEX idx_message_urls_run         ON message_urls (audit_run_id);
CREATE INDEX idx_message_urls_listings    ON message_urls USING GIN (listings);

-- Per-run roll-up of the URL-reputation checks for the sample.
CREATE TABLE url_check_results (
    id                BIGSERIAL PRIMARY KEY,
    audit_run_id      BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id         BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    sample_id         BIGINT REFERENCES content_sample_messages(id) ON DELETE SET NULL, -- nullable
    total_links       INTEGER NOT NULL DEFAULT 0,
    unique_domains    INTEGER NOT NULL DEFAULT 0,
    listed_domains    INTEGER NOT NULL DEFAULT 0,
    shortener_count   INTEGER NOT NULL DEFAULT 0,
    http_count        INTEGER NOT NULL DEFAULT 0,
    ip_literal_count  INTEGER NOT NULL DEFAULT 0,
    punycode_count    INTEGER NOT NULL DEFAULT 0,
    offbrand_count    INTEGER NOT NULL DEFAULT 0,
    inconclusive      BOOLEAN NOT NULL DEFAULT FALSE, -- zone(s) unavailable / paid feed / probe disabled
    weighted_worst    TEXT,                 -- 'ok'|'info'|'warning'|'critical'
    checked_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_url_check_results_domain ON url_check_results (domain_id);
CREATE INDEX idx_url_check_results_run    ON url_check_results (audit_run_id);
```

**Zone reuse.** URI zones are rows of `dnsbl_zones WHERE kind='domain'` (DBL, SURBL, ivmURI, URIBL) plus their `requires_registration`/`is_paid`/`return_code_map`/`weight`/`delisting_url` ŌĆö no separate zone table. The shortener domain list and Safe Browsing key live in `config.yaml` (`content.url.shorteners`, `secrets.safe_browsing_key`), not in a DB table (or a small `url_shortener_domains` reference table if operators want UI editing).

**File-store mapping.** Today each audit writes `~/.email_delivery_hero/audits/<domainId>/<auditId>.json` under a `content.url` key: `{ "content": { "url": { "sampleId": 7, "summary": { "totalLinks":12, "uniqueDomains":8, "listedDomains":1, "shortenerCount":1, "httpCount":0, "inconclusive":false, "weightedWorst":"critical" }, "links": [ { "url":"https://bit.ly/x", "linkDomain":"bit.ly", "finalDomain":"promo.example.com", "isShortener":true, "isHttps":true, "isIpLiteral":false, "isPunycode":false, "redirectHops":2, "listings":[], "aligned":true }, { "url":"http://hacked-blog.example/p", "linkDomain":"hacked-blog.example", "finalDomain":null, "isShortener":false, "isHttps":false, "isIpLiteral":false, "isPunycode":false, "listings":[{"zone":"dbl.spamhaus.org","listed":true,"code":"127.0.1.4","bit":"phish"}], "aligned":false } ] } } }`. Each `links[]` element becomes one `message_urls` row and the `summary` object becomes the one `url_check_results` row (both keyed to the enclosing `audit_run_id`/`domain_id`); derived `findings` land in the shared `findings` table/array via the normal CheckerŌåÆFinding contract. Sample bytes live in the content-scoring sample store, referenced by `sample_id`.

## 6. When we run the check

**On-demand.** Runs on every manual audit (`POST /api/domains/:id/audit`) and any Spam & Content category-scoped run **provided a sample message is available**; otherwise it emits the single `info` "add a sample" finding. After a user submits a URI delisting request or fixes a link, a fresh manual run against the same (or new) sample is the "am I clean yet?" tool.

**Periodic.** The in-process scheduler re-runs per domain at the `config.yaml` cadence (default every 6h for Spam & Content ŌĆö URI listings are volatile like IP/domain blacklists). It re-uses the domain's **latest pinned sample**; if none, it stays `info` and is skipped for regression purposes. Each run **diffs against the previous audit**: a link domain flipping `listed:false ŌåÆ true`, a `warning ŌåÆ critical` bit escalation, or a newly introduced shortener/raw-IP/homograph link is a **new problem** eligible for alerting (per [../scheduled_checks.mdx](../scheduled_checks.mdx)); a `listed:true ŌåÆ false` transition is reported resolved. Because the sample can change between runs, the diff is computed on the *link-domain* set with the sample id noted, and inconclusiveŌåölisted transitions are ignored to avoid resolver-blockage false alarms.

**Cost / rate-limit / back-off.** Query count per audit = (unique link + final domains) ├Ś (enabled URI zones); dedup keeps this small (a typical newsletter has < 10 unique domains). Cap concurrency (~8), reuse a dedicated non-public resolver for zones that block open resolvers, honor TTL caching, back off with jitter on `SERVFAIL`/timeout, and auto-degrade a zone whose RFC 5782 test point fails. Redirect-following and Safe Browsing add outbound HTTP/API cost ŌĆö both gated behind config, rate-limited, and cached per final domain per run.

## 8. Acceptance criteria

1. Given a sample message, the checker extracts every link from both HTML (`href`, `area`, `img`/`background`, CSS `url()`, AMP) and text parts, parses each per RFC 3986, and reduces the host to its **Public Suffix List** registrable domain, de-duplicating to a domain set before any DNS query.
2. With no sample message present, every `content.url_*` sub-check yields exactly one `info` "add a sample" finding ŌĆö never a false pass or false listing ŌĆö and the audit records no `message_urls` rows.
3. For each unique link domain, RHSBL zones are queried as `<linkdomain>.<zone>` with **no reversal**; a `127.0.0.x`/`127.0.1.x` `A` answer sets `listed=true` with the exact return code decoded (SURBL/URIBL by bitmask) via the zone's `return_code_map`; `NXDOMAIN` sets `listed=false`.
4. Per-URL flags are set correctly: `is_https` (scheme), `is_ip_literal` (IPv4/IPv6 host literal), `is_punycode` (`xn--` label), `is_shortener` (configured shortener list), and `aligned` (link domain vs. sender/org/allowlist).
5. High-trust URI listings (DBL `.2-.6`, SURBL phish/malware bit, URIBL black) and raw-IP/brand-homograph links map to **critical**; shortener/http/redirect/alignment/abused-legit map to **warning**; count/reachability advisories to **info**; the roll-up uses the worst **weighted** severity and info alone does not turn the Spam & Content cell amber.
6. The RFC 5782 test-point probe runs once per URI zone per audit; on failure the zone's results are `inconclusive` (info), never false-listed; zones flagged `requires_registration`/`is_paid` without credentials are skipped with an explanatory `info` note and a UI fix link.
7. Every non-ok finding's `remediation` is concrete ŌĆö a correct per-zone `delisting_url` for a listing (with return code/reason), or the exact link-hygiene edit (replace shortener with full URL, switch to `https`, remove IP-literal, use the real ASCII domain) ŌĆö never a generic message.
8. Redirect expansion, `content.url_reachable`, and `content.url_safe_browsing` are disabled in first round (`final_domain` null, findings `info`-gated) and, when enabled, respect the hop cap, timeout, no-cookie/no-JS policy, and per-final-domain caching.
9. `message_urls` and `url_check_results` exist with the specified PKs, FKs (`audit_run_id`, `domain_id`, nullable `sample_id`), the GIN index on `listings`, and B-tree indexes on `(domain_id)` and `(link_domain)`; URI zones are read from `dnsbl_zones WHERE kind='domain'` (no duplicate zone table), and each result serializes into the audit JSON under the `content.url` key round-tripping 1:1.
10. The scheduler diff (over the pinned sample's link-domain set) flags any cleanŌåÆlisted, warningŌåÆcritical, or newly-introduced shortener/raw-IP/homograph link as a new problem and any listedŌåÆclean as resolved, ignoring inconclusive transitions.
