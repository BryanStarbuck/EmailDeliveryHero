# Blacklists (DNSBL / RHSBL / IP & Domain Reputation Blocklists)

A DNS-based blocklist (DNSBL) is a list of IP addresses — and, for right-hand-side blocklists (RHSBL), of domain names — that a receiving mail server queries over ordinary DNS to decide whether to reject, quarantine, or score a message. If your sending IP or your domain (or a domain that appears in your message From/Return-Path/URLs) is listed on a zone a receiver honors, your mail is throttled, junked, or hard-bounced regardless of how perfect your SPF/DKIM/DMARC are. This is the single fastest way for a good sender to fall out of the inbox, and the single check that customers ask about first ("am I blacklisted?"). This spec is the flagship of the **Blacklists** dashboard category; every sub-check id uses the **`dnsbl`** prefix.

## 1. What it checks

DNSBLs have no single RFC that defines the *lists* themselves, but the query mechanism is standardized by **RFC 5782 (DNS Blacklists and Whitelists)**, which formalizes the two families:

- **IPv4 DNSBL** — reverse the four octets of the target IP, append the zone host, and issue an `A` query. A listing is signaled by an answer in `127.0.0.0/8` (conventionally `127.0.0.2`–`127.0.0.255`); the specific low-order octet is the **return code** that tells you *why* it is listed. A `TXT` query at the same name returns a human-readable reason and usually a removal URL. RFC 5782 also mandates that a well-behaved zone answer `127.0.0.2` for the test address `127.0.0.2` and `NXDOMAIN` for `127.0.0.1`, which we use as a liveness probe.
- **RHSBL / domain DNSBL** — the domain is queried **directly, with no reversal**: `<domain>.<zone>`, `A` lookup, same `127.0.0.x` semantics. Spamhaus DBL, SURBL, and ivmURI are of this kind.

**Worked IP example.** To test `203.0.113.24` against `zen.spamhaus.org`:
1. Reverse octets: `24.113.0.203`.
2. Form the query name: `24.113.0.203.zen.spamhaus.org`.
3. `A` lookup. `NXDOMAIN` → not listed. An answer such as `127.0.0.4` → listed (Zen packs SBL, CSS, XBL, and PBL sub-lists into distinct return codes: `127.0.0.2`/`.3` = SBL/CSS, `127.0.0.4`–`.7` = XBL, `127.0.0.10`/`.11` = PBL).
4. `TXT` lookup at the same name → the reason string and delisting link.

**Worked domain example.** To test `example.com` against `dbl.spamhaus.org`: query `example.com.dbl.spamhaus.org` directly (no reversal). `127.0.1.2` = spam domain, `127.0.1.4`–`.6` = phishing/malware/botnet C&C, `127.0.1.102`–`.106` = "abused legit" categories.

Why receivers care: DNSBL lookups are cheap (one cached DNS round-trip) and catch the majority of botnet and snowshoe spam at connection time, so nearly every MTA and spam engine (Postfix `reject_rbl_client`, SpamAssassin `RCVD_IN_*`, Rspamd, commercial gateways) consults several. The common real-world failure mode is a **shared-IP or ASN-wide listing you did not cause** — a neighbor on your hosting provider's IP block, a compromised mailbox on your subnet, or a provider whose whole ASN got swept into UCEPROTECT L3. A single listing on a high-trust zone (Spamhaus SBL/XBL/Zen) can drop inbox placement to near zero at the large mailbox providers; a listing on a low-trust or ASN-wide zone (UCEPROTECT L2/L3, some SORBS sub-zones) may cost only a few percent, which is exactly why per-zone **weighting** is essential — a naive "you're on a list!" alarm produces false panic.

## 3. How it is measured

**Targets.** For IP zones the checker iterates `ctx.sendingIps` (from `sending_ips`); if empty, it derives candidate IPs by resolving the domain's MX hosts and SPF `a`/`mx`/`ip4`/`ip6` mechanisms and testing those (an unexpected listing on an SPF-authorized IP is still your deliverability problem). For RHSBL/domain zones the target is `ctx.domain` and any additional envelope domains (Return-Path, DKIM `d=`).

**Query construction.**
- IPv4: split the dotted quad, reverse, join with `.`, append `zone_host`. `203.0.113.24` + `zen.spamhaus.org` → `24.113.0.203.zen.spamhaus.org`.
- IPv6: expand to the 32 nibbles, reverse them, dot-separate, append the zone (RFC 5782). Most zones only carry IPv4 data; IPv6 lookups usually `NXDOMAIN` and are skipped unless the zone documents IPv6 support.
- Domain: `<domain>.<zone_host>` verbatim, lowercased, IDNA/punycode-encoded, no reversal.

**Resolution.** First round uses `node:dns/promises` `resolve4()` for the `A` answer and `resolveTxt()` for the reason. `NXDOMAIN` (`ENOTFOUND`/`ENODATA`) = **not listed**. Any `127.0.0.0/8` (or documented `127.0.1.0/8`, `127.0.4.0/8`) answer = **listed**; the full answer set is stored as `return_code` (e.g. `127.0.0.4`) and decoded against the zone's code map to a sub-list label and severity. A `TXT` answer is stored verbatim as `reason_txt` and mined for a removal URL; if none is present we fall back to `dnsbl_zones.delisting_url`.

**Zone liveness / test-point probe.** Before trusting an `NXDOMAIN` as "clean", the runner (once per zone per audit) issues RFC 5782 sanity queries: `2.0.0.127.<zone>` must return `127.0.0.2` and `1.0.0.127.<zone>` must return `NXDOMAIN`. If the test point fails (dead zone, blocked resolver, or — critically — a zone that returns `127.0.0.2` for *everything* because your resolver is being rate-limited/blocked), the sub-check yields **info** "zone unavailable / inconclusive", never a false listing.

**Resolver policy & rate limits.** Public mirrors of Spamhaus, SURBL, URIBL, Abusix, and others **block or garbage-answer queries from large open resolvers (Google 8.8.8.8, Cloudflare 1.1.1.1, Quad9, OpenDNS) and cloud-provider resolvers** and from high-volume clients. Refusals are signaled **in-band**: Spamhaus answers `127.255.255.252` (bad query) / `.254` (public-resolver blocked) / `.255` (rate-limited); URIBL answers `127.0.0.1` (`URIBL_BLOCKED`). The checker MUST decode these as **"query refused — inconclusive"**, never as a listing (see problem state [PS-9](#ps-9--queries-refused--resolver-blocked)). Consequences the checker must handle: (1) **Barracuda requires free registration** of the querying resolver IP before `b.barracudacentral.org` resolves — until registered the sub-check is disabled/`info`. (2) **Abusix requires a free API key** embedded in the query name; (3) **invaluement is paid** and needs a licensed resolver/DNS feed. (4) Spamhaus public DNS is free only for low-volume, non-commercial use; commercial/high-volume users need the Spamhaus Data Query Service (DQS) key, which changes the zone host to `<dqs-key>.zen.dq.spamhaus.net` etc. The `dnsbl_zones` table carries `requires_registration` and `is_paid` so the engine skips or degrades a zone cleanly and the UI explains *why* a zone is inconclusive rather than silently omitting it. Queries are cached for the audit run and honor DNS TTL; a small concurrency cap (e.g. 8 in-flight) plus jittered retry avoids tripping rate limits.

**Severity mapping.** Per-zone `weight` (0.0-1.0) in `dnsbl_zones` scales the finding: high-trust IP/domain zones (Spamhaus Zen/SBL/XBL/DBL, Barracuda, SURBL, DroneBL) map a positive listing to **critical**; PBL, SpamCop (auto-expiring), PSBL, WPBL, Mailspike-BL, SpamRATS, Lashback map to **warning**; UCEPROTECT L2/L3, Backscatterer, geographic policy lists, and pure-reputation advisories (Sender Score, Mailspike-rep) map to **info** unless the score crosses a configured threshold. The category roll-up (`dnsbl.aggregate`) takes the worst *weighted* severity, so ASN-wide collateral alone cannot turn the Blacklists cell red.

## 5. Information schema (database schema)

Reference the shared core tables (owned by engineering, do **not** redefine): `domains`, `sending_ips`, `dkim_selectors`, `audit_runs`, `findings`. This check adds one **reference/config** table and one **result** table.

```sql
-- Reference/config: the configurable zone list (today lives in config.yaml).
CREATE TABLE dnsbl_zones (
    id                    SERIAL PRIMARY KEY,
    zone_host             TEXT NOT NULL UNIQUE,          -- e.g. 'zen.spamhaus.org', 'dbl.spamhaus.org'
    kind                  TEXT NOT NULL CHECK (kind IN ('ip','domain')),
    weight                NUMERIC(3,2) NOT NULL DEFAULT 1.00 CHECK (weight BETWEEN 0 AND 1),
    delisting_url         TEXT NOT NULL,                 -- per-zone removal portal
    description           TEXT,
    return_code_map       JSONB,                         -- { "127.0.0.2":"SBL", "127.0.0.4":"XBL", ... }
    enabled               BOOLEAN NOT NULL DEFAULT TRUE,
    requires_registration BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. Barracuda resolver reg, Abusix key
    is_paid               BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. invaluement, Spamhaus DQS commercial
    created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_zones_enabled ON dnsbl_zones (enabled);
CREATE INDEX idx_dnsbl_zones_kind    ON dnsbl_zones (kind);

-- Result: one row per (audit_run, target, zone) tuple actually queried.
CREATE TABLE dnsbl_check_results (
    id            BIGSERIAL PRIMARY KEY,
    audit_run_id  BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id     BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    zone_id       INTEGER NOT NULL REFERENCES dnsbl_zones(id),
    target        TEXT NOT NULL,        -- the exact IP or domain tested
    target_kind   TEXT NOT NULL CHECK (target_kind IN ('ip','domain')),
    listed        BOOLEAN NOT NULL,
    return_code   TEXT,                 -- '127.0.0.4' (null when not listed)
    sub_list      TEXT,                 -- decoded label from return_code_map, e.g. 'XBL'
    reason_txt    TEXT,                 -- raw TXT answer
    inconclusive  BOOLEAN NOT NULL DEFAULT FALSE, -- zone unavailable / rate-limited / not registered
    query_ms      INTEGER,              -- resolution latency for diagnostics
    checked_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_results_domain_listed ON dnsbl_check_results (domain_id, listed);
CREATE INDEX idx_dnsbl_results_zone          ON dnsbl_check_results (zone_id);
CREATE INDEX idx_dnsbl_results_run           ON dnsbl_check_results (audit_run_id);
```

**Config.yaml → table mapping.** Today the zone catalog lives in `~/.email_delivery_hero/config.yaml` under a `dnsbl.zones` list; each entry (`zone_host`, `kind`, `weight`, `delisting_url`, `description`, `enabled`, `requires_registration`, `is_paid`, `return_code_map`) is exactly one future `dnsbl_zones` row — the DB migration is a straight load of that list, and until then the admin "Blocklist Zones" panel writes back to `config.yaml`. Paid/registration secrets (Spamhaus DQS key, Abusix key, Barracuda-registered resolver IP) live under `config.yaml`'s `secrets:`/`resolvers:` blocks, not in `dnsbl_zones`.

**Result → file store mapping.** Each audit's structured results are written to `~/.email_delivery_hero/audits/<domainId>/<auditId>.json` under a `dnsbl` key: `{ "dnsbl": { "results": [ { "zone":"zen.spamhaus.org", "target":"203.0.113.24", "targetKind":"ip", "listed":true, "returnCode":"127.0.0.4", "subList":"XBL", "reasonTxt":"...", "inconclusive":false, "queryMs":42, "checkedAt":"..." } ], "summary": { "listedCount":2, "cleanCount":45, "inconclusiveCount":0, "weightedWorst":"critical" } } }`. Each array element becomes one `dnsbl_check_results` row (`audit_run_id`/`domain_id` from the enclosing audit); the derived `findings` still land in the shared `findings` table/`findings[]` array via the normal Checker→Finding contract. The `(domain_id, listed)` index backs the dashboard "any listing?" query; the `(zone_id)` index backs "which domains does zone X hit?" reporting.

## 6. When we run the check

**On-demand.** Runs on every manual audit (`POST /api/domains/:id/audit`) and category-scoped run. Because listings appear and clear on their own (SpamCop expires in ~24h, XBL clears minutes after cleanup), a fresh manual run is the primary "am I delisted yet?" tool after the user submits a removal request.

**Periodic.** The in-process scheduler re-runs per domain at the `config.yaml` cadence (default every 6h for the Blacklists category — more frequent than static DNS checks because reputation is volatile). Each run **diffs against the previous audit**: a zone flipping `listed:false → true`, or a `warning → critical` sub-list escalation, is a **new problem** surfaced by regression detection and (per [../scheduled_checks.mdx](../scheduled_checks.mdx)) eligible for alerting; a `listed:true → false` transition is reported as a resolved/improved item. Inconclusive→listed and listed→inconclusive transitions are *not* treated as regressions to avoid resolver-blockage false alarms.

**Cost / rate-limit / back-off.** Total queries per audit = (IP zones × sending IPs) + (domain zones × domains); with ~15 zones and a handful of IPs this is well under 100 DNS lookups. To stay inside free-tier limits: cap concurrency (~8), reuse a dedicated non-public resolver where zones block open resolvers, honor TTL caching within a run, apply exponential back-off with jitter on `SERVFAIL`/timeout, and **auto-disable** a zone for the run if its RFC 5782 test point indicates blocking (record `inconclusive=true`). Paid/registration-gated zones (`is_paid`/`requires_registration` without configured credentials) are skipped, not errored.

## 8. Acceptance criteria

1. For an IPv4 target the checker constructs the query name by reversing octets and appending `zone_host` (verified: `203.0.113.24` + `zen.spamhaus.org` → `24.113.0.203.zen.spamhaus.org`); for a domain/RHSBL zone it queries `<domain>.<zone_host>` with **no** reversal.
2. An `A` answer inside `127.0.0.0/8` (or a zone's documented `127.0.1.x`/`127.0.4.x` space) sets `listed=true`, stores the exact `return_code`, and decodes it to `sub_list` via the zone's `return_code_map`; `NXDOMAIN` sets `listed=false`.
3. A `TXT` answer at the same name is stored as `reason_txt`; when it contains a removal URL that is used, otherwise the finding falls back to `dnsbl_zones.delisting_url`.
4. Every non-ok finding's `remediation` contains a concrete, correct per-zone delisting URL plus the reason/return code and the underlying-cause guidance — never a generic message.
5. Per-zone `weight` maps listings to severity as specified: high-trust zones → critical, expiring/policy zones → warning, ASN-wide (UCEPROTECT L2/L3) and pure-reputation advisories → info; the category roll-up uses the worst **weighted** severity, and info/collateral alone does not turn the Blacklists cell amber or red.
6. The RFC 5782 test-point probe (`2.0.0.127.<zone>`→`127.0.0.2`, `1.0.0.127.<zone>`→`NXDOMAIN`) runs once per zone per audit; on failure the zone's results are marked `inconclusive=true` (info), never false-listed.
7. Zones flagged `requires_registration` or `is_paid` without configured credentials are skipped with an `inconclusive`/`info` note explaining why, and the UI links the fix.
8. `dnsbl_zones` and `dnsbl_check_results` exist with the specified PKs, FKs (`audit_run_id`, `domain_id`, `zone_id`), and indexes on `(domain_id, listed)` and `(zone_id)`; the `config.yaml` `dnsbl.zones` list round-trips 1:1 to `dnsbl_zones`, and each result serializes into the audit JSON under the `dnsbl` key.
9. The scheduler diff flags any clean→listed or warning→critical transition as a new problem and any listed→clean as resolved, while ignoring inconclusive transitions.
10. The audit-detail view groups `dnsbl` findings with severity badge, target, observed return code + reason, a working [Delist ↗] link, and a copy-to-clipboard of the full fix; clean zones are collapsed; the admin Blocklist Zones panel edits weight/enabled/registration state.
11. The zone catalog shipped as default config matches [§9](#9-the-complete-blocklist-catalog--where-every-list-is-and-how-to-reach-it): no zone from the dead-zone registry (§9.5) is ever queried, refusal codes (`127.255.255.x`, `URIBL_BLOCKED`) are decoded as inconclusive, and each catalog row carries a working lookup URL and delisting URL.
12. Every audit writes the [test_results.yaml](#12-test-results--the-yaml-we-persist-per-run) structure (schema §12) and every non-healthy run maps to at least one named [problem state](#16-problem-states--the-catalog-of-healthy-and-unhealthy-situations) (§16), which the UI names verbatim.
13. The Blacklists summary card (§13.1) shows the pass/fail roll-up and worst problem, and its chevron opens the full-page Blacklists technology view (§13.2) with the layout of §14.
14. Each problem-state deep-dive page (§16) shows the concept explanation, the observed data fields, the diagnose-it-yourself commands, and the "progress forward" checklist for that state.

## 11. The tests we want to perform

The Blacklists audit is a fixed sequence; every numbered test emits rows into `test_results.yaml` (§12) and maps onto the sub-checks of §2.

1. **Target discovery** — assemble the target set: registered `sending_ips`, IPs resolved from MX hosts, IPs authorized by SPF (`a`/`mx`/`ip4`/`ip6` expansion), the sending domain, Return-Path and DKIM `d=` domains. Record where each target came from (`source`).
2. **Zone-health preflight (RFC 5782)** — per enabled zone, once per audit: `2.0.0.127.<zone>` must answer `127.0.0.2`; `1.0.0.127.<zone>` must be NXDOMAIN. Classify each zone `ok | dead | wildcarding | blocked | slow`. Dead/wildcarding zones are excluded and flagged for the catalog maintainer ([PS-10](#ps-10--dead-or-wildcarding-zone-false-listing)).
3. **Resolver-sanity test** — detect refusal codes: any Spamhaus answer in `127.255.255.0/24`, any URIBL answer `127.0.0.1` → the resolver is blocked/rate-limited ([PS-9](#ps-9--queries-refused--resolver-blocked)); switch to the configured fallback resolver and re-probe.
4. **IP listing sweep** — every (sending IP × enabled IP zone): reversed-octet `A` query; on a hit, `TXT` at the same name; decode `return_code` → sub-list via the zone's `return_code_map`.
5. **Domain listing sweep** — every (domain target × enabled RHSBL zone): direct `A`/`TXT` query, same decoding (`127.0.1.x` DBL categories, SURBL/URIBL bitmasks).
6. **Positive-reputation checks** — DNSWL presence (`list.dnswl.org` — absence is `info`, presence records category/trust), Sender Score (`score.senderscore.com` — decode 0-100), Mailspike rep zone.
7. **Root-cause context probes** (attached to any hit, to power the deep-dive pages): PTR/FCrDNS of the listed IP (`dig -x`), ASN/org (`mmdblookup`/Team Cymru), domain age (`whois` creation date), open-relay quick test (future: `swaks`/nmap `smtp-open-relay` behind explicit user consent — it talks to the MX).
8. **Provider-portal checklist** — not queryable; emit one checklist row per §9.7 portal with its check URL and last-acknowledged state so the user can mark "verified clean at Google Postmaster / SNDS".
9. **Diff against previous run** — clean→listed = new problem (alertable), listed→clean = resolved, sub-list escalation = escalation; inconclusive transitions are ignored (§6).
10. **Oracle cross-check (CI only, not in the product runtime)** — the Jest suite validates our checker against `dig +noall +comments` output and, for the auth-adjacent checks, against `checkdmarc` JSON, using recorded fixtures so tests run offline.

## 13. User interface — summary card and the chevron full page

The Blacklists technology follows the product-wide pattern: **one summary surface on the main view, a chevron (`›`) that opens a full page dedicated purely to this one technology.** (This grows §4; the design language — colors, StatusCell, finding rows — is locked by [../ui.mdx](../ui.mdx).)

### 13.1 The summary card (main view)

The most important single fact a user needs at a glance is: **"am I listed anywhere that matters, and is that new?"** The card answers exactly that and nothing else:

```
┌──────────────────────────────────────────────────────────────┐
│  ⛔ Blacklists                                          ›    │
│                                                              │
│  ● 2 listed · 92 clean · 2 unknown        [as of 09:30]      │
│  Worst: Spamhaus XBL — 203.0.113.24 (compromised host)       │
│  ▲ NEW since yesterday's run                                 │
└──────────────────────────────────────────────────────────────┘
```

* Card background follows the four status colors (§ui 1.2): red here because the worst weighted finding is critical.
* Line 1: the roll-up counts from `summary` (listed / clean / inconclusive), plus run recency.
* Line 2: the single worst listing — zone, target, decoded sub-list in plain words. Never more than one problem on the card.
* Line 3 (conditional): the diff chip — `▲ NEW`, `▼ resolved`, or nothing when unchanged. New-listing state is the one thing worth interrupting a glance for.
* The chevron `›` (whole card is clickable) routes to the full Blacklists page. No other interaction lives on the card.

### 13.2 The full page (`/blacklists/:domain` — one technology, everything about it)

The full page shows **every pass and every fail**, with the debugging values (the exact DNS query name, the raw return code, the TXT answer, the target IP, the PTR, the resolver used) visible without leaving the page. Layout detail is §14; content blocks top-to-bottom:

1. **Verdict header** — status color band, `2 listed · 92 clean · 2 unknown across 47 zones`, domain, run timestamp, **[Re-check now]** button, and the diff chips (`+1 new`, `−1 resolved`).
2. **Problems table** — only the listed/inconclusive rows (§14 defines columns). Each row expands (chevron per row) to the **debug drawer**: the exact query name we sent (`24.113.0.203.zen.spamhaus.org`), raw answer (`127.0.0.4`), decoded sub-list, full TXT, resolver used, query latency, target's PTR/ASN context, and the matched [problem state](#16-problem-states--the-catalog-of-healthy-and-unhealthy-situations) with a **"Explain & fix this →"** link to that state's deep-dive page (§16).
3. **Fix panel** — the prioritized recommendation list generated per §15 for this run's findings (root cause first, then delist links in weight order).
4. **Provider portals checklist** — the §9.7 link-out rows with per-portal "mark verified" state, because clean public zones ≠ clean at Gmail/Microsoft.
5. **All zones (passed too)** — the full pass/fail matrix, collapsed by default behind "Show all 47 zones ▸": every zone × target with a green ✓ / red ✕ / gray ? and its query evidence on hover. This is where "show what passed" lives — visible, but never above the problems.
6. **History strip** — listed-count over the last 30 runs (sparkline), fed by the per-run `test_results.yaml` files; clicking a point loads that run's page read-only.

**States.** *Never-run:* gray page with a single [Run blacklist check] CTA. *Running:* the §progress_ui dock shows "querying 47 zones…" with per-zone ticks. *All clean:* green verdict header, "No listings across 47 zones", the fix panel is replaced by the **prevention** recommendations (§15.3), and the zones matrix is expanded by default (there is nothing above it to read). *Inconclusive-heavy:* amber banner naming the cause (resolver blocked / registration missing) linking PS-9's deep dive.

## 14. UX design — what goes where, and why

Spoken as the UX designer. The page is a **triage funnel**: severity decides vertical position; evidence hides one level down; navigation deeper is always a chevron. The user we design for arrives anxious ("my mail is bouncing") — the page must answer *am I in trouble?* in the first 100px, *what exactly is wrong?* in the first scroll, and *what do I do?* without leaving.

**Vertical order (top → bottom), fixed:**

1. **Verdict band** (full-width, ~96px) — status color, one sentence, counts, [Re-check now] right-aligned. Highest position because it's the answer to the question that brought the user here. Nothing decorative above it.
2. **Problems table** — the only table on the first screenful. Sorted worst-first by weighted severity, then tier, then zone name. Columns, left→right in scanning order: `Severity` (badge) · `List` (zone label + tier chip HIGH/MED/LOW) · `Target` (IP or domain, monospace) · `Meaning` (decoded sub-list in words: "compromised host", "policy: dynamic range") · `Since` (first-seen run, relative) · `Fix` ([Delist ↗] + [Copy fix]) · row-chevron `›`. Rationale: severity and list identity are what the user reports to their boss; the fix affordance is at line-end where the eye lands after reading the row. Row height 44px minimum; the whole row toggles the debug drawer.
3. **Debug drawer** (inside a row, indented, `bg-muted`) — two-column key/value grid, all values monospace and copy-on-click: `Query`, `Answer`, `TXT`, `Resolver`, `Latency`, `PTR`, `ASN`, `Domain age` where relevant. Terminates with the "Explain & fix this →" deep-dive link. Debug values are one level down because they support the row's claim but would destroy scannability inline.
4. **Fix panel** ("How to fix", card) — ordered list, never more than 5 visible ("+N more" expands). Each item: one imperative sentence + its link. Sits below the problems because fixing follows understanding, above the pass matrix because action beats reassurance.
5. **Provider portals checklist** — compact rows: portal name · [Check ↗] · state pill (`Unverified` gray / `Clean` green / `Problem` red, user-set). Placed here because it is a *manual* extension of the fix workflow.
6. **All-zones matrix** (collapsed) — table of zone × target cells (✓/✕/?), tier-grouped with HIGH first, sticky first column. Collapsed because 90+ green cells are emotional comfort, not information; expanded automatically only in the all-clean state.
7. **History sparkline** — last, because trend is reflection, not triage.

**Horizontal rules.** One content column, max-width ~1100px — tables need width; no right rail (the debug drawer replaces it, keeping evidence adjacent to its claim). All external links (`Delist ↗`, `Check ↗`) open new tabs with the ↗ affix; internal drill-downs use the chevron `›` exclusively — chevron always means "deeper inside EmailDeliveryHero", ↗ always means "leaving".

**Typography & color.** Zone names, IPs, DNS answers, TXT strings: monospace. Severity uses the locked four-color system only — tier chips are neutral outline chips so tier never visually competes with severity. `info` findings never render red or amber anywhere on this page.

**Empty/edge behaviors.** >10 problems: table paginates at 10 with severity groups kept intact. A single inconclusive zone renders as gray `?` in the matrix, not in the problems table, unless *every* HIGH-tier zone is inconclusive — then PS-9 escalates it to the problems table as one row (cause, not 47 symptoms). Mobile (~<768px): the problems table reflows to stacked cards, `Fix` actions stay visible without horizontal scroll; the matrix becomes per-zone accordion lists.

## 15. Recommendations — how we tell the user to fix it

The recommendation engine turns `results` + `problem_states` into an ordered, concrete playbook. Rules:

### 15.1 Ordering (always)

1. **Root cause before delisting** — every listing recommendation is preceded by its cause fix (a delist request with the cause live gets re-listed and burns credibility with the operator). The cause comes from the decoded sub-list: XBL/DroneBL → find and clean the compromised host; SCBL/PSBL/WPBL traps → list hygiene + stop the source; SpamRATS NoPtr/Dyna → fix PTR/FCrDNS; DBL abused-legit → patch the hacked site/open redirect.
2. **Highest weighted impact first** — Spamhaus/Barracuda/SURBL before medium zones; LOW-tier collateral last and explicitly labeled "will not measurably affect Gmail/Outlook/Yahoo".
3. **One delist link per listing, the right one** — from `reason_txt` when the TXT carries the canonical URL, else the catalog's `delist_url`. Deep-link the target where the portal supports it.

### 15.2 Non-negotiable advisories (baked into the copy)

* **Never pay for delisting.** UCEPROTECT "express delisting"/whitelisted.org, Backscatterer express, and paid-expedite flows are flagged with: "This list auto-expires (7 days–4 weeks). Paying is unnecessary and funds a practice the industry considers abusive (RFC 6471)." SPFBL's paid-expedite gets the same treatment with "wait for the free path".
* **Auto-expiring lists: consider waiting.** SpamCop ~24h, PSBL 2-4 weeks, WPBL ~10 days, Backscatterer 4 weeks — the recommendation states the expiry window so the user can choose "fix cause + wait" over form-filling.
* **Collateral listings: you may not be the problem.** UCEPROTECT L2/L3, SEM netbl, ASN-wide states get: "caused by your network neighbors; escalate to your hosting provider's abuse desk (whois `OrgAbuseEmail`: <value>) or plan an IP/provider move if placement is actually suffering."
* **After any delist request:** "re-check in EmailDeliveryHero after the operator's stated processing window; the scheduler will alert you when it clears" — the re-audit loop from §6.

### 15.3 Prevention recommendations (shown when clean, and appended after fixes)

Stay-clean guidance rendered in the fix panel's place on all-clean runs: keep FCrDNS valid on every sending IP; keep SPF/DKIM/DMARC green (cross-link those check pages); register at **DNSWL** (free, positive signal); enroll in **Microsoft SNDS/JMRP** and **Google Postmaster Tools** so provider-side reputation is observable; keep the 6-hour scheduled cadence on so new listings alert within hours; warm up new IPs/domains gradually (ties to PS-11); monitor the `emails/` ingested DMARC/TLS-RPT reports for early bounce signals.

## 16. Problem states — the catalog of healthy and unhealthy situations

Every run maps to one or more named states (`summary.problem_states`). PS-0 is the healthy state; the rest are unhealthy or degraded. Each state below defines: **data fields** (from `test_results.yaml`), **diagnose & debug** (the exact commands the deep-dive page shows), **tools**, **further health metrics** to test, and the **UI deep page** — because from the full page (§13.2) the user can keep diving: problems-table row → debug drawer → "Explain & fix this →" → `/blacklists/:domain/state/:psId`, a page purely about that one state that explains the concept, shows this domain's evidence, and lays out how to progress forward.

Severity shown is the default after §3 weighting.

| State | Name | Severity | One-line trigger |
|---|---|---|---|
| PS-0 | All clean | ok | Every enabled zone queried, zero listings, refusals none |
| PS-1 | High-trust manual listing (SBL/CSS-class) | critical | Spamhaus SBL/CSS or Barracuda lists a sending IP |
| PS-2 | Compromised-host listing (XBL/DroneBL-class) | critical | Exploit/botnet zone lists a sending IP |
| PS-3 | Policy listing — dynamic/consumer IP space (PBL-class) | warning | PBL/SpamRATS-Dyna say this IP shouldn't send direct-to-MX |
| PS-4 | Sending domain on a domain blocklist (DBL/SURBL/URIBL) | critical | RHSBL lists the domain itself |
| PS-5 | Trap-driven auto-expiring listing (SCBL/PSBL/WPBL-class) | warning | Spam-trap list hit; expires on its own |
| PS-6 | Collateral neighborhood/ASN listing | info | UCEPROTECT L2/L3, SEM-netbl, range listings you didn't cause |
| PS-7 | rDNS/FCrDNS defect driving listings | warning | Missing/generic PTR; SpamRATS NoPtr/Dyna and PBL follow from it |
| PS-8 | Provider-side block, all public lists clean | warning | Bounces from Gmail/Microsoft/Yahoo while §9 zones are green |
| PS-9 | Queries refused / resolver blocked | info (degraded) | Spamhaus `127.255.255.x` or `URIBL_BLOCKED` seen |
| PS-10 | Dead or wildcarding zone (false listing) | info | RFC 5782 probe fails; zone lists 127.0.0.1 / the world |
| PS-11 | New-domain zero reputation | info | Domain <30 days old; ZRD/SEM-FRESH class signals |
| PS-12 | No positive reputation established | info | Not on DNSWL; Sender Score absent/low but no listings |
| PS-13 | Pay-to-delist trap encountered | info (advisory) | The only listings are on paid-expedite operators |

### PS-0 — All clean (the healthy state)

**Concept.** Every enabled zone answered, nothing listed, no refusals — the public-blocklist layer says receivers have no reason to block on reputation lists. **Data fields:** `summary.listed=0`, `inconclusive=0`, `zone_health[*].status=ok`, `positive_reputation.*`. **Diagnose:** nothing to debug; the deep page becomes the *prevention* page (§15.3) plus the positive-reputation panel (DNSWL status, Sender Score trend). **Further health:** provider portals still unverified → prompt to enroll (Postmaster Tools, SNDS). **UI deep page:** green hero, "what keeps you clean" checklist with per-item state, the 30-run history strip, and the next scheduled check time.

### PS-1 — High-trust manual listing (SBL/CSS-class)

**Concept explained to the user.** A human or high-confidence system at Spamhaus/Barracuda decided traffic from this IP is a spam operation or persistent problem — this is not automated collateral; it will not expire quietly, and it devastates placement. **Data fields:** `results[]` row with `tier=high`, `sub_list∈{SBL,CSS}` or zone `b.barracudacentral.org`; `reason_txt` (carries the SBL case reference); target's `ptr`, `asn`. **Diagnose & debug:** `dig +short 24.113.0.203.zen.spamhaus.org` (expect `127.0.0.2/.3`); `dig +short TXT <same>` for the case URL; `whois 203.0.113.24` for netblock owner (is the listing range-wide?); check neighboring IPs with `ipcalc <cidr>` + loop to see if the whole range is swept. **Tools:** dig/doggo, whois, ipcalc, check.spamhaus.org lookup. **Further health metrics:** volume/complaint history (ingested DMARC reports), open-relay test on the MX, outbound queue inspection for the flagged traffic. **Progress forward (deep page):** ① identify and stop the cited operation (the TXT case link says what they saw), ② document the fix, ③ submit the SBL/Barracuda removal with that documentation, ④ status auto-rechecks each scheduled run; expect human review latency (SBL) or <24h (Barracuda).

### PS-2 — Compromised-host listing (XBL/DroneBL-class)

**Concept.** The IP behaved like malware: botnet callbacks, proxy abuse, or exploit traffic. Something on (or NATed behind) this IP is infected or open. Delisting before cleaning guarantees re-listing. **Data fields:** `sub_list=XBL` (`127.0.0.4-.7`) or DroneBL class code (`.3/.8-.10/.13/.14/.16` — the code says drone vs proxy vs DDoS vs breached); `ptr`, `asn`; NAT note when the IP fronts many machines. **Diagnose & debug:** `dig +short TXT 24.113.0.203.zen.spamhaus.org` (XBL TXT links the detection detail incl. last-seen timestamp and port); on the host: unexpected outbound 25/6667/SOCKS connections (`lsof -i -nP | grep -E ':25|:1080|:6667'`), mail queue runaway, cron/webshell audit; `nmap -p25,465,587 --script smtp-open-relay <ip>` for open relay. **Tools:** dig, nmap (`smtp-open-relay`, `dns-blacklist`), host-level AV/EDR, provider flow logs. **Further health metrics:** re-scan all §9.1 exploit-class zones after cleanup; watch `z.mailspike.net` zero-hour for recurrence. **Progress forward:** ① find the infected machine/service (the deep page lists the TXT's port/timestamp hints), ② clean/patch/credential-rotate, ③ self-delist (XBL allows immediate self-removal once clean; DroneBL via lookup page), ④ if it recurs, assume re-infection not list error.

### PS-3 — Policy listing: dynamic/consumer IP space (PBL-class)

**Concept.** Not an accusation of spam: the network's owner (or Spamhaus policy) declared this range shouldn't emit direct-to-MX mail — typical for home/cloud-dynamic IPs. Receivers reject on it because botnets live in such space. **Data fields:** `sub_list=PBL` (`127.0.0.10` ISP-maintained / `.11` Spamhaus-maintained — the code says who to talk to), SpamRATS `.36`; `asn.org` (consumer/cloud provider?), `ptr` (generic-looking `ip-203-0-113-24.provider.net`?). **Diagnose & debug:** `dig +short 24.113.0.203.pbl.spamhaus.org`; `dig +short -x 203.0.113.24` (generic PTR confirms policy space); `mmdblookup … autonomous_system_organization`. **Tools:** dig, mmdblookup/whois. **Further health metrics:** FCrDNS check ([./reverse_dns.mdx](./reverse_dns.mdx)), swaks send test through the proper smarthost. **Progress forward — a fork the deep page draws explicitly:** *(a) this IP shouldn't send mail* → route outbound through the provider's smarthost or an ESP, done — the listing is then correct and harmless; *(b) this is a legitimate static MTA* → get a proper PTR from the provider, then PBL self-exclusion at the check.spamhaus.org flow (`.10` listings may also need the ISP's own form since the ISP owns that entry).

### PS-4 — Sending domain on a domain blocklist (DBL/SURBL/URIBL)

**Concept.** The *name*, not the IP, is listed — every message carrying the domain in From/Return-Path/URLs is suspect no matter which IP sends it. Causes split into: the domain really spams/phishes, or "abused-legit" (site hacked, open redirect, compromised shortener). **Data fields:** RHSBL `results[]` rows; DBL code distinguishes `127.0.1.2-.6` (bad domain) vs `.102-.106` (**abused-legit** — different fix); SURBL/URIBL bitmask; `targets.domains[*].source` (was it the primary domain or a Return-Path/DKIM domain?); domain `created` date. **Diagnose & debug:** `dig +short whitehatengineering.com.dbl.spamhaus.org` (never query an IP at DBL — `127.0.1.255` is the you-did-it-wrong code); `dig +short TXT <same>`; `dnstwist --registered --format json <domain>` (lookalike confusion); site-side: check for injected pages/open redirects (Google Search Console security tab). **Tools:** dig, dnstwist, SURBL/URIBL lookup pages, Search Console. **Further health metrics:** [./link_url_reputation.mdx](./link_url_reputation.mdx) body-URL sweep (are other domains you link also listed?), DKIM `d=` and Return-Path domains individually. **Progress forward:** abused-legit codes → ① find and close the compromise (hacked CMS, open redirect), ② purge injected content, ③ delist via the zone's flow with the fix described; bad-domain codes on a domain you own legitimately → contest via the same flow with evidence; then re-audit and watch the domain-side zones for 2 weeks.

### PS-5 — Trap-driven auto-expiring listing (SCBL/PSBL/WPBL-class)

**Concept.** Mail from this IP hit a spam trap — an address that never opted in. Usually list-hygiene rot (purchased lists, ancient addresses, scraping) rather than infection. These lists forgive: they expire on their own once the traffic stops. **Data fields:** zone ∈ {`bl.spamcop.net`, `psbl.surriel.com`, `db.wpbl.info`, `truncate.gbudb.net`}; `diff` (recurring or first-time?); expiry window per zone (SCBL ~24h, PSBL 2-4wk, WPBL ~10d) carried in catalog metadata. **Diagnose & debug:** `dig +short 24.113.0.203.bl.spamcop.net`; SpamCop's lookup page shows report ages; cross-reference send logs around the listing time; check for a specific campaign/segment as the source. **Tools:** dig, SpamCop bl.shtml lookup, your ESP's suppression/bounce logs, ingested DMARC reports ([../emails.mdx](../emails.mdx)) for the volume spike. **Further health metrics:** bounce-rate trend, list-acquisition audit, confirm no PS-2 co-listing (trap + exploit zones together = infection, not hygiene). **Progress forward:** ① identify the sending stream that hit the trap, ② remove non-engaged/unverified addresses (or the whole acquired segment), ③ either wait out the expiry (the page shows the countdown) or self-delist (PSBL is instant), ④ recurrence = the hygiene fix didn't take.

### PS-6 — Collateral neighborhood/ASN listing

**Concept.** A range containing your IP — /24, allocation, or the whole ASN — got listed because of *neighbors*. You personally may be clean. Impact at major receivers is near zero (Gmail/Microsoft/Yahoo don't use UCEPROTECT L2/L3), but small self-hosted receivers occasionally do. **Data fields:** zone ∈ {`dnsbl-2/3.uceprotect.net`, `netbl.spameatingmonkey.net`, ivmSIP/24}; `asn`; whois `OrgAbuseEmail`. **Diagnose & debug:** `dig +short <rev>.dnsbl-2.uceprotect.net`; uceprotect.net rblcheck page names the allocation and the "poison" IPs; `whois <ip>` → abuse contact; `ipcalc` the allocation to see its width. **Tools:** dig, whois, ipcalc, uceprotect lookup. **Further health metrics:** confirm your own IP is clean at L1 and all HIGH zones (if yes, this is pure collateral); measure actual placement via bounce logs before acting. **Progress forward:** ① verify you're clean everywhere that matters (the page shows the HIGH-tier matrix for reassurance), ② forward the evidence to the provider's abuse desk (pre-drafted email with the whois contact filled in — [Copy] button), ③ only if real placement suffers: plan an IP/provider move; ④ **never pay** the express-delist (auto-links PS-13 advisory).

### PS-7 — rDNS/FCrDNS defect driving listings

**Concept.** The IP has no PTR, a generic PTR, or PTR↛A mismatch. This is both directly rejected by many receivers *and* the trigger for SpamRATS NoPtr/Dyna and PBL-class policy listings — one DNS fix clears a family of problems. **Data fields:** `targets.ips[*].ptr`, `fcrdns_ok=false`; SpamRATS `.36/.37` rows. **Diagnose & debug:** `dig +short -x 203.0.113.24` (empty or `dynamic-…`/`ip-…` pattern = defect); forward-confirm: `dig +short A <ptr-value>` must include the IP; `swaks --server mx… --quit-after EHLO` to see the HELO/banner vs PTR mismatch a receiver sees. **Tools:** dig, swaks; the fix itself happens at the IP's owner (hosting provider console or ticket — PTR is delegated to whoever owns the reverse zone). **Further health metrics:** the full [./reverse_dns.mdx](./reverse_dns.mdx) check; HELO-name alignment ([./smtp_security.mdx](./smtp_security.mdx)). **Progress forward:** ① set PTR to the mail host's FQDN (e.g. `mail.example.com`) via the provider, ② ensure that FQDN's A record points back (FCrDNS), ③ align the MTA's HELO to the same name, ④ then delist SpamRATS (removal.php requires the PTR to already be fixed) and PBL if applicable — the deep page orders these dependencies explicitly.

### PS-8 — Provider-side block with all public lists clean

**Concept.** The user's real complaint ("Gmail junks me", "Microsoft 550 5.7.606") with a green §9 matrix. The blocking list is private — Gmail domain reputation, Microsoft's internal lists, Cloudmark/Proofpoint at carriers. Public-zone cleanliness is necessary, not sufficient. **Data fields:** `summary.listed=0` while the user reports bounces; `provider_portals[*].user_state`; ingested bounce/DMARC evidence (SMTP codes: Microsoft `5.7.606-649`, Gmail `550-5.7.1`, carrier Cloudmark text). **Diagnose & debug:** read the actual bounce (the deep page shows a paste-the-NDR box that pattern-matches the code to the provider and its exact delist path); check Postmaster Tools domain/IP reputation graphs; check SNDS filter verdict and trap hits; `https://csi.cloudmark.com/en/reset/` and `ipcheck.proofpoint.com` lookups for carrier blocks. **Tools:** the §9.7 portal links, paste-NDR classifier, swaks (reproduce the rejection: `swaks --to probe@outlook.com --server <their-mx>` — future, consent-gated), Postmaster/SNDS dashboards. **Further health metrics:** spam-rate from Postmaster Tools vs the 0.1%/0.3% thresholds, DMARC alignment health (Gmail bulk-sender rules), List-Unsubscribe presence ([./list_unsubscribe.mdx](./list_unsubscribe.mdx)). **Progress forward:** ① classify the bounce → provider, ② use that provider's delist/escalation channel (sender.office.com for 5.7.606, SNDS for consumer, Gmail escalation form, Cloudmark reset…), ③ fix the reputation inputs the portal shows (complaints, volume spikes, auth), ④ mark the portal row `verified_clean`/`problem_reported` so the checklist tracks it.

### PS-9 — Queries refused / resolver blocked

**Concept (about *us*, not the domain).** The blocklists refused to answer our resolver — public/open resolvers and heavy users get in-band error codes, not real answers. Results this run are partially blind; nothing here says the domain is unhealthy. **Data fields:** `resolver.mode/server`, `refusals_detected=true`, per-row `refusal_code` (`127.255.255.252/.254/.255` Spamhaus, `127.0.0.1` URIBL), `zone_health[*].status=blocked`. **Diagnose & debug:** `dig +short 2.0.0.127.zen.spamhaus.org @8.8.8.8` (returns the refusal code — demonstrates the problem) vs the same query `@<local-resolver>` (returns `127.0.0.2`); confirm what `resolver.server` is actually set to. **Tools:** dig with explicit `@server`, the app's Settings → resolver config. **Further health metrics:** per-zone `probe_ms` (rate-limiting shows as `.255`/timeouts under load); query-volume estimate vs Spamhaus's free-tier bounds. **Progress forward:** ① point the checker at a real recursive resolver (local `unbound`/`knot-resolver`, or the ISP's), not 8.8.8.8/1.1.1.1; ② for volume, get the free Spamhaus DQS key and Abusix key and set them in Settings (the zone hosts switch to `<key>.zen.dq.spamhaus.net` automatically); ③ register the resolver IP with Barracuda; ④ re-run — refused zones re-query automatically.

### PS-10 — Dead or wildcarding zone (false listing)

**Concept.** DNSBLs die, and dying zones sometimes wildcard — answering "listed" for every query (AHBL, SpamCannibal). A "listing" from such a zone is noise, and reporting it would be a false alarm; this state exists so the product *explains* a suppressed hit rather than silently hiding it. **Data fields:** `zone_health[*]` row with `status=dead|wildcarding` (`negative_probe` returned an answer for `127.0.0.1`); any suppressed `results[]` rows for that zone. **Diagnose & debug:** `dig +short 1.0.0.127.<zone>` (an answer = wildcarding); `dig +short 2.0.0.127.<zone>` (NXDOMAIN = dead/empty); check the zone against §9.5 and dnsbl.com. **Tools:** dig, multirbl.valli.org zone page, dnsbl.com registry. **Further health metrics:** none for the domain — this is catalog hygiene. **Progress forward:** the zone is auto-disabled for the run and flagged in the admin Blocklist Zones panel with the probe evidence; the operator confirms retirement (moves it to the dead registry) or re-enables after the operator's outage resolves. Users see: "zone X is dead/misbehaving and was excluded — this does not affect your standing."

### PS-11 — New-domain zero reputation

**Concept.** The domain is simply *young*. Spamhaus ZRD lists everything <24h old by definition; SEM-FRESH tracks fresh registrations; many receivers throttle new domains regardless. Not an accusation — a maturity gate every domain passes through. **Data fields:** `targets.domains[*].created` (whois), ZRD hour-code (`127.0.2.<h>`), SEM-FRESH hits, DKIM/SPF/DMARC status from sibling checks. **Diagnose & debug:** `whois <domain> | grep -i 'creation'`; `dig +short <domain>.fresh.spameatingmonkey.net`; confirm the listing ages out (ZRD at 24h). **Tools:** whois, dig. **Further health metrics:** authentication completeness (new domains need perfect SPF/DKIM/DMARC to build reputation fast), warm-up volume curve from send logs. **Progress forward:** ① wait out the age gates (the page shows domain age and each gate's threshold), ② send low, consistent volume to engaged recipients first, ③ ensure auth is fully green before scaling, ④ enroll Postmaster Tools/SNDS on day one so reputation formation is observable.

### PS-12 — No positive reputation established

**Concept.** Nothing is *wrong* — but nothing vouches for you either: not on DNSWL, no meaningful Sender Score, unknown at Mailspike-rep. Positive signals are a buffer against future gray-area filtering. **Data fields:** `positive_reputation.dnswl.listed=false`, `senderscore.score` null or <70, `mailspike_rep`. **Diagnose & debug:** `dig +short <rev>.list.dnswl.org` (NXDOMAIN = not registered); `dig +short <rev>.score.senderscore.com` (last octet = score); review volume consistency (Sender Score needs sustained volume to exist at all). **Tools:** dig, dnswl.org self-registration, senderscore.org. **Further health metrics:** complaint and bounce trends from ingested reports. **Progress forward:** ① register the MTA at dnswl.org (free tier, category "email marketing"/"corporate" as appropriate), ② keep volume steady so Sender Score materializes, ③ treat this page as done when DNSWL shows `trust≥1` and score ≥70 — severity stays `info` throughout; the UI never nags.

### PS-13 — Pay-to-delist trap encountered

**Concept (consumer-protection state).** The only open listings are on operators that sell "express" removal (UCEPROTECT/whitelisted.org, Backscatterer express, some SPFBL flows). The industry position (RFC 6471) is that payment-for-removal is abusive; these listings auto-expire and barely affect major-receiver placement. This state exists so the product actively stops the user from paying out of panic. **Data fields:** all open `results[]` rows have catalog flag `paid_delist_offered=true`; expiry windows; `diff` recurrence. **Diagnose & debug:** confirm no HIGH-tier co-listing (if Spamhaus is also red, *that* is the real problem — this state only fires when paid-trap zones are the *only* hits); verify the auto-expiry clock from the operator's lookup page. **Tools:** dig, the operator lookup pages, §9.1 matrix. **Further health metrics:** placement evidence (bounces naming these lists are rare — ask for real NDRs before caring). **Progress forward:** ① do not pay — the page says so in bold with the RFC 6471 citation, ② fix any genuine cause (backscatter → stop bounce-after-accept on the MTA, [./smtp_security.mdx](./smtp_security.mdx)), ③ wait for auto-expiry (countdown shown), ④ if a real correspondent's server uses these lists, hand them the RFC 6471 context via a pre-drafted note.
