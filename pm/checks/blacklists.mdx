# Blacklists (DNSBL / RHSBL / IP & Domain Reputation Blocklists)

A DNS-based blocklist (DNSBL) is a list of IP addresses вҖ” and, for right-hand-side blocklists (RHSBL), of domain names вҖ” that a receiving mail server queries over ordinary DNS to decide whether to reject, quarantine, or score a message. If your sending IP or your domain (or a domain that appears in your message From/Return-Path/URLs) is listed on a zone a receiver honors, your mail is throttled, junked, or hard-bounced regardless of how perfect your SPF/DKIM/DMARC are. This is the single fastest way for a good sender to fall out of the inbox, and the single check that customers ask about first ("am I blacklisted?"). This spec is the flagship of the **Blacklists** dashboard category; every sub-check id uses the **`dnsbl`** prefix.

## 1. What it checks

DNSBLs have no single RFC that defines the *lists* themselves, but the query mechanism is standardized by **RFC 5782 (DNS Blacklists and Whitelists)**, which formalizes the two families:

- **IPv4 DNSBL** вҖ” reverse the four octets of the target IP, append the zone host, and issue an `A` query. A listing is signaled by an answer in `127.0.0.0/8` (conventionally `127.0.0.2`вҖ“`127.0.0.255`); the specific low-order octet is the **return code** that tells you *why* it is listed. A `TXT` query at the same name returns a human-readable reason and usually a removal URL. RFC 5782 also mandates that a well-behaved zone answer `127.0.0.2` for the test address `127.0.0.2` and `NXDOMAIN` for `127.0.0.1`, which we use as a liveness probe.
- **RHSBL / domain DNSBL** вҖ” the domain is queried **directly, with no reversal**: `<domain>.<zone>`, `A` lookup, same `127.0.0.x` semantics. Spamhaus DBL, SURBL, and ivmURI are of this kind.

**Worked IP example.** To test `203.0.113.24` against `zen.spamhaus.org`:
1. Reverse octets: `24.113.0.203`.
2. Form the query name: `24.113.0.203.zen.spamhaus.org`.
3. `A` lookup. `NXDOMAIN` вҶ’ not listed. An answer such as `127.0.0.4` вҶ’ listed (Zen packs SBL, CSS, XBL, and PBL sub-lists into distinct return codes: `127.0.0.2`/`.3` = SBL/CSS, `127.0.0.4`вҖ“`.7` = XBL, `127.0.0.10`/`.11` = PBL).
4. `TXT` lookup at the same name вҶ’ the reason string and delisting link.

**Worked domain example.** To test `example.com` against `dbl.spamhaus.org`: query `example.com.dbl.spamhaus.org` directly (no reversal). `127.0.1.2` = spam domain, `127.0.1.4`вҖ“`.6` = phishing/malware/botnet C&C, `127.0.1.102`вҖ“`.106` = "abused legit" categories.

Why receivers care: DNSBL lookups are cheap (one cached DNS round-trip) and catch the majority of botnet and snowshoe spam at connection time, so nearly every MTA and spam engine (Postfix `reject_rbl_client`, SpamAssassin `RCVD_IN_*`, Rspamd, commercial gateways) consults several. The common real-world failure mode is a **shared-IP or ASN-wide listing you did not cause** вҖ” a neighbor on your hosting provider's IP block, a compromised mailbox on your subnet, or a provider whose whole ASN got swept into UCEPROTECT L3. A single listing on a high-trust zone (Spamhaus SBL/XBL/Zen) can drop inbox placement to near zero at the large mailbox providers; a listing on a low-trust or ASN-wide zone (UCEPROTECT L2/L3, some SORBS sub-zones) may cost only a few percent, which is exactly why per-zone **weighting** is essential вҖ” a naive "you're on a list!" alarm produces false panic.

## 3. How it is measured

**Targets.** For IP zones the checker iterates `ctx.sendingIps` (from `sending_ips`); if empty, it derives candidate IPs by resolving the domain's MX hosts and SPF `a`/`mx`/`ip4`/`ip6` mechanisms and testing those (an unexpected listing on an SPF-authorized IP is still your deliverability problem). For RHSBL/domain zones the target is `ctx.domain` and any additional envelope domains (Return-Path, DKIM `d=`).

**Query construction.**
- IPv4: split the dotted quad, reverse, join with `.`, append `zone_host`. `203.0.113.24` + `zen.spamhaus.org` вҶ’ `24.113.0.203.zen.spamhaus.org`.
- IPv6: expand to the 32 nibbles, reverse them, dot-separate, append the zone (RFC 5782). Most zones only carry IPv4 data; IPv6 lookups usually `NXDOMAIN` and are skipped unless the zone documents IPv6 support.
- Domain: `<domain>.<zone_host>` verbatim, lowercased, IDNA/punycode-encoded, no reversal.

**Resolution.** First round uses `node:dns/promises` `resolve4()` for the `A` answer and `resolveTxt()` for the reason. `NXDOMAIN` (`ENOTFOUND`/`ENODATA`) = **not listed**. Any `127.0.0.0/8` (or documented `127.0.1.0/8`, `127.0.4.0/8`) answer = **listed**; the full answer set is stored as `return_code` (e.g. `127.0.0.4`) and decoded against the zone's code map to a sub-list label and severity. A `TXT` answer is stored verbatim as `reason_txt` and mined for a removal URL; if none is present we fall back to `dnsbl_zones.delisting_url`.

**Zone liveness / test-point probe.** Before trusting an `NXDOMAIN` as "clean", the runner (once per zone per audit) issues RFC 5782 sanity queries: `2.0.0.127.<zone>` must return `127.0.0.2` and `1.0.0.127.<zone>` must return `NXDOMAIN`. If the test point fails (dead zone, blocked resolver, or вҖ” critically вҖ” a zone that returns `127.0.0.2` for *everything* because your resolver is being rate-limited/blocked), the sub-check yields **info** "zone unavailable / inconclusive", never a false listing.

**Resolver policy & rate limits.** Public mirrors of Spamhaus, Barracuda, SORBS, and others **block queries from large open resolvers (Google 8.8.8.8, Cloudflare 1.1.1.1)** and from high-volume clients. Consequences the checker must handle: (1) **Barracuda requires free registration** of the querying resolver IP before `b.barracudacentral.org` resolves вҖ” until registered the sub-check is disabled/`info`. (2) **Abusix requires a free API key** embedded in the query name; (3) **invaluement is paid** and needs a licensed resolver/DNS feed. (4) Spamhaus public DNS is free only for low-volume, non-commercial use; commercial/high-volume users need the Spamhaus Data Query Service (DQS) key, which changes the zone host to `<dqs-key>.zen.dq.spamhaus.net` etc. The `dnsbl_zones` table carries `requires_registration` and `is_paid` so the engine skips or degrades a zone cleanly and the UI explains *why* a zone is inconclusive rather than silently omitting it. Queries are cached for the audit run and honor DNS TTL; a small concurrency cap (e.g. 8 in-flight) plus jittered retry avoids tripping rate limits.

**Severity mapping.** Per-zone `weight` (0.0-1.0) in `dnsbl_zones` scales the finding: high-trust IP/domain zones (Spamhaus Zen/SBL/XBL/DBL, Barracuda, SURBL) map a positive listing to **critical**; PBL, SpamCop (auto-expiring), Mailspike-BL, SORBS sub-zones map to **warning**; UCEPROTECT L2/L3 and pure-reputation advisories (Sender Score, Mailspike-rep) map to **info** unless the score crosses a configured threshold. The category roll-up (`dnsbl.aggregate`) takes the worst *weighted* severity, so ASN-wide collateral alone cannot turn the Blacklists cell red.

Reference the shared core tables (owned by engineering, do **not** redefine): `domains`, `sending_ips`, `dkim_selectors`, `audit_runs`, `findings`. This check adds one **reference/config** table and one **result** table.

```sql
-- Reference/config: the configurable zone list (today lives in config.yaml).
CREATE TABLE dnsbl_zones (
    id                    SERIAL PRIMARY KEY,
    zone_host             TEXT NOT NULL UNIQUE,          -- e.g. 'zen.spamhaus.org', 'dbl.spamhaus.org'
    kind                  TEXT NOT NULL CHECK (kind IN ('ip','domain')),
    weight                NUMERIC(3,2) NOT NULL DEFAULT 1.00 CHECK (weight BETWEEN 0 AND 1),
    delisting_url         TEXT NOT NULL,                 -- per-zone removal portal
    description           TEXT,
    return_code_map       JSONB,                         -- { "127.0.0.2":"SBL", "127.0.0.4":"XBL", ... }
    enabled               BOOLEAN NOT NULL DEFAULT TRUE,
    requires_registration BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. Barracuda resolver reg, Abusix key
    is_paid               BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. invaluement, Spamhaus DQS commercial
    created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_zones_enabled ON dnsbl_zones (enabled);
CREATE INDEX idx_dnsbl_zones_kind    ON dnsbl_zones (kind);

-- Result: one row per (audit_run, target, zone) tuple actually queried.
CREATE TABLE dnsbl_check_results (
    id            BIGSERIAL PRIMARY KEY,
    audit_run_id  BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id     BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    zone_id       INTEGER NOT NULL REFERENCES dnsbl_zones(id),
    target        TEXT NOT NULL,        -- the exact IP or domain tested
    target_kind   TEXT NOT NULL CHECK (target_kind IN ('ip','domain')),
    listed        BOOLEAN NOT NULL,
    return_code   TEXT,                 -- '127.0.0.4' (null when not listed)
    sub_list      TEXT,                 -- decoded label from return_code_map, e.g. 'XBL'
    reason_txt    TEXT,                 -- raw TXT answer
    inconclusive  BOOLEAN NOT NULL DEFAULT FALSE, -- zone unavailable / rate-limited / not registered
    query_ms      INTEGER,              -- resolution latency for diagnostics
    checked_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_results_domain_listed ON dnsbl_check_results (domain_id, listed);
CREATE INDEX idx_dnsbl_results_zone          ON dnsbl_check_results (zone_id);
CREATE INDEX idx_dnsbl_results_run           ON dnsbl_check_results (audit_run_id);
```

**Config.yaml вҶ’ table mapping.** Today the zone catalog lives in `~/.email_delivery_hero/config.yaml` under a `dnsbl.zones` list; each entry (`zone_host`, `kind`, `weight`, `delisting_url`, `description`, `enabled`, `requires_registration`, `is_paid`, `return_code_map`) is exactly one future `dnsbl_zones` row вҖ” the DB migration is a straight load of that list, and until then the admin "Blocklist Zones" panel writes back to `config.yaml`. Paid/registration secrets (Spamhaus DQS key, Abusix key, Barracuda-registered resolver IP) live under `config.yaml`'s `secrets:`/`resolvers:` blocks, not in `dnsbl_zones`.

**Result вҶ’ file store mapping.** Each audit's structured results are written to `~/.email_delivery_hero/audits/<domainId>/<auditId>.json` under a `dnsbl` key: `{ "dnsbl": { "results": [ { "zone":"zen.spamhaus.org", "target":"203.0.113.24", "targetKind":"ip", "listed":true, "returnCode":"127.0.0.4", "subList":"XBL", "reasonTxt":"...", "inconclusive":false, "queryMs":42, "checkedAt":"..." } ], "summary": { "listedCount":2, "cleanCount":45, "inconclusiveCount":0, "weightedWorst":"critical" } } }`. Each array element becomes one `dnsbl_check_results` row (`audit_run_id`/`domain_id` from the enclosing audit); the derived `findings` still land in the shared `findings` table/`findings[]` array via the normal CheckerвҶ’Finding contract. The `(domain_id, listed)` index backs the dashboard "any listing?" query; the `(zone_id)` index backs "which domains does zone X hit?" reporting.

## 6. When we run the check

**On-demand.** Runs on every manual audit (`POST /api/domains/:id/audit`) and category-scoped run. Because listings appear and clear on their own (SpamCop expires in ~24h, XBL clears minutes after cleanup), a fresh manual run is the primary "am I delisted yet?" tool after the user submits a removal request.

**Periodic.** The in-process scheduler re-runs per domain at the `config.yaml` cadence (default every 6h for the Blacklists category вҖ” more frequent than static DNS checks because reputation is volatile). Each run **diffs against the previous audit**: a zone flipping `listed:false вҶ’ true`, or a `warning вҶ’ critical` sub-list escalation, is a **new problem** surfaced by regression detection and (per [../scheduled_checks.mdx](../scheduled_checks.mdx)) eligible for alerting; a `listed:true вҶ’ false` transition is reported as a resolved/improved item. InconclusiveвҶ’listed and listedвҶ’inconclusive transitions are *not* treated as regressions to avoid resolver-blockage false alarms.

**Cost / rate-limit / back-off.** Total queries per audit = (IP zones Г— sending IPs) + (domain zones Г— domains); with ~15 zones and a handful of IPs this is well under 100 DNS lookups. To stay inside free-tier limits: cap concurrency (~8), reuse a dedicated non-public resolver where zones block open resolvers, honor TTL caching within a run, apply exponential back-off with jitter on `SERVFAIL`/timeout, and **auto-disable** a zone for the run if its RFC 5782 test point indicates blocking (record `inconclusive=true`). Paid/registration-gated zones (`is_paid`/`requires_registration` without configured credentials) are skipped, not errored.

## 8. Acceptance criteria

1. For an IPv4 target the checker constructs the query name by reversing octets and appending `zone_host` (verified: `203.0.113.24` + `zen.spamhaus.org` вҶ’ `24.113.0.203.zen.spamhaus.org`); for a domain/RHSBL zone it queries `<domain>.<zone_host>` with **no** reversal.
2. An `A` answer inside `127.0.0.0/8` (or a zone's documented `127.0.1.x`/`127.0.4.x` space) sets `listed=true`, stores the exact `return_code`, and decodes it to `sub_list` via the zone's `return_code_map`; `NXDOMAIN` sets `listed=false`.
3. A `TXT` answer at the same name is stored as `reason_txt`; when it contains a removal URL that is used, otherwise the finding falls back to `dnsbl_zones.delisting_url`.
4. Every non-ok finding's `remediation` contains a concrete, correct per-zone delisting URL plus the reason/return code and the underlying-cause guidance вҖ” never a generic message.
5. Per-zone `weight` maps listings to severity as specified: high-trust zones вҶ’ critical, expiring/policy zones вҶ’ warning, ASN-wide (UCEPROTECT L2/L3) and pure-reputation advisories вҶ’ info; the category roll-up uses the worst **weighted** severity, and info/collateral alone does not turn the Blacklists cell amber or red.
6. The RFC 5782 test-point probe (`2.0.0.127.<zone>`вҶ’`127.0.0.2`, `1.0.0.127.<zone>`вҶ’`NXDOMAIN`) runs once per zone per audit; on failure the zone's results are marked `inconclusive=true` (info), never false-listed.
7. Zones flagged `requires_registration` or `is_paid` without configured credentials are skipped with an `inconclusive`/`info` note explaining why, and the UI links the fix.
8. `dnsbl_zones` and `dnsbl_check_results` exist with the specified PKs, FKs (`audit_run_id`, `domain_id`, `zone_id`), and indexes on `(domain_id, listed)` and `(zone_id)`; the `config.yaml` `dnsbl.zones` list round-trips 1:1 to `dnsbl_zones`, and each result serializes into the audit JSON under the `dnsbl` key.
9. The scheduler diff flags any cleanвҶ’listed or warningвҶ’critical transition as a new problem and any listedвҶ’clean as resolved, while ignoring inconclusive transitions.
10. The audit-detail view groups `dnsbl` findings with severity badge, target, observed return code + reason, a working [Delist вҶ—] link, and a copy-to-clipboard of the full fix; clean zones are collapsed; the admin Blocklist Zones panel edits weight/enabled/registration state.
