# Blacklists (DNSBL / RHSBL / IP & Domain Reputation Blocklists)

A DNS-based blocklist (DNSBL) is a list of IP addresses ŌĆö and, for right-hand-side blocklists (RHSBL), of domain names ŌĆö that a receiving mail server queries over ordinary DNS to decide whether to reject, quarantine, or score a message. If your sending IP or your domain (or a domain that appears in your message From/Return-Path/URLs) is listed on a zone a receiver honors, your mail is throttled, junked, or hard-bounced regardless of how perfect your SPF/DKIM/DMARC are. This is the single fastest way for a good sender to fall out of the inbox, and the single check that customers ask about first ("am I blacklisted?"). This spec is the flagship of the **Blacklists** dashboard category; every sub-check id uses the **`dnsbl`** prefix.

## 1. What it checks

DNSBLs have no single RFC that defines the *lists* themselves, but the query mechanism is standardized by **RFC 5782 (DNS Blacklists and Whitelists)**, which formalizes the two families:

- **IPv4 DNSBL** ŌĆö reverse the four octets of the target IP, append the zone host, and issue an `A` query. A listing is signaled by an answer in `127.0.0.0/8` (conventionally `127.0.0.2`ŌĆō`127.0.0.255`); the specific low-order octet is the **return code** that tells you *why* it is listed. A `TXT` query at the same name returns a human-readable reason and usually a removal URL. RFC 5782 also mandates that a well-behaved zone answer `127.0.0.2` for the test address `127.0.0.2` and `NXDOMAIN` for `127.0.0.1`, which we use as a liveness probe.
- **RHSBL / domain DNSBL** ŌĆö the domain is queried **directly, with no reversal**: `<domain>.<zone>`, `A` lookup, same `127.0.0.x` semantics. Spamhaus DBL, SURBL, and ivmURI are of this kind.

**Worked IP example.** To test `203.0.113.24` against `zen.spamhaus.org`:
1. Reverse octets: `24.113.0.203`.
2. Form the query name: `24.113.0.203.zen.spamhaus.org`.
3. `A` lookup. `NXDOMAIN` ŌåÆ not listed. An answer such as `127.0.0.4` ŌåÆ listed (Zen packs SBL, CSS, XBL, and PBL sub-lists into distinct return codes: `127.0.0.2`/`.3` = SBL/CSS, `127.0.0.4`ŌĆō`.7` = XBL, `127.0.0.10`/`.11` = PBL).
4. `TXT` lookup at the same name ŌåÆ the reason string and delisting link.

**Worked domain example.** To test `example.com` against `dbl.spamhaus.org`: query `example.com.dbl.spamhaus.org` directly (no reversal). `127.0.1.2` = spam domain, `127.0.1.4`ŌĆō`.6` = phishing/malware/botnet C&C, `127.0.1.102`ŌĆō`.106` = "abused legit" categories.

Why receivers care: DNSBL lookups are cheap (one cached DNS round-trip) and catch the majority of botnet and snowshoe spam at connection time, so nearly every MTA and spam engine (Postfix `reject_rbl_client`, SpamAssassin `RCVD_IN_*`, Rspamd, commercial gateways) consults several. The common real-world failure mode is a **shared-IP or ASN-wide listing you did not cause** ŌĆö a neighbor on your hosting provider's IP block, a compromised mailbox on your subnet, or a provider whose whole ASN got swept into UCEPROTECT L3. A single listing on a high-trust zone (Spamhaus SBL/XBL/Zen) can drop inbox placement to near zero at the large mailbox providers; a listing on a low-trust or ASN-wide zone (UCEPROTECT L2/L3, some SORBS sub-zones) may cost only a few percent, which is exactly why per-zone **weighting** is essential ŌĆö a naive "you're on a list!" alarm produces false panic.

## 3. How it is measured

**Targets.** For IP zones the checker iterates `ctx.sendingIps` (from `sending_ips`); if empty, it derives candidate IPs by resolving the domain's MX hosts and SPF `a`/`mx`/`ip4`/`ip6` mechanisms and testing those (an unexpected listing on an SPF-authorized IP is still your deliverability problem). For RHSBL/domain zones the target is `ctx.domain` and any additional envelope domains (Return-Path, DKIM `d=`). A fourth IP source is the **email-report pipeline** ([┬¦19](#19-email-derived-targets--ips-and-domains-mined-from-ingested-report-emails)): source IPs mined from ingested DMARC aggregate reports in `emails/` that authenticated as this domain join the target set with `source: email_report` ŌĆö these are IPs *actually observed sending your mail* that no static config listed.

**Query construction.**
- IPv4: split the dotted quad, reverse, join with `.`, append `zone_host`. `203.0.113.24` + `zen.spamhaus.org` ŌåÆ `24.113.0.203.zen.spamhaus.org`.
- IPv6: expand to the 32 nibbles, reverse them, dot-separate, append the zone (RFC 5782). Most zones only carry IPv4 data; IPv6 lookups usually `NXDOMAIN` and are skipped unless the zone documents IPv6 support.
- Domain: `<domain>.<zone_host>` verbatim, lowercased, IDNA/punycode-encoded, no reversal.

**Resolution.** First round uses `node:dns/promises` `resolve4()` for the `A` answer and `resolveTxt()` for the reason. `NXDOMAIN` (`ENOTFOUND`/`ENODATA`) = **not listed**. Any `127.0.0.0/8` (or documented `127.0.1.0/8`, `127.0.4.0/8`) answer = **listed**; the full answer set is stored as `return_code` (e.g. `127.0.0.4`) and decoded against the zone's code map to a sub-list label and severity. A `TXT` answer is stored verbatim as `reason_txt` and mined for a removal URL; if none is present we fall back to `dnsbl_zones.delisting_url`.

**Zone liveness / test-point probe.** Before trusting an `NXDOMAIN` as "clean", the runner (once per zone per audit) issues RFC 5782 sanity queries: `2.0.0.127.<zone>` must return `127.0.0.2` and `1.0.0.127.<zone>` must return `NXDOMAIN`. If the test point fails (dead zone, blocked resolver, or ŌĆö critically ŌĆö a zone that returns `127.0.0.2` for *everything* because your resolver is being rate-limited/blocked), the sub-check yields **info** "zone unavailable / inconclusive", never a false listing.

**Resolver policy & rate limits.** Public mirrors of Spamhaus, SURBL, URIBL, Abusix, and others **block or garbage-answer queries from large open resolvers (Google 8.8.8.8, Cloudflare 1.1.1.1, Quad9, OpenDNS) and cloud-provider resolvers** and from high-volume clients. Refusals are signaled **in-band**: Spamhaus answers `127.255.255.252` (bad query) / `.254` (public-resolver blocked) / `.255` (rate-limited); URIBL answers `127.0.0.1` (`URIBL_BLOCKED`). The checker MUST decode these as **"query refused ŌĆö inconclusive"**, never as a listing (see problem state [PS-9](#ps-9--queries-refused--resolver-blocked)). Consequences the checker must handle: (1) **Barracuda requires free registration** of the querying resolver IP before `b.barracudacentral.org` resolves ŌĆö until registered the sub-check is disabled/`info`. (2) **Abusix requires a free API key** embedded in the query name; (3) **invaluement is paid** and needs a licensed resolver/DNS feed. (4) Spamhaus public DNS is free only for low-volume, non-commercial use; commercial/high-volume users need the Spamhaus Data Query Service (DQS) key, which changes the zone host to `<dqs-key>.zen.dq.spamhaus.net` etc. The `dnsbl_zones` table carries `requires_registration` and `is_paid` so the engine skips or degrades a zone cleanly and the UI explains *why* a zone is inconclusive rather than silently omitting it. Queries are cached for the audit run and honor DNS TTL; a small concurrency cap (e.g. 8 in-flight) plus jittered retry avoids tripping rate limits.

**Severity mapping.** Per-zone `weight` (0.0-1.0) in `dnsbl_zones` scales the finding: high-trust IP/domain zones (Spamhaus Zen/SBL/XBL/DBL, Barracuda, SURBL, DroneBL) map a positive listing to **critical**; PBL, SpamCop (auto-expiring), PSBL, WPBL, Mailspike-BL, SpamRATS, Lashback map to **warning**; UCEPROTECT L2/L3, Backscatterer, geographic policy lists, and pure-reputation advisories (Sender Score, Mailspike-rep) map to **info** unless the score crosses a configured threshold. The category roll-up (`dnsbl.aggregate`) takes the worst *weighted* severity, so ASN-wide collateral alone cannot turn the Blacklists cell red.

## 5. Information schema (database schema)

Reference the shared core tables (owned by engineering, do **not** redefine): `domains`, `sending_ips`, `dkim_selectors`, `audit_runs`, `findings`. This check adds one **reference/config** table and one **result** table.

```sql
-- Reference/config: the configurable zone list (today lives in config.yaml).
CREATE TABLE dnsbl_zones (
    id                    SERIAL PRIMARY KEY,
    zone_host             TEXT NOT NULL UNIQUE,          -- e.g. 'zen.spamhaus.org', 'dbl.spamhaus.org'
    kind                  TEXT NOT NULL CHECK (kind IN ('ip','domain')),
    weight                NUMERIC(3,2) NOT NULL DEFAULT 1.00 CHECK (weight BETWEEN 0 AND 1),
    delisting_url         TEXT NOT NULL,                 -- per-zone removal portal
    description           TEXT,
    return_code_map       JSONB,                         -- { "127.0.0.2":"SBL", "127.0.0.4":"XBL", ... }
    enabled               BOOLEAN NOT NULL DEFAULT TRUE,
    requires_registration BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. Barracuda resolver reg, Abusix key
    is_paid               BOOLEAN NOT NULL DEFAULT FALSE, -- e.g. invaluement, Spamhaus DQS commercial
    created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_zones_enabled ON dnsbl_zones (enabled);
CREATE INDEX idx_dnsbl_zones_kind    ON dnsbl_zones (kind);

-- Result: one row per (audit_run, target, zone) tuple actually queried.
CREATE TABLE dnsbl_check_results (
    id            BIGSERIAL PRIMARY KEY,
    audit_run_id  BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id     BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    zone_id       INTEGER NOT NULL REFERENCES dnsbl_zones(id),
    target        TEXT NOT NULL,        -- the exact IP or domain tested
    target_kind   TEXT NOT NULL CHECK (target_kind IN ('ip','domain')),
    listed        BOOLEAN NOT NULL,
    return_code   TEXT,                 -- '127.0.0.4' (null when not listed)
    sub_list      TEXT,                 -- decoded label from return_code_map, e.g. 'XBL'
    reason_txt    TEXT,                 -- raw TXT answer
    inconclusive  BOOLEAN NOT NULL DEFAULT FALSE, -- zone unavailable / rate-limited / not registered
    query_ms      INTEGER,              -- resolution latency for diagnostics
    checked_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_dnsbl_results_domain_listed ON dnsbl_check_results (domain_id, listed);
CREATE INDEX idx_dnsbl_results_zone          ON dnsbl_check_results (zone_id);
CREATE INDEX idx_dnsbl_results_run           ON dnsbl_check_results (audit_run_id);
```

**Config.yaml ŌåÆ table mapping.** Today the zone catalog lives in `~/.email_delivery_hero/config.yaml` under a `dnsbl.zones` list; each entry (`zone_host`, `kind`, `weight`, `delisting_url`, `description`, `enabled`, `requires_registration`, `is_paid`, `return_code_map`) is exactly one future `dnsbl_zones` row ŌĆö the DB migration is a straight load of that list, and until then the admin "Blocklist Zones" panel writes back to `config.yaml`. Paid/registration secrets (Spamhaus DQS key, Abusix key, Barracuda-registered resolver IP) live under `config.yaml`'s `secrets:`/`resolvers:` blocks, not in `dnsbl_zones`.

**Result ŌåÆ file store mapping.** Each audit's structured results are written to `~/.email_delivery_hero/audits/<domainId>/<auditId>.json` under a `dnsbl` key: `{ "dnsbl": { "results": [ { "zone":"zen.spamhaus.org", "target":"203.0.113.24", "targetKind":"ip", "listed":true, "returnCode":"127.0.0.4", "subList":"XBL", "reasonTxt":"...", "inconclusive":false, "queryMs":42, "checkedAt":"..." } ], "summary": { "listedCount":2, "cleanCount":45, "inconclusiveCount":0, "weightedWorst":"critical" } } }`. Each array element becomes one `dnsbl_check_results` row (`audit_run_id`/`domain_id` from the enclosing audit); the derived `findings` still land in the shared `findings` table/`findings[]` array via the normal CheckerŌåÆFinding contract. The `(domain_id, listed)` index backs the dashboard "any listing?" query; the `(zone_id)` index backs "which domains does zone X hit?" reporting. **Note:** this audit-JSON layout is the in-memory/DB result contract; the on-disk persistence users and the UI read is the per-run YAML file of [┬¦12](#12-test-results-yaml--the-blacklists-section-of-the-run-file) (`~/.email_delivery_hero/runs/<domain>/ŌĆ”`, owned by [../storage.mdx](../storage.mdx)).

## 6. When we run the check

**On-demand.** Runs on every manual audit (`POST /api/domains/:id/audit`) and category-scoped run. Because listings appear and clear on their own (SpamCop expires in ~24h, XBL clears minutes after cleanup), a fresh manual run is the primary "am I delisted yet?" tool after the user submits a removal request.

**Periodic.** The in-process scheduler re-runs per domain at the `config.yaml` cadence (default every 6h for the Blacklists category ŌĆö more frequent than static DNS checks because reputation is volatile). Each run **diffs against the previous audit**: a zone flipping `listed:false ŌåÆ true`, or a `warning ŌåÆ critical` sub-list escalation, is a **new problem** surfaced by regression detection and (per [../scheduled_checks.mdx](../scheduled_checks.mdx)) eligible for alerting; a `listed:true ŌåÆ false` transition is reported as a resolved/improved item. InconclusiveŌåÆlisted and listedŌåÆinconclusive transitions are *not* treated as regressions to avoid resolver-blockage false alarms.

**Cost / rate-limit / back-off.** Total queries per audit = (IP zones ├Ś sending IPs) + (domain zones ├Ś domains); with ~15 zones and a handful of IPs this is well under 100 DNS lookups. To stay inside free-tier limits: cap concurrency (~8), reuse a dedicated non-public resolver where zones block open resolvers, honor TTL caching within a run, apply exponential back-off with jitter on `SERVFAIL`/timeout, and **auto-disable** a zone for the run if its RFC 5782 test point indicates blocking (record `inconclusive=true`). Paid/registration-gated zones (`is_paid`/`requires_registration` without configured credentials) are skipped, not errored.

## 8. Acceptance criteria

1. For an IPv4 target the checker constructs the query name by reversing octets and appending `zone_host` (verified: `203.0.113.24` + `zen.spamhaus.org` ŌåÆ `24.113.0.203.zen.spamhaus.org`); for a domain/RHSBL zone it queries `<domain>.<zone_host>` with **no** reversal.
2. An `A` answer inside `127.0.0.0/8` (or a zone's documented `127.0.1.x`/`127.0.4.x` space) sets `listed=true`, stores the exact `return_code`, and decodes it to `sub_list` via the zone's `return_code_map`; `NXDOMAIN` sets `listed=false`.
3. A `TXT` answer at the same name is stored as `reason_txt`; when it contains a removal URL that is used, otherwise the finding falls back to `dnsbl_zones.delisting_url`.
4. Every non-ok finding's `remediation` contains a concrete, correct per-zone delisting URL plus the reason/return code and the underlying-cause guidance ŌĆö never a generic message.
5. Per-zone `weight` maps listings to severity as specified: high-trust zones ŌåÆ critical, expiring/policy zones ŌåÆ warning, ASN-wide (UCEPROTECT L2/L3) and pure-reputation advisories ŌåÆ info; the category roll-up uses the worst **weighted** severity, and info/collateral alone does not turn the Blacklists cell amber or red.
6. The RFC 5782 test-point probe (`2.0.0.127.<zone>`ŌåÆ`127.0.0.2`, `1.0.0.127.<zone>`ŌåÆ`NXDOMAIN`) runs once per zone per audit; on failure the zone's results are marked `inconclusive=true` (info), never false-listed.
7. Zones flagged `requires_registration` or `is_paid` without configured credentials are skipped with an `inconclusive`/`info` note explaining why, and the UI links the fix.
8. `dnsbl_zones` and `dnsbl_check_results` exist with the specified PKs, FKs (`audit_run_id`, `domain_id`, `zone_id`), and indexes on `(domain_id, listed)` and `(zone_id)`; the `config.yaml` `dnsbl.zones` list round-trips 1:1 to `dnsbl_zones`, and each result serializes into the audit JSON under the `dnsbl` key.

### 10.4 How a test run executes the tools

One Blacklists run (for one domain, inside one dashboard **Run**) executes the tool sequence below. Every external invocation ŌĆö in-process `dnsbl.batch()` included, logged with its equivalent call string ŌĆö appends one entry to **`blacklists.tool_runs[]`** in the run file (┬¦12) with the locked shape `{tool, command, started_at, duration_ms, exit_code, output_format, parsed, error}`. `command` is the **exact argv string with every input argument** (reversed IP, zone host, domain, resolver, timeouts) so any entry can be replayed verbatim in a terminal; `parsed` is the captured/parsed output; `error` is `null` or the failure string. In-process calls record `exit_code: 0` on resolution (including NXDOMAIN ŌĆö "clean" is a successful answer) and `1` on library error.

**Etiquette rules that bind every step** (they are why these exact flags exist): per-query timeout **5 s** (`dnsbl` default; `dig +time=3 +tries=1`), global concurrency cap **8** in-flight (never the library default 64), jittered exponential back-off on SERVFAIL/timeout, per-run answer cache honoring TTL, and **never** a large open resolver (8.8.8.8 / 1.1.1.1 / 9.9.9.9 / OpenDNS ŌĆö the npm default!) ŌĆö Spamhaus public mirrors refuse them in-band (`127.255.255.252/.254/.255`) and URIBL answers `127.0.0.1` (`URIBL_BLOCKED`); any refusal code parses as **inconclusive, never listed**, and flips the run into [PS-9](#ps-9--queries-refused--resolver-blocked). Spamhaus public zones are free only for low-volume, non-commercial use; with a DQS key configured the zone host in every command swaps to `<key>.zen.dq.spamhaus.net` (etc.) automatically.

## 11. The tests we want to perform

The Blacklists audit is a fixed sequence; every numbered test emits rows into the run file's `blacklists` section (┬¦12 ŌĆö `results[]`, `tool_runs[]`, `tests[]`) and maps onto the sub-checks of ┬¦2. The exact tool command line each step executes is specified in ┬¦10.4.

1. **Target discovery** ŌĆö assemble the target set: registered `sending_ips`, IPs resolved from MX hosts, IPs authorized by SPF (`a`/`mx`/`ip4`/`ip6` expansion), the sending domain, Return-Path and DKIM `d=` domains. Record where each target came from (`source`).
2. **Zone-health preflight (RFC 5782)** ŌĆö per enabled zone, once per audit: `2.0.0.127.<zone>` must answer `127.0.0.2`; `1.0.0.127.<zone>` must be NXDOMAIN. Classify each zone `ok | dead | wildcarding | blocked | slow`. Dead/wildcarding zones are excluded and flagged for the catalog maintainer ([PS-10](#ps-10--dead-or-wildcarding-zone-false-listing)).
3. **Resolver-sanity test** ŌĆö detect refusal codes: any Spamhaus answer in `127.255.255.0/24`, any URIBL answer `127.0.0.1` ŌåÆ the resolver is blocked/rate-limited ([PS-9](#ps-9--queries-refused--resolver-blocked)); switch to the configured fallback resolver and re-probe.
4. **IP listing sweep** ŌĆö every (sending IP ├Ś enabled IP zone): reversed-octet `A` query; on a hit, `TXT` at the same name; decode `return_code` ŌåÆ sub-list via the zone's `return_code_map`.
5. **Domain listing sweep** ŌĆö every (domain target ├Ś enabled RHSBL zone): direct `A`/`TXT` query, same decoding (`127.0.1.x` DBL categories, SURBL/URIBL bitmasks).
6. **Positive-reputation checks** ŌĆö DNSWL presence (`list.dnswl.org` ŌĆö absence is `info`, presence records category/trust), Sender Score (`score.senderscore.com` ŌĆö decode 0-100), Mailspike rep zone.
7. **Root-cause context probes** (attached to any hit, to power the deep-dive pages): PTR/FCrDNS of the listed IP (`dig -x`), ASN/org (`mmdblookup`/Team Cymru), domain age (`whois` creation date), open-relay quick test (future: `swaks`/nmap `smtp-open-relay` behind explicit user consent ŌĆö it talks to the MX).
8. **Provider-portal checklist** ŌĆö not queryable; emit one checklist row per ┬¦9.7 portal with its check URL and last-acknowledged state so the user can mark "verified clean at Google Postmaster / SNDS".
9. **Diff against previous run** ŌĆö cleanŌåÆlisted = new problem (alertable), listedŌåÆclean = resolved, sub-list escalation = escalation; inconclusive transitions are ignored (┬¦6).
10. **Oracle cross-check (CI only, not in the product runtime)** ŌĆö the Jest suite validates our checker against `dig +noall +comments` output and, for the auth-adjacent checks, against `checkdmarc` JSON, using recorded fixtures so tests run offline.

## 15. Recommendations ŌĆö how we tell the user to fix it

The recommendation engine turns `results` + `problem_states` into an ordered, concrete playbook. Rules:

### 15.1 Ordering (always)

1. **Root cause before delisting** ŌĆö every listing recommendation is preceded by its cause fix (a delist request with the cause live gets re-listed and burns credibility with the operator). The cause comes from the decoded sub-list: XBL/DroneBL ŌåÆ find and clean the compromised host; SCBL/PSBL/WPBL traps ŌåÆ list hygiene + stop the source; SpamRATS NoPtr/Dyna ŌåÆ fix PTR/FCrDNS; DBL abused-legit ŌåÆ patch the hacked site/open redirect.
2. **Highest weighted impact first** ŌĆö Spamhaus/Barracuda/SURBL before medium zones; LOW-tier collateral last and explicitly labeled "will not measurably affect Gmail/Outlook/Yahoo".
3. **One delist link per listing, the right one** ŌĆö from `reason_txt` when the TXT carries the canonical URL, else the catalog's `delist_url`. Deep-link the target where the portal supports it.

### 15.2 Non-negotiable advisories (baked into the copy)

* **Never pay for delisting.** UCEPROTECT "express delisting"/whitelisted.org, Backscatterer express, and paid-expedite flows are flagged with: "This list auto-expires (7 daysŌĆō4 weeks). Paying is unnecessary and funds a practice the industry considers abusive (RFC 6471)." SPFBL's paid-expedite gets the same treatment with "wait for the free path".
* **Auto-expiring lists: consider waiting.** SpamCop ~24h, PSBL 2-4 weeks, WPBL ~10 days, Backscatterer 4 weeks ŌĆö the recommendation states the expiry window so the user can choose "fix cause + wait" over form-filling.
* **Collateral listings: you may not be the problem.** UCEPROTECT L2/L3, SEM netbl, ASN-wide states get: "caused by your network neighbors; escalate to your hosting provider's abuse desk (whois `OrgAbuseEmail`: <value>) or plan an IP/provider move if placement is actually suffering."
* **After any delist request:** "re-check in EmailDeliveryHero after the operator's stated processing window; the scheduler will alert you when it clears" ŌĆö the re-audit loop from ┬¦6.

### 15.3 Prevention recommendations (shown when clean, and appended after fixes)

Stay-clean guidance rendered in the fix panel's place on all-clean runs: keep FCrDNS valid on every sending IP; keep SPF/DKIM/DMARC green (cross-link those check pages); register at **DNSWL** (free, positive signal); enroll in **Microsoft SNDS/JMRP** and **Google Postmaster Tools** so provider-side reputation is observable; keep the 6-hour scheduled cadence on so new listings alert within hours; warm up new IPs/domains gradually (ties to PS-11); monitor the `emails/` ingested DMARC/TLS-RPT reports for early bounce signals.

### PS-0 ŌĆö All clean (the healthy state)

**Concept.** Every enabled zone answered, nothing listed, no refusals ŌĆö the public-blocklist layer says receivers have no reason to block on reputation lists. **Data fields:** `summary.listed=0`, `inconclusive=0`, `zone_health[*].status=ok`, `positive_reputation.*`. **Diagnose:** nothing to debug; the deep page becomes the *prevention* page (┬¦15.3) plus the positive-reputation panel (DNSWL status, Sender Score trend). **Further health:** provider portals still unverified ŌåÆ prompt to enroll (Postmaster Tools, SNDS). **UI deep page:** green hero, "what keeps you clean" checklist with per-item state, the 30-run history strip, and the next scheduled check time.

### PS-1 ŌĆö High-trust manual listing (SBL/CSS-class)

**Concept explained to the user.** A human or high-confidence system at Spamhaus/Barracuda decided traffic from this IP is a spam operation or persistent problem ŌĆö this is not automated collateral; it will not expire quietly, and it devastates placement. **Data fields:** `results[]` row with `tier=high`, `sub_listŌłł{SBL,CSS}` or zone `b.barracudacentral.org`; `reason_txt` (carries the SBL case reference); target's `ptr`, `asn`. **Diagnose & debug:** `dig +short 24.113.0.203.zen.spamhaus.org` (expect `127.0.0.2/.3`); `dig +short TXT <same>` for the case URL; `whois 203.0.113.24` for netblock owner (is the listing range-wide?); check neighboring IPs with `ipcalc <cidr>` + loop to see if the whole range is swept. **Tools:** dig/doggo, whois, ipcalc, check.spamhaus.org lookup. **Further health metrics:** volume/complaint history (ingested DMARC reports), open-relay test on the MX, outbound queue inspection for the flagged traffic. **Progress forward (deep page):** ŌæĀ identify and stop the cited operation (the TXT case link says what they saw), ŌæĪ document the fix, Ōæó submit the SBL/Barracuda removal with that documentation, ŌæŻ status auto-rechecks each scheduled run; expect human review latency (SBL) or <24h (Barracuda).

### PS-2 ŌĆö Compromised-host listing (XBL/DroneBL-class)

**Concept.** The IP behaved like malware: botnet callbacks, proxy abuse, or exploit traffic. Something on (or NATed behind) this IP is infected or open. Delisting before cleaning guarantees re-listing. **Data fields:** `sub_list=XBL` (`127.0.0.4-.7`) or DroneBL class code (`.3/.8-.10/.13/.14/.16` ŌĆö the code says drone vs proxy vs DDoS vs breached); `ptr`, `asn`; NAT note when the IP fronts many machines. **Diagnose & debug:** `dig +short TXT 24.113.0.203.zen.spamhaus.org` (XBL TXT links the detection detail incl. last-seen timestamp and port); on the host: unexpected outbound 25/6667/SOCKS connections (`lsof -i -nP | grep -E ':25|:1080|:6667'`), mail queue runaway, cron/webshell audit; `nmap -p25,465,587 --script smtp-open-relay <ip>` for open relay. **Tools:** dig, nmap (`smtp-open-relay`, `dns-blacklist`), host-level AV/EDR, provider flow logs. **Further health metrics:** re-scan all ┬¦9.1 exploit-class zones after cleanup; watch `z.mailspike.net` zero-hour for recurrence. **Progress forward:** ŌæĀ find the infected machine/service (the deep page lists the TXT's port/timestamp hints), ŌæĪ clean/patch/credential-rotate, Ōæó self-delist (XBL allows immediate self-removal once clean; DroneBL via lookup page), ŌæŻ if it recurs, assume re-infection not list error.

### PS-3 ŌĆö Policy listing: dynamic/consumer IP space (PBL-class)

**Concept.** Not an accusation of spam: the network's owner (or Spamhaus policy) declared this range shouldn't emit direct-to-MX mail ŌĆö typical for home/cloud-dynamic IPs. Receivers reject on it because botnets live in such space. **Data fields:** `sub_list=PBL` (`127.0.0.10` ISP-maintained / `.11` Spamhaus-maintained ŌĆö the code says who to talk to), SpamRATS `.36`; `asn.org` (consumer/cloud provider?), `ptr` (generic-looking `ip-203-0-113-24.provider.net`?). **Diagnose & debug:** `dig +short 24.113.0.203.pbl.spamhaus.org`; `dig +short -x 203.0.113.24` (generic PTR confirms policy space); `mmdblookup ŌĆ” autonomous_system_organization`. **Tools:** dig, mmdblookup/whois. **Further health metrics:** FCrDNS check ([./reverse_dns.mdx](./reverse_dns.mdx)), swaks send test through the proper smarthost. **Progress forward ŌĆö a fork the deep page draws explicitly:** *(a) this IP shouldn't send mail* ŌåÆ route outbound through the provider's smarthost or an ESP, done ŌĆö the listing is then correct and harmless; *(b) this is a legitimate static MTA* ŌåÆ get a proper PTR from the provider, then PBL self-exclusion at the check.spamhaus.org flow (`.10` listings may also need the ISP's own form since the ISP owns that entry).

### PS-4 ŌĆö Sending domain on a domain blocklist (DBL/SURBL/URIBL)

**Concept.** The *name*, not the IP, is listed ŌĆö every message carrying the domain in From/Return-Path/URLs is suspect no matter which IP sends it. Causes split into: the domain really spams/phishes, or "abused-legit" (site hacked, open redirect, compromised shortener). **Data fields:** RHSBL `results[]` rows; DBL code distinguishes `127.0.1.2-.6` (bad domain) vs `.102-.106` (**abused-legit** ŌĆö different fix); SURBL/URIBL bitmask; `targets.domains[*].source` (was it the primary domain or a Return-Path/DKIM domain?); domain `created` date. **Diagnose & debug:** `dig +short whitehatengineering.com.dbl.spamhaus.org` (never query an IP at DBL ŌĆö `127.0.1.255` is the you-did-it-wrong code); `dig +short TXT <same>`; `dnstwist --registered --format json <domain>` (lookalike confusion); site-side: check for injected pages/open redirects (Google Search Console security tab). **Tools:** dig, dnstwist, SURBL/URIBL lookup pages, Search Console. **Further health metrics:** [./link_url_reputation.mdx](./link_url_reputation.mdx) body-URL sweep (are other domains you link also listed?), DKIM `d=` and Return-Path domains individually. **Progress forward:** abused-legit codes ŌåÆ ŌæĀ find and close the compromise (hacked CMS, open redirect), ŌæĪ purge injected content, Ōæó delist via the zone's flow with the fix described; bad-domain codes on a domain you own legitimately ŌåÆ contest via the same flow with evidence; then re-audit and watch the domain-side zones for 2 weeks.

### PS-5 ŌĆö Trap-driven auto-expiring listing (SCBL/PSBL/WPBL-class)

**Concept.** Mail from this IP hit a spam trap ŌĆö an address that never opted in. Usually list-hygiene rot (purchased lists, ancient addresses, scraping) rather than infection. These lists forgive: they expire on their own once the traffic stops. **Data fields:** zone Ōłł {`bl.spamcop.net`, `psbl.surriel.com`, `db.wpbl.info`, `truncate.gbudb.net`}; `diff` (recurring or first-time?); expiry window per zone (SCBL ~24h, PSBL 2-4wk, WPBL ~10d) carried in catalog metadata. **Diagnose & debug:** `dig +short 24.113.0.203.bl.spamcop.net`; SpamCop's lookup page shows report ages; cross-reference send logs around the listing time; check for a specific campaign/segment as the source. **Tools:** dig, SpamCop bl.shtml lookup, your ESP's suppression/bounce logs, ingested DMARC reports ([../emails.mdx](../emails.mdx)) for the volume spike. **Further health metrics:** bounce-rate trend, list-acquisition audit, confirm no PS-2 co-listing (trap + exploit zones together = infection, not hygiene). **Progress forward:** ŌæĀ identify the sending stream that hit the trap, ŌæĪ remove non-engaged/unverified addresses (or the whole acquired segment), Ōæó either wait out the expiry (the page shows the countdown) or self-delist (PSBL is instant), ŌæŻ recurrence = the hygiene fix didn't take.

### PS-6 ŌĆö Collateral neighborhood/ASN listing

**Concept.** A range containing your IP ŌĆö /24, allocation, or the whole ASN ŌĆö got listed because of *neighbors*. You personally may be clean. Impact at major receivers is near zero (Gmail/Microsoft/Yahoo don't use UCEPROTECT L2/L3), but small self-hosted receivers occasionally do. **Data fields:** zone Ōłł {`dnsbl-2/3.uceprotect.net`, `netbl.spameatingmonkey.net`, ivmSIP/24}; `asn`; whois `OrgAbuseEmail`. **Diagnose & debug:** `dig +short <rev>.dnsbl-2.uceprotect.net`; uceprotect.net rblcheck page names the allocation and the "poison" IPs; `whois <ip>` ŌåÆ abuse contact; `ipcalc` the allocation to see its width. **Tools:** dig, whois, ipcalc, uceprotect lookup. **Further health metrics:** confirm your own IP is clean at L1 and all HIGH zones (if yes, this is pure collateral); measure actual placement via bounce logs before acting. **Progress forward:** ŌæĀ verify you're clean everywhere that matters (the page shows the HIGH-tier matrix for reassurance), ŌæĪ forward the evidence to the provider's abuse desk (pre-drafted email with the whois contact filled in ŌĆö [Copy] button), Ōæó only if real placement suffers: plan an IP/provider move; ŌæŻ **never pay** the express-delist (auto-links PS-13 advisory).

### PS-7 ŌĆö rDNS/FCrDNS defect driving listings

**Concept.** The IP has no PTR, a generic PTR, or PTRŌåøA mismatch. This is both directly rejected by many receivers *and* the trigger for SpamRATS NoPtr/Dyna and PBL-class policy listings ŌĆö one DNS fix clears a family of problems. **Data fields:** `targets.ips[*].ptr`, `fcrdns_ok=false`; SpamRATS `.36/.37` rows. **Diagnose & debug:** `dig +short -x 203.0.113.24` (empty or `dynamic-ŌĆ”`/`ip-ŌĆ”` pattern = defect); forward-confirm: `dig +short A <ptr-value>` must include the IP; `swaks --server mxŌĆ” --quit-after EHLO` to see the HELO/banner vs PTR mismatch a receiver sees. **Tools:** dig, swaks; the fix itself happens at the IP's owner (hosting provider console or ticket ŌĆö PTR is delegated to whoever owns the reverse zone). **Further health metrics:** the full [./reverse_dns.mdx](./reverse_dns.mdx) check; HELO-name alignment ([./smtp_security.mdx](./smtp_security.mdx)). **Progress forward:** ŌæĀ set PTR to the mail host's FQDN (e.g. `mail.example.com`) via the provider, ŌæĪ ensure that FQDN's A record points back (FCrDNS), Ōæó align the MTA's HELO to the same name, ŌæŻ then delist SpamRATS (removal.php requires the PTR to already be fixed) and PBL if applicable ŌĆö the deep page orders these dependencies explicitly.

### PS-8 ŌĆö Provider-side block with all public lists clean

**Concept.** The user's real complaint ("Gmail junks me", "Microsoft 550 5.7.606") with a green ┬¦9 matrix. The blocking list is private ŌĆö Gmail domain reputation, Microsoft's internal lists, Cloudmark/Proofpoint at carriers. Public-zone cleanliness is necessary, not sufficient. **Data fields:** `summary.listed=0` while the user reports bounces; `provider_portals[*].user_state`; ingested bounce/DMARC evidence (SMTP codes: Microsoft `5.7.606-649`, Gmail `550-5.7.1`, carrier Cloudmark text). **Diagnose & debug:** read the actual bounce (the deep page shows a paste-the-NDR box that pattern-matches the code to the provider and its exact delist path); check Postmaster Tools domain/IP reputation graphs; check SNDS filter verdict and trap hits; `https://csi.cloudmark.com/en/reset/` and `ipcheck.proofpoint.com` lookups for carrier blocks. **Tools:** the ┬¦9.7 portal links, paste-NDR classifier, swaks (reproduce the rejection: `swaks --to probe@outlook.com --server <their-mx>` ŌĆö future, consent-gated), Postmaster/SNDS dashboards. **Further health metrics:** spam-rate from Postmaster Tools vs the 0.1%/0.3% thresholds, DMARC alignment health (Gmail bulk-sender rules), List-Unsubscribe presence ([./list_unsubscribe.mdx](./list_unsubscribe.mdx)). **Progress forward:** ŌæĀ classify the bounce ŌåÆ provider, ŌæĪ use that provider's delist/escalation channel (sender.office.com for 5.7.606, SNDS for consumer, Gmail escalation form, Cloudmark resetŌĆ”), Ōæó fix the reputation inputs the portal shows (complaints, volume spikes, auth), ŌæŻ mark the portal row `verified_clean`/`problem_reported` so the checklist tracks it.

### PS-9 ŌĆö Queries refused / resolver blocked

**Concept (about *us*, not the domain).** The blocklists refused to answer our resolver ŌĆö public/open resolvers and heavy users get in-band error codes, not real answers. Results this run are partially blind; nothing here says the domain is unhealthy. **Data fields:** `resolver.mode/server`, `refusals_detected=true`, per-row `refusal_code` (`127.255.255.252/.254/.255` Spamhaus, `127.0.0.1` URIBL), `zone_health[*].status=blocked`. **Diagnose & debug:** `dig +short 2.0.0.127.zen.spamhaus.org @8.8.8.8` (returns the refusal code ŌĆö demonstrates the problem) vs the same query `@<local-resolver>` (returns `127.0.0.2`); confirm what `resolver.server` is actually set to. **Tools:** dig with explicit `@server`, the app's Settings ŌåÆ resolver config. **Further health metrics:** per-zone `probe_ms` (rate-limiting shows as `.255`/timeouts under load); query-volume estimate vs Spamhaus's free-tier bounds. **Progress forward:** ŌæĀ point the checker at a real recursive resolver (local `unbound`/`knot-resolver`, or the ISP's), not 8.8.8.8/1.1.1.1; ŌæĪ for volume, get the free Spamhaus DQS key and Abusix key and set them in Settings (the zone hosts switch to `<key>.zen.dq.spamhaus.net` automatically); Ōæó register the resolver IP with Barracuda; ŌæŻ re-run ŌĆö refused zones re-query automatically.

### PS-10 ŌĆö Dead or wildcarding zone (false listing)

**Concept.** DNSBLs die, and dying zones sometimes wildcard ŌĆö answering "listed" for every query (AHBL, SpamCannibal). A "listing" from such a zone is noise, and reporting it would be a false alarm; this state exists so the product *explains* a suppressed hit rather than silently hiding it. **Data fields:** `zone_health[*]` row with `status=dead|wildcarding` (`negative_probe` returned an answer for `127.0.0.1`); any suppressed `results[]` rows for that zone. **Diagnose & debug:** `dig +short 1.0.0.127.<zone>` (an answer = wildcarding); `dig +short 2.0.0.127.<zone>` (NXDOMAIN = dead/empty); check the zone against ┬¦9.5 and dnsbl.com. **Tools:** dig, multirbl.valli.org zone page, dnsbl.com registry. **Further health metrics:** none for the domain ŌĆö this is catalog hygiene. **Progress forward:** the zone is auto-disabled for the run and flagged in the admin Blocklist Zones panel with the probe evidence; the operator confirms retirement (moves it to the dead registry) or re-enables after the operator's outage resolves. Users see: "zone X is dead/misbehaving and was excluded ŌĆö this does not affect your standing."

### PS-11 ŌĆö New-domain zero reputation

**Concept.** The domain is simply *young*. Spamhaus ZRD lists everything <24h old by definition; SEM-FRESH tracks fresh registrations; many receivers throttle new domains regardless. Not an accusation ŌĆö a maturity gate every domain passes through. **Data fields:** `targets.domains[*].created` (whois), ZRD hour-code (`127.0.2.<h>`), SEM-FRESH hits, DKIM/SPF/DMARC status from sibling checks. **Diagnose & debug:** `whois <domain> | grep -i 'creation'`; `dig +short <domain>.fresh.spameatingmonkey.net`; confirm the listing ages out (ZRD at 24h). **Tools:** whois, dig. **Further health metrics:** authentication completeness (new domains need perfect SPF/DKIM/DMARC to build reputation fast), warm-up volume curve from send logs. **Progress forward:** ŌæĀ wait out the age gates (the page shows domain age and each gate's threshold), ŌæĪ send low, consistent volume to engaged recipients first, Ōæó ensure auth is fully green before scaling, ŌæŻ enroll Postmaster Tools/SNDS on day one so reputation formation is observable.

### PS-12 ŌĆö No positive reputation established

**Concept.** Nothing is *wrong* ŌĆö but nothing vouches for you either: not on DNSWL, no meaningful Sender Score, unknown at Mailspike-rep. Positive signals are a buffer against future gray-area filtering. **Data fields:** `positive_reputation.dnswl.listed=false`, `senderscore.score` null or <70, `mailspike_rep`. **Diagnose & debug:** `dig +short <rev>.list.dnswl.org` (NXDOMAIN = not registered); `dig +short <rev>.score.senderscore.com` (last octet = score); review volume consistency (Sender Score needs sustained volume to exist at all). **Tools:** dig, dnswl.org self-registration, senderscore.org. **Further health metrics:** complaint and bounce trends from ingested reports. **Progress forward:** ŌæĀ register the MTA at dnswl.org (free tier, category "email marketing"/"corporate" as appropriate), ŌæĪ keep volume steady so Sender Score materializes, Ōæó treat this page as done when DNSWL shows `trustŌēź1` and score Ōēź70 ŌĆö severity stays `info` throughout; the UI never nags.

### PS-13 ŌĆö Pay-to-delist trap encountered

**Concept (consumer-protection state).** The only open listings are on operators that sell "express" removal (UCEPROTECT/whitelisted.org, Backscatterer express, some SPFBL flows). The industry position (RFC 6471) is that payment-for-removal is abusive; these listings auto-expire and barely affect major-receiver placement. This state exists so the product actively stops the user from paying out of panic. **Data fields:** all open `results[]` rows have catalog flag `paid_delist_offered=true`; expiry windows; `diff` recurrence. **Diagnose & debug:** confirm no HIGH-tier co-listing (if Spamhaus is also red, *that* is the real problem ŌĆö this state only fires when paid-trap zones are the *only* hits); verify the auto-expiry clock from the operator's lookup page. **Tools:** dig, the operator lookup pages, ┬¦9.1 matrix. **Further health metrics:** placement evidence (bounces naming these lists are rare ŌĆö ask for real NDRs before caring). **Progress forward:** ŌæĀ do not pay ŌĆö the page says so in bold with the RFC 6471 citation, ŌæĪ fix any genuine cause (backscatter ŌåÆ stop bounce-after-accept on the MTA, [./smtp_security.mdx](./smtp_security.mdx)), Ōæó wait for auto-expiry (countdown shown), ŌæŻ if a real correspondent's server uses these lists, hand them the RFC 6471 context via a pre-drafted note.

## 18. The checked-in registry ŌĆö `blacklists.yaml` as the single source of truth

The complete ┬¦9 research ŌĆö every blocklist we know, where it lives, how to query it, what its answers mean, how to get off it ŌĆö is **data, not code**, and it changes on operators' schedules, not ours. It therefore lives in one hierarchical YAML file **checked into the source tree**:

```
code/packages/backend/src/modules/blacklists/registry/blacklists.yaml
```

Rules: reviewed like code (a zone edit is a normal PR/diff), shipped with the app (no network fetch to know the catalog), loaded and validated at backend boot, and **never** edited at runtime ŌĆö runtime/operator overrides still go to `<stateDir>/blacklist_zones.yaml` (merged by `zone`, override wins), preserving the ┬¦5 config contract. `zones.ts` becomes a *loader* (read ŌåÆ parse ŌåÆ validate ŌåÆ freeze); a registry that fails validation aborts boot with the offending entry named ŌĆö a silently-empty catalog would report every domain "clean."

### 18.1 Schema (`registry_version: 1`)

Top level is `blacklists:` ŌĆö the list of queryable lists ŌĆö plus three sibling blocks that complete the ┬¦9 research: `dead_zones:`, `aggregators:`, `provider_portals:`.

```yaml
registry_version: 1
compiled: 2026-07              # when the research snapshot was verified
sources:                       # where the research came from (S9 provenance)
  - https://multirbl.valli.org
  - https://www.dnsbl.com
  - operator sites (verified per entry)

blacklists:
  - name: Spamhaus ZEN                     # REQUIRED - human name of the list
    url: https://www.spamhaus.org          # REQUIRED - operator's home page
    type: ip                               # REQUIRED - ip | domain | both
    description: >-                        # REQUIRED - one paragraph: what it
      Combined SBL+CSS+XBL+PBL zone;       # lists, who uses it, why a hit hurts
      the single highest-impact IP list.
    # ---- how to query it ----------------------------------------------------
    zone: zen.spamhaus.org                 # DNS zone we query (omit for web-only)
    query:
      method: dnsbl                        # dnsbl (reversed-IP A) | rhsbl (direct domain A) | web_only
      record_types: [A, TXT]
      ipv6: false                          # zone documents IPv6 nibble support
      rfc5782_probe: true                  # liveness test-point pair applies
    # ---- how to interpret it -------------------------------------------------
    return_codes:
      "127.0.0.2":  { label: "SBL (manual spam listing)", severity: critical, problem_state: PS-1 }
      "127.0.0.4":  { label: "XBL (compromised host)",    severity: critical, problem_state: PS-2 }
      "127.0.0.10": { label: "PBL (ISP dynamic policy)",  severity: warning,  problem_state: PS-3 }
    refusal_codes: ["127.255.255.252", "127.255.255.254", "127.255.255.255"]  # inconclusive, never a listing
    # ---- how much it matters -------------------------------------------------
    tier: high                             # high | medium | low  (drives default weight, S3)
    weight: 1.0
    severity: critical                     # default when no return-code match
    enabled: true                          # default-on set = S2; rest ship disabled
    # ---- how to check and get off it -----------------------------------------
    lookup_url: https://check.spamhaus.org
    delist_url: https://check.spamhaus.org
    auto_expires: null                     # e.g. "~24h after reports stop" (SpamCop)
    paid_delist_offered: false             # true triggers the PS-13 never-pay advisory
    # ---- how to reach it ------------------------------------------------------
    access:
      cost: free                           # free | registration | paid
      notes: "Free low-volume via own resolver; DQS key otherwise"
      dqs_zone: "{key}.zen.dq.spamhaus.net"  # host template when a key is configured
    status: { alive: true }                # liveness bookkeeping (S9.5)

dead_zones:                                # the do-not-check registry (S9.5) - engine hard-blocks these
  - zone: dnsbl.sorbs.net                  # one entry per dead zone host (all 18 SORBS zones enumerated)
    name: SORBS
    died: 2024-06
    reason: "Proofpoint decommissioned; dead zones may wildcard and list the world"
  - zone: ix.dnsbl.manitu.net
    name: NiX Spam
    died: 2025-01
    reason: shut down

aggregators:                               # S9.6 link-out services
  - name: MultiRBL (valli.org)
    url: https://multirbl.valli.org
    description: "~892 active + ~588 dead zones tracked; per-zone liveness"

provider_portals:                          # S9.7 invisible blacklists
  - provider: google_postmaster
    name: Google Postmaster Tools
    check_url: https://postmaster.google.com
    delist_url: https://support.google.com/mail/contact/gmail_bulk_sender_escalation
    description: "Domain/IP reputation and spam rate; no delist portal - fix reputation"
```

Field rules: `name`, `url`, `type`, `description` are required on every entry ŌĆö they are the four the user asked of the research ("name, URL, type, description"); everything else exists so the engine can *query and understand* a listing without touching code: `zone`+`query` say how to ask, `return_codes`+`refusal_codes` say what the answer means, `tier`/`weight`/`severity` say how much to care, `lookup_url`/`delist_url`/`auto_expires`/`paid_delist_offered` say how to get off, `access` says what credentials gate it, `status`/`dead_zones` say whether to ask at all. Entries with `query.method: web_only` (Cloudmark, Proofpoint, invaluement's public face) carry no `zone` and surface as link-outs, never DNS queries. `type: both` entries are queried on both sweeps (IP and domain) ŌĆö SPFBL, 0spam, NordSpam, Suomispam are of this kind.

### 18.2 Loader contract & serving

- **Load once at boot** (sync read + YAML parse via the shared yaml-store), validate every entry (unique `zone`, valid `type`/`tier`/`severity`, parseable return codes, required fields present, no entry whose `zone` also appears in `dead_zones`), then freeze.
- **Merge order:** registry defaults ŌåÉ `<stateDir>/blacklist_zones.yaml` operator overrides (same row shape, matched by `zone`; `enabled: false` disables a default; unknown zones append ŌĆö but a zone on `dead_zones` can never be enabled from anywhere).
- **Serve it:** `GET /api/blacklists/zones` returns the effective merged registry (entries + dead-zone count + per-zone last liveness) ŌĆö this feeds the ┬¦17 registry-health panel and the ┬¦4 Blocklist Zones admin panel; the panel writes overrides to the state-dir file, never to the checked-in registry.
- **Keep ┬¦9 and the file in lockstep:** ┬¦9 is the human research narrative; `blacklists.yaml` is its machine twin. A change to one without the other is a spec violation (checked in review, noted in maintenance).

## 19. Email-derived targets ŌĆö IPs and domains mined from ingested report emails

The `emails/` directory at the repo root (and the [../emails.mdx](../emails.mdx) ingestion pipeline behind it) receives DMARC aggregate (rua) and TLS-RPT report messages ŌĆö Google, Microsoft, and Outlook.com routinely deliver these today. Every DMARC rua row names a **`source_ip` that actually transmitted mail claiming to be your domain**. That is exactly the population the blacklist check exists to protect, and it routinely contains IPs *no static config knows about* ŌĆö an ESP's rotating pool, a forgotten marketing tool, a misconfigured relay. This section closes that loop.

### 19.1 What we extract

From each ingested report (parsed per [../emails.mdx](../emails.mdx); `parsedmarc` is the ┬¦10 oracle):

| Extracted | From | Becomes |
|---|---|---|
| `source_ip` of rows where SPF **or** DKIM passed and aligned | DMARC rua `<row>` | **Blacklist IP target**, `source: email_report` ŌĆö this is *our* mail stream; a listing here is our delisting problem |
| `source_ip` of rows that failed both | DMARC rua `<row>` | **Context only** ŌĆö spoofers/phishers sending as us; recorded on the Reports page and the ┬¦16 deep dives, but *not* delist-targeted (their listings are good news). A high-volume failing IP cross-referenced as *listed* is corroborating evidence for the DMARC story, shown as `info` |
| DKIM `d=` / SPF envelope domains seen authenticating for us | rua `<auth_results>` | **RHSBL domain targets** when they are ours (subdomains, ESP bounce domains we own); third-party ESP domains are checked `info`-only |
| Receiving-MX names and their IPs | TLS-RPT reports | **Context** for [./tls_transport.mdx](./tls_transport.mdx); not blacklist targets (inbound, not outbound) |

**Windowing & hygiene.** Only IPs seen in reports covering the last **30 days** join the sweep (older pools rotate away); each carries `first_seen`/`last_seen`/`message_count` so the UI can say "203.0.113.99 ŌĆö seen in 4 reports, 1,240 messages, last 2026-06-28". Private/reserved ranges (RFC 1918, CGNAT) are dropped. The per-audit target cap stays bounded: email-derived IPs are deduped against configured `sending_ips` (a duplicate keeps the stronger `sending_ips` source tag) and capped at the top N (default 20) by message volume, with the truncation logged ŌĆö never silent.

### 19.2 How it flows into the check

Target discovery (┬¦3, test 1 of ┬¦11) gains the fourth source: `sending_ips` Ōł¬ `mx_resolved` Ōł¬ `spf_authorized` Ōł¬ `email_report`. Email-derived IPs are swept against every enabled IP zone **identically** to configured IPs ŌĆö same return-code decoding, same weighting, same problem states, same delisting remediation. The run file's `blacklists.targets.ips[*]` (┬¦12) records them with `source: email_report` plus the windowing fields above. The diff engine treats a newly-observed listed IP as a **new problem** (that's the headline scenario this section exists for: "your ESP quietly moved you onto a listed IP ŌĆö here is the delisting path and the evidence trail from your own DMARC reports").

### 19.3 Where the user sees it

- **┬¦17 dashboard, Email-derived targets strip** ŌĆö the fleet rollup (IPs seen / listed) with drill-down.
- **┬¦13.2 per-domain full page** ŌĆö email-derived targets appear in the problems table and the all-zones matrix like any target, tagged with a small `via DMARC reports` chip; the debug drawer shows the report evidence (reporter, window, message count) alongside the DNS evidence.
- **Reports page** ([../left_bar.yaml](../left_bar.yaml) `reports`) ŌĆö the inverse view: per report, each source IP's blacklist verdict as one column, cross-linking here.

**Fixes.** For a listed aligned IP the remediation is the standard per-zone delisting playbook (┬¦15) *plus* the stream-level advice: if the IP belongs to an ESP/shared pool, open a ticket with the ESP citing the zone + return code (pre-drafted, [Copy]); if it's yours, run the ┬¦16 state's cause-first flow. IP delisting portals are the same per-zone `delist_url`s ŌĆö there is no separate "IP removal" universe; the registry (┬¦18) already carries every one.
