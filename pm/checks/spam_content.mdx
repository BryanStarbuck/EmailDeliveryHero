# EmailDeliveryHero — Spam & Content Check

**Spam & Content** is the message layer: everything a receiver evaluates *after* the plumbing
(DNS & Infrastructure) and the identity checks (SPF/DKIM/DMARC) have passed. Two senders with
identical, perfect authentication are separated by exactly the things this category tests — does
the message body look like the spam corpus, does the bulk mail carry the RFC 8058 one-click
unsubscribe the **2024 Gmail/Yahoo bulk-sender rules** made mandatory, are the domains linked
*inside* the message on a URI blocklist, is the complaint rate under Gmail's hard **0.3%** line,
and — the ground truth — where does the mail actually land at Gmail, Outlook, Yahoo, and Apple.
A single blocklisted link domain or a complaint spike can zero out inbox placement while every DNS
check in the app still shows green.

This check is the **Spam & Content** dashboard category; all sub-check ids use the **`content.`**
checkId prefix. It is the **category-level spec**: it defines the roll-up, the per-run results
section (`spam_content` in the run YAML), the full-page Spam & Content run view, the tool
execution contract, and the problem-state catalog. The seven per-topic specs below own the
per-sub-check detail; this file never contradicts them.

## 1. What the category covers

**The standards.** Message format RFC 5322 and MIME RFC 2045–2049 (what content scoring parses);
`List-Unsubscribe` RFC 2369 + one-click `List-Unsubscribe-Post` RFC 8058 and `List-Id` RFC 2919
(list management); DNSBL/RHSBL query mechanics RFC 5782 and URL syntax RFC 3986 (link reputation);
ARF complaint reports RFC 5965 (feedback loops); the BIMI, AVP, and VMC IETF drafts (brand
indicators). The enforcement floor is the **2024 Gmail/Yahoo bulk-sender rules** (RFC 8058
one-click unsubscribe honored within 2 days, aligned `From:`, spam-complaint rate < 0.3%) plus
Microsoft's 2025 follow-on for 5,000+/day senders.

**Why receivers care.** Content and reputation are the *post-authentication* discriminators.
Botnet spam rotates IPs but must always drive the recipient to a payload URL — so the **link
domain** is the most stable spam fingerprint (SURBL, Spamhaus DBL). Complaint rate is the purest
"recipients don't want this" signal and Gmail throttles on it directly. And seed-list placement is
the only measurement of the outcome every other check merely predicts.

**Common real-world failure modes.** (1) A marketing template — HTML-only, image-heavy, trigger
phrases, ALL-CAPS subject — scores past SpamAssassin's 5.0 line even though the domain
authenticates perfectly. (2) A bulk stream missing `List-Unsubscribe-Post` fails the Gmail/Yahoo
gate outright. (3) A campaign links through `bit.ly` or a hacked blog that sits on the DBL — the
message inherits the link's reputation. (4) A cold list drives complaints past 0.3% and Gmail
silently spam-folders everything for weeks. (5) A brand publishes BIMI while DMARC is still
`p=none`, so the logo never renders anywhere. (6) DNS-side auth is green but the receiver's own
`Authentication-Results` says `dkim=fail` because the ESP rewrites the signed body — only a seed
test catches it.

**The finding-id map.** Every sub-check id is prefixed `content.`; the family a finding belongs to
is derived from its id by this prefix table (many ids carry a `.<domain>`/`.<provider>` suffix, so
matching is by **id prefix**):

**First round vs gated.** Only the pure-DNS and own-data sub-checks run unconditionally: the BIMI
DNS rows (`bimi_present/syntax/dmarc_prereq/selector/l_present/svg_url/vmc/vmc_url/single/dns_health`),
the **report-email analysis** rows (`report_*` — pure local-file work over the corpus directory and
report store; an empty corpus emits a single info, never a fabricated failure),
the reputation advisories (`fbl_enrollment`, `reputation_data_available`, `postmaster_verified`),
and `blocklist_history` (trends this app's own stored [Blacklists](./blacklists.mdx) runs).
Everything else is gated: content scoring, list management, and link reputation need an uploaded
**sample message** (`.eml`); GPT/FBL/ESP metrics need a connected integration; inbox placement
needs seeds or a seed-service credit. A gated family with no input emits a single **info** finding
("no sample message — upload one" / "no reputation source connected" / "no seed test yet"), never
a fabricated failure — `info` alone never turns the category cell amber.

**Severity mapping (category-wide).** Delivery-killing now → **critical**: linked domain on a
high-trust URI zone, raw-IP or brand-homograph link, complaint rate ≥ 0.3%, GPT reputation
BAD/LOW, bulk stream missing one-click unsubscribe, `From:` aligned with neither SPF nor DKIM,
SpamAssassin ≥ 5.0, hidden text or a risky attachment, majority Spam/Missing at a provider,
receiver-observed `dmarc=fail`, BIMI published while DMARC is not enforcing, report-observed
**spoofing** (a source failing both SPF and DKIM alignment with real volume) or **own mail
quarantined/rejected** by receivers. Reputation-eroding →
**warning**: score 2.0–5.0, shortener/`http:` links, missing `mailto:` fallback, complaint
0.1–0.3%, GPT MEDIUM, warmup/volume anomalies, Gmail Promotions-tab placement, 50–79% inbox,
SVG-only BIMI (no VMC), report-observed alignment fragility (a stream passing DMARC on one
mechanism only), a real-world DMARC pass rate below ~100%, orphan reports for an unmonitored
domain, a stale report corpus. Visibility and adoption gaps → **info**: no sample, no integration,
no seeds, no BIMI for a domain not opting in, declined BIMI record, an empty report corpus.

## 2. "View one category run" UI — the Spam & Content page

Every run persists the full category result (§4), so the Spam & Content page is **run-scoped**:
it renders **one run's** Spam & Content section, with navigation between the domain's stored runs.

**Routes.**

| Page | Route |
| --- | --- |
| This category, one specific run | `/domains/:domainId/runs/:runId/spam-content` |
| This category, the newest run (alias) | `/domains/:domainId/spam-content` |
| Whole run report (all six tests) | `/domains/:domainId/runs/:runId` (newest alias `/domains/:domainId`) |
| **Check detail (family explainer), one specific run** | `/domains/:domainId/runs/:runId/spam-content/check/:checkKey` (§8) |
| **Check detail, the newest run (alias)** | `/domains/:domainId/spam-content/check/:checkKey` |
| Problem-state drill-down, one specific run | `/domains/:domainId/runs/:runId/spam-content/:problemId` |
| Problem-state drill-down, the newest run (alias) | `/domains/:domainId/spam-content/:problemId` |

`:checkKey` slugs are fixed (§8): `bimi`, `report-emails`, `content-score`, `list-unsubscribe`,
`link-reputation`, `sender-reputation`, `inbox-placement`. The static `check` segment outranks the dynamic
`:problemId` segment (the router's static-over-dynamic rule), so
`…/spam-content/check/bimi` never matches the problem-state route. The legacy
`/domains/:domainId/content` content-scoring page remains as a redirect alias of
`…/spam-content/check/content-score` (interior owned by
[./content_scoring.mdx](./content_scoring.mdx)).

**How the page is reached.**
- **Domain-health table (Table 1):** clicking the **Spam & Content cell** of a domain row opens
  the domain's **newest** run's Spam & Content page (the alias route). Clicking anywhere else in
  the row opens the whole newest run report.
- **Runs table (Table 2):** clicking the Spam & Content cell of a run row opens **that run's**
  Spam & Content page. Clicking anywhere else in the row opens that run's whole report.
- **Run report:** the Spam & Content category chip carries a chevron (›) into this page for the
  run being viewed.
- The dashboard **cell label** for this category is the run's **inbox-placement percentage**
  (e.g. `92%`, per [../dashboard.mdx](../dashboard.mdx) §5.2). When no placement test has ever run
  the label falls back to the standard `K of M fail` / `Healthy` computed from the other content
  sub-tests; never-run stays gray `Never run`. Cell color is the worst severity across all
  `content.*` findings in that run (dark green healthy / dark red critical / amber warning /
  light gray never-run).

**Contents, top to bottom** (exact layout below). The page must answer: *is my mail landing in
inboxes right now? What did we observe (the score, the headers, the links, the rates, what the
receivers' own DMARC/TLS reports say)? Which of
the seven families pass/fail, sub-test by sub-test? How do I fix each failure, in what order?*

1. **Header** — back link to the run report, domain name, "Spam & Content" title, the category
   StatusCell, the **run timestamp** (`run of <startedAt>`), **‹ prev / next ›** run navigation
   (steps through the domain's stored runs in `~/.email_delivery_hero/runs/<domain>/`, disabled at
   either end; navigating swaps `:runId` in the route), and a **Re-run** button
   (`POST /api/audit/run/:domainId` — creates a *new* run and navigates to it when done).
2. **Inbox-placement hero** — the metric the dashboard cell shows, writ large: the overall
   inbox-placement percentage with a per-provider strip (Gmail / Outlook / Yahoo / Apple /
   long-tail — inbox/spam/promotions/missing mini-bars from `inbox_placement.providers[]`), the
   test's sample label ("campaign_2026-06-28.eml" vs "default template"), and the trend vs the
   previous run (▲/▼ n pts). When placement has never run the hero shows `—` with one line —
   "No placement test yet — connect seeds to measure where mail actually lands" — and the strip
   collapses. Beneath the hero: one plain-language verdict line and the **primary next step** from
   the §5 fix-order ladder (one accent CTA maximum).
3. **Family status strip** — seven compact chips (BIMI · Reports · Content · List-Unsub · Links ·
   Reputation · Placement), each colored by that family's worst severity with a ✓/✗/⚠/ⓘ glyph.
   Clicking a chip scrolls to that family's group in the results table.
4. **Sample & sources panel** (two-up left) — what this run measured against: the sample message
   (filename, date captured, subject, "change sample" link), the connected reputation sources
   (GPT ✓/–, FBL ✓/–, ESP ✓/–, each with last-sync freshness), and the seed-list summary (seeds
   per provider). This panel is where "flying blind" (PS-09) becomes visible at a glance.
5. **Reputation panel** (two-up right) — from `sender_reputation`: complaint rate vs the 0.3%/0.1%
   lines (mini-gauge), hard-bounce rate, GPT domain/IP reputation bands, warmup/volume note, and
   the recurring-blocklist flag. Values gray out with a "connect …" hint when the source is absent.
6. **Sub-test results table** — one row per finding, **grouped by family** in the §1 table order,
   fail-first within each group; each family group header repeats the family chip and its
   pass/fail count **and carries a "Full ⟨family⟩ report ›" chevron** into that family's §8 check
   detail page (run-scoped — the same `:runId` the table is showing). Row anatomy: status icon (✗ red / ⚠ amber / ✓ green / ⓘ slate), finding id in
   small-caps mono, plain-language title, the **observed value** (score, header, link domain +
   return code, rate — monospace), chevron accordion. Expanded = *observed* (exact evidence: the
   raw header, the fired rules with points, the DNS answer from the URI zone) / *expected* / *why
   it matters* (one sentence) / **Fix** strip with `[copy fix]` carrying the exact header, record,
   or edit — never prose.
7. **Problem states** — cards for each matched §6 problem state with a one-paragraph hook and a
   **"Learn more ›"** chevron opening `/domains/:domainId/spam-content/:problemId`.

**Empty / never-run / loading states.** Never-run → gray header cell + "No run yet — run one" CTA.
Loading → skeleton hero + skeleton rows. A run with **no sample and no integrations** is *not*
empty — it renders the BIMI results plus the info advisories, the hero shows `—`, and the
Sample & sources panel becomes the page's call to action. A missing run id → "run not found,
open the newest" link.

**Layout** (desktop-first, 1200 px content max-width, app-wide left nav stays; same shadcn/ui +
lucide vocabulary as the SPF/DMARC/DNS pages — the four category pages must feel like siblings):

Placement rules and rationale:

- **Highest: the outcome.** Unlike the record-shaped categories, Spam & Content has a natural
  hero metric — the placement percentage *is* the product's promise ("are we in the inbox?"), and
  it is the same number the dashboard cell shows, so the click-through never surprises. The
  verdict line under it names the worst provider and the driving cause in plain language.
- **Second band, two-up:** left = **inputs** (what we measured against — sample, sources, seeds;
  the "connect more data" surface); right = **reputation** (the rates and bands operators watch
  weekly). On narrow widths the right panel stacks below the left.
- **Main table grouped by family, not one flat list** — 50+ findings would drown a flat table.
  Within a group: fail → warn → info → pass (never alphabetical). Copy buttons always copy the
  publishable header/record/edit. Family chips anchor-link to their group headers.
- **Lowest: problem-state cards** — deliberate repetition of what the table flagged: the table is
  for operators, the cards for learners.
- **Type & color:** severity colors identical to the Dashboard cells; monospace
  (`font-mono text-xs`) for every header, rule name, link domain, and rate; lucide icons
  (`ShieldCheck`/`ShieldAlert`/`Info`/`ChevronRight`/`ChevronLeft`/`Copy`/`Mail`/`Link2`).
- **Drill-down pages** (`/domains/:domainId/spam-content/:problemId`) reuse the same header
  chrome, then: **Concept** (2–3 paragraphs), **Your data** (the matched findings + snapshot
  fields rendered live from the viewed run), **Diagnose it yourself** (copyable §3 terminal
  commands with the domain/sample substituted in), **Tools**, **Path forward** (numbered steps).
  Back chevron returns to the Spam & Content page anchor.

## 3. Testing toolbox — Brew / npm tools

Verified against `formulae.brew.sh` and the npm registry **2026-07**. These tools are the engine
for the sample-message families (scoring, headers, links) and the debug bench for everything else;
they are shelled out to via the [../run_checks.mdx](../run_checks.mdx) ToolRunner (`execFile`,
bounded timeout + `maxBuffer`, never a shell string). Install the bench with:

```bash
brew install bind doggo checkdmarc swaks lychee tidy-html5 curl perl cpanminus
cpanm --notest Mail::SpamAssassin        # SpamAssassin is NOT a brew formula — see below
npm install mailauth mailparser nodemailer tldts     # same TS/Node stack as the app
```

> **Verified NOT on Homebrew (do not write `brew install …` for these):**
> - **`spamassassin`** — there is **no** SpamAssassin formula in homebrew-core (confirmed
>   2026-07; formulae.brew.sh has no entry). On macOS it installs via Perl:
>   `brew install perl cpanminus && cpanm --notest Mail::SpamAssassin` (or MacPorts
>   `p5-mail-spamassassin`). The `spamassassin` and `spamc` binaries then land in the Perl bin dir.
> - **`rspamd`** — packaged for Linux distros, FreeBSD/OpenBSD ports, and MacPorts, but **not**
>   Homebrew. We do not depend on it; SpamAssassin (above) is our scoring engine.
> - **`dog`** — gone from Homebrew (house rule: use `doggo`).
> - **`oletools`** — not a brew formula (pipx-only); macro-attachment analysis stays future.
> - **npm `spamc`** — exists but abandoned (v0.0.5, last published 2014) — do **not** depend on
>   it; we parse `spamassassin` output ourselves. There is **no** npm package that embeds the
>   SpamAssassin rule engine.

### How a test run executes the tools

Every external invocation goes through the ToolRunner and is recorded — argv, timing, exit code,
and parsed output — as one entry in **`spam_content.tool_runs[]`** in the run file (§4), so a run
is fully reproducible and debuggable after the fact. The contract per tool
(`<sample>` = the stored sample path
`~/.email_delivery_hero/samples/<domain>/<file>.eml`; all timeouts are wall-clock, process killed
and `error` recorded on breach):

| Tool | Exact command line (every input arg) | Output asked for | Timeout | Parsed how → feeds |
| --- | --- | --- | --- | --- |
| SpamAssassin | `spamassassin --test-mode --local <sample>` | text (message + `X-Spam-Status` header and rule report — SA has no JSON mode) | 60 s | Regex the `X-Spam-Status: … score=N required=5.0 tests=…` line, then the per-rule `pts rule description` report block into `{rule, score, description}[]` → `content_score.{score,threshold,rules_fired}` and the scoring sub-tests |
| mailauth report | `npx mailauth report --client-ip <sending-ip> --sender <mail-from> <sample>` | **json** (stdout) | 30 s | `JSON.parse` → `spf/dkim/dmarc/arc/bimi` verdict objects → `list_unsubscribe.from_alignment`, `bimi.message_side`, cross-checks vs the DNS-side categories |
| mailauth vmc *(future)* | `npx mailauth vmc -a <a=-url-from-bimi-record> -d <domain>` | **json** (stdout) | 30 s | `JSON.parse` → issuer, validity window, logo hash match → `bimi.vmc` and `bimi_vmc_valid`/`bimi_vmc_logo_match` |
| checkdmarc | `checkdmarc <domain> -f json` | **json** (stdout) | 60 s | `JSON.parse`; the `bimi` object cross-validates our `bimi_*` DNS rows (test-suite oracle, not the primary detector) |
| dig (per URI zone × link domain) | `dig +short +time=3 +tries=1 A <linkdomain>.<zone>` (TXT re-query on a hit: `dig +short +time=3 +tries=1 TXT <linkdomain>.<zone>`) | text (one `127.0.x.y` per line, or empty = not listed) | 5 s | Return code decoded via the zone's `return_code_map` (shared with [blacklists.mdx](./blacklists.mdx)) → `link_reputation.flagged[]` and the `url_dbl/url_surbl/url_uribl` sub-tests. First-round primary path is in-process `node:dns/promises` — `dig` runs are the recorded fallback/debug form |
| lychee *(future probe)* | `lychee --format json --timeout 10 --max-redirects 5 --max-concurrency 8 --no-progress <urls.txt>` (`urls.txt` = the extracted, deduped URL list written to the run's scratch dir) | **json** (stdout) | 120 s | `JSON.parse` per-link status/redirect map → `link_reputation.links[].{status,final_domain,hops}` and `url_reachable`/`url_redirect_chain` |
| curl (per shortener link) *(future probe)* | `curl -sIL --max-redirs 5 --max-time 10 -o /dev/null -w '%{url_effective}\t%{num_redirects}\t%{http_code}' <url>` | text (single tab-separated line) | 12 s | Split on tabs → final URL (PSL-reduced to `final_domain`), hop count, status → fed back through the URI zones |
| swaks (per seed) *(future)* | `swaks --server <smtp-host> --port 587 --tls --auth-user <user> --from probe@<domain> --to <seed+token@provider> --header "X-EDH-Test-Id: <token>" --data @<sample> --timeout 30` | text (SMTP transcript) | 60 s | Transcript parsed for the final `250`/`4xx`/`5xx` per seed → `inbox_placement.sends[]`; folder read-back is the IMAP/Graph connector, not a CLI |
| tidy-html5 | `tidy -quiet -errors <part.html>` (HTML part extracted to the scratch dir) | text (stderr lint lines; exit 0/1/2) | 15 s | Warning/error count + lines → advisory evidence on `content.html_hygiene` |

Rules that bind all rows: stdout and stderr are captured separately with a 10 MB `maxBuffer`;
JSON is preferred and requested wherever the tool offers it (`output_format: json`), text
otherwise with the parser documented above; a non-zero exit or timeout records
`error: <message>` on the `tool_runs[]` entry and downgrades the dependent sub-tests to a
**warning** "tool failed — re-run" (never a fabricated pass or critical); a missing binary emits
the standing **info** finding naming the §3 install line (e.g. "SpamAssassin not installed —
`cpanm Mail::SpamAssassin`"). `mailparser`/`tldts`/`node:dns` work is in-process and is **not**
recorded in `tool_runs[]` (it is not an external tool).

## 5. Recommendations — the fix-order ladder

Every failing sub-test carries its concrete one-line remediation (family specs own the exact
text). Above the per-test fixes, the page computes one **primary recommendation** from the
fix-order ladder — content fixes have a natural dependency order, and telling the user to buy a
VMC while their complaint rate is spiking is malpractice. First matching rung wins:

| Rung | Condition (first match wins) | Primary recommendation |
| --- | --- | --- |
| 0 | A provider shows majority **Missing** (`seed_missing`) or a linked domain you own is on a high-trust URI zone (`url_dbl`/`url_surbl` critical) | You are blocked, not filtered: pause sending to that provider, request delisting via the zone's portal, and clean the compromised site/shortener before anything else. |
| 1 | Receivers are actively dropping or spoofers are actively using your domain (`report_enforcement` critical, `report_spoofing` critical) | Act on what the receivers themselves reported: authorize the failing own-stream (SPF `include:` / DKIM selector) before the next send, or confirm the spoofing source is being rejected and report high-volume abuse. |
| 2 | Complaint rate ≥ 0.3% or GPT reputation `BAD`/`LOW` (`complaint_rate`, `gpt_spam_rate`, `gpt_domain_reputation`) | Stop the bleeding: pause the offending segment today, suppress every complainer from FBL/GPT data, and cut volume to the most-engaged recipients until the rate is < 0.1%. |
| 3 | Bulk-compliance gate failing (`list_unsubscribe`, `list_unsub_oneclick`, `list_unsub_https`, `from_alignment` critical) | Ship the Gmail/Yahoo requirements: RFC 8058 one-click header + https unsubscribe endpoint, and a `From:` that aligns with your DKIM `d=` — these are hard gates, not scores. |
| 4 | Content scoring critical (`spamassassin_score` ≥ 5, `html_hygiene` hidden text, `mime_valid` broken, `attachment_risk`) | Fix the message: address the highest-scoring fired rules first and re-score until the total is < 2.0. |
| 5 | Link hygiene (`url_shortener`, `url_https`, `url_ip_literal`, `url_punycode`, `url_redirect_chain`, abused-legit zone hits) | Clean the links: full destination URLs on your own https domains, no shorteners, no raw IPs; re-check the final domains against the URI zones. |
| 6 | Report-observed fragility or TLS failures (`report_fragility`, `report_pass_rate` < ~100%, `report_tls` warning) | De-risk the streams the reports flagged: set `aspf=r` or brand the Return-Path so the ESP stream dual-aligns; fix the MX cert / TLSA / MTA-STS layer TLS-RPT named — before a key rotation or cert expiry turns the warning into bounces. |
| 7 | Placement warnings (50–79% inbox, `seed_tab_placement` Promotions, `placement_longtail`) | Tune per provider: SNDS/JMRP for Outlook, engagement pruning for Yahoo, plainer markup for Gmail Primary — the per-provider rows name the lever. |
| 8 | Warmup/volume anomalies (`warmup`, `volume_consistency`) | Smooth the sending pattern: ramp new IPs/domains over 4–8 weeks and spread campaigns instead of burst-and-silence. |
| 9 | BIMI absent/incomplete while DMARC is enforcing (`bimi_*` warnings) | Collect the reward: publish the BIMI record, serve an SVG Tiny-PS logo, and add a VMC/CMC so Gmail/Apple actually render it. |
| 10 | Visibility gaps only (`reputation_data_available`, `postmaster_verified`, `seed_coverage`, no-sample / empty-corpus / stale-corpus infos, `report_domain_attribution` orphans) | You're flying blind: upload a representative `.eml`, verify the domain in Google Postmaster Tools, enroll FBLs, add seeds, and point the report mailbox / analysis directory here (add orphan-report domains as monitored domains) — every future run gets sharper. |
| 11 | All healthy | Keep it: re-run on schedule, watch the placement trend and complaint band — content reputation regresses quietly (a template edit, a new vendor link, one bad list import). |

Standing advisories surfaced as `info` when applicable: keep the sample message current (a
6-month-old sample audits a campaign you no longer send); never A/B test your way past a filter by
obfuscation — obfuscation itself scores; placement probes cost seeds/credits, so they run on the
slow cadence, not every 6 h.

## 6. Problem states

The health situations a domain can be in for Spam & Content. Each maps from finding ids (matched
by **id prefix**, since many carry suffixes), each has a drill-down page
(`/domains/:domainId/spam-content/:problemId`). PS-00 is the healthy goal state. A state matches
when any listed prefix matches a finding at warning/critical severity (PS-09 matches on `info`).

| Id | Problem state | Detected by (`content.` elided) | Severity |
| --- | --- | --- | --- |
| PS-00 | Message layer healthy end to end | all families pass | ok |
| PS-01 | Mail is blocked or vanishing at a provider | `seed_missing`, `placement_gmail`/`placement_outlook`/`placement_yahoo`/`placement_apple`/`placement_longtail` (majority Spam/Missing), `seedlist_overall` < 50% | critical |
| PS-02 | The message itself scores as spam | `spamassassin_score`, `spammy_phrases`, `html_hygiene`, `mime_valid`, `header_sanity`, `attachment_risk`, `subject`, `image_text_ratio`, `multipart`, `encoding` | critical / warning |
| PS-03 | Bulk-sender compliance gap (Gmail/Yahoo 2024) | `list_unsubscribe`, `list_unsub_oneclick`, `list_unsub_https`, `list_unsub_syntax`, `list_unsub_reachable`, `from_alignment` | critical |

Drill-down pages follow the app-wide pattern (§2): **Concept**, **Your data** (the matched

### PS-00 — Message layer healthy end to end

The goal state: a scored sample under 2.0 with clean structure, both unsubscribe headers present
and DKIM-covered, every link full-URL https on domains you control and clean on the URI zones,
complaint rate under 0.1% with sources connected, ≥ 90% inbox placement, BIMI rendering (or
consciously declined).
- **Data fields:** `families.*.status = ok`, all seven §4 snapshots clean, a current `sample_id`.
- **Diagnose/debug:** nothing to fix — `spamassassin --test-mode --local <sample>` and a
  mail-tester.com pass (9–10/10) should agree with us.
- **Tools:** the §3 bench, run on schedule rather than ad hoc.
- **More health metrics:** the run-over-run trend is the health metric now — placement pts,
  complaint band, score drift on the *same* sample (ruleset updates re-price rules).
- **UI:** all-green strip; the drill-down teaches *keeping* it green — templates rot (a designer
  ships an image-only hero), lists age (complaints creep), a vendor link changes hands.

### PS-01 — Mail is blocked or vanishing at a provider

- **Concept:** blocked ≠ filtered. Spam-folder placement means the message was accepted and
  judged; **Missing** means it was rejected in SMTP or silently dropped — a different, more
  urgent problem. Since the enforcement waves this is visible in bounce text: Gmail
  `421 4.7.26` (unauthenticated rate-limit) and `550 5.7.x` permanent policy rejects
  (Nov 2025 ramp), Microsoft `550 5.7.515` (May 2025, 5k+/day). A majority-Missing provider
  while others inbox fine points at that provider's gate, not at content.
- **Data fields:** `inbox_placement.providers[]` (`missing` counts), `seed_missing`,
  `seedlist_overall`; the swaks `tool_runs[]` transcripts carry the SMTP reply per seed.
- **Diagnose/debug:** read the per-seed SMTP transcript (`tool_runs[]` → final `250`/`4xx`/`5xx`
  line); re-send one probe by hand: `swaks --server <smtp-host> --port 587 --tls --from
  probe@<domain> --to <seed@provider> --data @<sample>`; grep ingested bounces for the sentinel
  codes above.
- **Tools:** swaks (transcript), the provider postmaster pages (Gmail sender guidelines,
  Microsoft delist portal), [../emails.mdx](../emails.mdx) bounce ingestion.
- **More health metrics:** future — sentinel-code counters per provider per run, so enforcement
  onset is timestamped.
- **UI:** the hero's provider strip shows Missing in dark red; the drill-down explains
  block-vs-filter and links the provider's unblock/delist path; fix = clear the named gate
  (auth, complaint, or listing) before resuming volume to that provider.

### PS-02 — The message itself scores as spam

- **Concept:** rule engines price every structural defect and sum the points; at SpamAssassin's
  default 5.0 the message is tagged spam, and real inboxes are tuned tighter — the safe target
  is **< 2.0**. The score is fully diagnosable: every fired rule has a name, a weight, and a
  concrete fix, so remediation is "fix the biggest number first", never "make it less spammy".
- **Data fields:** `content_score.{score,threshold,engine,rules_fired[]}` and the per-rule
  `{rule, score, description}` rows; `sample_id` names what was graded.
- **Diagnose/debug:** `spamassassin --test-mode --local <sample>` → read the rule report
  points-descending → fix the named defect (add the `text/plain` part, cut the image share,
  sentence-case the subject, drop the attachment) → re-score until < 2.0.
- **Tools:** SpamAssassin via `cpanm Mail::SpamAssassin` (§3 — never `brew install
  spamassassin`); tidy-html5 for the HTML part; mail-tester.com as the external second opinion.
- **More health metrics:** score trend on an unchanged sample = ruleset drift (`sa-update`);
  a jump after a template edit pinpoints the edit; future — a second engine (Rspamd via Docker,
  or npm `spamscanner`) for band-edge cross-validation.
- **UI:** the fired-rules table sorted by points, each row carrying its sub-check id and copy
  fix; the score renders as `1.4 / 5.0` with the 2.0 warning line marked.

### PS-03 — Bulk-sender compliance gap (Gmail/Yahoo 2024)

- **Concept:** the two-header one-click pair is a hard gate for bulk senders, not a score:
  `List-Unsubscribe: <https://…>` (HTTPS URI, opaque per-recipient token) **and**
  `List-Unsubscribe-Post: List-Unsubscribe=One-Click` (RFC 8058). The endpoint must accept the
  bare form-encoded POST with 2xx — no redirect, no login — and be honored within 2 days. DKIM's
  `h=` must cover both headers so they can't be stripped in transit. `From:` must align with an
  authenticated domain. Transactional-only senders are exempt — the `isBulkSender` flag decides
  the severity.
- **Data fields:** `list_unsubscribe.{header_present,one_click,https_uri,mailto_uri,
  is_bulk_sender,from_alignment}`.
- **Diagnose/debug:** read the sample's headers verbatim; `npx mailauth report <sample>` →
  check `dkim[].signingHeaders` includes both `List-Unsubscribe*` names and the `from_alignment`
  verdicts; opt-in endpoint probe: `curl -s -o /dev/null -w '%{http_code} %{redirect_url}' -X
  POST -H 'Content-Type: application/x-www-form-urlencoded' -d 'List-Unsubscribe=One-Click'
  '<uri>'` — **only with a synthetic recipient token; a real token really unsubscribes someone.**
- **Tools:** mailauth (npm), curl; the actual fix site is usually the ESP's header settings.
- **More health metrics:** future — `list_unsub_honored`: re-probe a synthetic address after 48 h
  to verify the unsubscribe took effect.
- **UI:** observed/expected prints both headers verbatim (present green, absent red with the
  exact line to add, copyable); the bulk toggle is linked from the card so a transactional
  sender can downgrade the state in one click.

### PS-04 — A linked domain is blocklisted

- **Concept:** receivers extract every URL, reduce it to the registrable domain, and query the
  URI zones — one listed domain (a partner, a tracker, a shortener endpoint, your own hacked
  blog) filters the message everywhere, regardless of the sending domain's health. Return codes
  carry the diagnosis: DBL `127.0.1.2/4/5/6` = spam/phish/malware/botnet, `127.0.1.102–.106` =
  **abused-legitimate** (a compromised legit site — secure it first, then delist); SURBL last
  octet is a bitmask (8 = phish, 16 = malware, 64 = abuse, 128 = cracked); URIBL 2 = black,
  4 = grey, and **1 means your resolver is blocked** — an inconclusive result, never a listing.
- **Data fields:** `link_reputation.zone_hits[]` (`domain`, `zone`, `code`, `reason`,
  `delisting_url`), `links_total`, `unique_domains`.
- **Diagnose/debug:** per flagged domain: `dig +short A <domain>.dbl.spamhaus.org` /
  `<domain>.multi.surbl.org` / `<domain>.multi.uribl.com`, then the TXT re-query for the reason
  string — through the system/own resolver (the zones' public mirrors block open resolvers like
  8.8.8.8); decode the return code before acting on it.
- **Tools:** dig/doggo (§3), the per-zone delisting portals (check.spamhaus.org, surbl.org,
  uribl.com) carried in the finding.
- **More health metrics:** re-query flagged domains every run — abused-legit listings drop after
  cleanup; a *new* hit on an unchanged template means the linked site changed, not your mail.
- **UI:** one row per hit naming domain, zone, decoded meaning, and the correct delisting URL;
  abused-legit codes swap in the "secure the compromised site first" fix variant.

### PS-05 — Link hygiene looks phish-shaped

- **Concept:** short of a listing, links carry their own penalty signals: raw-IP URLs
  (near-certain filtering), `http:` scheme, punycode/homograph hosts impersonating a brand,
  public shorteners (filters expand them anyway — SpamAssassin 4.x DecodeShortURLs — so they
  hide nothing and cost trust), deep redirect chains, and simply too many links or domains.
  All computable locally from the sample.
- **Data fields:** `link_reputation.flagged[]` (`domain`/`url`, `kind`, `final_domain`),
  `links_total`, `unique_domains`.
- **Diagnose/debug:** eyeball the flagged URLs (scheme, host, look-alike characters); expand a
  shortener: `curl -sIL --max-redirs 5 -o /dev/null -w '%{url_effective} %{num_redirects}'
  <url>` → PSL-reduce the final domain and re-check it against the §3 zones.
- **Tools:** curl (single URL), lychee (bulk, JSON), tldts in-process for the PSL reduction.
- **More health metrics:** future probes — `url_reachable` (dead links score too),
  `url_redirect_chain` depth, `url_safe_browsing` (needs an API key).
- **UI:** one row per offense with the exact URL in monospace; fix = full destination URLs on
  your own https domains, a branded short domain if shortening is unavoidable, prune the count.

### PS-06 — Complaints/bounces over the line

- **Concept:** user-reported spam is the purest "recipients don't want this" signal, and Gmail
  throttles on it directly: **< 0.1%** is the stated ideal, **≥ 0.3%** the hard line — at or
  above it Gmail withholds mitigation until **7 consecutive days** back under. Hard bounces ride
  along as the list-hygiene signal. Neither is probeable from outside: the numbers come from
  Google Postmaster Tools, Microsoft SNDS/JMRP, and FBLs — which is why PS-09 gates this state.
- **Data fields:** `sender_reputation.{complaint_rate,complaint_trend,bounce_rate,
  gpt_spam_rate,sources}`.
- **Diagnose/debug:** Postmaster Tools spam-rate dashboard (populates at ≥ 5,000 msgs/day to
  Gmail); SNDS filter-verdict color per IP; correlate the spike date with the campaign/segment
  that caused it; suppress every complainer from FBL data.
- **Tools:** the GPT connector (§3 API sources), SNDS/JMRP enrollment,
  [../emails.mdx](../emails.mdx) FBL ingestion.
- **More health metrics:** complaint trend vs send volume (spikes correlate with one segment);
  engagement/sunsetting policy; future — per-campaign complaint attribution.
- **UI:** the Reputation panel's mini-gauge with the 0.1%/0.3% lines; the drill-down is the
  suppress-then-slow playbook, with the 7-clean-days mitigation rule spelled out.

### PS-07 — Reputation decay in progress

- **Concept:** the slow-burn state: GPT domain/IP reputation sliding to MEDIUM, engagement
  falling, warmup skipped on a new IP/domain, burst-and-silence volume patterns, or this app's
  own [Blacklists](./blacklists.mdx) history showing repeat listings. Nothing is on fire yet —
  the point of the state is to act before PS-01/PS-06 arrive.
- **Data fields:** `sender_reputation.{gpt_domain_reputation,gpt_ip_reputations[],
  complaint_trend,recurring_blocklists}`, `inbox_placement` `seed_trend`.
- **Diagnose/debug:** read the GPT band history (falling = which campaign/segment started it?);
  check volume day-over-day for spikes; review the blocklist-history trend across stored runs.
- **Tools:** GPT connector, the app's own run history (the differ is the detector here).
- **More health metrics:** placement trend (▲/▼ pts on the hero) is the leading indicator;
  future — engagement metrics from the ESP connector.
- **UI:** trend arrows on the Reputation panel and hero; fix = ramp discipline (4–8 week
  warmup), steady volume, prune the coldest segment — behavioral fixes, no record to publish.

### PS-08 — BIMI published but not rendering

- **Concept:** BIMI displays the brand logo — and doubles as a forcing function for DMARC
  enforcement, its hard prerequisite (`p=quarantine`/`reject`). The chain is: enforcing DMARC →
  `default._bimi` TXT (`v=BIMI1` first, `l=` HTTPS SVG Tiny-PS URL) → and for Gmail/Apple, an
  `a=` VMC/CMC from a Mark Verifying Authority. A record published while DMARC is `p=none` is
  the critical variant (advertises a logo no one will render and signals config drift);
  SVG-only is the common warning; an intentionally empty `l=`/`a=` is declined BIMI — a valid
  opt-out (info).
- **Data fields:** `bimi.{selector,record_found,raw_record,dmarc_enforcing,logo_url,vmc_url,
  vmc}`.
- **Diagnose/debug:** `dig +short TXT default._bimi.<domain>` (tag order, values) →
  `dig +short TXT _dmarc.<domain>` (policy must be quarantine/reject) → `curl -sI <logo-url>`
  (200, `image/svg+xml`) → `npx mailauth vmc -a <vmc-url> -d <domain>` for issuer/validity/logo
  hash (future probe).
- **Tools:** dig, curl, mailauth vmc, checkdmarc (cross-check), the BIMI Group SVG conformance
  checker; MVA portals (DigiCert/Entrust) for the VMC purchase.
- **More health metrics:** future — `bimi_vmc_valid` expiry watch (a VMC nearing expiry is a
  regression the differ flags before the logo disappears).
- **UI:** the BIMI chip's row group reads as a chain (DMARC ✓ → record ✓ → SVG ✓ → VMC ✗) so
  the first broken link is obvious; fix strip carries the exact TXT record; the drill-down
  walks the DMARC prerequisite and the VMC path — deliberately last in the §5 ladder
  (branding, not deliverability).

### PS-09 — Flying blind — no sample / sources / seeds

- **Concept:** not a failure — the onboarding state. Scoring, list-management, and link checks
  grade a real message and cannot run without an uploaded `.eml`; complaint and reputation
  numbers exist only inside Postmaster Tools/SNDS/FBLs the sender must enroll in; placement
  needs seeds. The category must present each gap as *unlock* ("upload one sample → ~40 more
  sub-tests next run"), never as a red mark: all PS-09 findings are `info` and the cell stays
  green.
- **Data fields:** `sample_id` absent, `sender_reputation.sources` = absent,
  `inbox_placement` never tested, `postmaster_verified`, `fbl_enrollment`, `seed_coverage`.
- **Diagnose/debug:** none — the fixes are actions: get an `.eml` (Gmail ⋮ → "Show original" →
  Download, or the ESP's "view source"); verify the domain in Postmaster Tools
  (`google-site-verification` TXT — the one part we can see in DNS); register sending IPs at
  SNDS and turn on JMRP; enroll FBLs and route them into [../emails.mdx](../emails.mdx); add
  seed accounts or a seed-service key.
- **Tools:** the upload panel (drag-drop/paste, size-capped per
  [./content_scoring.mdx](./content_scoring.mdx)); the provider consoles linked stepwise from
  the drill-down.
- **More health metrics:** once connected, PS-06/PS-07's metrics populate — this state is the
  gateway to the whole behavior layer.
- **UI:** the Sample & sources panel is the page CTA; each absent source is a quiet
  "Connect ›" row; the drill-down is the onboarding checklist in fix-order.

### PS-10 — Delivered, but to the wrong place (Gmail tabs)

- **Concept:** Promotions is a tab, not a spam folder — mail there was accepted, authenticated,
  and shown; open rates differ but deliverability is intact. The state exists to *prevent
  panic*: it is a warning at most, and chasing Primary-tab placement with tricks (fake personal
  tone, stripped styling that harms the brand) costs more than it earns. Slow delivery
  (`seed_delivery_latency`) rides along here — minutes-late mail at one provider hints at
  greylisting or throttling.
- **Data fields:** `inbox_placement.providers[]` (`promotions` counts), `seed_tab_placement`,
  `seed_delivery_latency`.
- **Diagnose/debug:** compare the Promotions share across runs (a jump correlates with a
  template change); test a plainer variant (fewer images, fewer links, lighter markup) through
  the same seeds and diff the tab split.
- **Tools:** the seed loop itself is the instrument; no external tool applies.
- **More health metrics:** tab split trend per provider; latency percentiles per provider.
- **UI:** Promotions renders amber-lite with an explicit "this is not spam-foldering" note;
  fix = content-weight levers, offered as optional tuning, never as a compliance item.

### PS-11 — Receivers' reports flag your mail stream

- **Concept:** DMARC aggregate and TLS-RPT reports are the receivers **telling you what they
  saw** — the only evidence that arrives without probing anything. Four escalating stories hide
  in them: a stream that passes DMARC on **one mechanism only** (e.g. an ESP subdomain envelope
  failing SPF alignment under `aspf=s`, rescued by DKIM — one key rotation from rejection under
  `p=reject`); a **spoofer** using your domain (rows failing both alignments); your **own mail
  being quarantined/rejected** right now; and **inbound TLS quietly failing** for senders that
  enforce MTA-STS/DANE. The run-time test ([../emails.mdx §13](../emails.mdx)) walks the report
  corpus on every run, attributes each report to its domain by the payload's policy domain, and
  scores this catalog.
- **Data fields:** the whole `report_emails` §4 snapshot (`attribution`, `dmarc.pass_rate_pct`,
  `dkim_only`/`spf_only`/`both_fail`, `quarantined`/`rejected`, `tlsrpt.sessions_failed`,
  `window`), plus the per-source-IP table from the Reports view.
- **Diagnose/debug:** open the expandable per-source table (source IP · count · disposition ·
  SPF/DKIM eval + alignment · envelope); confirm a fragile stream's envelope
  (`em2598.<domain>` ⇒ the ESP's default Return-Path); check the published record:
  `dig +short TXT _dmarc.<domain>` (read `aspf=`/`adkim=`/`p=`); after the fix, watch the next
  window's reports confirm dual alignment.
- **Tools:** the [../emails.mdx](../emails.mdx) ingestion + Reports view ("Ingest now"), dig for
  the record edits; the ESP's authenticated-domain / custom Return-Path settings page is where
  the branded-bounce fix actually ships.
- **More health metrics:** pass-rate trend across windows; new-source appearances
  (`dmarc.report_new_source`); TLS failure-rate trend — all fed by the same store.
- **UI:** the family group reads observed-first (the counts and IPs from the reports, with the
  "from reports" chip); fragility rows carry the two-option fix (`aspf=r` **or** branded
  Return-Path) as a copyable record edit; orphan-report rows link to "Add domain"; the healthy
  state prints the §12 emails.mdx numbers ("96.8% dual-aligned · 0 spoofed · 0 rejected ·
  TLS 7/0") so green is quantified, never blank.

## 9. "Run this check now" — the single-check re-run

Every check detail page (§8), and each family group header's overflow on the category page,
offers a **single-family re-run** — the affordance that closes the fix loop: publish the record /
add the header / clean the link, then re-test *just that* in seconds instead of a full audit.

**Endpoint.**

```
POST /api/audit/run/:domainId/check/:checkerId       → 200 AuditResult (the merged latest)
```

`:checkerId` is the **checker registry id** (§8 map: `content.bimi`, `content.report_emails`,
`content.scoring`, `content.list_unsubscribe`, `content.url`, `content.reputation`,
`content.inbox_placement`). Unknown domain or checker id → 404. A full audit already in flight
for the domain (the `inFlight` map in `audit.service.ts`) → 409 with
`{ reason: "audit_in_flight" }` and the UI falls back to "audit already running — watch the dock".

**Backend shape.** This generalizes the **already-implemented** content-scoring re-score
(`AuditService.rescoreContent`, exposed today as
`POST /api/audit/content-sample/:domainId/rescore`) into `rerunChecker(domainId, checkerId)`,
keeping its exact semantics:

- runs the one checker with the domain's config and `previousResults` from the latest result;
- **surgical merge** — swap only this checker's findings/payload (matched by the §1 id-prefix
  table, exactly as `isContentScoringFinding` does today), keep every other check's output;
- re-summarize score/status/counts and the `isNew` new-problem count;
- **latest-cache-only persist** — refreshes the latest-result store and the dashboard but
  **never rewrites the immutable per-run history** (pm/storage.mdx §7.2); a single-check re-run
  is a patch to "now", not a new run;
- with no prior run at all, a full audit runs instead (nothing to merge into);
- a checker crash merges the standing `<checkerId>.error` warning finding, never a fabricated
  pass;
- external tools invoked by the re-run still log `tool_runs[]` entries per §3 into the latest
  result's execution log.

The legacy `POST /api/audit/content-sample/:domainId/rescore` remains as a thin alias delegating
to `rerunChecker(domainId, "content.scoring")` — the upload-panel flow keeps working unchanged.

**UX rules.**

- Button label **"Run this check now ⟳"**; while running: spinner + "Checking…", button disabled;
  on completion the results query invalidates and the page re-renders from the merged latest —
  no navigation.
- **On a historical run view** the button relabels to **"Re-run on newest ⟳"** with a tooltip
  ("re-running updates the newest result — this historical run is immutable"); on completion it
  navigates to the newest-run alias of the same page so the user is looking at what they just
  produced.
- **Gated family, input absent** — the button still runs (the checker is cheap and re-emits the
  info finding) but the hover names the missing input; the real CTA is the §8 unlock panel.
- **Inbox placement** is the one costed family: the button opens a confirm dialog stating the
  seed/credit cost ("sends 1 probe to 46 seeds — uses 1 placement credit") before posting;
  placement re-runs are never auto-triggered.
- The full **"Re-run" (whole audit)** stays where §2 put it — the category-page header — and is
  never replaced by the single-check button; the two are visually distinct (accent vs outline).

## 10. History & trends across runs

The run store keeps every run per domain (§4), so trend rendering is a **client-side derivation
over `GET /api/audit/runs?domainId=…`** (newest-first, full results included) — no new endpoint.
The frontend derives, per run, each family's status (worst severity over the §1 prefix match) and
its **headline metric**:

| Family | Headline metric (per run) | Sparkline shape |
| --- | --- | --- |
| BIMI | chain depth reached (DMARC → record → SVG → VMC, 0–4) | step line |
| Report emails | real-world DMARC pass rate over the window | line |
| Content scoring | SpamAssassin total (same-sample runs annotated) | line with 2.0/5.0 bands |
| List-Unsubscribe | one-click present (the bulk gate) | ✓/✗ boolean strip |
| Link reputation | zone-hit count (flagged count secondary) | bar |
| Sender reputation | complaint rate | line with 0.1%/0.3% bands |
| Inbox placement | overall inbox % | line, per-provider on hover |

**Where trends render.**

1. **Category page (§2)** gains a **History band** between the two-up panels and the sub-test
   table: one small sparkline tile per family (label + current value + Δ vs previous run);
   clicking a tile opens that family's §8 detail page. Collapsed by default when the domain has
   fewer than 2 runs.
2. **Check detail page (§8, band 8)** — the family's sparkline plus the last-N-runs mini-table
   (timestamp, chip, metric, Δ); row click swaps `:runId`.
3. **The hero (§2)** keeps its ▲/▼-vs-previous-run marker — same derivation, previous run only.

Trend **semantics** stay owned by the §4 regression differ: a band crossing (score crosses
2.0/5.0, complaint crosses 0.1%/0.3%, placement drops a band, a new zone hit, a lost one-click
header, a VMC entering its expiry window) is what flags `isNew` — the sparklines *visualize*, the
differ *decides*. One-run domains render the tiles with "first run — trends appear after the next
run" and no fabricated flat line.

## 12. Acceptance criteria

1. All seven family checkers run in every run and emit pass **and** fail findings with the
   `content.` prefix; input-gated families with no sample/integration/seeds emit a single **info**
   finding each (never a fabricated pass, warning, or critical); the report-email family with an
   empty corpus emits a single **info** the same way.
2. The run file at `~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml` contains
   a top-level `spam_content` key with `status`, the seven family snapshot keys
   (`bimi`, `report_emails`, `content_score`, `list_unsubscribe`, `link_reputation`,
   `sender_reputation`, `inbox_placement`), `tool_runs[]`, `tests[]`, and `problem_states` exactly
   per §4; the file's `run:` block and sibling category keys are owned by
   [../storage.mdx](../storage.mdx).
3. Every external tool invocation appends one `tool_runs[]` entry with
   `tool, command (exact argv string with every input arg), started_at, duration_ms, exit_code,
   output_format (json|text), parsed, error` — including failed and timed-out invocations
   (`error` non-null, dependent sub-tests downgraded to a "re-run" warning).
4. The toolbox is honest: no spec, code, or UI text ever says `brew install spamassassin` or
   `brew install rspamd` (neither formula exists); the SpamAssassin install path is
   `brew install perl cpanminus && cpanm --notest Mail::SpamAssassin`, and a missing binary
   yields the info finding naming that exact line.
5. A bulk-flagged domain (`isBulkSender`) whose sample lacks `List-Unsubscribe-Post` yields a
   **critical** with the copyable header as the fix; a non-bulk domain yields a **warning**.
6. A sample linking a domain listed on DBL/SURBL/URIBL yields a **critical** naming the link
   domain, the zone, the decoded return code, and the zone's delisting URL; shortener/http/raw-IP/
   punycode links yield the §1 severities with the offending URL in the evidence.
7. SpamAssassin totals map to severity as < 2.0 → ok, 2.0–4.99 → warning, ≥ 5.0 → critical, and
   the finding evidence lists every fired rule with its points.
8. BIMI: a record published while this run's DMARC is not at `p=quarantine`/`p=reject` yields a
   **critical**; SVG-only (no `a=`) yields a **warning**; a domain with no BIMI and no opt-in
   yields **info** only.
9. The Dashboard `spamContent` cell shows the run's inbox-placement percentage (e.g. `92%`) when
   a placement result exists, else the `K of M fail`/`Healthy` fallback; color = worst `content.*`
   severity; `info`-only never turns it amber; the cell click opens the newest run's category page
   from Table 1 and that run's page from Table 2, while clicks elsewhere in the row open the
   whole run report.
10. The category page renders at `/domains/:domainId/runs/:runId/spam-content` (newest alias
    `/domains/:domainId/spam-content`): header with run timestamp + working ‹ prev / next ›
    run navigation + Re-run, the inbox-placement hero with per-provider strip (and the `—`
    never-tested state), the seven-chip family strip with anchor links, the Sample & sources and
    Reputation panels, the family-grouped fail-first sub-test table with observed values,
    accordion evidence, and copyable fixes, and the matched problem-state cards.
11. Problem-state drill-downs (`/domains/:domainId/spam-content/:problemId`) render concept, live
    data from the viewed run, copyable diagnose-commands, tools, and the numbered path forward
    for PS-01…PS-11.
12. The primary recommendation is computed by the §5 ladder (first matching rung), so a provider
    block outranks a complaint spike, which outranks compliance headers, and BIMI advice never
    appears while mail is being junked.
13. A later scheduled run that sees a placement drop, a new URI-zone hit, a complaint band change,
    a lost one-click header, or a VMC nearing expiry flags each as a **new problem** via the
    run-file regression diff.
14. Check detail pages render at `/domains/:domainId/runs/:runId/spam-content/check/:checkKey`

15. The §8 slug map is frozen (slug ↔ checker registry id ↔ §4 section key); the static `check`

16. `POST /api/audit/run/:domainId/check/:checkerId` re-runs exactly one checker and responds
