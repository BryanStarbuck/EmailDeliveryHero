# EmailDeliveryHero — SPF Check

**SPF (Sender Policy Framework, [RFC 7208](https://www.rfc-editor.org/rfc/rfc7208))** is a DNS TXT
record at a domain's apex that enumerates which hosts may send mail using that domain in the SMTP
`MAIL FROM` (envelope-from / Return-Path). Receivers evaluate it — walking `ip4`/`ip6`/`a`/`mx`/
`include`/`exists` mechanisms left-to-right — to decide whether the connecting IP is authorized.
SPF is also the **SPF half of DMARC alignment**: a broken or over-limit record silently drops the
domain's DMARC pass rate. The single most common real-world failure is not "no SPF" but a record
that **exceeds the 10-DNS-lookup limit** and therefore evaluates to `permerror` — under DMARC that
counts as an SPF *fail* for every message the domain sends.

This check belongs to the **SPF** dashboard category; all sub-check ids use the **`spf`** checkId
prefix. It is implemented as `spf.check.ts`.

## 1. What it checks

**The standard.** RFC 7208 (April 2014, obsoletes RFC 4408). A sender publishes exactly one TXT
record beginning `v=spf1`; a receiver, given the envelope-from domain and the connecting IP, walks
the record's **mechanisms** (`ip4`, `ip6`, `a`, `mx`, `ptr`, `exists`, `include`, `all`) and
**modifiers** (`redirect=`, `exp=`) left-to-right, returning the qualifier of the first matching
mechanism: `+` pass (default), `-` fail, `~` softfail, `?` neutral. Evaluation yields exactly one
of: `none`, `neutral`, `pass`, `fail`, `softfail`, **`temperror`** (transient DNS failure), or
**`permerror`** (the record cannot be interpreted). The old DNS **SPF type-99 RR is deprecated** —
only TXT counts.

**The hard limits the auditor enforces** (all RFC 7208 §4.6.4 unless noted):

- **10 DNS lookups** total, counted **recursively**: `include`, `a`, `mx`, `ptr`, `exists`, and
  `redirect=` each cost one; `ip4`, `ip6`, `all`, `exp=` cost nothing. Exceeding 10 → `permerror`.
- **2 void lookups**: queries returning NXDOMAIN or an empty answer; a third → `permerror`.
- Each `mx` (and `ptr`) may itself evaluate at most **10** names.
- **One record**: two or more `v=spf1` strings → `permerror` (§4.5) — receivers ignore SPF entirely.
- **255 bytes** per TXT character-string (RFC 1035); long records are split into multiple quoted
  strings inside ONE record. Whole answers past ~**512 bytes** risk UDP truncation (§3.4).
- Subdomains do **not** inherit SPF — every envelope-from domain needs its own record.

**Why receivers care.** SPF `pass` with alignment is one of the two ways to satisfy DMARC and one
of the Gmail/Yahoo/Microsoft bulk-sender requirements (SPF or DKIM mandatory, both recommended).
`permerror`, `softfail`-everything, or `none` all degrade reputation and inbox placement; `+all`
makes the domain trivially spoofable — and is itself scored as a spam signal.

**Common real-world failure modes.** (1) The record grew past 10 lookups as ESPs were added →
`permerror`. (2) Two `v=spf1` records (one from the registrar wizard, one from an ESP wizard) →
SPF ignored entirely. (3) A stale `include:` to a decommissioned vendor → void lookups / dead
include `permerror`. (4) `+all`/`?all`/no `all` leaves the domain effectively open. (5) The real
sending IP is not covered by any mechanism, so mail fails despite a "valid-looking" record.
(6) The ESP's default bounce domain (`em1234.sendgrid.net`) passes raw SPF but is **unaligned**
with the `From:` domain, so SPF contributes nothing to DMARC.

## 2. Tests we perform (sub-checks)

The comprehensive first-round set for the `spf` category. Every headline test emits a finding —
**pass (`ok`) findings included** — so the SPF page renders an explicit passed/failed row per test.
Each remediation is the exact record/term/step to apply.

| Sub-check (finding id) | What it verifies | How measured | Typical critical / warning finding | Concrete remediation |
| --- | --- | --- | --- | --- |
| `spf.present` / `spf.missing` | Exactly one `v=spf1` TXT record exists at the apex. | `resolveTxt(domain)`; join string chunks; prefix-match `v=spf1`. | **critical:** no record (`spf.missing`). | Publish at the apex: `v=spf1 include:_spf.google.com ~all` (swap the include for your ESP). |
| `spf.multiple` | Not more than one `v=spf1` record (≥2 = `permerror`, §4.5). | Count matching TXT strings. | **critical:** two or more records → SPF ignored entirely. | Merge into one record; delete the extras so exactly one `v=spf1` remains. |
| `spf.syntax` | Record parses against the RFC 7208 grammar: known mechanisms, valid qualifiers, valid CIDR (ip4 0–32, ip6 0–128), values where required, `all` takes none. | Tokenize + validate each term. | **critical:** any grammar violation (each is a `permerror`). | Fix the quoted offending term(s); re-validate before publishing. |
| `spf.macro` | `%`-macros are well-formed: `%%`/`%_`/`%-` or `%{<letter><digits?><r?><delims?>}` with a valid letter (`s l o d i p v h`, plus `c r t` in exp). | Scan every `%` in domain-specs (§7). | **critical:** stray `%`, unterminated or unknown macro → `permerror`. | Correct the macro to a valid `%{…}` form or remove it. |
| `spf.lookups` | Recursive expansion stays within the **10-lookup budget**. | Walk include/redirect graph counting `include`/`a`/`mx`/`ptr`/`exists`/`redirect`. | **critical:** > 10 → `permerror`; **warning:** 8–10 (one vendor away from breaking). | Replace `a`/`mx` with explicit `ip4:`/`ip6:`, drop unused includes, or flatten (dynamically — see §8). |
| `spf.void` | Not more than **2 void lookups** (NXDOMAIN / empty answers). | Track void answers during expansion. | **critical:** > 2 → `permerror`; **warning:** 1–2 (fragile, usually a dead vendor). | Delete the `include:`/`a`/`mx` terms pointing at names with no records (the include tree shows which). |
| `spf.all` | Terminal `all` with a sane qualifier: `-all` or `~all`; **never `+all`**; `?all`/absent flagged. | Read the qualifier of the `all` term. | **critical:** `+all` (authorizes the internet); **warning:** `?all` or no `all` (and no redirect). | Use `~all` with DMARC deployed, `-all` on strict/non-sending domains; remove any `+all`. |
| `spf.all_terminal` | Nothing follows `all` (later terms are dead code, ignored by receivers). | Check token position. | **warning:** mechanisms after `all` never evaluate. | Move `all` to the end; delete the unreachable terms. |
| `spf.ptr` | The deprecated `ptr` mechanism is not used (§5.5 "SHOULD NOT"; slow, unreliable, burns a lookup, some receivers skip it). | Scan for `ptr`. | **warning:** `ptr` present. | Delete `ptr`; replace with explicit `ip4:`/`ip6:` or the vendor's `include:`. |
| `spf.redirect` | `redirect=` sanity: not combined with `all` (then it is ignored, §6.1); target publishes a valid single `v=spf1`; counted as a lookup. | Parse modifier; resolve target during expansion. | **warning:** `redirect` + `all` together; **critical:** target has no/broken SPF (via `spf.include_resolves`). | Remove one of `redirect=`/`all`; point `redirect=` at a domain with one valid record. |
| `spf.exp` | `exp=` (if present) targets a resolvable TXT — not lookup-counted, but the explanation must exist when triggered. | Resolve the exp target TXT. | **warning:** `exp=` target missing/unresolvable. | Publish the explanation TXT, or drop `exp=`. |
| `spf.include_resolves` | Every `include:`/`redirect=` target publishes exactly one syntactically valid `v=spf1` (a nested `permerror` propagates upward). | Recurse into each target; re-run presence/single/syntax there. | **critical:** target with no record, multiple records, or invalid syntax; **warning:** transient lookup failure (`temperror`-flavored). | Remove the dead include (decommissioned vendor) or have the target's operator fix their record. |
| `spf.recursion_depth` | The include/redirect graph has no cycles and sane nesting. | `visited` set during expansion; depth cap. | **critical:** include/redirect loop → `permerror`. | Break the cycle; includes must not reference back into the chain. |
| `spf.dup_mechanisms` | No duplicate terms (copy-paste rot; each dup include also wastes a lookup). | Dedup the apex term list. | **warning:** the same term appears twice. | Remove the duplicates; each network/vendor appears once. |
| `spf.cidr_scope` | Declared ranges are not absurdly broad. | Prefix size per `ip4`/`ip6` term. | **critical:** `/0` (equivalent to `+all`); **warning:** ip4 < /8 or ip6 < /16. | Narrow the CIDR to the actual sending ranges; never authorize a `/0`. |
| `spf.length` | The record does not risk UDP truncation. | Byte length of the joined record. | **warning:** > ~450 bytes (truncation/TCP fallback → `temperror` at strict resolvers). | Shorten: drop unused includes, replace `a`/`mx` with explicit ranges, split streams onto subdomains. |
| `spf.dns_type` | No stale deprecated **SPF type-99** RR lingering next to the TXT. | `dig <domain> SPF` (Brew `bind`; degrades gracefully when dig is absent). | **info:** a type-99 record answers (stale config; receivers ignore it). | Delete the SPF-type (99) record; publish TXT only. |
| `spf.ttl` | The apex SPF TXT RRset **TTL is operationally sane** — an emergency fix must propagate before the day is out, but sub-minute TTLs hammer resolvers for nothing. | Bench `dig +noall +answer TXT <domain>` captures the TTL (the only first-round path to one; skipped when `dig` absent). | **warning:** TTL > 86 400 s (a broken record stays broken a day+ after the fix); **info:** TTL < 300 s (migration-grade churn — fine short-term, wasteful as steady state); **ok** otherwise, quoting the seconds. | Set the SPF TXT record's TTL to ~3600 s (1 hour) at the DNS console once the record is stable. |
| `spf.cross_resolver` | The **local resolver and a public resolver (8.8.8.8) see the same `v=spf1` answer** — catches propagation lag after an edit, split-horizon views, and stale caches that make "works for me" lie. | Compare the local `node:dns` `v=spf1` set against bench `dig @8.8.8.8 +short TXT <domain>` (skipped when `dig` absent). | **warning:** the two views disagree — one resolver sees a record (or version of it) the other does not; **ok:** identical `v=spf1` sets. | Wait out the record's TTL after an edit and re-run; if it persists, check for split-horizon DNS or authoritative servers serving different zone versions. |
| `spf.cross_check` | Our parser agrees with the **`checkdmarc` conformance oracle** (record string, validity verdict, recursive lookup count, void count). | Compare `checkdmarc <domain> -f json` fields against our own observation (skipped when `checkdmarc` absent). | **info:** any disagreement ("ours n, checkdmarc m") — surfaces parser bugs on either side instead of hiding them; agreement emits the **ok** row. | Nothing to change in DNS — a mismatch is a bug in one of the two parsers; the tool-run panel holds both answers verbatim. |
| `spf.exists_usage` | `exists:` is only used in its sane form — **with a macro**. A macro-less `exists:` matches whenever the fixed name resolves, i.e. for **every** connecting IP. | Inspect each `exists` term for `%{…}` (row emitted only when `exists:` is present). | **warning:** macro-less `exists:` (an accidental pass-everything, one DNS entry away from `+all`); **info:** macro `exists:` (legal; counted but not statically expandable — verify with a live-message evaluation). | Replace a macro-less `exists:` with the intended `ip4:`/`include:`; keep macro `exists:` only if you operate a purpose-built DNS zone for it. |
| `spf.include_qualifier` | Lookup mechanisms carry the default `+` qualifier. **`-include:` is a semantic trap** — `include` only ever *matches* on the target's `pass`, so `-include:` means "hard-fail exactly the IPs your vendor authorizes". | Read each `include`/`a`/`mx`/`exists` term's qualifier (row emitted only when a non-`+` qualifier is present). | **warning:** `-include:`; **info:** `~`/`?` on any lookup mechanism (legal, rarely what was meant). | Drop the qualifier (plain `include:`) unless you truly mean to fail the target's authorized ranges. |
| `spf.ipv6` | **IPv6 completeness**: infrastructure that receives on IPv6 usually sends on it too (Gmail prefers AAAA), so a v4-only pass-set silently fails the v6 path. | Count IPv6 entries in the expanded pass-set; probe the MX exchanges for AAAA (audit-side lookups, never SPF-budget-counted). | **warning:** zero IPv6 coverage while the domain's MX hosts publish AAAA; **ok:** v6 authorized, or no v6 mail signal observed. | Add the sending hosts' `ip6:` ranges (or confirm the vendor `include:` already carries them — the include tree shows it). |
| `spf.mail_posture` | SPF matches the domain's **declared mail role** (RFC 7505): a non-sending domain should publish exactly `v=spf1 -all` + Null MX + DMARC `p=reject`; a Null-MX domain with an authorizing record is send-only (legit — confirm intent). | Read the MX RRset (memoized, shared with the MX check) next to the parsed record (row emitted when either signal is present). | **info:** record is exactly `v=spf1 -all` but Null MX is missing (non-sending trio incomplete); **info:** Null MX + an authorizing record (send-only posture — confirm it is deliberate); locked-down trio emits the **ok** row. | Non-sending: publish `v=spf1 -all`, Null MX `0 .`, DMARC `p=reject`. Send-only: keep the record, document the posture. |
| `spf.alignment_prep` | The DNS-visible half of DMARC/SPF **alignment posture**: the `_dmarc` record's `aspf=` tag. With `aspf=s` (strict), ESP return-path subdomains (`em1234.<domain>`) contribute **nothing** to DMARC even when raw SPF passes. | `resolveTxt(_dmarc.<domain>)`; read the `aspf=` tag (relaxed is the RFC 7489 default). | **warning:** `aspf=s` — observed live on act3ai.com: the SendGrid stream passed raw SPF at `em2598.act3ai.com` yet every Google/Microsoft rua report scored `policy_evaluated` SPF=fail, leaving DKIM to carry DMARC alone; **info:** no DMARC record yet (alignment not in play); **ok:** relaxed (`aspf=r` or tag absent). | Use relaxed alignment: delete `aspf=s` (or set `aspf=r`) in the `_dmarc` record — keep strict only under a hard requirement, and then make the envelope-from domain exactly match `From:`. |
| `spf.ip_coverage` | Every configured sending IP (`ctx.sendingIps`, managed on the Domains page) is inside the expanded pass-set. | CIDR-match each IP against collected `ip4`/`ip6`/`a`/`mx` results. | **critical:** a known sending IP is uncovered — mail from it fails SPF. | Add the exact missing `ip4:<ip>` / `ip6:<ip>` (or the vendor's `include:`) before the `all` term. |
| `spf.lookup_failed` | DNS answered at all (SERVFAIL/timeout is transient, not "no record"). | Distinguish NXDOMAIN/NODATA from resolver errors. | **warning:** could not resolve — re-run. | Retry; verify the domain's authoritative nameservers. |
| `spf.alignment` *(future)* | The SPF-authenticated envelope-from domain **aligns** with the visible `From:` domain per message — the message-level half of what `spf.alignment_prep` already measures in DNS. | Needs message-level data (`mailauth` on a live message, or DMARC rua ingestion — [../emails.mdx](../emails.mdx)). | **warning:** SPF passes for the ESP's bounce domain but is unaligned → contributes nothing to DMARC. | Configure the ESP's custom return-path/bounce subdomain (e.g. `bounce.<domain>` CNAME) so envelope-from aligns. |
| `spf.flatten` *(future automation)* | Flags records that would benefit from flattening; the automated rewrite (includes → pinned IPs, kept in sync) is a future maintenance job. | Heuristic on lookup count + include depth. | **info:** fragile / near the ceiling. | Prefer trimming; if flattening, it must be **dynamically maintained** — static flattened lists break silently when providers rotate ranges. |

### How a test run executes the tools

First-round detection is in-process `node:dns/promises` — those queries land in the structured
observation (`record:`, `include_tree`, `pass_set`) and are **not** tool runs. Whenever a run
shells out to an external binary, the invocation is captured verbatim as one entry in
**`spf.tool_runs[]`** inside the per-run YAML file (§5, [../storage.mdx](../storage.mdx)):
`{tool, command, started_at, duration_ms, exit_code, output_format, parsed, error}`. Rules that
apply to every invocation:

- **The exact argv string is stored in `command`** — every input argument (domain, IP, resolver)
  fully substituted, never templated, so the user can copy-paste the line into a terminal and
  reproduce the answer byte-for-byte. This is the same string the drill-down pages' "Diagnose it
  yourself" blocks display.
- **JSON output is requested wherever the tool offers it** (`output_format: json`); stdout is
  `JSON.parse`d into `parsed`. Text-only tools store trimmed stdout lines as-is
  (`output_format: text`).
- Each child process is started with `spawn`/`execFile` and an **argv array** (never a shell
  string — no injection surface from domain names), runs under a **hard timeout** (SIGTERM, then a
  grace SIGKILL), stdout/stderr captured separately. A non-zero exit, kill-by-timeout, or
  unparseable stdout sets `error` (a string; `null` on success) and downgrades the dependent
  sub-test to **warning**/skip ("tool unavailable — re-run"), never a false critical.
- A **missing binary is not an error**: the entry is still written — `exit_code: null` and
  `error: "ENOENT — not installed (brew install <formula>)"` — so the UI can offer the install
  hint, and the dependent sub-test degrades as documented (e.g. `spf.dns_type`, `spf.ttl` and
  `spf.cross_resolver` are skipped when `dig` is absent; `spf.cross_check` when `checkdmarc` is).
  This matches the DKIM bench convention ([./dkim.mdx](./dkim.mdx) §3).
- Binaries are resolved to **absolute paths** via the shared ToolLocator (Stage-0 `ctx.tools`
  when provided, else the same PATH + conventional-locations search) — scheduled launchd runs do
  not carry Homebrew's PATH.

The invocations, in run order:

| # | When | Tool | Exact command line | Output | Timeout | Parsed into |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | Every run (when `dig` present) | **dig** | `dig +time=2 +tries=1 +short <domain> SPF` | text | 5 000 ms | Each non-empty stdout line is a stale **type-99** answer → `parsed: {answers:[<lines>]}`; any line fires the `spf.dns_type` info finding with the lines as evidence. Empty stdout → empty answers, test passes. |
| 2 | Every run (when `dig` present) | **dig** | `dig +time=3 +tries=1 +noall +answer TXT <domain>` | text | 5 000 ms | The full-answer form is the only way to see the **TTL** that `+short` hides: the SPF row's TTL lands in `record.ttl` and drives `spf.ttl` (warn > 86 400 s, info < 300 s). |
| 3 | Every run (when `dig` present) | **dig** | `dig @8.8.8.8 +time=2 +tries=1 +short TXT <domain>` | text | 5 000 ms | The **public-resolver view**: stdout's `v=spf1` strings are compared with the local answer → `record.cross_resolver` (`agrees`, `public_records[]`) and the `spf.cross_resolver` row. Doubles as the missing/temperror disambiguation when doggo (row 5) is absent. |
| 4 | Every run (when `checkdmarc` present) | **checkdmarc** | `checkdmarc <domain> -f json` | json | 30 000 ms | The conformance cross-check: `parsed.spf.record`, `.valid`, `.dns_lookups`, `.dns_void_lookups`, `.warnings[]`, `.error` are compared against our own `raw_record`, `lookup_count`, `void_count`, `eval_result`. Agreement emits the ok `spf.cross_check` row; any disagreement emits the `info` finding ("cross-check mismatch — ours n, checkdmarc m") so parser bugs on either side surface instead of hiding. |
| 5 | Only when the apex TXT answer was missing or `temperror` | **doggo** | `doggo <domain> TXT @8.8.8.8 --json` | json | 5 000 ms | `parsed.responses[0].answers[]` from a public resolver disambiguates: an answer containing `v=spf1` here but not locally → `spf.lookup_failed` **warning** (local-resolver problem, "re-run"); no answer anywhere → `spf.missing` stands. The tool_run is the evidence either way. |
| 6 | *Future* — per configured sending IP | **mailauth** | `npx mailauth spf --sender postmaster@<domain> --ip <ip> --helo <domain>` | json | 15 000 ms | `parsed.status.result` (`pass`/`fail`/`softfail`/`neutral`/`permerror`/`temperror`) is the live `check_host()` verdict cross-validating each `ip_coverage[]` row. Preferred form is the **in-process** `spf({sender, ip, helo})` call (same Node process, no child); it is still logged as a `tool_runs` entry with `command` set to the equivalent CLI line above so the user can reproduce it. |

Everything else in the §3 tables — `kdig`, `drill`, `q`, `dnsx`, `dnsviz`, `swaks`, `spf-parse`,
`pyspf`, `spf-tools` — is **debug bench only**: surfaced as copyable commands on the drill-down
pages and used in our own test suites, but never executed by an audit run (swaks in particular
sends real mail and must stay user-initiated).

Where the captured output lands: every entry is appended to `spf.tool_runs[]` in the run file in
execution order; row 1 feeds the `spf.dns_type` test row, row 2 feeds `record.ttl` and the
`spf.ttl` row, row 3 feeds `record.cross_resolver` and the `spf.cross_resolver` row, row 4 feeds
the `spf.cross_check` row and hardens `lookup_count`/`void_count`/`eval_result`, row 5 feeds the
`spf.lookup_failed`-vs-`spf.missing` decision, and row 6 (future) feeds `ip_coverage[].covered`
confirmation. The UI's tool-run panel (§6.2) renders the list verbatim.

## 4. How it is measured

**First-round detection (pure `node:dns/promises`; `dig` only for the optional type-99 probe).**

1. **Fetch.** `resolveTxt(domain)` returns each record's 255-byte string chunks; join them without
   a separator, then filter to strings starting `v=spf1` (case-insensitive; `v=spf10` does not
   match). Zero → `spf.missing` critical. Two or more → `spf.multiple` critical, and evaluation
   stops (a multi-record domain has no usable SPF).
2. **Tokenize + parse.** Split on whitespace. Each term is either a **modifier** (`name=value`;
   only `redirect`/`exp` are meaningful, unknown modifiers are legally ignored) or a **mechanism**
   (optional qualifier `+ - ~ ?` defaulting to `+`, a name from the known set, an optional
   `:value` and `/cidr`). CIDR prefixes range-checked (ip4 0–32, ip6 0–128); `all` takes no value;
   `include`/`exists` require a domain. Any violation → `spf.syntax` critical. Every `%` in a
   domain-spec must begin `%%`, `%_`, `%-`, or a valid `%{…}` macro → else `spf.macro` critical.
3. **Simulate + count (the core algorithm).** A recursive walker expands the mechanism graph,
   maintaining `lookup_count`, `void_count`, a `visited` set (cycle guard), and the accumulated
   **pass-set** of concrete CIDRs:
   - `include:X` / `redirect=X` → **+1 lookup**; `resolveTxt(X)`; the target must hold exactly one
     valid `v=spf1` (else `spf.include_resolves` critical); recurse into it.
   - `a[:X]` → **+1 lookup**; resolve A+AAAA of X (default: current domain); IPs join the pass-set
     (with the term's dual-CIDR suffix applied when present).
   - `mx[:X]` → **+1 lookup**; resolve MX, then each exchange's A records (capped at 10 hosts, not
     budget-counted) into the pass-set.
   - `exists:X` → **+1 lookup**; `resolve4(X)`. `ptr` → **+1 lookup**, deprecated, not evaluated.
   - `ip4:`/`ip6:` join the pass-set at **no** lookup cost; `all`/`exp=` cost nothing.
   - An NXDOMAIN/empty answer increments `void_count` and marks the tree node `is_void`.
   - Macro-bearing targets (`%{i}` etc.) are **counted but not expanded** (they need a live
     message); noted on the tree node.
   - Stop descending once the budget is blown (the overrun count is still reported); a `visited`
     hit → `spf.recursion_depth` critical without infinite recursion.
4. **Severity mapping.** `lookup_count > 10` → critical; 8–10 → warning; else an `ok` finding with
   the count ("4/10"). `void_count > 2` → critical; 1–2 → warning. Terminal `+all` → critical;
   `?all`/absent → warning; `-all`/`~all` → ok. Each uncovered `ctx.sendingIps` entry → critical
   quoting the IP and the exact `ip4:`/`ip6:` line to add.
5. **Posture reads (cheap, memoized).** Alongside the expansion, four side observations complete
   the picture — none of them SPF-budget-counted:
   - **TTL** — bench dig `+noall +answer TXT` (§3 row 2) captures the apex TXT RRset TTL into
     `record.ttl` (`null` when `dig` is absent) → `spf.ttl`.
   - **Cross-resolver** — bench dig `@8.8.8.8 +short TXT` (§3 row 3) yields the public view; the
     `v=spf1` string sets are compared verbatim → `record.cross_resolver`, `spf.cross_resolver`.
   - **Mail posture** — `resolveMx(domain)` (memoized; the MX check resolves it anyway) detects
     Null MX (`0 .`, RFC 7505) and whether MX exists at all; combined with "is the record exactly
     `v=spf1 -all`" → `record.mail_posture`, `spf.mail_posture`. The MX exchanges' AAAA records
     (first 3 exchanges probed) also feed `record.ipv6.mx_has_aaaa` → `spf.ipv6`, judged against
     the count of IPv6 entries in the expanded pass-set.
   - **Alignment prep** — `resolveTxt(_dmarc.<domain>)` (memoized; the DMARC check resolves it
     anyway); the `aspf=` tag (default `r`) lands in `record.alignment` → `spf.alignment_prep`.
6. **eval_result.** `permerror` when any of: syntax/macro errors, multiple records, lookups > 10,
   voids > 2, a cycle, or a critical include failure. `temperror` on apex resolver failure.
   `none` when missing. Otherwise `valid`.

**Edge cases.** Case-insensitive mechanism names (flag nothing; receivers accept them); `+`
vs implicit qualifier equivalence; multi-string TXT concatenation without inserted spaces; CNAME
chains under the apex name (the resolver follows them — still one lookup); a bare `v=spf1` with no
terms (neutral for everything → the no-`all` warning); dual A/AAAA coverage; IPv4-mapped IPv6
(`::ffff:1.2.3.4`) in coverage matching; trailing dots tolerated.

**Caching / rate limits.** Pure DNS; worst case ~10–30 queries bounded by the lookup rule + MX
host cap. The per-run `visited` memo prevents re-querying the same include twice. Transient
resolver failures are reported as `warning` ("re-run"), never a false critical.

## 5. Test results (YAML)

The information we extract from the tests. Every run persists as **one YAML file** in the runs
directory (directory layout, filename grammar, and the `run:` block are owned by
[../storage.mdx](../storage.mdx) — referenced here, not redefined):

```
~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml
```

e.g. `~/.email_delivery_hero/runs/example.com/2026_07_01_09_30AM.yaml` — `<domain>` is the domain
name lowercased; the filename is the run's **start** time in the local timezone, 12-hour clock,
zero-padded, with an AM/PM suffix. The file opens with a `run:` block (`run_id`, `domain`,
`started_at`, `finished_at` ISO-UTC, `score`, `status`) followed by **one top-level key per
category**: `spf`, `dkim`, `dmarc`, `blacklists`, `dns_infra`, `spam_content`.

**This spec owns the top-level `spf` key.** It holds the worst-severity `status`, the structured
observation produced by the checker (the `record:` block below — also exposed in-process as
`results.spf`, see §10), the **`tool_runs[]`** capture of every external tool invocation (§3
"How a test run executes the tools"), the per-test rows, and the matched problem states. This is
the canonical shape every SPF UI surface and the regression differ consume:


Field-name mapping: `record.*` keys are 1:1 with the `spf_check_results` columns in §10
(`raw_record`, `mechanisms`, `lookup_count`, `void_count`, `all_qualifier`, `has_redirect`,
`byte_length`, `ttl`, `eval_result`), with the tree under `include_tree` (maps to
`spf_include_tree`), per-IP rows under `ip_coverage` (maps to `spf_ip_coverage`), and the four
posture blocks (`cross_resolver`, `ipv6`, `mail_posture`, `alignment`) as nested objects (JSONB
column `posture` in a future DB). `tests[]` rows are the `findings` projected into on-disk
vocabulary (`result: pass|fail|warn|info` ⇔ severity `ok|critical|warning|info`; `fix` ⇔
`remediation`), and `status`/`problem_states` are the worst severity and the §9 mapping — all
three are **computed by the checker at run time from its own findings** and written with the
payload; the canonical findings stay in the `run:` block, so these keys are read-only projections
for the UI and the differ never diffs them. **`tool_runs[]` lives only in the run file** — it is
run forensics (what exactly was executed, when, and what came back), not queryable state, so it
has no §10 table; the §3 execution rules define every field.

## 6. User interface — the summary surface and the full SPF page

Two surfaces, following the app-wide pattern in [../ui.mdx](../ui.mdx): a **summary cell** on the
main view, and a **full page dedicated to SPF** — which is a **"view one category run" page**: it
shows the SPF test results **of one specific run** (a run = one execution of all checks for one
domain, per the Run → Test → Sub-test vocabulary in [../dashboard.mdx](../dashboard.mdx) §1).

**Routes (LOCKED).**

| Page | Route |
| --- | --- |
| SPF category run page (one run's SPF results) | `/domains/:domainId/runs/:runId/spf` |
| Newest-run alias (always the domain's latest run) | `/domains/:domainId/spf` |
| Problem-state drill-down | `/domains/:domainId/runs/:runId/spf/:problemId` (newest alias `/domains/:domainId/spf/:problemId`) |

`:conceptId` comes from the §6.4 concept catalog (`lookup-limit`, `mechanism-include`, …).

### 6.1 The summary surface (main view)

SPF is one of the six category columns on the Dashboard. The SPF cell for a domain is the **worst
severity** among all `spf.*` findings in the latest run, and shows the single most important fact
at a glance:

- **Color** — dark green (valid & within budget), amber (worst = warning, e.g. 9/10 lookups or
  `?all`), dark red (any critical — `permerror`, missing record, uncovered sending IP), light gray
  (never run). `info`-only findings keep the cell green.
- **Label** — the one-line summary of the most decision-relevant metric: the **evaluation state
  and budget**. `Valid · 4/10` / `permerror · 11/10` / `+all` / `No record` / `K of M fail`.
- **Hover** — tooltip with the top open problem's title (e.g. "SPF permerror — 11/10 lookups").
- **Cell click → inside the test** (the one exception to whole-row navigation,
  [../dashboard.mdx](../dashboard.mdx) §6). Clicking the **SPF cell on a Domain-health row**
  opens the domain's **newest run's** SPF page (`/domains/:domainId/spf`, the newest-run alias).
  Clicking the **SPF cell on a Runs-table row** opens **that run's** SPF page
  (`/domains/:domainId/runs/:runId/spf`). Clicking anywhere else in either row still opens the
  whole run report as today.

### 6.2 The full SPF page — a "view one category run" page

`/domains/:domainId/runs/:runId/spf` — newest-run alias `/domains/:domainId/spf`.

One page, purely about SPF **for one specific run of one domain** — everything on it (record,
tree, table, tool runs) was observed during that run, at that run's `started_at`. It must answer,
top to bottom: *Did my SPF evaluate cleanly in this run? What exactly did we observe (the record,
the lookup budget, the expanded tree)? Which tests passed/failed? How do I fix each failure?*
Every failed test shows the relevant debugging values — the DNS name queried, the raw TXT answer,
the offending term, the IP address at issue. Data comes from the run's YAML file's `spf` key (§5).

**How the page is reached.**

1. The **SPF cell** on a Dashboard Domain-health row → the newest run's SPF page (§6.1).
2. The **SPF cell** on a Dashboard Runs-table row → that run's SPF page (§6.1).
3. The **SPF section chevron** on a run report (`/domains/:domainId/runs/:runId`) → that same
   run's SPF page.
4. The **‹ prev / next ›** controls in the run context strip (below) → the adjacent run's SPF
   page for the same domain.

Contents, in order (exact layout in §7):

   **"Newest" indicator** — a badge shown when this run is the domain's latest (always shown on

4. **Include tree panel** — the §5 `include_tree` rendered as an indented, expandable tree: each

5. **Test results table** — one row per §2 test: pass/fail/warn/info icon, test name, the observed

6. **Sending-IP coverage panel** — one row per configured sending IP: covered ✓/✗ and the pass-set

7. **Tool runs panel** — a collapsed-by-default accordion listing this run's `spf.tool_runs[]`

navigation affordance into the §6.4 explainer layer — the full click map is the table at the end
of §6.4. Hover keeps the quick tooltip; **click always navigates to a run-scoped page** (URL-
addressable and shareable), never to a modal that vanishes on refresh.

### 6.3 The run report's SPF section — how the cell grows

On the run report (`/domains/:domainId/runs/:runId`, `RunDetailPage`) the SPF category today
renders a chip plus a flat findings list. It grows to:

1. **Run-scoped chevron.** The SPF chip's chevron routes to **that run's** SPF page
   (`/domains/:domainId/runs/:runId/spf`) whenever the report shows a historical run — exactly the
   pattern DKIM/DMARC/DNS already follow in `RunDetailPage.tsx`; on the newest-run route the alias
   (`/domains/:domainId/spf`) is equivalent.
2. **Summary strip** at the top of the expanded SPF section, rendered from `results.spf`: the mini
   lookup gauge (`n/10`), the void count (`n/2`), the `all`-qualifier chip, and the `eval_result`
   chip — the same four facts as the SPF page's budget hero, at half size. Each element deep-links
   to its §6.4 concept page.
3. **Per-test status list.** One compact row per §2 test present in the run (icon + plain-language
   title + observed value), fail-first. **Each row's title links to the run-scoped test explainer**
   (`…/spf/tests/:testId`). Non-`ok` rows keep the inline Fix block and Copy-fix button exactly as
   today — the link adds depth, it removes nothing.
4. **"Open full SPF page ›"** link at the section's foot → the same run's SPF page.

### 6.5 "Run this check now" — the scoped re-run

The section-5 affordance on every explainer page, plus a secondary **"Re-run SPF only"** button in
the SPF page header (next to the existing Re-run, which starts a full six-category run).

**UX.** Click → button shows a spinner ("Running SPF…", disabled; the shared scan dock shows the
domain as running); on completion the app navigates to the **new run's** version of the current
page — same `testId`/`conceptId`/`problemId` path with the new `:runId` substituted — so the user
watches their fix take effect in place. On failure → toast with the error; the page stays on the
old run. Concurrent scoped runs of the same domain coalesce onto the in-flight run (the existing
per-domain in-flight guard in `audit.service.ts`).

**Granularity.** The smallest runnable unit is the **whole SPF category**: the §2 sub-tests share
one recursive DNS expansion, so a single test cannot be re-measured in isolation. Every test
page's section 5 says so explicitly ("re-runs all SPF tests — they share one DNS expansion").

**API.** `POST /api/audit/run/:domainId?checks=spf` — a small extension of the existing
`POST /api/audit/run/:domainId` ([audit.controller.ts](../../code/packages/backend/src/modules/audit/audit.controller.ts)
today runs all six categories; the surgical single-check merge already has a precedent in the
content-scoring re-score path). Semantics of `checks` (comma-separated category checkId prefixes):

- Only the named checkers execute; the run produces a **new run file** (§5 / [../storage.mdx](../storage.mdx)).
- The five sibling category keys are **carried forward verbatim** from the domain's previous run,
  each annotated `carried_forward: true` and keeping its original `checked_at` — the run file is
  always complete, and every consumer can tell measured from copied data.
- The `run:` block's `score`/`status` are recomputed over the merged findings, and the block gains
  `checks: [spf]` so the Runs table can badge the run **"SPF only"**.
- The regression differ ([../scheduled_checks.mdx](../scheduled_checks.mdx)) **skips
  carried-forward categories** — they were not re-measured, so they can neither regress nor
  improve.
- No `checks` param → today's behavior, unchanged. Unknown prefix → 400.

### 6.6 History & trends across runs

Because every run persists as one YAML file (§5), history is a read over the domain's run list —
`GET /api/audit/runs?domainId=…` already returns it newest-first; the frontend derives everything
below client-side in the first round. If run payloads grow heavy, a slim future endpoint
(`GET /api/audit/history/:domainId?category=spf&limit=20` returning per-run
`{runId, started_at, status, eval_result, lookup_count, void_count, tests:[{id,result}]}`)
replaces the client-side derivation without changing the UI.

- **Cross-run history band** on the SPF page, directly under the budget hero: one dot per run
  (newest right, last ~10), colored by that run's SPF status, over a **lookup-count sparkline**
  (the creep toward 10 is the single most valuable trend — §11). Hover → run timestamp,
  `eval_result`, `n/10`, `all_qualifier`. **Click a dot → that run's SPF page** (swaps `:runId`).
  Runs where SPF was carried forward (§6.5) render as hollow dots and are skipped by the
  sparkline. Notable transitions get a tick mark: `all` weakened, a record appearing/vanishing,
  lookup count crossing 8 or 10.
- **Per-test history strip** on each `tests/:testId` page (below section 5): the last ~10 runs'
  pass/warn/fail dots **for that one test id**; hover → the run timestamp + that run's observed
  value; click → the same test page under that run's `:runId`. This is where "did my fix work?"
  is answered in one glance.
- **States.** Loading → a skeleton band. One run only → "history appears after your next run"
  (muted, with a link to [../scheduled_checks.mdx](../scheduled_checks.mdx)-driven scheduling in
  Settings). Fetch error → muted "history unavailable" line — history never blocks the main page.

Placement rules and rationale:

- **Highest: the verdict.** The header row carries identity + status; the **budget hero** directly
  under it is the one visual a non-expert needs — am I `valid` or `permerror`, how close to the
  10-lookup cliff, and what's my next move. One accent CTA maximum on the page.
- **Run context strip** — directly under the identity row, before any results. It makes it
  unmistakable that everything below belongs to **one run**: the run's `started_at` in local time
  (matching the run-file name, §5), **‹ prev / next ›** stepping chronologically through this
  domain's runs (each swaps `:runId` in the URL, refetches, and scrolls to top; disabled with a
  muted tooltip at either end of the history), and the **NEWEST** badge on the latest run — on an
  older run it becomes a **"Jump to newest ›"** link. Muted styling (`text-slate-500`, small);
  it orients, it never competes with the verdict.
- **Second band, two-up:** left = **Published record** (the ground truth users paste elsewhere;
  monospace, copy button top-right, mechanism table beneath with lookup costs); right = **Include
  tree** (the most-diagnostic SPF visual — the void/over-budget node is visible at a glance rather
  than buried in prose). On narrow widths the tree stacks below the record.
- **Main table: test results.** A table, not cards — users scan many rows; sort order is fixed
  fail → warn → info → pass (never alphabetical). Each row: status icon (✗ red / ⚠ amber / ✓ green
  / ⓘ slate), finding id in small caps, plain-language title, right-aligned chevron `[v]`
  accordion. Expanded row shows *observed* (the exact term / raw record, monospace), *why it
  matters* (one sentence), and the **Fix** strip: the exact term/record edit with `[copy fix]`.
  Copy buttons always copy the publishable value, never prose.
- **Coverage band** below the table: per-IP verdicts are operational data the user acts on
  (add-this-line), distinct from record hygiene.
- **Tool-runs band** — collapsed accordion under the coverage band: one row per `tool_runs[]`
  entry showing the tool, the exact command line (monospace, `[copy]`), duration and exit code;
  expanding a row reveals the `parsed` output as a code block (and `error` when set, in red).
  Collapsed by default because it is forensics, not triage — but always present, because "what
  exactly did you run?" is the first question a skeptical operator asks.
- **Lowest: problem-state cards** — education and deep dives; two per row, title + one-line hook +
  chevron to `/domains/:domainId/runs/:runId/spf/:problemId` (newest alias
  `/domains/:domainId/spf/:problemId`). They repeat information the table already flagged on
  purpose: the table is for operators, the cards for learners.
- **Type & color:** shadcn/ui primitives (`Card`, `Table`, `Accordion`, `Badge`), lucide icons

- **Drill-down pages** (`/domains/:domainId/runs/:runId/spf/:problemId`, newest alias

## 8. Recommendations — how we tell the user to fix SPF

Every failing test carries its concrete one-line remediation (§2). Above the per-test fixes, the
page computes one **primary recommendation** — the hero CTA — from this precedence ladder (first
match wins):

| Current state | Primary recommendation |
| --- | --- |
| No record (PS-01) | Publish `v=spf1 include:<your-ESP> ~all` today. Build the include list from your known senders, not guesses. |
| Multiple records (PS-02) | Merge into one record first — until then receivers ignore SPF entirely. |
| Syntax/macro errors (PS-05) | Fix the quoted terms; a `permerror` record protects nothing. |
| Over the lookup/void limits (PS-03/04) | Cut lookups: delete dead/unused includes first (free wins), then replace `a`/`mx` with explicit `ip4:`/`ip6:`, then consider **dynamically maintained** flattening — never a static IP dump. |
| `+all` or `/0` (PS-06) | Remove it immediately — the domain is open to the entire internet. |
| Uncovered sending IP (PS-08) | Add the exact `ip4:`/`ip6:` line (copy button holds it) before the `all` term. |
| `?all` / no `all` | Append `~all`. |
| Healthy `~all`, DMARC enforcing | Keep `~all` — with DMARC at quarantine/reject, softfail lets forwarded mail still be evaluated by DKIM/DMARC instead of being rejected at SMTP time. |
| Healthy `~all`, **no** DMARC | Consider `-all` (stronger without DMARC), or better: deploy DMARC ([./dmarc.mdx](./dmarc.mdx)). |
| Parked / non-sending domain | Publish exactly `v=spf1 -all`, a Null MX (`0 .`, RFC 7505), and DMARC `p=reject`. |

Additional standing recommendations surfaced as `info` when applicable: keep the record under ~450
bytes; subdomains that send mail need their own records (no inheritance); recognized ESP includes
(Google `_spf.google.com` — flattened by Google to direct `ip4`/`ip6` blocks, 1 lookup as verified
2026-07, Microsoft `spf.protection.outlook.com` ≈2–3, SendGrid `sendgrid.net`, Mailchimp
`servers.mcsv.net`, Amazon SES `amazonses.com`) each have a live lookup cost — shown next to the
include in the tree, never assumed from stale docs; prefer aligned **DKIM** on every stream (it
survives forwarding; SPF does not).

## 9. Problem states

The health situations a domain can be in for SPF. Each maps from finding ids (detection is
automatic), and each has a deep-dive below — and a drill-down UI page (§7). PS-00 is the healthy
goal state; the rest are unhealthy in descending typical severity.

| Id | Problem state | Detected by | Severity |
| --- | --- | --- | --- |
| PS-00 | Valid and within budget | all tests pass, `eval_result=valid` | ok |
| PS-01 | No SPF record at all | `spf.missing` | critical |
| PS-02 | Multiple SPF records | `spf.multiple` | critical |
| PS-03 | Over the lookup limit (permerror) | `spf.lookups` | critical |
| PS-04 | Dead includes / void lookups | `spf.void`, `spf.include_resolves` | critical |
| PS-05 | Syntax or macro errors (permerror) | `spf.syntax`, `spf.macro` | critical |
| PS-06 | Open door (`+all`, `?all`, missing `all`, `/0` range) | `spf.all`, `spf.cidr_scope` | critical |
| PS-07 | Dead weight & deprecated constructs (`ptr`, terms after `all`, duplicates, ignored `redirect`) | `spf.ptr`, `spf.all_terminal`, `spf.dup_mechanisms`, `spf.redirect` | warning |
| PS-08 | Sending IP not covered | `spf.ip_coverage` | critical |
| PS-09 | Include/redirect loop | `spf.recursion_depth` | critical |
| PS-10 | Oversized record / stale type-99 | `spf.length`, `spf.dns_type` | warning |
| PS-11 | Unaligned SPF (ESP bounce domain) *(future: message data)* | `spf.alignment` | warning |
| PS-12 | Fragile flattened record *(future advisory)* | `spf.flatten` | info |

### PS-00 — Valid and within budget

The goal state: one record, clean grammar, comfortably under 10 lookups, zero voids, `~all` (with
DMARC) or `-all`, every sending IP covered.
- **Data fields:** `eval_result=valid`, `lookup_count`, `void_count`, `all_qualifier`,
  `ip_coverage[].covered` all true, `tests[]` all pass.
- **Diagnose/debug:** nothing to fix — `checkdmarc <domain> -f json` should agree (no SPF errors
  or warnings).
- **Tools:** `dig +short TXT <domain>` for a sanity glance.
- **More health metrics:** the scheduled-run regression diff (lookup count creeping toward 10, a
  new include appearing, `-all` weakened); future `spf.alignment` once message data exists.
- **UI:** green budget gauge, all-green table; drill-down explains *keeping* it healthy — every
  new vendor tool nudges the lookup count, so re-audit after onboarding anything that sends mail.

### PS-01 — No SPF record at all

- **Concept:** receivers get result `none`: they cannot verify any sender, your mail rides on
  weaker signals, and the Gmail/Yahoo/Microsoft bulk-sender rules (SPF or DKIM mandatory) penalize
  you. SPF can also never contribute to DMARC.
- **Data fields:** `record_found=false`, `record_count=0`, `query_name`, `eval_result=none`.
- **Diagnose/debug:** `dig +short TXT <domain>` (no `v=spf1` line) → confirm against a public
  resolver `kdig @8.8.8.8 +short TXT <domain>` → check whether a subdomain you actually send from
  is the real envelope-from (subdomains need their own records).
- **Tools:** dig/kdig/doggo; `checkdmarc <domain>` reports the missing record.
- **More health metrics:** none until the record exists.
- **UI:** budget hero at `none`; single critical row with the copyable starter record
  `v=spf1 include:<your-ESP> ~all`; drill-down explains building the include list from known
  senders.

### PS-02 — Multiple SPF records

- **Concept:** two or more `v=spf1` strings → `permerror` (§4.5) — receivers ignore ALL of them.
  Classic after a provider migration: the registrar wizard's record lingers next to the ESP's.
- **Data fields:** `record_count ≥ 2`, `raw_record` (all strings joined with ` | `),
  `eval_result=permerror`.
- **Diagnose/debug:** `dig TXT <domain> +multiline` and count the `v=spf1` answers; find which DNS
  console entry created each.
- **Tools:** dig; `doggo --json` makes the answer count explicit for scripts.
- **More health metrics:** regression diff — flag if the count ever rises above 1 again.
- **UI:** critical row listing every observed string; fix = a single merged record (the copy
  button holds the union of all mechanisms with one `v=spf1` and one `all`).

### PS-03 — Over the lookup limit (permerror)

- **Concept:** more than 10 DNS-querying mechanisms across the whole recursive expansion →
  `permerror` for **every** message, even though each individual include looks fine. The single
  most common enterprise SPF failure; it appears silently when one more vendor is added or when a
  vendor's own include grows.
- **Data fields:** `lookup_count`, `include_tree` (per-node `cost_lookups`), `mechanisms`.
- **Diagnose/debug:** read the include tree on the SPF page — it shows exactly which branch costs
  what; cross-check with `checkdmarc <domain> -f json` (it counts recursively too).
- **Tools:** checkdmarc; dmarcian SPF Surveyor for a second visual; `dig +short TXT <include>` to
  walk a branch manually.
- **More health metrics:** lookup-count trend across scheduled runs — creeping toward 10 is the
  early warning (we flag at 8).
- **UI:** red gauge past 10; the tree highlights the heaviest branches; fix ladder = delete dead
  includes → replace `a`/`mx` with `ip4:`/`ip6:` → split streams onto subdomains → dynamic
  flattening (last resort).

### PS-04 — Dead includes / void lookups

- **Concept:** an `include:`/`a`/`mx` pointing at a name with no records is a **void lookup**;
  more than 2 → `permerror`. An `include:` whose target publishes **no SPF at all** is an
  immediate `permerror`. Usually a decommissioned vendor — and a repurposed vendor domain is also
  a takeover-style risk.
- **Data fields:** `void_count`, `include_tree` nodes with `is_void=true`, `spf.include_resolves`
  findings.
- **Diagnose/debug:** the tree marks each void node; confirm with
  `dig +short TXT <include-target>` (empty answer).
- **Tools:** dig/doggo; `checkdmarc` flags dead includes.
- **More health metrics:** re-probed every scheduled run; a vendor include going dark between runs
  is a regression finding.
- **UI:** the void badge in the tree; fix = delete the exact term (copy button holds the record
  minus that term).

### PS-05 — Syntax or macro errors (permerror)

- **Concept:** an unknown mechanism (`inlcude:`), a bad CIDR (`/33`), a value on `all`, comma
  separators, stray `%` or an invalid `%{x}` macro — any single violation makes receivers discard
  the whole record; the domain silently has no SPF while the owner believes it does.
- **Data fields:** `raw_record`, `mechanisms` (what survived parsing), the `spf.syntax`/`spf.macro`
  findings quoting each offending term.
- **Diagnose/debug:** read `raw_record` character-by-character in the monospace panel (smart
  quotes and doubled spaces are visible there); `checkdmarc` names the offending term; npm
  `spf-parse` gives per-term validity as a second opinion.
- **Tools:** checkdmarc; spf-parse (npm); EasyDMARC's raw checker to validate a corrected record
  *before* publishing.
- **More health metrics:** regression diff on `raw_record` — any change to the string is surfaced.
- **UI:** the mechanism table renders unknown/invalid terms with a red badge in place; fix block
  shows a corrected full record assembled from the valid parts.

### PS-06 — Open door (`+all`, `?all`, missing `all`, `/0` range)

- **Concept:** `+all` (or a bare trailing `all`, or `ip4:0.0.0.0/0`) authorizes the entire
  internet — worse than useless, since some receivers score it as a spam signal. `?all` or no
  `all` gives receivers no default policy: functionally close to having no SPF.
- **Data fields:** `all_qualifier`, `mechanisms` (the CIDR terms), `pass_set`.
- **Diagnose/debug:** read the last term of `raw_record`; check every `ip4:`/`ip6:` prefix.
- **Tools:** dig; any §3 checker flags `+all` instantly.
- **More health metrics:** regression diff — `-all`/`~all` weakening to `?all`/`+all` between runs
  is a high-value regression.
- **UI:** critical row; fix = the same record with the qualifier corrected (copy button); the
  drill-down explains the `~all` vs `-all` decision (see §8 — with DMARC, `~all`; without, `-all`;
  non-sending domains, exactly `v=spf1 -all`).

### PS-07 — Dead weight & deprecated constructs

- **Concept:** `ptr` is deprecated (§5.5) — slow, unreliable, burns a lookup, and some receivers
  treat it as no-match. Terms after `all` never evaluate. Duplicate terms waste lookups. A
  `redirect=` next to an `all` is silently ignored (§6.1). None of these break SPF today; all of
  them mislead the next person who edits the record.
- **Data fields:** `mechanisms` (order + duplicates), `has_redirect`, `all_qualifier` position,
  the `spf.ptr`/`spf.all_terminal`/`spf.dup_mechanisms`/`spf.redirect` findings.
- **Diagnose/debug:** the mechanism table strikes through dead terms and badges `ptr`; nothing
  else needed.
- **Tools:** none required — this is pure record hygiene.
- **More health metrics:** lookup head-room reclaimed after cleanup (the gauge drops).
- **UI:** warning rows; fix = the cleaned record with `ptr`/duplicates/dead terms removed and
  `all` moved last (one copy button for the full corrected record).

### PS-08 — Sending IP not covered

- **Concept:** the record parses and evaluates fine, but a host you actually send from is not in
  the expanded pass-set — every message from it fails SPF (and, at DMARC enforcement, gets
  quarantined/rejected unless DKIM saves it). The classic "we added a new relay and forgot SPF".
- **Data fields:** `ip_coverage[]` (`ip`, `covered`, `matched_by`), `pass_set`, the domain's
  `sendingIps` config (Domains page).
- **Diagnose/debug:** the coverage panel shows the exact uncovered IP; verify the IP really is a
  legitimate sender (mail server config, ESP dashboard), then confirm live with
  `npx mailauth spf --sender a@<domain> --ip <ip>` (a true `check_host()` simulation).
- **Tools:** mailauth (npm); swaks to send a live probe and read `Authentication-Results` at a
  seed mailbox.
- **More health metrics:** coverage re-checked every scheduled run; an IP falling out of coverage
  (a vendor rotating ranges out from under a static list) is a regression finding.
- **UI:** critical row + the coverage panel row with the exact `ip4:<ip>`/`ip6:<ip>` line,
  copyable; drill-down explains envelope-from vs `From:` and why the IP must be in the pass-set.

### PS-09 — Include/redirect loop

- **Concept:** record A includes B which includes A (directly or via a longer chain) — evaluation
  never terminates, so receivers abort with `permerror`. Rare, but catastrophic and invisible to
  eyeball review across DNS zones.
- **Data fields:** `include_tree` (the node whose `resolved_to` says "cycle"), the
  `spf.recursion_depth` finding naming the looping domain.
- **Diagnose/debug:** walk the tree; confirm each hop with `dig +short TXT <target>` until the
  repeat appears.
- **Tools:** dig; checkdmarc (its recursive parse also detects loops).
- **More health metrics:** none — fix it.
- **UI:** critical row naming the loop edge; fix = restructure so the chain is a tree (usually:
  replace one back-reference with the concrete `ip4:` ranges).

### PS-10 — Oversized record / stale type-99

- **Concept:** past ~450 bytes the whole DNS answer risks classic-UDP truncation → TCP retries or
  `temperror` at strict resolvers. Separately, a leftover **SPF type-99** RR (deprecated since
  RFC 7208; registrars are dropping the type entirely) signals stale config — receivers only read
  TXT.
- **Data fields:** `byte_length`, the `spf.length`/`spf.dns_type` findings.
- **Diagnose/debug:** `dig +short TXT <domain> | wc -c`; `dig <domain> SPF +short` (any answer =
  the stale type-99).
- **Tools:** dig; drill for DNSSEC-aware size tracing.
- **More health metrics:** byte-length trend as vendors are added.
- **UI:** warning/info rows; fix = shorten (drop includes, explicit ranges, subdomain streams) and
  delete the type-99 RR at the DNS console.

### PS-11 — Unaligned SPF (ESP bounce domain) *(future — message data)*

- **Concept:** the ESP sends with its own Return-Path (`em1234.sendgrid.net`), which passes raw
  SPF for *the ESP's* domain — but under DMARC, SPF only counts when the envelope-from domain
  **aligns** with the visible `From:` domain (relaxed = same org-domain). Unaligned SPF
  contributes nothing; the stream lives or dies on DKIM alone.
- **Data fields:** future — envelope-from per stream from rua ingestion
  ([../emails.mdx](../emails.mdx)) or a live message via `mailauth`.
- **Diagnose/debug:** `npx mailauth report message.eml` on a message from each sending system and
  read the SPF **alignment** verdict (not just pass/fail); check the ESP dashboard for a "custom
  return-path / bounce domain" setting.
- **Tools:** mailauth; the ESP's own docs (SendGrid/Mailchimp both support custom bounce CNAMEs).
- **More health metrics:** share of DMARC passes carried by SPF vs DKIM per stream (future rua
  data).
- **UI:** advisory card (future); fix = configure the ESP's custom return-path subdomain (e.g.
  `bounce.<domain>` CNAME) so envelope-from aligns.

### PS-12 — Fragile flattened record *(future advisory)*

- **Concept:** replacing includes with a static dump of their current IPs "solves" the lookup
  limit and then **breaks silently** the day the provider rotates ranges. Flattening is only safe
  when dynamically re-synced.
- **Data fields:** heuristic — a long `ip4:`/`ip6:` list where a known vendor include used to be;
  `byte_length`.
- **Diagnose/debug:** compare the flattened ranges against the vendor's current include
  (`dig +short TXT _spf.google.com` and follow).
- **Tools:** spf-tools (GitHub) for reference flattening; checkdmarc to re-verify.
- **More health metrics:** drift between the flattened list and the live vendor include — the
  future `spf.flatten` maintenance job re-checks it on schedule.
- **UI:** info card; recommendation = prefer trimming/includes; if flattening, automate the
  re-sync (future feature).

## 10. Information schema (database schema)

Reference the shared core tables (owned by engineering, **not** redefined here): `domains`,
`sending_ips`, `audit_runs`, `findings`. Human-readable SPF findings land in the shared `findings`
table with `category='spf'` and `check_id` = the sub-check id. The structured, machine-parsed
observation for each run lives in the check-specific tables below.

```sql
-- One row per audit run per domain: the parsed SPF observation and budget accounting.
CREATE TABLE spf_check_results (
    id             BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    audit_run_id   BIGINT NOT NULL REFERENCES audit_runs(id) ON DELETE CASCADE,
    domain_id      BIGINT NOT NULL REFERENCES domains(id)    ON DELETE CASCADE,
    query_name     TEXT    NOT NULL,       -- the apex name queried
    record_found   BOOLEAN NOT NULL,
    record_count   INT     NOT NULL DEFAULT 0,  -- >1 => spf.multiple critical
    raw_record     TEXT,                    -- exact joined v=spf1 string (all strings when multiple)
    mechanisms     JSONB,                   -- apex terms [{qualifier,type,value,lookup,raw}]
    lookup_count   INT     NOT NULL DEFAULT 0,  -- recursive; >10 => permerror
    void_count     INT     NOT NULL DEFAULT 0,  -- NXDOMAIN/empty answers; >2 => permerror
    all_qualifier  TEXT,                    -- '-all' | '~all' | '?all' | '+all' | NULL
    has_redirect   BOOLEAN NOT NULL DEFAULT FALSE,
    byte_length    INT     NOT NULL DEFAULT 0,
    eval_result    TEXT    NOT NULL,        -- 'valid' | 'permerror' | 'temperror' | 'none'
    checked_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX uq_spf_results_run_domain ON spf_check_results(audit_run_id, domain_id);
CREATE INDEX ix_spf_results_domain            ON spf_check_results(domain_id);
CREATE INDEX ix_spf_results_eval              ON spf_check_results(eval_result);

-- The recursively expanded include/a/mx/exists/redirect graph, one row per resolved node.
CREATE TABLE spf_include_tree (
    id             BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    spf_result_id  BIGINT NOT NULL REFERENCES spf_check_results(id) ON DELETE CASCADE,
    parent_id      BIGINT REFERENCES spf_include_tree(id) ON DELETE CASCADE,  -- NULL = apex node
    depth          INT     NOT NULL,        -- 0 = apex
    term           TEXT    NOT NULL,        -- e.g. 'include:_spf.google.com'
    resolved_to    JSONB,                   -- child record / IPs / notes this node resolved to
    cost_lookups   INT     NOT NULL DEFAULT 0,
    is_void        BOOLEAN NOT NULL DEFAULT FALSE,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ix_spf_tree_result ON spf_include_tree(spf_result_id, depth);

-- One row per configured sending IP tested against the expanded pass-set.
CREATE TABLE spf_ip_coverage (
    id             BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    spf_result_id  BIGINT NOT NULL REFERENCES spf_check_results(id) ON DELETE CASCADE,
    ip             INET    NOT NULL,
    covered        BOOLEAN NOT NULL,
    matched_by     TEXT,                    -- the pass-set entry that covered it (NULL when uncovered)
    checked_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ix_spf_coverage_result ON spf_ip_coverage(spf_result_id);
```

**Mapping onto today's file store.** There is **no database in the first round**. Each run
persists as one YAML file — `~/.email_delivery_hero/runs/<domain>/<timestamp>.yaml`
([../storage.mdx](../storage.mdx)) — whose top-level `spf` key is the §5 document: the
human-readable findings are the `tests[]` rows (`checkId:"spf"` in-process), and the checker's
structured payload (exposed in-process as **`results.spf`**) is the `record:` block, mapping 1:1
to `spf_check_results`, with the tree nested under `include_tree` (maps to `spf_include_tree`
rows via parent pointers), the authorized CIDRs under `pass_set`, and the per-IP rows under
`ip_coverage[]` (maps to `spf_ip_coverage`). **`tool_runs[]` is deliberately file-only** — run
forensics with no DB table (§5). Because the store module fronts all of this, the file→DB move is
a single-module change.

## 11. When we run the check

- **On-demand.** Runs on every manual audit (`POST /api/audit/run/:domainId` and the run-all
  endpoint). SPF is cheap (worst case ~10–30 DNS queries, bounded by the 10-lookup rule + the MX
  host cap), so it always runs inline with the rest of the engine — no feature gate.
- **Scoped on-demand.** The §6.5 "Run this check now" affordance triggers
  `POST /api/audit/run/:domainId?checks=spf`: only the SPF checker executes, the other five
  category keys are carried forward from the previous run (`carried_forward: true`), and a new
  run file is written. Cheap enough to click after every DNS edit — that is its purpose.
- **Periodic.** The in-process scheduler re-audits each monitored domain on the configured
  cadence. Each run's `spf.*` findings are diffed against the previous run; any newly appearing or
  worsened finding is a **new problem** surfaced on the Dashboard. High-value regressions: the
  lookup count crossing 10 (or reaching 8), a second `v=spf1` appearing, `-all`/`~all` weakening,
  a vendor include going void, or a sending IP falling out of coverage.
- **First-round vs future.** All §2 sub-checks except `spf.alignment` and `spf.flatten` automation
  are first-round; everything is pure `node:dns/promises` except the optional `spf.dns_type`
  type-99 probe, which shells out to Brew `dig` and degrades gracefully when it is absent.
- **Cost / rate-limit / back-off.** No paid feeds or provider APIs. The per-run `visited` memo
  prevents duplicate include queries; transient resolver failures are reported as a **warning**
  ("could not resolve — re-run"), never a false critical.
- **Regression detection** applies exactly as in [../scheduled_checks.mdx](../scheduled_checks.mdx):
  compare the latest `results.spf` and `findings[]` to the prior run by finding id + severity.

## 13. Acceptance criteria

1. Resolving the apex correctly joins multi-string TXT chunks and identifies `v=spf1` records
   (case-insensitive; `v=spf10` does not match); absence yields `spf.missing` **critical** with a
   concrete publish-this remediation and `eval_result=none`.
2. Two or more `v=spf1` records yield `spf.multiple` **critical**, `eval_result=permerror`, and no
   false mechanism findings (the record set is treated as unusable).
3. Grammar violations (unknown mechanism, bad CIDR, value on `all`, missing required domain) each
   yield `spf.syntax` **critical** quoting the offending term; malformed `%`-macros yield
   `spf.macro` **critical**; a valid record with valid macros parses without error and emits an
   `ok` syntax finding.
4. The recursive expansion counts `include`/`a`/`mx`/`ptr`/`exists`/`redirect` (and **not**
   `ip4`/`ip6`/`all`/`exp`): > 10 yields `spf.lookups` **critical** with the exact count; 8–10
   yields **warning**; otherwise an **ok** finding with "n/10".
5. Void answers are tracked: > 2 yields `spf.void` **critical**; 1–2 yields **warning**; an
   `include:` whose target has no SPF yields `spf.include_resolves` **critical**; a transient
   include lookup failure yields **warning**, never critical.
6. Include/redirect cycles are detected (`spf.recursion_depth` **critical**) without infinite
   recursion; each target is queried at most once per run.
7. Terminal `+all` yields `spf.all` **critical**; `?all` or missing `all` (without `redirect`)
   yields **warning**; `-all`/`~all` yields **ok**. Terms after `all` yield `spf.all_terminal`
   **warning** listing the dead terms; `redirect=` alongside `all` yields `spf.redirect`
   **warning**.
8. `ptr` yields `spf.ptr` **warning** with a concrete replacement; duplicate terms yield
   `spf.dup_mechanisms` **warning**; `/0` ranges yield `spf.cidr_scope` **critical** and very
   broad ranges **warning**; records past ~450 bytes yield `spf.length` **warning**.
9. `spf.ip_coverage` marks **critical** any configured sending IP not covered by the expanded
   pass-set (ip4 and ip6, including CIDR-suffixed `a`/`mx` results), quoting the IP and the exact
   line to add; full coverage emits an **ok** finding.
10. Pass results are emitted as `ok` findings so the SPF page renders an explicit passed/failed
    row per test, and the category cell label reflects the eval state and lookup budget.
11. The structured payload is persisted under the **top-level `spf` key** of the per-run YAML file
    `~/.email_delivery_hero/runs/<domain>/<YYYY_MM_DD_hh_mm{AM|PM}>.yaml`
    ([../storage.mdx](../storage.mdx)) with all §5 `record:` fields (snake_case, matching
    `spf_check_results` column names), including `include_tree`, `pass_set`, `ip_coverage[]`,
    `tool_runs[]`, `tests[]`, and `problem_states`; the same object is exposed in-process as
    `results.spf`.
12. Every non-`ok` finding carries a concrete `remediation` (the exact term/record/step), never a
    generic "check your DNS".
13. The Dashboard SPF cell reflects the worst severity; `info`-only never turns it amber; clicking
    the SPF cell on a **Domain-health** row opens the domain's **newest run's** SPF page
    (`/domains/:domainId/spf`), clicking the SPF cell on a **Runs-table** row opens **that run's**
    SPF page (`/domains/:domainId/runs/:runId/spf`), and clicking anywhere else in a row opens the
    whole run report; the run report's SPF section chevron opens the same run-scoped SPF page.
14. The full SPF page is **run-scoped** (`/domains/:domainId/runs/:runId/spf`, newest-run alias
    `/domains/:domainId/spf`) and renders: the header with Re-run (starts a **new** run) and the
    **run context strip** (run `started_at` timestamp, working ‹ prev / next › navigation across
    the domain's runs, NEWEST badge on the latest run / "Jump to newest ›" on older ones), the
    budget hero (eval verdict, lookup gauge, void count, all-chip), the raw + mechanism-table
    record panel, the expandable include tree with per-node lookup costs and void badges, the
    fail-first test-results table with copyable fixes (the heart of the page), the sending-IP
    coverage panel, the collapsed tool-runs panel, and the matched problem-state cards linking to
    run-scoped drill-down pages (`/domains/:domainId/runs/:runId/spf/:problemId`, newest alias
    `/domains/:domainId/spf/:problemId`).
15. A second scheduled run that sees a dropped record, a new duplicate, the lookup count crossing
    8 or 10, a weakened `all`, a new void include, or an IP falling out of coverage flags each as
    a **new problem** via the regression diff.
16. Every external tool invocation during a run is captured as one `spf.tool_runs[]` entry —
    `tool`, the **exact argv string** in `command` (every input argument substituted,
    copy-paste reproducible), `started_at`, `duration_ms`, `exit_code`,
    `output_format: json|text` (JSON requested wherever the tool supports it), the captured
    `parsed` output, and `error` (`null` on success) — per the §3 execution table (dig type-99
    probe, checkdmarc cross-check, conditional doggo public-resolver probe); processes are
    spawned with argv arrays under hard timeouts, and a missing binary or timeout degrades to
    **warning**/skip, never a false critical.
17. The run report's SPF section (§6.3) renders the summary strip (mini gauge, voids,
