# EmailDeliveryHero — Report-Email Ingestion (DMARC aggregate & TLS-RPT)

The mailbox providers your domain sends to — Google, Microsoft, Yahoo, Outlook.com — send **report
emails back to you**. Those emails are not human-readable prose; they are machine reports, one
compressed attachment per message, describing **who sent mail as your domain and whether it
authenticated** (DMARC aggregate reports) and **whether inbound TLS to your MX succeeded or failed**
(TLS-RPT reports). They are the single most direct evidence of a deliverability problem: they are the
receivers telling you, in their own words, what they saw.

This spec is the authoritative description of **those emails and how EmailDeliveryHero turns them
into problems-and-fixes**. It is the shared ingestion machinery that the two "future ingestion" stubs
already scoped in the per-check specs plug into — `dmarc.real_pass_rate` /
[`dmarc_aggregate_reports`](./checks/dmarc.mdx#5-information-schema-database-schema) and
`infra.tls_rpt_reports_ingested` / [`tls_rpt_reports`](./checks/tls_rpt.mdx#5-information-schema-database-schema).
Where the per-check specs audit the **DNS record** that *requests* these reports (`rua=` on the DMARC
and TLS-RPT records), **this spec covers the reports that come back**.

> example throughout (§12). That directory is the **fixture set / sample corpus**, not a runtime

## 1. What these emails are

There are **three** kinds of report email a monitored domain receives. All three arrive because a DNS
record you published named a `rua=`/`ruf=` mailbox and asked receivers to report to it. Two of the
three are in scope for the first ingestion round; the third is future.

| Report kind | Standard | Requested by DNS record | Attachment media type | Payload | Answers the question |
| --- | --- | --- | --- | --- | --- |
| **DMARC aggregate (RUA)** | RFC 7489 / DMARCbis RFC 9990 | `_dmarc.<domain>` `rua=mailto:…` | `application/zip`, `application/gzip` (also `application/x-zip-compressed`) | XML (`<feedback>…`) | *Who sent mail as me, from which IPs, and did SPF/DKIM pass **and align**?* |
| **TLS-RPT (SMTP TLS Reporting)** | RFC 8460 | `_smtp._tls.<domain>` `rua=mailto:…`/`https:…` | `application/tlsrpt+gzip` (email) / `application/tlsrpt+json` (HTTPS POST) | JSON (`{ organization-name, policies[] … }`) | *Did inbound TLS to my MX succeed, or did MTA-STS/DANE/STARTTLS fail silently?* |
| **DMARC failure/forensic (RUF)** *(future)* | RFC 7489 / DMARCbis RFC 9991 | `_dmarc.<domain>` `ruf=mailto:…; fo=1` | `message/feedback-report` + `message/rfc822` | ARF | *Show me a redacted sample of one message that failed alignment.* |

**Crucial classification fact (learned from the real corpus).** The subject line and the reporting-
org name are **not** reliable type discriminators. Of the 10 `act3ai.com` reports in `emails/`, three
carry a `microsoft.com` submitter and a "Report Domain: act3ai.com" subject **identical in shape** to
the DMARC ones, yet they are **TLS-RPT reports** (`application/tlsrpt+gzip`), not DMARC. The
classifier **must key off the attachment media type and the decompressed payload's root element**
(`<feedback>` XML vs. TLS-RPT JSON), never the filename or subject. See §4.2.

Google Postmaster Tools reputation and provider feedback-loop (FBL) complaint reports are **separate
feeds** (a web dashboard / a distinct ARF stream) and belong to
[checks/reputation_metrics.mdx](./checks/reputation_metrics.mdx) — they are noted here only so nobody
tries to route them through this parser.

## 2. Anatomy of a report email — how to read a `.eml`

A `.eml` is a raw RFC 822 message. The report is an **attachment**, and the useful data is inside a
compressed part that must be decoded before it can be parsed. The pipeline never trusts the human-
visible body; it walks the MIME tree.

1. **Parse MIME.** Split headers from body; if `Content-Type: multipart/*`, split on the boundary and
   walk every part (recurse into nested `multipart/mixed` / `multipart/report`).
2. **Find the report part** by `Content-Type` (the media types in §1's table), falling back to the
   `Content-Disposition: attachment; filename=…` extension (`.zip`, `.gz`, `.xml`, `.json`) only as a
   secondary hint.
3. **Base64-decode** the part per its `Content-Transfer-Encoding` (almost always `base64`).
4. **Decompress** by magic bytes, not by filename: `PK\x03\x04` → ZIP (Google); `\x1f\x8b` → gzip
   (Outlook, Microsoft TLS-RPT); raw `<?xml`/`{` → already plain.
5. **Parse** the resulting XML (DMARC) or JSON (TLS-RPT) — §4.3.

**Filename convention (informational only).** Providers name the attachment
`<reporting-domain>!<audited-domain>!<begin-epoch>!<end-epoch>[!unique].xml.(gz|zip)` for DMARC, e.g.
`google.com!act3ai.com!1750809600!1750896000.zip`. The window and reporter can be read from the
filename, but the authoritative values are **inside** the payload — parse those, treat the filename as
a hint only.

**Robustness rules.** Tolerate: multiple attachments in one email (parse each); an attachment that is
already-decompressed XML/JSON; `application/octet-stream` with a `.gz`/`.zip` filename (sniff magic
bytes); oversized reports (stream-decompress with a hard cap, default 25 MB uncompressed, to avoid a
zip-bomb); a non-report email in the mailbox (no matching part → skip, log `info`, do not error).

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/mime.ts` — functions `extractReportPayloads()` (MIME walk + base64 decode), `decompressPayload()` (magic-byte ZIP/gzip, capped at `MAX_UNCOMPRESSED_BYTES`), and `classifyPayload()` (§4.2 media-type/payload-root classification).

## 3. What problems these reports reveal

This is the heart of the spec: the catalog of **deliverability problems detectable from the reports**,
each with the concrete fix. Every problem below becomes a `Finding` (§5) rolled into the DMARC or DNS
& Infrastructure dashboard category (§6). Problems are grouped by report kind.

### 3.1 From DMARC aggregate (RUA) reports

DMARC passes for a message if SPF **or** DKIM passes **and** the passing identity **aligns** with the
`From:` domain (relaxed = same org domain; strict = exact). The report gives, per source IP: the
message `count`, the receiver's applied `disposition` (none/quarantine/reject), and the SPF/DKIM
**evaluated** and **aligned** results. From that we detect:

| Problem | What in the report signals it | Why it hurts deliverability | Concrete fix |
| --- | --- | --- | --- |
| **Unauthorized / unknown sender** | A source IP with meaningful `count` where **both** SPF and DKIM fail alignment, and the IP is not a known ESP of the domain. | Either spoofing/abuse of your domain, or a legitimate sender you forgot to authorize — both erode reputation and, under `p=reject`, get rejected. | If yours: add the sender to SPF (`include:`) and enable DKIM for it. If not: it is being rejected already under `p=reject`; monitor and, if volume is high, report the abuse. |
| **SPF-alignment gap on a real stream** | A known ESP source where DKIM aligns+passes but **SPF alignment fails** because the envelope (Return-Path/mfrom) domain is a subdomain/ESP domain, under `aspf=s` (strict). | The stream passes DMARC **only** via DKIM. One DKIM key rotation/breakage and the whole stream fails DMARC and is rejected under `p=reject`. Single point of failure. | Relax SPF alignment (`aspf=s` → `aspf=r`) **or** configure a branded/custom Return-Path under the org domain (e.g. `bounces.<domain>` CNAME'd at the ESP) so the envelope aligns. |
| **DKIM-alignment gap on a real stream** | A known ESP source where SPF aligns+passes but DKIM fails/doesn't align (unsigned, or signed with an ESP `d=` that isn't your domain), under `adkim=s`. | Symmetric single-point-of-failure: the stream leans entirely on SPF; a Return-Path change or forward breaks it. | Enable DKIM signing with a selector under your domain (`selector._domainkey.<domain>`) and/or set `adkim=r`. |
| **Forwarded / indirect mail failing** | Rows with `header_from=<domain>` but envelope `spf:` domain of a forwarder/gmail.com/etc., SPF non-aligned, "rescued" only by DKIM pass. | Forwarders rewrite the envelope; only DKIM (or ARC) survives. If DKIM also breaks, forwarded mail is rejected. | Ensure DKIM is robust; consider ARC-aware handling ([checks/arc.mdx](./checks/arc.mdx)). Usually benign when DKIM aligns. |
| **Active enforcement (mail being quarantined/rejected)** | Any row with `disposition=quarantine` or `reject` and non-trivial `count`. | Your **own** legitimate mail is being dropped by receivers right now. | Identify the failing source, authorize it (SPF/DKIM), and confirm alignment before it recurs. |
| **Policy weaker than believed** | The `<policy_published>` block shows `p=none` (or `sp=none`) while the owner thinks they enforce. | Monitor-only = zero spoofing protection despite the reports arriving. | Raise to `p=quarantine` → `p=reject` once reports confirm all legit sources align. (See [checks/dmarc.mdx](./checks/dmarc.mdx) `dmarc.policy`.) |
| **Real-world pass rate too low to enforce** | Aggregate: % of volume that is DMARC-aligned-pass across all sources over the window. | Raising to `p=reject` while a meaningful slice fails would reject legitimate mail. | Do not tighten `p` until aligned-pass rate is ~100% for known sources; fix the failing streams first. This is the data behind `dmarc.real_pass_rate`. |
| **New source appeared** | A source IP present in this window's report but absent from prior windows. | Could be a new legit ESP (authorize it) or the onset of spoofing (watch it). | Reconcile against known senders; add to SPF/DKIM if yours, else monitor. |

### 3.2 From TLS-RPT reports

TLS-RPT reports summarize, per reporting org per day, whether the sender could negotiate TLS to your
MX under your MTA-STS/DANE policy: `policies[]` (with `policy-type` `sts`/`tlsa`/`no-policy-found`),
`summary.total-successful-session-count` / `total-failure-session-count`, and `failure-details[]`.

| Problem | What in the report signals it | Why it hurts deliverability | Concrete fix |
| --- | --- | --- | --- |
| **TLS negotiation failures to your MX** | `total-failure-session-count > 0`, `failure-details[].result-type` = `starttls-not-supported`, `certificate-host-mismatch`, `certificate-expired`, `validation-failure`, `tlsa-invalid`, etc. | Senders enforcing MTA-STS/DANE **bounce or downgrade** mail to you; silent for days without these reports. | Fix the reported layer: renew/replace the MX certificate (host-mismatch/expired), repair the TLSA record after a key roll (`tlsa-invalid`), ensure STARTTLS is offered. See [checks/tls_transport.mdx](./checks/tls_transport.mdx). |
| **Failure-rate spike / regression** | A day's failure count jumps vs. the prior baseline. | An MTA-STS/DANE change just broke inbound TLS. | Roll back the offending policy/cert change; re-check MTA-STS mode and MX cert. |
| **`no-policy-found` where a policy is expected** | `policy-type: no-policy-found` from a reporter while you believe MTA-STS is published. | Your MTA-STS/TLSA isn't being seen — policy not actually protecting you. | Verify `_mta-sts` TXT + the HTTPS policy file ([checks/mta_sts.mdx](./checks/mta_sts.mdx)) or the TLSA record. |
| **Healthy baseline (informational)** | `total-failure-session-count == 0` across reporters, `mode: enforce`. | Confirms inbound TLS is working — a green signal, not a problem. | None; record the healthy baseline so a later regression is detectable. |

### 3.3 What the reports do NOT tell you

State the limits so the audit doesn't over-claim: DMARC/TLS-RPT reports carry **no blacklist/DNSBL
status**, **no content spam score**, and **no inbox-vs-spam placement**. Those come from separate
checks ([checks/blacklists.mdx](./checks/blacklists.mdx),
[checks/content_scoring.mdx](./checks/content_scoring.mdx),
[checks/inbox_placement.mdx](./checks/inbox_placement.mdx)). Report ingestion complements, never
replaces, the DNS/probe checks.

## 4. The ingestion pipeline

Report emails arrive **asynchronously** (daily-ish, on each receiver's schedule), independent of the
on-demand/scheduled DNS audits. Ingestion is a distinct stage that reads the report mailbox, parses
what it finds, and writes normalized rows that the audit engine and UI read.

```
receive  →  classify  →  decode  →  parse  →  normalize  →  derive findings  →  store  →  diff/surface
(mailbox)   (media type) (b64+     (XML/JSON) (per-source   (report.* into the  (reports/  (regression +
             /payload)    gunzip)             rows)          dmarc/infra cats)   *.json)    Reports UI)
```

### 4.1 Receive (source of the emails)

First round supports two sources, both admin-configured (§8 / [settings.mdx](./settings.mdx)):

- **IMAP mailbox poll** — the `rua` mailbox (e.g. `dmarc@<domain>`, `tls-reports@<domain>`) is polled
  on a cadence (default hourly). Uses the same Google Workspace identity the app already authenticates
  with via OpenAuthFederated where possible; otherwise an app-password/OAuth mailbox configured in
  Admin settings. Processed messages are marked read / moved to a `Processed` folder (idempotent — see
  dedupe in §4.5).
- **Drop folder** — a local directory of `.eml`/`.xml`/`.json`/`.gz`/`.zip` files (this is exactly the
  repo `emails/` corpus, and how a user can hand-feed reports without wiring a mailbox). The folder
  path is admin-configurable; new files are ingested and then moved to a processed subdir.

> **Runtime, per the charter.** Ingestion is **TypeScript on Node** like everything else — the poller
> is the same kind of Node worker the launchd scheduler already invokes
> (`code/deploy/launchd/trigger-scheduler.mjs`), not a shell/`curl` job.

### 4.2 Classify (never trust the filename)

For each candidate part, in order: (1) `Content-Type` media type → DMARC-agg vs TLS-RPT vs RUF; (2)
if ambiguous (`application/octet-stream`), sniff magic bytes then peek the decompressed payload — root
`<feedback>` element ⇒ DMARC aggregate; a JSON object with `organization-name`/`policies` ⇒ TLS-RPT.
A part matching nothing is skipped with an `info` log. **This step is why the three `microsoft.com`
files in the corpus are correctly handled as TLS-RPT despite DMARC-looking subjects.**

### 4.3 Decode & parse

- **DMARC XML** → parse `<report_metadata>` (`org_name`, `report_id`, `date_range.begin/end`),
  `<policy_published>` (`domain`, `p`, `sp`, `adkim`, `aspf`, `pct`, `np`), and every `<record>`:
  `<row><source_ip>`, `<count>`, `<policy_evaluated>` (`disposition`, `dkim`, `spf`),
  `<identifiers><header_from>`, and `<auth_results>` (`<dkim><domain><result>`,
  `<spf><domain><result>`). Use a dependency-light XML parser; never `eval`.
- **TLS-RPT JSON** → parse `organization-name`, `date-range`, and each `policies[]` entry's
  `policy.policy-type`, `summary.total-successful-session-count`,
  `summary.total-failure-session-count`, and `failure-details[]` (`result-type`, `failed-session-count`).

### 4.4 Normalize

Reduce to a common shape the engine can reason over:
- **DMARC** → one normalized row per `<record>`: `{ reporterOrg, reportId, windowStart, windowEnd,
  sourceIp, count, disposition, spfEvaluated, dkimEvaluated, spfAligned, dkimAligned, dmarcPass,
  headerFrom, envelopeSpfDomain, dkimSigningDomains[] }`. `dmarcPass = (spfAligned || dkimAligned)`.
- **TLS-RPT** → one normalized row per reporter-per-policy: `{ reporterOrg, reportDate, policyType,
  successCount, failureCount, failureDetails[] }`.

### 4.5 Dedupe

Reports are re-sent and windows overlap. Key each report by `(reporterOrg, reportId)` (DMARC) or
`(reporterOrg, reportDate, policyType)` (TLS-RPT); ignore a report whose key is already stored. This
makes both the IMAP poll and the drop-folder scan **idempotent** — re-ingesting the same `emails/`
corpus produces no duplicates.

### 4.6 Derive findings & store

Normalized rows are aggregated over a rolling window (default 7 days), scored against §3's catalog
into `Finding`s (§5), and persisted (§9). The findings attach to the domain's audit surface so they
appear on the dashboard and in the Reports view (§7).

Implemented in code:
Parsing is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/dmarc-xml.ts` — function `parseDmarcAggregateXml()` and `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/tlsrpt-json.ts` — function `parseTlsRptJson()`. Aggregation + §3-catalog scoring into findings is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/derive-findings.ts` — functions `aggregateDmarc()`, `aggregateTlsRpt()`, `deriveDmarcReportFindings()`, `deriveTlsRptFindings()`. The pipeline is orchestrated by `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/reports.service.ts` — class `ReportsService`, method `ingest()`.

## 5. Findings emitted (rolled into existing categories)

To honor the **six locked dashboard categories** (no seventh is added — see
[dashboard.mdx](./dashboard.mdx)), report-derived findings use existing `checkId` prefixes: DMARC-
aggregate findings under **`dmarc.*`**, TLS-RPT findings under **`infra.*`**. Each is a standard
`Finding { checkId, severity, title, detail, remediation }` ([engineering.mdx](./engineering.mdx)).
In addition, the **run-time corpus test** (§13) emits aggregate **`content.report_*`** findings into
the existing **Spam & Content** category — the corpus-analysis surface (did we scan, did every
report land on the right domain, what does the mail stream show) — while the per-source detail
stays in the two tables below. One parse, three surfaces; severities always agree.

| checkId | Source | Severity rule | Title / detail | Remediation |
| --- | --- | --- | --- | --- |
| `dmarc.real_pass_rate` | DMARC agg | **warning** if aligned-pass rate < ~100% of known-sender volume while `p<reject`, or any non-trivial unaligned volume before enforcing; **info** at 100%. | "N% of mail is DMARC-aligned; M msgs from K sources fail alignment." | Fix/authorize the failing sources (SPF `include:` / DKIM selector / alignment) **before** raising policy. |
| `dmarc.report_unaligned_source` | DMARC agg | **critical** if a source fails **both** SPF+DKIM alignment with real volume and isn't a known sender; **warning** if it's a known ESP with a one-sided alignment gap. | "Source `<ip>` sent `<count>` as `<header_from>` — SPF `<r>`/aligned `<b>`, DKIM `<r>`/aligned `<b>`." | Per §3.1: authorize (SPF/DKIM) if yours; else monitor — it's already being rejected under `p=reject`. |
| `dmarc.report_alignment_fragility` | DMARC agg | **warning** — a real stream passes on only one mechanism (e.g. SendGrid via DKIM only, SPF strict-misaligned). | "Stream from `<ip>`/`<envelope-domain>` passes DMARC via DKIM only; SPF alignment fails under `aspf=s`." | `aspf=s` → `aspf=r` **or** brand the Return-Path (`bounces.<domain>`), so the stream has dual-auth resilience. |
| `dmarc.report_enforcement` | DMARC agg | **critical** if own mail is `disposition=reject`/`quarantine`; **info** if all `none`. | "`<count>` msgs from `<ip>` were `<disposition>` by `<reporterOrg>`." | Identify + authorize the failing source; confirm alignment. |
| `dmarc.report_new_source` | DMARC agg | **info** (watch) — a source IP unseen in prior windows. | "New sending source `<ip>` (`<count>` msgs) appeared this window." | Reconcile against known senders; add to SPF/DKIM if yours. |
| `infra.tls_rpt_reports_ingested` | TLS-RPT | **warning** on any `failure-details` volume / failure-rate spike; **info** at zero failures. | "`<reporterOrg>`: `<success>` ok / `<failure>` failed TLS sessions (`<result-type>`)." | Fix the reported layer: MX cert (host-mismatch/expired), TLSA (`tlsa-invalid`), STARTTLS, or MTA-STS policy. |

Every non-`ok` finding carries a **concrete** remediation (the exact tag edit or step) — never a
generic "check your DNS," per [engineering.mdx §6.4](./engineering.mdx).

## 6. How report findings roll up onto the dashboard

The six category columns are **LOCKED**; report findings simply feed the two relevant columns:

- **DMARC column** ← all `dmarc.report_*` and `dmarc.real_pass_rate` findings (worst severity wins the
  cell color, exactly as for the DNS-based `dmarc.*` sub-checks).
- **DNS & Infrastructure column** ← `infra.tls_rpt_reports_ingested` (alongside the TLS-RPT record
  audit and MTA-STS/DANE).

Because reports arrive asynchronously, a report-derived finding can appear on the **most recent
audit's** category cell between DNS audits (the store attaches the latest report-window findings to
the domain; the dashboard's daily row reflects them). `info`-only report findings (healthy TLS
baseline, new-source watch) never turn a cell amber — same rule as everywhere.

  Count | Disposition | SPF (eval/align) | DKIM (eval/align) | DMARC | Header-From | Envelope`, sorted
  by count desc. A TLS-RPT finding expands to a **per-reporter/day table** with success/failure counts

  reports without waiting for the poll.

## 9. Information schema (on-disk store)

There is **no database** in the first round ([engineering.mdx §7](./engineering.mdx),
[storage.mdx](./storage.mdx)). Ingested reports persist as JSON under the state root, fronted by the
`store` module, and map 1:1 to the future SQL tables already declared in the per-check specs
(`dmarc_aggregate_reports` in [dmarc.mdx](./checks/dmarc.mdx), `tls_rpt_reports` in
[tls_rpt.mdx](./checks/tls_rpt.mdx)).

```
~/.email_delivery_hero/                       # state root (storage.mdx; EDH_STATE_DIR override)
└── reports/
    └── <domainId>/
        ├── dmarc/<reporterOrg>-<reportId>.json     # one parsed DMARC aggregate report (deduped by reportId)
        ├── tlsrpt/<reporterOrg>-<reportDate>.json  # one parsed TLS-RPT report (deduped by reporter+date)
        └── ingest-state.json                        # last poll time, processed message ids, cursor
```

A parsed DMARC report file holds `{ reporterOrg, reportId, window:{begin,end}, policyPublished:{domain,
p,sp,adkim,aspf,pct,np}, rows:[ { sourceIp, count, disposition, spf:{evaluated,aligned},
dkim:{evaluated,aligned}, dmarcPass, headerFrom, envelopeSpfDomain, dkimSigningDomains[] } ] }` — the
same fields as `dmarc_aggregate_reports`. A parsed TLS-RPT file holds `{ reporterOrg, reportDate,
policies:[ { policyType, successCount, failureCount, failureDetails:[{resultType,count}] } ] }` — the
same fields as `tls_rpt_reports`. The derived `report.*`/`dmarc.report_*`/`infra.*` findings land in
the normal audit document's `findings[]` (with a `source:"report"` marker) so the dashboard and run
detail read them with no special path. Because the store module abstracts all of this, the file→DB
move is a single-module change.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/report-store.ts` — functions `saveDmarcReport()` / `saveTlsRptReport()` (deduped writes under `reports/<domainId>/`), `listDmarcReports()` / `listTlsRptReports()`, `hasAnyReports()`, and `readIngestState()` / `writeIngestState()` (the `ingest-state.json` cursor). The on-demand "Ingest now" API is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/reports/reports.controller.ts` — class `ReportsController`, method `ingest()` (`POST /api/domains/:id/reports/ingest`).

## 10. When we run it & regression detection

- **Ingestion cadence** is **report-driven**, not audit-driven: the poller runs on its own schedule
  (default hourly) because reports trickle in daily per receiver. This is deliberately decoupled from
  the DNS-audit cadence in [scheduled_checks.mdx](./scheduled_checks.mdx).
- **On-demand** ingest via the Reports view's **Ingest now** and `POST /api/domains/:id/reports/ingest`.
- **Regression detection** mirrors the rest of the app ([engineering.mdx §8](./engineering.mdx)): each
  new rolling-window aggregation is diffed against the prior one. New problems to surface: a **new
  unaligned source** appears, the **aligned-pass rate drops**, own mail starts being **rejected/
  quarantined**, or **TLS failure sessions appear/spike**. Each becomes a "new problem" on the
  Dashboard the same way a DNS regression does.

## 12. Worked example — the 10 real reports for act3ai.com

The `emails/` corpus (used as the fixture set) resolves to exactly the picture the ingestion is built
to produce:

- **Classification.** 5× `google.com` (DMARC agg, `application/zip`), 2× `protection.outlook.com`
  (DMARC agg, `application/gzip`), **3× `microsoft.com` (TLS-RPT, `application/tlsrpt+gzip`)** — the
  three TLS-RPT files are the reason §4.2 keys off media type, not subject.
- **Published policy (all 7 DMARC reports agree):** `p=reject; sp=reject; adkim=s; aspf=s; pct=100` —
  the strictest posture. So DMARC's own record checks are **healthy**; the problems are in the field
  data, not the record.
- **Volume:** 1,195 messages over 2026-06-24→30. **DMARC-aligned pass: 1,157 (96.8%). Fail: 38
  (3.2%).**
- **The one real problem → `dmarc.report_alignment_fragility` (warning).** ~34 messages from SendGrid
  IPs (`149.72.*`, `159.183.*`) with envelope `em2598.act3ai.com`: **DKIM passes and aligns, but SPF
  alignment fails** under `aspf=s` (envelope is a subdomain). The stream passes DMARC **via DKIM
  only** — one DKIM hiccup from being rejected under `p=reject`. **Fix:** set `aspf=r`, or brand the
  Return-Path (`bounces.act3ai.com` CNAME'd at SendGrid) so SPF aligns too. This is the single
  actionable finding a good audit must surface.
- **No spoofing → `dmarc.report_unaligned_source` is clear.** Zero rows fail *both* SPF and DKIM; all
  traffic traces to Google Workspace or the SendGrid tenant. The handful of `gmail.com` /
  `whitehatengineering.com` envelope rows are benign forwards, DKIM-authenticated.
- **`dmarc.real_pass_rate`:** 96.8% aligned; because `p=reject` is already enforced and the 3.2% is a
  known stream rescued by DKIM, the finding is a **warning** (fix the SendGrid alignment) rather than a
  block-the-enforcement critical. Do **not** loosen `p`.
- **TLS-RPT → `infra.tls_rpt_reports_ingested` (info/healthy).** Microsoft reports show MTA-STS
  `mode: enforce`, MX = `aspmx.l.google.com` + alt1-4, and **0 failed sessions** across all three
  days. Inbound TLS is healthy; record the baseline so a future regression is caught.
- **Out of scope from these reports:** blacklist status must come from
  [checks/blacklists.mdx](./checks/blacklists.mdx) against the Google Workspace + SendGrid IPs above.

## 13. Run-time corpus analysis — the Spam & Content `report_emails` test

The §4 pipeline is **asynchronous** (a poller on its own cadence). This section adds the
**synchronous, in-run test** the user sees the moment they run checks: every audit run also walks
the report-email corpus, attributes each email to its domain, and analyzes the problems. It is
**family #7 of the Spam & Content category** — section key `report_emails`, checker id
`content.report_emails` — specified at the category level in
[checks/spam_content.mdx](./checks/spam_content.mdx) §1/§4; this section owns the behavior.

### 13.1 What the test does, step by step

1. **Scan the corpus, in place.** The checker reads every `.eml`/`.xml`/`.json`/`.gz`/`.zip` file in
   the **analysis directory** — admin-configurable (`reports.analyzeDir`, §8); empty = the §4.1 drop
   folder (`<state>/reports/inbox`). Pointing it at the repo's `emails/` corpus is exactly how the
   §12 worked example runs in development. Unlike the poller's drop-folder ingest, the run-time scan
   is **read-only — it never moves or renames files** (a corpus must survive unlimited re-runs), and
   §4.5's dedupe keys make every re-scan idempotent: already-stored reports count as duplicates,
   never as new data.
2. **Decode exactly like §2.** Same MIME walk, same base64 + magic-byte decompression (`PK` → ZIP,
   `\x1f\x8b` → gzip), same §4.2 classification by media type and payload root — never by filename
   or subject. A file with no report payload is skipped and counted; a file that fails to decode is
   counted as an error and surfaces in the `content.report_corpus` finding (warning), never thrown.
3. **Attribute each report to its domain — the routing rule.** The report names its own subject:
   `<policy_published><domain>` (DMARC) / `policies[].policy.policy-domain` (TLS-RPT). Matching
   against the monitored-domain list is **exact name first, else the closest monitored parent** (a
   report for `em2598.act3ai.com` rolls up to `act3ai.com`). Three outcomes:
   - **This run's domain** → stored (§9 store, deduped) and analyzed in step 4.
   - **Another monitored domain** → stored under **that** domain — the report surfaces in *that*
     domain's runs and Reports view, never mixed into this one. Right results, right domain.
   - **No monitored domain** → an **orphan**: counted and named in
     `content.report_domain_attribution` (the mailbox is receiving reports for a domain nobody
     monitors — usually a domain that should be added).
4. **Analyze the run's domain over the rolling window.** Aggregate the stored reports (window per
   §4.6, anchored on the newest report so a historical corpus still analyzes) and score them
   against the §3 problem catalog. Emit the `content.report_*` findings below plus the
   `spam_content.report_emails` snapshot in the run YAML.

**Ordering.** In the run graph the `content.report_emails` scan runs **before**
`dmarc.reports` and the TLS-RPT derivation, so the per-category findings (§5) read a store the
same run just refreshed — scan once, analyze everywhere.

Implemented in code:
The run-time corpus scan is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/checks/report-emails/report-emails.check.ts` — `reportEmailsCheck` (the `Checker`, id `"content.report_emails"`), `run()` method. The DMARC-category derivation that reads the same refreshed store is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/audit/checks/dmarc-reports/dmarc-reports.check.ts` — `dmarcReportsCheck` (id `"dmarc.reports"`), `run()` method.

## 14. Proactive health checks that hunt these problems (future)

The reports are **retrospective** — they tell you what receivers already saw. The audit engine should
also **proactively hunt** the same problems so it flags them before (or without) a report. For every
sending source and stream, the DNS/probe checks reconcile against what the reports would show:

1. **Enumerate known senders** per domain (SPF `include:` targets, DKIM selectors, configured ESPs,
   and — once ingested — the source IPs/envelope domains observed in reports).
2. **Alignment simulation (the core future check).** For each known stream, compute what DMARC would
   conclude given the domain's `adkim`/`aspf`: does the envelope domain align (SPF) and does a DKIM
   `d=` under the org domain exist and align? A stream that would pass on **only one** mechanism is
   flagged as *fragile* — the same `dmarc.report_alignment_fragility` problem, detected from DNS
   alone. (For act3ai.com this would flag the SendGrid subdomain under `aspf=s` without needing the
   report.)
3. **Enforcement-readiness cross-check** (already scoped as `dmarc.enforcement_readiness` in
   [checks/dmarc.mdx](./checks/dmarc.mdx)): only advise raising `p=none → reject` when every known
   stream is dual-aligned.
4. **TLS posture probe** ([checks/tls_transport.mdx](./checks/tls_transport.mdx),
   [checks/mta_sts.mdx](./checks/mta_sts.mdx), [checks/dane_tlsa.mdx](./checks/dane_tlsa.mdx)):
   actively connect to each MX, verify STARTTLS, certificate validity/name-match, and TLSA/MTA-STS
   consistency — pre-empting the `starttls-not-supported`/`certificate-host-mismatch`/`tlsa-invalid`
   failures TLS-RPT would otherwise report after the fact.
5. **Blacklist sweep** ([checks/blacklists.mdx](./checks/blacklists.mdx)) over every source IP the
   reports reveal — closing the gap in §3.3.
6. **New-source alerting** — when ingestion surfaces a source IP unseen in prior windows, cross-check
   it against SPF and known ESPs; unknown + volume ⇒ escalate.

This makes report ingestion and proactive DNS auditing **two views of one truth**: the reports confirm
in the field what the health checks predict from DNS, and each backstops the other.

## 15. Acceptance criteria

1. A `.eml` in `emails/` (or a report mailbox) is parsed by walking its MIME tree; the report part is
   found by **media type / payload sniffing**, base64-decoded, and decompressed by **magic bytes**
   (ZIP `PK`, gzip `\x1f\x8b`) — never by filename or subject.
2. The **three `microsoft.com` files are classified as TLS-RPT**, and the Google/Outlook files as
   DMARC aggregate, regardless of their DMARC-looking subjects.
3. DMARC XML parsing yields, per `<record>`: source IP, count, disposition, SPF/DKIM evaluated **and
   aligned** results, header-from, and envelope/DKIM identities; `dmarcPass = spfAligned || dkimAligned`.
4. TLS-RPT JSON parsing yields per-reporter-per-policy success/failure counts and `failure-details`.
5. A source failing **both** SPF and DKIM alignment with real volume emits
   `dmarc.report_unaligned_source` (**critical** if unknown sender); a stream passing on only one
   mechanism emits `dmarc.report_alignment_fragility` (**warning**) with the exact `aspf=r`/branded-
   Return-Path fix.
6. `dmarc.real_pass_rate` reports the aligned-pass percentage and is **warning** when a non-trivial
   share is unaligned before enforcement, **info** at ~100%.
7. `infra.tls_rpt_reports_ingested` is **warning** on any failure sessions with the layer-specific fix,

23. Each clickable sub-concept on the explainer navigates per §16.5's concept map (tags →
    sibling dmarc explainers, SPF/DKIM columns → their category pages, source IPs → the
    blacklist affordance, TLS-RPT cross-link → the Reports view), and the references footer
    links RFC 7489 §7.2 / RFC 9990, the reporter postmaster pages, and a DMARC-XML reader tool.
24. The explainer implements all §16.6 states — loading skeletons, ingestion-disabled,
    no-reports-yet (with the copyable `rua=` record), not-measured on an old run, stale-corpus
    freshness banner, and fetch-error with Retry — and the viewed run stays a **snapshot**: an
    older `:runId` never silently shows newer report data without the "live data" chip of §16.3.

## 16. The DMARC sub-test explainer page — `checkKey: reports`

The DMARC (+ARC) category's grow-every-click pass ([checks/dmarc.mdx](./checks/dmarc.mdx) §6.3)
registers this spec's content as the **`reports` unit** — band row "Ingested aggregate reports",
one-liner *"What receivers actually saw: real pass rates and spoof sources."* dmarc.mdx owns the
band row and the shared page chrome; **this section owns everything rendered inside the explainer
page** for the unit, filling the §6.4 five-block contract on the locked route pattern. The
`dmarc-reports` checker (`code/…/audit/checks/dmarc-reports/dmarc-reports.check.ts`, checker id
`dmarc.reports`) is the data source: it reads the §9 store and emits the §5 findings — every one
carries `checkId: "dmarc.reports"` and `source: "report"`, which is exactly how the page selects
its rows from the run.

### 16.1 Route, chrome, and the status chip

```
/domains/:domainId/runs/:runId/dmarc/check/reports     (canonical, run-scoped)
/domains/:domainId/dmarc/check/reports                 (newest-run alias)
```

Shared chrome per dmarc.mdx §6.4: the category header row, the **run context strip** (back link →
the run's DMARC page `/domains/:domainId/runs/:runId/dmarc`, ‹ prev / next › run paging that swaps
`:runId`, the "newest run" chip on the alias), and the unit-scoped run-history strip + references
footer below the blocks. The static `check` segment keeps this disjoint from the
`/dmarc/:problemId` (`PS-nn`) drill-downs; the router registers `check/reports` (with the other
`check/:checkKey` routes) **ahead of** the `:problemId` param route.

**Status chip derivation (locked).** The chip = the worst severity among the viewed run's findings
with `checkId: "dmarc.reports"`, using the checker's actual severity values:

| Worst severity present | Chip |
| --- | --- |
| `critical` (`dmarc.report_unaligned_source.<ip>` unknown sender, `dmarc.report_enforcement.<ip>`) | ✗ red — "Failing" |
| `warning` (pass-rate < ~100%, fragility, own both-fail stream) | ⚠ amber — "Needs attention" |
| `info` only (healthy baselines, no-reports-yet, ingestion disabled, new-source watch) | ⓘ slate — "Informational" |
| `ok` present and nothing worse (`…unaligned_source` / `…fragility` clear rows) | ✓ green — "Healthy" |
| no `dmarc.reports` findings in the run at all (run predates the checker) | gray — "Not measured" |

This is the same chip the category page's band row and the run report's mini-list show for the
unit — one derivation, three surfaces.

### 16.2 The five blocks

Wireframe (the §12 corpus, worst = warning):

1. **What this is.** 2–4 plain-language paragraphs, fixed copy (this block never depends on run
   data — the concept never disappears): what a DMARC aggregate (rua) report is (§1 row 1), that
   the `rua=` tag on `_dmarc.<domain>` *requests* them, that they are the receivers' own record of
   *who sent mail as your domain, from which IPs, and whether SPF/DKIM passed **and aligned***,
   and one concrete example ("Google's daily zip for act3ai.com covering 06-24"). One-line gloss
   for "alignment" on first use; a "how the sausage is made" link to §2's decode pipeline for the
   curious. Cross-link: the tag that requests these reports is audited by the
   [`reporting` unit](./checks/dmarc.mdx#63-the-sub-tests-band-and-the-checkkey-registry).
2. **Your current state.** Everything here comes from **the viewed run** (snapshot rule, §16.3):
   - The large status chip (§16.1) with the aggregation window and report count as the sub-line.
   - The **raw record**: the run's `_dmarc.<domain>` TXT answer (monospace), with the unit's
     input tag — `rua=` — visually highlighted (the other tags render muted; clicking any tag
     navigates per the §16.5 concept map).
   - The **scoped aggregate breakdown table** — this unit's "parsed field-by-field" view. Because
     the unit parses reports rather than record tags, the parsed fields are the `DmarcAggregate`
     fields the checker computes (derive-findings.ts): `reportCount`, `reporters`,
     `window.begin/end`, `totalMessages`, `alignedPassMessages` (dual-aligned),
     `dmarcPassMessages`, `passRatePct`, DKIM-only / SPF-only / both-fail volumes, quarantined /
     rejected volumes, and `policyPublished` (`p`, `sp`, `adkim`, `aspf`, `pct` — each value
     rendered with its meaning, defaults grayed when absent). Every row: value + one-line meaning.
   - **One anchored row per finding** the checker emitted for this run — id anchor
     `#finding-<finding-id>` (e.g. `#finding-dmarc.report_alignment_fragility.em2598.act3ai.com`)
     — rendering exactly the checker's fields: severity chip, `title`, `detail` (the observed
     block), `evidence` in monospace when present, and `remediation` (block 4 repeats it as
     steps). Per-IP findings (`…unaligned_source.<ip>`, `…enforcement.<ip>`) group under one
     family heading with a count.
   - The expandable **per-source-IP table** (§7.1 columns: Source IP | Count | Disposition |
     SPF eval/align | DKIM eval/align | DMARC | Header-From | Envelope | Reporters, count-desc)
     — the `rows[]` of the aggregate; collapsed by default, remembered per session.
3. **What it means.** Deliverability impact keyed to the current worst state, reusing §3.1's "why
   it hurts" copy verbatim per firing finding: the passing paragraph ("these reports confirm every
   real stream authenticates — receivers see no spoofing surface") plus one paragraph per firing
   state (pass-rate gap = enforcement risk; unaligned source = spoofing or a forgotten sender;
   fragility = one DKIM hiccup from rejection under `p=reject`; enforcement = your own mail is
   being dropped **right now**; new source = watch item). The §3.3 limits line closes the block
   ("these reports carry no blacklist or content-score signal — see those checks").
4. **What you can do about it.** Numbered, copy-pasteable steps per firing finding, copy buttons
   carrying **only publishable values** (never prose):
   - *Fragility (DKIM-only)* → step 1: the corrected TXT `v=DMARC1; p=reject; sp=reject;
     adkim=s; aspf=r; rua=mailto:…` `[Copy]`; step 2 (alternative): the branded Return-Path CNAME
     (`bounces.<domain> → <esp-return-path-host>`) `[Copy]`; step 3: where to paste (DNS
     provider's TXT editor), the propagation wait, and the copyable verify command
     (`dig +short TXT _dmarc.<domain>`).
   - *Unaligned/unknown source* → if yours: the SPF `include:<esp-spf-host>` mechanism `[Copy]`
     and the DKIM CNAME/selector name `selector._domainkey.<domain>` `[Copy]`; if not yours: the
     monitor-and-report guidance (already rejected under `p=reject`).
   - *Enforcement* → authorize-the-source steps (same snippets), flagged "before your next send".
   - *No reports yet* (empty state) → the full `rua=` record to publish `[Copy]` plus the
     Settings → Admin mailbox pointer (§7.3 copy).
   - Healthy run → the block collapses to "Nothing to do — keep it this way", plus the standing
     recommendation (keep the rua mailbox monitored; §13.2's freshness rule guards it).
5. **Check for new reports.** This unit's block 5 is the **scope exception** locked in
   [checks/dmarc.mdx §6.5](./checks/dmarc.mdx): freshness here belongs to the ingestion pipeline,
   not a DNS run, so the primary button reads **"Check for new reports"** and calls the existing
   `POST /api/domains/:id/reports/ingest` (reports.controller.ts). While in flight it spins and
   disables; on success a toast reports "N new report(s) ingested" and, when N > 0, the page
   navigates to its **newest alias** so fresh findings resolve; N = 0 ⇒ "No new reports — newest
   is <date>". A secondary text action **"Re-run all checks"** calls the v1 full-run endpoint
   `POST /api/audit/run/:domainId` for users who want the run YAML to reflect the fresh reports
   immediately. Keyboard: `r` triggers the primary button (same input-field guard as the pager).

### 16.3 What the run records — the snapshot rule (small checker addition)

Today the checker persists only its `findings[]` into the run. For the explainer, that is almost
enough — block 2's finding rows read the run's findings verbatim — but the aggregate breakdown
table and per-source table would have to read the **live store**, which breaks the "an older run
stays a snapshot" rule. Two-part contract:

- **Proposed addition (small, preferred).** The `dmarc.reports` checker also writes a
  `dmarc.reports` snapshot into the run file mirroring the `DmarcAggregate` it already computes:
  `{ report_count, reporters[], window: {begin, end}, window_days, total_messages,
  aligned_pass_messages, dmarc_pass_messages, pass_rate_pct, dkim_only, spf_only, both_fail,
  quarantined, rejected, policy_published: {p, sp, adkim, aspf, pct, np}, rows[] }` (`rows[]` =
  the §7.1 per-source columns). No new computation — it serializes what derive-findings.ts already
  holds in memory. This makes every block-2 element run-scoped and lets §16.4's history strip plot
  real pass-rate history.
- **v1 fallback (until the snapshot lands).** Block 2 renders the finding rows from the run and
  the breakdown/per-source tables from the live `GET /api/domains/:id/reports` view. When the
  viewed run is **not** the domain's newest, those two tables carry a small **"live data — not
  from this run"** chip so the snapshot rule is never silently violated.

### 16.4 Run-history strip — did my fix work?

Below block 5, the unit-scoped strip per dmarc.mdx §6.7: the domain's last 10 runs, one chip per
run — colored by that run's worst `dmarc.reports` severity (the §16.1 derivation applied
per run), gray for runs with no `dmarc.reports` findings. Each chip tooltips the run timestamp +
pass-rate (when the §16.3 snapshot exists) and **clicking a chip swaps `:runId`** to that run's
explainer. Derived client-side from the existing `GET /api/audit/runs` list; the optional compact
endpoint `GET /api/audit/history/:domainId?category=dmarc&unit=reports` (dmarc.mdx §6.7) serves it
in one call when it lands. Once snapshots exist, the strip upgrades to a **pass-rate sparkline**
(96.8 → 100 after the `aspf=r` fix is the "did my fix work?" picture) — future, gated on §16.3.

### 16.6 States

- **Loading** → five skeleton blocks + skeleton history strip (shared chrome behavior).
- **Ingestion disabled** (admin switch off, §8) → chip ⓘ slate; block 2 shows the single
  "Report ingestion disabled" finding the checker actually emits; block 4 = "Enable report
  ingestion under Settings → Admin"; block 5's button disabled with that tooltip.
- **No reports yet** → chip ⓘ slate; block 2 shows the checker's "No DMARC aggregate reports
  ingested yet" finding; block 4 = the copyable `rua=` record + the 24–72h expectation (§7.3);
  block 5 stays active ("Check for new reports" is exactly the right action).
- **Not measured** (the viewed run predates the checker — no `dmarc.reports` findings) → gray
  chip, blocks 1/3 render normally, block 2 = one line "This run did not include the ingested-
  reports check", block 4 defers to block 5's re-run.
- **Stale corpus** (newest report older than 2× the window — §13.2 freshness rule) → an amber
  banner above block 2: "Newest report is N days old — reports may have stopped arriving; check
  the rua= address and mailbox wiring", linking to `…/dmarc/check/reporting` and Settings.
- **Run-fetch error** → error strip with Retry above block 1 (shared chrome). **Ingest-call
  error** (block 5) → toast with the API error; the page stays on the current run.
- **Unknown `checkKey`** is owned by the shared chrome (dmarc.mdx §6.4's "No such sub-test"
  panel) — nothing extra here.
