# EmailDeliveryHero — Settings

This document specifies the **Settings page** for EmailDeliveryHero. Settings is where the
signed-in user tunes *how* the deliverability auditor behaves — which checks run, which DNS
blocklists to query, how periodic re-audits are scheduled, how the user is alerted to new problems,
where state lives on disk, and which company domains may sign in.

Settings follows the app's optional-login model (see [Security](./security.mdx)): the Settings page is
**reachable and usable while logged out**, where changes are attributed to the **`default`** user and
persist in `users/default/config.yaml`. Some settings are **all-users** (they change the current
user's own experience or trigger read-only actions); others are **admin-only**, enforced on the
backend by the `role:admin` claim (the `@RequireRole("admin")` guard), because they change global
behavior that affects every user and every scheduled run — and the `default` user, having no roles,
is refused those with `403`.

Related specs:

* [Overview](./overview.mdx) — the charter, personas, the left nav, and where Settings fits.
* [Engineering](./engineering.mdx) — the file store (`~/.email_delivery_hero/config.yaml`,
  `domains.yaml`), the NestJS module list, and the audit engine these settings drive.
* [Authentication](./authentication.mdx) — how `role:admin` gates the Admin sub-section.
* [Dashboard](./dashboard.mdx) — where notifications about *new* problems ultimately surface.
* [Scheduled checks](./scheduled_checks.mdx) — the full periodic-re-audit config that Settings
  summarizes and links out to.
* [Spam & deliverability checks](./spam_checks.mdx) — the full catalog of the six check categories
  that "Checks configuration" below enables/disables.

> **Two homes for domains.** The **Domains** left-bar area (see [Overview](./overview.mdx)) is the
> primary CRUD surface for the domains under audit and their per-domain detail. Settings hosts the
> **global defaults** (default DKIM selectors to probe, default enabled checks, default DNSBL zones)
> and a compact management view of the same domain list. Editing a domain in either place writes the
> same `domains.yaml`.

## 1. Monitored domains

Manage the set of sending domains under audit and the per-domain facts the checkers need. This is a
compact mirror of the **Domains** left-bar area; Settings additionally holds the **global defaults**
applied to newly added domains.

| Field | Type | Meaning | Persists to |
| --- | --- | --- | --- |
| Domain | string (FQDN) | The sending domain, e.g. `whitehatengineering.com`. Unique. | `domains.yaml[].domain` |
| Sending IPs | string[] (IPv4/IPv6) | The IPs mail is sent from — used by the PTR and DNSBL checkers. | `domains.yaml[].sendingIps` |
| DKIM selectors | string[] | Per-domain selectors to probe (e.g. `google`, `s1`). Overrides the global default. | `domains.yaml[].selectors` |
| Enabled | boolean | Whether this domain is included in scheduled runs. | `domains.yaml[].enabled` |
| Notes | string | Free-text operator note. | `domains.yaml[].notes` |
| **Default DKIM selectors** | string[] | Global fallback probed when a domain has no selectors of its own. | `config.yaml checks.dkim.defaultSelectors` |

Actions: **Add domain**, **Edit**, **Remove**, and **Audit now** (fires
`POST /api/audit/run/:domainId`, the same on-demand path as the Domains area). Adding or removing a
domain is available to **all users**; editing the **global default selectors** is **admin-only**
(it changes behavior for every domain that relies on the fallback).

## 2. Checks configuration

Controls the audit engine: which of the six **check categories** run, which DNS blocklists the
blacklist checker queries, the default DKIM selectors, and the scoring thresholds that map an audit
score to green/amber/red. The full behavior of each category is specified in
[Spam & deliverability checks](./spam_checks.mdx); this group only toggles and parameterizes them.

| Field | Type | Meaning | Persists to |
| --- | --- | --- | --- |
| Enabled categories | multi-select | Which of the six categories run: **SPF**, **DKIM**, **DMARC**, **Blacklists**, **DNS & Infrastructure** (MX/PTR), **Spam & Content**. | `config.yaml checks.enabled[]` **(admin-only)** |
| DNSBL zones | string[] | The blocklist zones the Blacklists category queries — e.g. `zen.spamhaus.org`, `b.barracudacentral.org`, `bl.spamcop.net`. Add/remove/reorder. | `config.yaml checks.dnsbl.zones[]` **(admin-only)** |
| Default DKIM selectors | string[] | Selectors probed when a domain defines none of its own. | `config.yaml checks.dkim.defaultSelectors[]` **(admin-only)** |
| Score thresholds | numbers | Green/amber/red cutoffs applied to the 0–100 audit score. | `config.yaml checks.thresholds` **(admin-only)** |
| Severity weights | numbers | Points deducted per finding severity (`critical`/`warning`/`info`) — feeds score derivation. | `config.yaml checks.weights` **(admin-only)** |
| SPF lookup limit | number | Max DNS lookups before SPF is flagged (RFC limit is 10). | `config.yaml checks.spf.maxLookups` **(admin-only)** |

Everything in this group is **admin-only** — it changes what every scheduled run does and how every
domain is scored. Non-admins see the values (read-only) so they understand how their domains are
judged, but cannot save changes.

## 3. Scheduling (the Scheduled-checks tab)

Route **`/settings/scheduling`** — the **Scheduling** item in the settings left bar
(`pm/left_bar.yaml`, settings bar → General → Scheduling). This tab **is** the scheduling
configuration surface: the Dashboard's top-right **scheduled toggle** mirrors the same on/off flag,
and its **chevron** navigates here (see [Dashboard](./dashboard.mdx) §7.2 and
[Scheduled checks](./scheduled_checks.mdx), which this section supersedes where they disagree on
defaults and permissions).

### 3.1 Use case

The manual **Run checks** button answers *"how healthy am I right now?"* — but deliverability
breaks **while nobody is looking**: an IP lands on a DNSBL overnight, a DKIM key gets rotated
badly during an unrelated DNS change, an expiring DNSSEC signature ticks toward failure on a
weekend. Scheduling exists so the app notices these **before the user's recipients do**, and
surfaces them as *new problems* on the [Dashboard](./dashboard.mdx) the next time the user looks.

Design intents, in priority order:

1. **Off by default, deliberately.** A fresh install must not start firing DNSBL queries, SMTP
   probes, and HTTP fetches on a timer the user never asked for — scheduled traffic against
   rate-limited public blocklists is something the user opts into, not discovers. `schedule.enabled`
   defaults to **`false`**.
2. **One flip to a working schedule.** Turning scheduling on must never dead-end in an unconfigured
   state. Enabling with no times configured **seeds the default schedule — `06:00`, every day** —
   so a single toggle flip (on the Dashboard *or* on this tab) yields a schedule that runs before
   the workday starts and has fresh results waiting at 9am. No second step is required.
3. **The user picks the time(s) of day.** A daily audit is most useful when it completes just
   before the user habitually checks the Dashboard, so the primary control is a **time-of-day
   picker**, not an opaque "every N hours" interval.
4. **More than once per day is a first-class case.** A sender actively fighting a blacklist
   listing, or one who sends both morning and evening campaigns, wants **multiple runs per day**
   (e.g. `06:00` *and* `18:00`) so a mid-day listing is caught within hours and a delisting is
   confirmed the same day. The times list accepts any number of entries.
5. **Enabling covers everything by default.** The global switch is ANDed with each domain's
   `scheduleEnabled` (see [Domains](./domains.mdx) §6), and per-domain toggles default **on** — so
   flipping the one global switch immediately covers every monitored domain, and the user opts
   *individual* domains out rather than opting each one in.
6. **Anyone may turn it on.** Scheduling is an **all-users** setting (including the logged-out
   `default` user per [Security](./security.mdx)) — it must be *easy* to turn on, and a
   scheduled run does nothing a manual "Run checks" click couldn't. This supersedes the earlier
   admin-only marking of this group.

### 3.2 User interface

```
├──────────────────────────────────────────────────────────────────────────┤

│  ⓘ Scheduling is off — checks only run when you press "Run checks".       │
│                                                                            │

│   [ 06:00 ]  (–)                                                           │
│   [ 18:00 ]  (–)                                                           │
│   [ + Add another time ]                                                   │
│   Times use this machine's local timezone.                                 │
│                                                                            │
│  ── Days ─────────────────────────────────────────────────────────────    │
│   [✓Mon] [✓Tue] [✓Wed] [✓Thu] [✓Fri] [✓Sat] [✓Sun]     (default: every day)│
│                                                                            │
│  ── Status ───────────────────────────────────────────────────────────    │
│   Next run:  —  (scheduling is off)                                        │
│   Last run:  today 06:00 · scheduled  →  view in Runs                      │
│   Covers:    4 of 5 monitored domains   (per-domain toggles → Domains)     │
│   Runs while the app is closed:  launchd agent installed ✓                 │
│                                                                            │

├──────────────────────────────────────────────────────────────────────────┤
│                                        [ Cancel ]   [ Save ]               │
└──────────────────────────────────────────────────────────────────────────┘
```

Controls, top to bottom:

* **Master toggle.** The single on/off for the whole feature (`schedule.enabled`, default
  **off**). It sits first and largest; it is the same flag as the Dashboard top-right toggle, and
  the two stay in sync (both read `GET /api/scheduler`, both write
  `PUT /api/scheduler/config`). The toggle **saves immediately** (optimistic, no Save click
  needed) so "turn it on" is truly one action. If no times are configured when it flips on, the
  backend seeds `times: ["06:00"]`, every day.
* **Off-state banner.** While off, a quiet info line states the consequence ("checks only run when
  you press Run checks"). The times/days form stays **fully editable while off** — a user can lay
  out the schedule first and flip the switch last; nothing fires until enabled.
* **Time(s) of day.** One or more `HH:MM` rows (native `<input type="time">`). **+ Add another
  time** appends a row — this is how the audit runs **multiple times per day**. Each row has a
  remove **(–)**; the last remaining row cannot be removed (at least one time is always
  configured). Duplicates are collapsed and the list is stored sorted. Times are interpreted in
  the machine's **local timezone** (an IANA timezone picker is the noted enhancement in
  [Scheduled checks](./scheduled_checks.mdx); this tab does not offer it first round).
* **Days.** Seven weekday chips, all selected by default (= every day). Deselecting lets a user
  skip weekends (or run *only* weekends). At least one day must remain selected. Daily times +
  weekday chips replace the older cadence enum ("every N hours" stays a
  [Scheduled checks](./scheduled_checks.mdx) enhancement, not a control on this tab).
* **Status block** (read-only, from `GET /api/scheduler`):
  * **Next run** — the computed next fire time, or "— (scheduling is off)". The Dashboard
    chevron's tooltip shows the same value.
  * **Last run** — timestamp and trigger source (`scheduled` / `manual`), linking to the
    Dashboard's Runs table.
  * **Covers** — "N of M monitored domains", reflecting the AND with per-domain
    `scheduleEnabled`; links to [Domains](./domains.mdx) where the per-domain toggles live.
  * **Runs while the app is closed** — whether the OS-level agent (launchd on macOS) is
    installed, per [Scheduled checks](./scheduled_checks.mdx). Read-only here; install/uninstall
    is `just install-agent` / the Scheduled-checks page.
* **Run a scheduled check now.** Fires `POST /api/scheduler/run` with `{ "force": true }` — runs
  the scheduled set immediately (even while the toggle is off) so the user can see exactly what a
  scheduled run will do before committing to one.
* **Save / Cancel.** The times and days form uses an explicit **Save** (dirty-state bar);
  validation (≥ 1 time, ≥ 1 day, `HH:MM` shape) happens on Save and inline.

### 3.3 Storage

**User intent** lives in the `schedule:` block of `config.yaml` under the state dir
(`EDH_STATE_DIR`, per [Storage](./storage.mdx)), written through the shared atomic YAML store
(`shared/yaml-store.ts` — temp-file + rename, hand-editable):

```yaml
# <state dir>/config.yaml
schedule:
  enabled: false            # DEFAULT OFF — the master switch (Dashboard toggle + this tab)
  times:                    # HH:MM, 24h, local time; one or more = multiple runs per day
    - "06:00"
  weekdays: [mon, tue, wed, thu, fri, sat, sun]   # all seven = every day (default)
```

**Runtime state** is *not* mixed into user intent — the scheduler keeps its telemetry in
`scheduler-state.json` next to it (`lastRunAt`, `lastTrigger: "in-process" | "os" | "manual"`); `nextRunAt` is
always **computed** from `times` × `weekdays`, never persisted. Scheduled runs write ordinary run
records to `audits.json` / `runs.json` exactly like manual runs.

**Which domains run:** a scheduled run audits the monitored domains whose
`domains.yaml scheduleEnabled` is `true` (global switch AND per-domain switch). Manual runs
(`POST /api/audit/run`, per-domain play buttons) ignore `scheduleEnabled` entirely.

REST contract (backend `scheduler` module, NestJS on **9312**):

| Method + route | Purpose | Access |
| --- | --- | --- |
| `GET /api/scheduler` | Status: `enabled`, `times`, `weekdays`, computed `nextRunAt`, `lastRunAt`, `lastTrigger`, `running`, and the covered-domain counts. Backs this tab's status block and the Dashboard toggle/tooltip. | all users |
| `GET /api/scheduler/config` | The raw `schedule:` block. | all users |
| `PUT /api/scheduler/config` | Write the `schedule:` block (whole or partial); validates, seeds `times: ["06:00"]` when enabling with none, persists to `config.yaml`, and **re-arms the in-process timer** to match. | all users |
| `POST /api/scheduler/run` | Trigger a scheduled run now. Honors the flag: when `enabled: false` it **skips** (`{ started: false, reason: "disabled" }`) unless the body carries `{ "force": true }`. Also skips if a scheduled run started within the last 5 minutes (dedupe — protects against the in-process timer and the OS agent firing together). | all users |

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/scheduler/scheduler.controller.ts` — class `SchedulerController` (routes `status()`, `getConfig()`, `updateConfig()`, `run()`), delegating to `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/scheduler/scheduler.service.ts` — class `SchedulerService`, methods `getConfig()`, `updateConfig()`, `status()`, `runNow()`. The computed `nextRunAt` is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/scheduler/next-run.ts` — function `computeNextRun()`; the `schedule:` block read/write is `~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/scheduler/schedule-config.store.ts` — `readScheduleConfig()` / `writeScheduleConfig()` / `defaultScheduleConfig()`.

Two scheduling layers consume this storage (details in
[Scheduled checks](./scheduled_checks.mdx)):

* **In-process** — the backend's scheduler service arms a timer for the next `times` ×
  `weekdays` slot whenever the config is loaded or PUT; it replaces the old
  `EDH_PERIODIC_AUDIT_MINUTES` env interval, which is retired.
* **OS-level (launchd)** — `just install-agent` installs
  `com.emaildeliveryhero.scheduler.plist`, which runs `code/deploy/launchd/trigger-scheduler.mjs`
  → `POST /api/scheduler/run`. Because that endpoint checks `enabled` and dedupes, the agent can
  stay installed while the toggle is off (it becomes a no-op) and can coexist with the in-process
  timer without double-running.

## 4. Notifications

How the user learns about **new** problems found by periodic runs — findings that newly appear or
worsen in severity since the domain's previous audit (the regression detection in
[Engineering](./engineering.mdx) §8). Notification *preferences* are **per-user**; the delivery
*channels* (webhook/SMTP) are **admin-only** because they are shared infrastructure.

| Field | Type | Meaning | Persists to |
| --- | --- | --- | --- |
| Desktop notifications | boolean | Browser/desktop notification when a new problem appears. Per-user. | `config.yaml notifications.users[<sub>].desktop` |
| Email me | boolean | Email this user on new problems (sent via the configured SMTP relay / `swaks`). Per-user. | `config.yaml notifications.users[<sub>].email` |
| Minimum severity | enum | Only notify at/above this severity: `info` / `warning` / `critical`. Per-user. | `config.yaml notifications.users[<sub>].minSeverity` |
| Digest vs. immediate | enum | `immediate` per run, or a `daily` roll-up. Per-user. | `config.yaml notifications.users[<sub>].mode` |
| Webhook / Slack URL | string (URL) | Incoming-webhook URL posted to on new problems (Slack-compatible payload). | `config.yaml notifications.webhook.url` **(admin-only)** |
| Webhook enabled | boolean | Master switch for the shared webhook channel. | `config.yaml notifications.webhook.enabled` **(admin-only)** |
| SMTP relay | object | Host/port/from used to send notification email. | `config.yaml notifications.smtp` **(admin-only)** |
| Send test notification | action | Fires a sample notification on the selected channels. | (action) |

Each user edits their **own** desktop/email/severity/mode preferences. The **webhook** and **SMTP**
settings are **admin-only**.

## Settings config model

Settings read and write the same on-disk files described in [Engineering](./engineering.mdx) §7 —
no database. Global and admin-owned settings live in `config.yaml`; per-domain facts live in
`domains.yaml`; per-user preferences are namespaced by the user's OpenAuthFederated `sub` inside
`config.yaml`.

```yaml
# ~/.email_delivery_hero/config.yaml  (read/written by the settings module)
stateDir: ~/.email_delivery_hero        # display only; real value resolved from EDH_STATE_DIR

checks:                                  # §2 — admin-only
  enabled: [spf, dkim, dmarc, dnsbl, dns_infra]   # subset of the six categories
  spf:
    maxLookups: 10
  dkim:
    defaultSelectors: [google, selector1, selector2, k1]
  dnsbl:
    zones:
      - zen.spamhaus.org
      - b.barracudacentral.org
      - bl.spamcop.net
  thresholds:                            # 0–100 score → colour
    green: 90                            # score >= green  → green
    amber: 70                            # green > score >= amber → amber; below → red
  weights:                               # points deducted per finding
    critical: 40
    warning: 15
    info: 0

schedule:                                # §3 — the Scheduling tab (all-users)
  enabled: false                         # DEFAULT OFF; one flip on the Dashboard or §3 enables it
  times: ["06:00"]                       # HH:MM local; multiple entries = multiple runs per day
  weekdays: [mon, tue, wed, thu, fri, sat, sun]   # all seven = every day (default)
```

### REST contract

The `settings` concern is served by the NestJS backend under `/api` (all routes behind the global
`JwtAuthGuard`; admin-only writes additionally behind the `role:admin` permissions guard). The
frontend calls the generated TanStack-Query hooks (see [Engineering](./engineering.mdx) §4).

| Method + route | Purpose | Guard |
| --- | --- | --- |
| `GET /api/settings` | Return the effective settings the current user may see (global + this user's per-user block + resolved tool status). | signed-in |
| `PUT /api/settings` | Update **all-users / per-user** fields for the current user (notifications prefs, appearance). | signed-in |
| `PUT /api/settings/admin` | Update **admin-only** global fields (checks, dnsbl zones, thresholds, schedule, shared channels, retention, tools, access). Rejected `403` without `role:admin`. | `role:admin` |
| `POST /api/settings/notifications/test` | Send a test notification on the selected channels. | signed-in (webhook/SMTP config edits still admin-only) |
| `POST /api/settings/tools/detect` | Re-probe the environment for `dig`/`swaks` and return status. | signed-in |
| `GET /api/settings/export` | Stream a zip of `config.yaml` + `domains.yaml` + `audits/`. | signed-in |
| `POST /api/settings/import` | Restore from an exported archive (validated on read). | `role:admin` |
| `POST /api/settings/reset` | Reset audit history or the whole app (body selects scope). | `role:admin` |

`GET/PUT` operate on the file store transactionally via the `store` module (atomic
write-temp-then-rename, per-file mutex, schema validation on read) — see
[Engineering](./engineering.mdx) §7.

## Acceptance criteria

The Settings page is correct when **all** of the following hold:

1. Settings renders a **left sub-nav of setting groups** and a **right content pane**, driven by the
   `settings` left bar in `pm/left_bar.yaml`, and every route is reachable only by a signed-in user.
2. **Monitored domains** (§1) lists the domains from `domains.yaml` and can add/edit/remove them and
   trigger "Audit now"; edits write the same `domains.yaml` the Domains area uses.
3. **Checks configuration** (§2) enables/disables the six check categories, edits the DNSBL zone
   list, default DKIM selectors, and score thresholds/weights, persisting to `config.yaml`; all of
   §2 is **admin-only** and non-admins see it read-only.
4. **Scheduling** (§3) lives at `/settings/scheduling`: scheduling defaults **off**
   (`schedule.enabled: false` on a fresh install); the master toggle saves immediately and enabling
   with no times configured seeds `06:00` every day, so one flip yields a working schedule; the
   user can set **one or more times of day** (multiple runs per day via "+ Add another time") and
   narrow the weekdays; the status block shows computed next run, last run + trigger source, and
   the covered-domain count; "Run a scheduled check now" fires
   `POST /api/scheduler/run` `{ force: true }`; intent persists to the `schedule:` block of
   `config.yaml`, runtime state to `scheduler-state.json`, and a scheduled run audits only domains with
   `scheduleEnabled: true`. All of §3 is **all-users**.
5. **Notifications** (§4) lets each user set their own desktop/email/severity/mode preferences
   (per-user block keyed by OAF `sub`), while the shared webhook and SMTP channels are admin-only;
   "Send test notification" works.
6. **Storage & data** (§5) shows the resolved state dir (honoring `EDH_STATE_DIR`), offers Open/
   Export to all users, and gates retention, Import, and the destructive Reset actions to admins.
7. **Tools & environment** (§6) reports detected `dig`/`swaks` status, and admin-only resolver/
   timeout/CLI-preference edits persist to `config.yaml`.
8. **Account & access** (§7) shows the signed-in identity and role, allows Sign out, and displays the
   allowed Workspace domains (editable only by admins, applied on the OpenAuthFederated side); the
   **Admin** sub-section at `/settings/admin` renders only with `role:admin`.
9. **Appearance** (§8) applies theme/density per-user and never affects audits or other users.
10. Reads use `GET /api/settings`; all-users/per-user writes use `PUT /api/settings`; admin-only
    writes use `PUT /api/settings/admin` (and the admin actions) and return `403` without
    `role:admin`. All persistence goes through the file store — `config.yaml` + `domains.yaml` under
    `~/.email_delivery_hero/` — with **no database**.
11. No setting is enforced by the frontend alone: for every admin-only field, the backend rejects an
    unauthorized write even if the UI would have allowed it.
