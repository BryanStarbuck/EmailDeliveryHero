# EmailDeliveryHero вҖ” Installing the Brew / OS Tools

The audit engine shells out to CLI tools. [run_checks.mdx В§5](./run_checks.mdx) names them вҖ” `dig`,
`doggo`, `kdig`, `drill`, `dnsx`, `swaks`, `openssl`, `checkdmarc`, `parsedmarc`, `testssl`,
`spamassassin`/`spamc`, `whois`, and more вҖ” and the per-check "Testing toolbox" sections
([checks/dns.mdx](./checks/dns.mdx), [checks/dkim.mdx](./checks/dkim.mdx),
[checks/spam_content.mdx](./checks/spam_content.mdx)вҖҰ) name the rest. On a fresh machine **most of
these are absent.**

The engine already degrades gracefully вҖ” a missing tool is a capability downgrade, never a failure
([run_checks.mdx В§5.3](./run_checks.mdx)). This spec is the **other half of that story**: rather than
quietly running a thinner audit forever, we give the user a **one-click path to a full-fidelity
audit** вҖ” a dedicated **Install** page that detects what's missing, lets them choose, installs it
through Homebrew, and then **resumes exactly the run they were starting.**

This file owns:

* The **shared Install page** (В§4) вҖ” the UI and flow used by **both** package managers. The
  companion [install_npm.mdx](./install_npm.mdx) owns the npm/pnpm mechanics and reuses this page.
* The **canonical Homebrew catalog** (В§3) вҖ” the single source of truth for *which formula provides
  which tool*, superseding the scattered "Testing toolbox" mentions across the check specs.
* Detection (В§5), installation (В§6), the API + child-process contract (В§7), and **continuing** вҖ”
  return-to-origin + resume-the-pending-run (В§8).

Siblings: [run_checks.mdx](./run_checks.mdx) (the `ToolRunner`/`ToolLocator` contract we reuse),
[settings.mdx В§6](./settings.mdx) (Tools & environment вҖ” the *status* surface this page makes
*actionable*), [errors.mdx](./errors.mdx) (log lines), [engineering.mdx](./engineering.mdx) (module
layout), [install_npm.mdx](./install_npm.mdx) (the npm sibling).

## 1. Where Install sits вҖ” preflight before a run

The trigger funnel from [run_checks.mdx В§1](./run_checks.mdx) grows **one gate** at the front. When
the user starts a run (Dashboard **Run checks**, a Domains **Run audit now**, a **Run all**, or a
per-check **Run this check now**), the frontend first asks the backend *what would this run need
that isn't installed?* If the answer is non-empty, we **divert to the Install page** carrying two
pieces of state вҖ” where we came from and what the user was trying to do вҖ” then come back and finish
the job.

```
 user clicks "Run all" on /            preflight: GET /api/install/preflight?scope=run-all
        в”Ӯ                                          в”Ӯ
        в–ј                                          в–ј
 nothing missing в”Җв”Җв–¶ run proceeds normally    missing = [doggo, checkdmarc, spamassassin]
                                                   в”Ӯ
                                                   в–ј
                              navigate /install?manager=brew&from=%2F&intent=run-all
                                                   в”Ӯ  user picks rows, clicks Install, waits
                                                   в–ј
                              installs finish в”Җв”Җв–¶ navigate back to `from` (/) в”Җв”Җв–¶ replay `intent` (run-all)
```

The gate is **advisory, never mandatory.** A run is always *allowed* to proceed with missing tools
(the engine degrades). The Install page is offered, not forced: it is reachable any time from
**Settings вҶ’ Tools & environment** (В§9), and the preflight diversion can be dismissed
("Run anyway") вҖ” see В§4.6.

## 2. Use cases

| # | Use case | Entry | Outcome |
| --- | --- | --- | --- |
| U1 | **Run diverted to install** вҖ” user starts a run; required tools are missing. | Any run trigger вҶ’ preflight вҶ’ `/install?...&from&intent` | User installs the checked tools; the app returns to `from` and **replays the pending run** (В§8). |
| U2 | **Proactive install from Settings** вҖ” user opens the Install page directly to top up their toolbox. | Settings вҶ’ Tools & environment вҶ’ **Install missing tools** (`/install?manager=brew`) | Tools install; no pending run вҖ” user stays on the page (or navigates away freely). |
| U3 | **Partial install** вҖ” user unchecks tools they don't want. | Install page checkboxes | Only checked rows install; unchecked rows are left absent and the audit keeps degrading for those (honestly reported). |
| U4 | **Homebrew itself missing** вҖ” `brew` is not on the machine. | Preflight/detection finds no `brew` | The page shows a **"Homebrew not installed"** banner with the official one-line install command to copy вҖ” we never try to install brew *for* them (В§6.5). |
| U5 | **Non-brew tool** вҖ” a needed tool has no Homebrew formula (SpamAssassin, rspamd, opendkimвҖҰ). | Catalog entry flagged `manager: special` | The row renders with its **real** install path (`cpanm`, `pipx`, Docker, apt) вҖ” auto-installable where safe (`cpanm`), copy-command where it needs root/Docker (В§3.3, В§6.4). |
| U6 | **Re-run detection** вҖ” user installed something in another terminal. | **Re-detect** button (В§4.5) | The list re-probes and drops the now-present tools. |
| U7 | **Install fails** вҖ” a formula errors mid-install. | Any install | The row goes red with the tail of `brew` stderr; other rows are unaffected; the pending run still offers to proceed with what succeeded (В§6.3, В§8.4). |

## 3. The canonical Homebrew catalog

This section is the **single source of truth** for the brew side. Every formula below was verified
against the live Homebrew API (`formulae.brew.sh/api/formula/<name>.json`) on **2026-07-02**. The
scattered "Testing toolbox" install lines in the check specs are downstream of this table вҖ” where
they disagree, **this table wins**, and В§3.4 records the drift to fix.

The catalog lives in the repo as data (В§9): `code/packages/backend/src/modules/install/catalog/brew.ts`.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/install/catalog/brew.ts` вҖ” the `BREW_CATALOG` array (the `ToolCatalogEntry[]` this section defines as data).

Each entry is:

```ts
export interface ToolCatalogEntry {
  id: string;              // stable id, e.g. 'doggo', 'spamassassin'
  manager: 'brew' | 'special';   // 'special' = installs via cpanm/pipx/docker/apt (В§3.3)
  formula: string;         // brew formula name (may differ from the binary! e.g. drillвҶ’ldns)
  binaries: string[];      // the executables it provides вҖ” what detection looks for on PATH
  category: Category;      // 'dns' | 'spf' | 'dkim' | 'dmarc' | 'blacklist' | 'spam' | 'tls' | 'general'
  tier: 'default' | 'extended';  // default = preselected baseline; extended = opt-in
  summary: string;         // one line shown on the row
  usedBy: string[];        // checkIds that shell out to it вҖ” drives per-scope preflight (В§5.3)
  installArgs?: string[];  // override for 'special' entries (e.g. cpanm invocation)
  notes?: string;          // the gotcha, shown as row subtext (keg-only, needs DockerвҖҰ)
}
```

The lean set that lights up the **entire first-round audit path** (DNS, SPF/DKIM/DMARC records,

are **checked by default** on the Install page.

### 3.4 Drift the catalog corrects (carry these into the check specs)

1. **`drill` is provided by `ldns`, not `drill`.** `brew install drill` installs an **unrelated Rust
   HTTP load-tester** (`desc: "HTTP load testing application written in Rust"`). The DNS `drill`
   tool ships in the `ldns` formula. The catalog's `ldns` entry owns the `drill` binary; the code
   must **never** emit `brew install drill`.
2. **`whois` is keg-only.** Homebrew's `whois` does not symlink into `/opt/homebrew/bin` (it would
   shadow the macOS system `whois`); the binary lives at `/opt/homebrew/opt/whois/bin/whois`.
   Detection (В§5) must check the keg path, and the ToolLocator fallback already knows it.
3. **SpamAssassin / rspamd are NOT brew formulas.** Any spec, code, or UI string that says
   `brew install spamassassin` or `brew install rspamd` is **wrong**. The canonical path is
   `cpanm --notest Mail::SpamAssassin` (SpamAssassin) or Docker (rspamd). This corrects the stale
   phrasing flagged in [checks/content_scoring.mdx](./checks/content_scoring.mdx) and
   [spam_checks.mdx](./spam_checks.mdx) (which still say "Homebrew spamassassin"); the authoritative
   path is [checks/spam_content.mdx В§3](./checks/spam_content.mdx).
4. **`dog` is gone** (disabled in Homebrew 2024-08-01) вҶ’ use **`doggo`**. **`mailhog` is gone** вҶ’
   use **`mailpit`**. **`opendkim`/`opendmarc`** were removed from brew. The catalog carries only the
   live successors.

### 4.2 The tool row (rounded card)

Each missing tool is a **rounded rectangle** (`rounded-lg border border-edh-border p-4`), able to
span **multiple lines**, with a **checkbox on the left**:

* **Checkbox** вҖ” left edge, vertically centered. Checked = "install this". **Every row we can
  auto-install starts checked** вҖ” default *and* extended tier alike. The guiding rule is **check them
  all unless there's a good reason not to**, and the only good reason is *"this tool can't be
  batch-installed"*: a `special` row that needs Docker, a password/root, or a missing `pipx`
  (`autoInstallable: false`) has **no checkbox at all** вҖ” it's replaced by a **Copy** button (last
  bullet below). So the set that starts checked is exactly the set that is one-click installable.
  Nothing installs without the user pressing Install, and they can uncheck any row first. Toggling
  recomputes the "N selected" counters in both action bars.
* **Line 1** вҖ” the tool name(s) (the `binaries`, e.g. `spamassassin  spamc`) in `font-medium`, with a
  right-aligned pair of chips: the **category** (DNS/SPF/DKIM/DMARC/BLACKLIST/SPAM/TLS) and the
  **manager** (`brew` / `cpanm` / `pipx` / `docker` / `os`).
* **Line 2** вҖ” the one-line `summary`.
* **Line 3** вҖ” the exact **install command** in a mono chip, and a **status dot**: `в—Ҹ missing`
  (grey) вҶ’ `в—җ installing` (amber, animated) вҶ’ `вң“ installed` (green) вҶ’ `вң— failed` (red).
* **Line 4** (only when present) вҖ” the `notes` gotcha as amber subtext (keg-only, "needs Docker",
  "may take minutes").
* When a row is `special` and **not auto-installable** (Docker/os/root), the checkbox is replaced by
  a **Copy** button and the row cannot be selected for the batch install (it's copy-and-run-yourself).

Rows are **grouped by category** with small section headers (DNS, then TLS, then SPAMвҖҰ), and within
a category, default tier before extended.

### 4.3 The Install action bars (top and bottom)

Two **identical** bars вҖ” one above the rows, one below вҖ” so the button is always in reach on a long
list. Each shows the live count ("Install 6 selected tools") and one primary **Install** button.
They are the same control; clicking either starts the same batch (В§6). While a batch runs, both
bars switch to a **progress state**: "Installing 3 of 6вҖҰ" with a determinate bar, and the button
becomes **Cancel** (aborts the *remaining* queue; the in-flight formula finishes or is killed on the
child-process timeout, В§7).

### 4.4 Live install output

When the batch runs, each installing row **expands** to show a live, auto-scrolling tail of that
formula's `brew` stdout/stderr (last ~12 lines, monospace, muted) вҖ” streamed from the backend (В§7.3).
On success the tail collapses to `вң“ installed`; on failure it stays open, red, showing the last
lines of stderr so the user sees *why*.

### 4.5 Re-detect

A **Re-detect** button (header, top-right) re-runs detection (В§5) and rebuilds the list вҖ” dropping
tools that appeared since the page loaded (U6) and re-adding any that vanished. It's also run
**automatically** once when every selected install has settled, so the list reflects reality before
the "continue" step (В§8).

### 4.6 The run-context footer (only under `?intent`)

When the page was reached from a diverted run (U1, i.e. `?intent` is present), a footer offers the
two escape hatches the advisory gate promises (В§1):

* **Skip вҖ” run anyway** вҖ” abandon installing and go straight to В§8's continue step with whatever is
  currently installed (the run degrades honestly).
* **Cancel run** вҖ” drop the pending intent and return to `from` **without** running.

When there is no `?intent` (U2, proactive), the footer is absent; the user simply navigates away
when done.

## 5. How we check what's installed

Detection answers two questions: *is the package manager itself present?* and *for each catalog
entry, is its tool on this machine?*

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/install/install.detect.ts` вҖ” functions `detectEntry()` / `detectAll()`, `brewPresent()` (В§5.1), and `pnpmEnablePath()`; the per-tool binary probe reuses `locateTool()` from `shared/tool-runner.ts`.

### 5.1 Is Homebrew present?

`locateTool('brew')` ([run_checks.mdx В§5.2](./run_checks.mdx)) вҖ” an in-process PATH search plus the
conventional fallback dirs (`/opt/homebrew/bin`, `/usr/local/bin`, Linuxbrew's
`/home/linuxbrew/.linuxbrew/bin`). If it resolves, we run `brew --version` once (cached) to confirm
it executes. If it does **not** resolve, the page shows the **U4** banner (В§6.5) and every `brew`
row becomes copy-only.

### 5.2 Is a given tool installed?

Two complementary signals, cheap-first:

1. **`locateTool(binary)`** вҖ” the authoritative check, because it's exactly what a *run* will use. A
   tool counts as installed iff at least one of its `binaries` resolves to an executable path
   (honoring the `whois` keg path, В§3.4). This is pure Node `fs.access`, no child process.
2. **`brew list --formula --versions` (one call, cached per detection)** вҖ” a single spawn that lists
   every installed formula with its version. We cross-reference by `formula` id to (a) show versions
   on already-installed tools in Settings, and (b) catch a formula that's installed but whose binary
   isn't yet on PATH (rare; relink needed). `locateTool` is truth for "can the run use it"; `brew
   list` enriches.

`special` entries detect by their **binary only** (`locateTool('spamc')`, `locateTool('rspamc')`,
`locateTool('olevba')`) вҖ” they're not brew formulas, so `brew list` never sees them.

### 5.3 What does *this run* need? (scoped preflight)

Preflight is **scope-aware** so we never nag about a tool the pending run won't touch. Each catalog
entry lists the `checkIds` that shell out to it (`usedBy`). Given a run scope we compute the set of
checks that will execute (all enabled checks for `run-all`/`run-domain`; the single family for
`run-check`), union their tools, subtract what's installed, and return the **missing** remainder.
A `run-check=spf` preflight therefore lists only SPF/DNS tools, never `swaks` or `spamassassin`.

`GET /api/install/preflight?manager=brew&scope=<scope>&domainId=<id?>` вҶ’
`{ brewPresent, missing: ToolStatus[], optional: ToolStatus[], installed: ToolStatus[] }`, where
`missing` is the uninstalled **default** tier and `optional` the uninstalled **extended** tier. Both
are shown as rows, and the page **pre-checks every auto-installable row across both** (В§4.2) вҖ” the
tier split only decides which absences **divert a run** (a missing `missing` entry does; a missing
`optional` entry does not), never which boxes start checked.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/install/install.service.ts` вҖ” class `InstallService`, method `preflight()` (scopeвҶ’category filtering via `scopeCategories()` in `catalog/index.ts`).

## 6. How we install them

### 6.1 The batch

`POST /api/install/run` with the selected `ids`. The backend installs them **one at a time**
(serial вҖ” brew's own locking makes parallel installs unsafe, and serial output is what the row-by-row
UI shows). For each id:

* **`brew` entries** вҶ’ `runTool('brew', ['install', formula], { timeoutMs, stream })` via the
  `ToolRunner` contract (В§7). No shell, ever вҖ” `formula` is a fixed catalog string, never user input.
* **`special` / cpanm** (`spamassassin`) вҶ’ first ensure its brew deps (`perl`, `cpanminus`) via the
  same brew path, then `runTool('cpanm', ['--notest', 'Mail::SpamAssassin'])`.
* **`special` / pipx** (`dkimpy`вҖҰ) вҶ’ if `locateTool('pipx')`, `runTool('pipx', ['install', pkg])`;
  else the row is copy-only.
* **`special` / docker В· os В· root** вҶ’ **not auto-run.** The row is copy-only (В§6.4).

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/install/install.service.ts` вҖ” class `InstallService`, methods `start()` (queues the job), `runJob()` (serial loop; one failure never sinks the batch, В§6.3), `resolveSpawns()` (pnpm/pipx fallbacks), and `runOne()` (the per-tool spawn).

### 6.2 Streaming per-row status

Each install streams progress to the UI (В§7.3) so rows animate `missing вҶ’ installing вҶ’ installed/
failed` live. The batch endpoint returns a summary `{ results: [{ id, ok, code, tailStderr }] }`
when the queue drains; the stream is the live view, the summary is the settled truth.

### 6.3 Isolation вҖ” one failure never sinks the batch

A formula that fails (non-zero `brew` exit, or the child-process timeout) marks **that row** failed
and the queue **continues** to the next id. The batch never throws; the user ends with a mix of вң“
and вң— rows, and В§8 offers to continue with whatever succeeded.

### 6.4 Root, Docker, and the honesty rule

The app runs as an unprivileged localhost process and **cannot** type a sudo password or start a
Docker daemon for the user. So the rule is: **anything that needs root, a password, or Docker is
never silently attempted** вҖ” the row is **copy-only**, showing the exact command to paste into a
terminal (`docker run вҖҰ`, `sudo apt install opendmarc`). Homebrew itself **never needs sudo** (that
is the whole point of brew), so the entire `brew` tier is genuinely one-click. `cpanm --notest
Mail::SpamAssassin` installs into the user's perl and needs no root either вҖ” hence it's the one
`special` path we automate.

### 6.5 Homebrew missing (U4)

We do **not** run the Homebrew bootstrap (a `curl | bash` from a web app is exactly the shell-piping
we forbid). Instead the page shows a banner with the official command to **copy**:

```
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

вҖҰa "Re-detect" button, and a short note that after installing brew and re-detecting, all rows become
one-click. On Linux without brew, the banner offers the distro path (`apt`/`dnf`) alongside
Linuxbrew.

## 7. How we call them вҖ” the API + child-process contract

### 7.1 REST surface (backend `install` module)

| Endpoint | Purpose | Access |
| --- | --- | --- |
| `GET /api/install/catalog?manager=brew` | The full catalog (В§3) with each entry's live status merged in. | signed-in |

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/modules/install/install.controller.ts` вҖ” class `InstallController`, methods `catalog()`, `preflight()`, `detect()`, `run()`, `jobStatus()`, `jobStream()` (the SSE endpoint).

All are **signed-in** (usable by the `default` user вҖ” the app is optional-login,
[security.mdx](./security.mdx)); none are admin-gated, because installing local CLI tools on your
own machine is the app's core job, not a privileged config change. *(If a deployment wants to lock
this down, `@RequirePermission('tools:install')` is the single-decorator hook вҖ” noted, not imposed.)*

### 7.2 The child-process contract (reused verbatim)

Every spawn вҖ” `brew`, `cpanm`, `pipx` вҖ” goes through the **existing `ToolRunner`**
([run_checks.mdx В§5.1](./run_checks.mdx), `shared/tool-runner.ts`): `execFile` with an **args array,
no shell**; a hard timeout (installs get a **generous budget вҖ” 10 min** default, since a cold
`brew install nmap` or `cpanm SpamAssassin` is slow вҖ” versus the 30s of a check); a `maxBuffer` cap;
and a **structured result** (`{ code, stdout, stderr, timedOut }`) where a non-zero exit is *data*
(the row failed) not an exception. `brew`/`cpanm` resolve through `locateTool` (В§5.1) so the absolute
path is used even under a minimal PATH.

Implemented in code:
`~/BGit/Bryan_git/EmailDeliveryHero/code/packages/backend/src/shared/tool-runner.ts` вҖ” `runTool()` (execFile, hard timeout + SIGKILL grace, `maxBuffer`, structured `ToolResult`) and `locateTool()` (override вҶ’ PATH вҶ’ conventional-dir resolution). Installs stream via `install.service.ts` `InstallService.runOne()`, which spawns directly for the live line reader.

### 7.3 Streaming

Installs are long, so unlike a check (which returns one structured result) an install **streams**.
The `install` service spawns with `execFile` but attaches `child.stdout`/`stderr` line readers and
publishes `{ id, phase: 'installing'|'done'|'failed', line }` events over an **SSE** channel
(`GET вҖҰ/stream`), keyed by a `jobId` the `POST /run` returns. This is additive to the ToolRunner
contract вҖ” the same `execFile`, with the streams tee'd to the SSE bus and still captured for the
final `ToolResult`. If SSE is unavailable, the UI falls back to polling `GET /api/install/run/:jobId`
for coarse per-row status.

## 8. Continuing вҖ” return to origin and resume the pending run

This is the payoff: the user started a run, we detoured through Install, and now we must land them
**back where they were, doing what they asked** вҖ” and **only after the installs finish.**

### 8.1 The two carried params

When the run funnel (В§1) diverts, it navigates to
**`/install?manager=brew&from=<encoded route>&intent=<intent>`**:

* **`from`** вҖ” the URL-encoded route the user was on (e.g. `%2F` for the Dashboard,
  `%2Fdomains%2Fabc123` for a domain page). One of ~20 app routes; we don't enumerate them, we just
  round-trip the string.
* **`intent`** вҖ” what to replay. A small, explicit vocabulary (not arbitrary code):
  `run-all` В· `run-domain:<domainId>` В· `run-check:<checkId>:<domainId>` В· `none`. The frontend
  builds it from the button that was clicked; the Install page treats it as opaque-but-typed data.

Because both are in the **URL**, the detour survives a refresh and is shareable/back-button-safe вҖ”
no hidden global state. (A copy is also mirrored to `sessionStorage` so an accidental param loss
still resumes.)

### 8.2 The wait

The continue step **must not fire until every selected install has settled** (В§6.2). The page tracks
the batch's settled promise; **Skip вҖ” run anyway** (В§4.6) is the only way to shortcut it, and even
that waits for any *in-flight* formula to stop (we never leave a `brew` child orphaned). Once
settled, we run one final **Re-detect** (В§4.5) so the resumed run sees the freshly installed tools
on PATH.

### 8.3 The replay

With installs settled and detection refreshed, the Install page:

1. Navigates to `from`.
2. Dispatches the `intent` on arrival вҖ” the exact same call the origin button would have made
   ([run_checks.mdx В§1](./run_checks.mdx)): `run-all` вҶ’ the Dashboard fan-out;
   `run-domain:<id>` вҶ’ `POST /api/audit/run/:id`; `run-check:<checkId>:<id>` вҶ’ the per-check run.
   The scan-progress dock ([progress_ui.mdx](./progress_ui.mdx)) lights up exactly as if the user had
   never left.

The replay uses a one-shot query param on `from` (`?resume=<intent>`) that the destination page reads
and clears, so a later manual refresh of `from` does **not** re-trigger the run.

### 8.4 Partial success

If some installs failed (В§6.3), the continue step still proceeds вҖ” the run degrades for the tools
that didn't make it, exactly as a normal run would, and the failures were already shown red on their
rows. We never block the user's audit on a formula that wouldn't install; the whole point of the
engine's graceful degradation is that a full toolbox is *better*, never *required*.

## 11. Security

* **No shell, ever.** Every install is `execFile('brew'|'cpanm'|'pipx', [args])` with a fixed catalog
  formula string. There is **no** interpolation of user input into a command line вҖ” the only inputs
  are catalog `id`s validated against the in-repo catalog (an unknown id is a `400`, never a spawn).
* **No `curl | bash`.** The one thing that would need it (installing Homebrew itself) is deliberately
  **copy-only** (В§6.5). We surface the command; we never pipe the internet into a shell on the user's
  behalf.
* **No privilege escalation.** The app never invokes `sudo`; root-needing installs are copy-only
  (В§6.4). The brew/cpanm tier runs entirely as the unprivileged user.
* **Localhost only.** These endpoints install software on the machine running the app; they inherit
  the app's localhost binding and auth ([security.mdx](./security.mdx)). The `tools:install`
  permission hook (В§7.1) exists for anyone who wants to gate them further.

## 12. Acceptance criteria

1. Starting a run whose enabled checks need an absent tool **diverts** to `/install?manager=brew&from=&intent=`;
   starting a run with everything present proceeds with **no** diversion.
2. The Install page lists exactly the **scope-missing** tools (В§5.3): a `run-check:spf` preflight
   shows SPF/DNS tools only, never `swaks`/`spamassassin`.
3. Each row is a rounded, multi-line card with a **left checkbox**, name/category/manager chips, the
   summary, the exact install command, a live status dot, and (where relevant) the gotcha note.
   **Every auto-installable row starts checked** вҖ” default *and* extended tier вҖ” per the "check them
   all unless there's a good reason not to" rule; the only rows that don't start checked are the ones
   that **can't be batch-installed** (Docker/root/os/pipx-absent), which show a **Copy** button
   instead of a checkbox.
4. There are **two** Install bars (top and bottom); both show the live selected-count and start the
   same batch; unchecked rows are **not** installed.
5. Installs run **serially** via `execFile` (no shell) through the existing `ToolRunner`, with a
   generous timeout; each row animates `missing вҶ’ installing вҶ’ installed/failed` from a **live
   stream**, and one failure never aborts the batch.
6. The catalog **never** emits `brew install drill` (uses `ldns`), `brew install spamassassin`
   (uses `cpanm Mail::SpamAssassin`), or `brew install rspamd`/`opendkim`/`opendmarc`; `whois` is
   detected at its keg path.
7. When `brew` is absent, the page shows the **copy-only** Homebrew bootstrap banner and turns every
   `brew` row copy-only вҖ” it never runs `curl | bash` itself.
8. Root/Docker/pip-less tools (`rspamd`, `opendmarc`, `opendkim`, `rblcheck`, `zonemaster-cli`) render
   as **copy-only** rows; `spamassassin` (cpanm) and pipx tools (when `pipx` is present) are
   one-click; **the entire default tier is one-click with zero root.**
9. **Continuing:** once every selected install settles (and a final re-detect runs), the page
   navigates back to `from` and **replays `intent`** (run-all / run-domain / run-check) exactly as the
   origin button would have вҖ” the scan dock lights up вҖ” and a later refresh of `from` does **not**
   re-trigger the run.
10. **Skip вҖ” run anyway** proceeds to the continue step with the current toolbox; **Cancel run**
    returns to `from` without running; a proactive visit (no `intent`) shows neither and just installs.
