# EmailDeliveryHero — Dashboard

The **Dashboard** is the landing view of EmailDeliveryHero and the screen users live in. It is a
brand header over **two TanStack Tables**:

1. **Domain health** (top) — one row per monitored domain; each row summarizes the domain's
   **newest run** across the six test columns.
2. **Runs** (below) — one row per **run** (a dated, per-domain execution of all checks); the
   history that lets a user open any past report.

From here a user runs a manual health check for everything or for one domain, toggles the recurring
scheduler, and clicks into any run's full findings.

This spec builds on the siblings and does not repeat them:

* [Overview](./overview.mdx) — the product charter and the list of checks.
* [Left bar](./leftbar.mdx) — the left navigation (data source of truth `pm/left_bar.yaml`); this
  document does not redefine it and only shows where it sits.
* [Engineering](./engineering.mdx) — the audit engine (findings severity `ok`/`info`/`warning`/`critical`),
  the scheduler endpoints, and the React 19 + Vite + Tailwind + lucide-react + TanStack
  Table/Router/Query frontend on port **4444**.
* [Scheduled checks](./scheduled_checks.mdx) — the recurring-checks configuration page the toggle's chevron opens.
* [Spam checks](./spam_checks.mdx) — the full catalog of sub-tests each of the six test columns groups.
* [View one health check run](./use_cases/view_health_check_run.mdx) — the run report a row click opens.

## 1. Vocabulary — Run → Test → Sub-test (LOCKED)

Three levels, used consistently across every spec and every screen:

| Level | Term | Definition |
| --- | --- | --- |
| 1 | **Run** | One execution of all checks for **one domain**. A run has a `runId`, the **domain** it ran against, a **`startedAt`** date-time (when it started) and a **`finishedAt`** date-time (when it stopped). Pressing "Run checks" over N domains creates **N runs** (one per domain). |
| 2 | **Test** | A **test area** inside a run — exactly the six locked categories: **SPF, DKIM, DMARC, Blacklists, DNS & Infrastructure, Spam & Content**. Every run contains these six tests. |
| 3 | **Sub-test** | An individual check inside a test (e.g. "SPF ≤10 DNS lookups", "selector `s1._domainkey` key present"). Each finding the engine emits belongs to one sub-test; a test's cell color rolls up its sub-tests' worst severity. |

```ts
// The run record the backend persists and the Runs table renders.
export interface AuditRun {
  runId: string;      // unique id for this run
  domainId: string;
  domain: string;     // domain name at run time
  startedAt: string;  // ISO date-time — when the run started
  finishedAt: string; // ISO date-time — when the run stopped
  score: number;
  status: "healthy" | "warning" | "critical";
  findings: Finding[];   // every sub-test outcome (see engineering.mdx)
  counts: SeverityCounts;
}
```

**Backend requirement.** The engine keeps the latest result per domain (`audits.json`) **and** a
run history (`runs.json`): every run is appended with its `runId`, `startedAt`, `finishedAt`,
capped at the newest **50 runs per domain**. Endpoints:

* `GET /api/audit/runs` — all kept runs, newest `startedAt` first (optional `?domainId=` filter).
* `GET /api/audit/runs/:runId` — one run in full (the run report's data).

## 3. The top header / brand

The header renders the wordmark **"Email Delivery Hero"** as **normal text** — large and bold, but
with **normal letter spacing and normal word spacing**. No tracked-out letters, no extra gaps
between the words: it reads exactly like the three words typed with single spaces.

| Aspect | Tailwind treatment | Intent |
| --- | --- | --- |
| Font size | `text-4xl md:text-5xl` | Large; still dominates the top of the content area. |
| Weight | `font-extrabold` | Confident, heavy wordmark. |
| Letter tracking | `tracking-normal` | **Normal text** — no extra spacing between letters. |
| Word spacing | plain string `Email Delivery Hero` | Regular single spaces between the three words; no `mx-*` spans, no `word-spacing`. |
| Casing / color | title-case, `text-slate-900` | High contrast on the page background (WCAG AA). |
| Layout | header row is `flex items-start justify-between`; wordmark left, controls right | Brand left, actions right. |

## 4. The two tables

Both are **TanStack Tables** ([Engineering §5](./engineering.mdx)) sharing one column vocabulary:
the six **LOCKED test columns**, in this order everywhere — **SPF, DKIM, DMARC, Blacklists,
DNS & Infrastructure, Spam & Content**.

### 4.1 Table 1 — Domain health (one row per domain)

* **One row = one monitored domain.** Each test cell rolls up the sub-tests of that test from the
  domain's **newest run** (the latest result per domain).
* Column order: **Domain** first, then the six test columns, then the trailing **row-controls**
  cell (play button + triple-dot menu).
* A domain never audited shows six gray never-run cells.
* **Row click** → the domain's **newest run report** (`/domains/$id`).

### 4.2 Table 2 — Runs (one row per run)

* **One row = one run** (§1): per domain, with its start date-time.
* Column order: **Date** first (the run's `startedAt`, formatted `YYYY-MM-DD HH:mm`, newest first),
  **Domain** second, then the **six test columns** showing **that run's** results, then the trailing
  **row-controls** cell (right chevron + triple-dot menu).
* Sorted by `startedAt` descending — the newest run is the top row.
* Fed by `GET /api/audit/runs` (§1). Shows the newest **100** runs across all domains.
* A section heading **"Runs"** separates it from Table 1.
* **Row click** → **that run's** report (`/domains/$domainId/runs/$runId`).

### 4.3 Row controls — Table 1 (Domain health)

Trailing cell on every row, right-aligned, two controls:

* **▶ Play button** (lucide `Play`) — kicks off a health-check run for **just that domain**
  (`POST /api/audit/run/:domainId`, through the scan runner so the per-domain progress card shows,
  per [Progress UI](./progress_ui.mdx)). While that domain is scanning the play button shows a
  spinner and is disabled. It does **not** navigate.
* **⋮ Triple-dot menu** (lucide `MoreVertical`) — a pull-down menu of actions for that domain:
  * **Run checks now** — same as the play button.
  * **Open newest report** — same as the row click.
  * **Edit domain** — navigates to the Domains page (`/domains`) with this domain's editor open.
  * **Scheduled checks on/off** — toggles the domain record's `scheduleEnabled`.
  * **Delete domain** — destructive, confirm dialog first (same behavior as the Domains page delete).

### 4.4 Row controls — Table 2 (Runs)

Trailing cell on every row, right-aligned, two controls:

* **› Right chevron** (lucide `ChevronRight`) — opens the row up: navigates to that run's report
  (the same place the row click goes). Every Runs row has it.
* **⋮ Triple-dot menu** (lucide `MoreVertical`) — actions for that run:
  * **Open report** — same as the row click / chevron.
  * **Run checks again** — a fresh run for the same domain (`POST /api/audit/run/:domainId`).
  * **Delete run** — removes this run from the history (confirm first).

Clicks on any row control **stop propagation** — they never trigger the whole-row navigation.

### 4.5 Types and `ColumnDef[]`

```ts
// pages/dashboard/columns.ts
import type { ColumnDef } from "@tanstack/react-table";

// The six locked TEST keys, in display order (identical in both tables).
export type TestKey =
  | "spf"
  | "dkim"
  | "dmarc"
  | "blacklists"
  | "dnsInfra"
  | "spamContent";

export type CellColor = "green" | "red" | "amber" | "gray";

export interface CellStatus {
  color: CellColor; // healthy | critical | warning | never-run
  label: string;    // short metric string, <=10 chars ideal (see §5.2)
}

// Table 1 row — one monitored domain, cells from its newest run.
export interface DomainHealthRow {
  domainId: string;
  domain: string;
  cells: Record<TestKey, CellStatus>;
}

// Table 2 row — one run.
export interface RunRow {
  runId: string;
  domainId: string;
  domain: string;
  startedAt: string;  // ISO — rendered YYYY-MM-DD HH:mm, newest first
  finishedAt: string;
  cells: Record<TestKey, CellStatus>;
}
```

## 5. Status cell

Every test cell is a solid color block with short text — a glanceable grid where color signals
health and the label gives the metric. Shared by both tables (and the run report's test chips).

### 5.1 Colors → meaning (LOCKED)

| Meaning | When | Tailwind classes |
| --- | --- | --- |
| **Healthy** | Every sub-test in the test passed (all `ok`). | `bg-green-800 text-white` |
| **Critical / very bad** | One or more sub-tests are `critical`. | `bg-red-800 text-white` |
| **Some problems** | Warnings present, nothing critical (worst is `warning`/`info`). | `bg-amber-500 text-black` |
| **Never run** | No data for that cell (test never ran). | `bg-gray-100 text-gray-500` |

Color is derived from the **worst finding severity** among that test's sub-tests
(`critical → red`, `warning`/`info → amber`, all `ok → green`, no data → `gray`). All four meet
WCAG AA.

### 5.2 Metric text format rules

Cell text is **metric-based and short (≤10 chars ideal)**:

* **Healthy (green):** the literal `Healthy`.
* **Never run (gray):** `Never run` (or `—` in the ultra-compact empty state, §7).
* **K of M sub-tests fail:** `K of M fail` (e.g. `2 of 7 fail`).
* **Naturally a percentage:** show the percent (e.g. `92%`).
* **Just a count of problems:** `N problems` / `1 problem`.

Per-test choice: **SPF / DKIM / DMARC / DNS & Infrastructure** → `K of M fail`; **Blacklists** →
`N problems` (number of DNSBL listings); **Spam & Content** → inbox-placement `NN%`.

## 6. Row click → the run report (and the one exception)

**Clicking anywhere in a row navigates to a run report** — the
[view one health check run](./use_cases/view_health_check_run.mdx) page showing every test, every
sub-test, its severity, and its remediation.

* **Table 1 (Domain health):** row click opens the domain's **newest** run report (`/domains/$id`).
* **Table 2 (Runs):** row click opens **that specific run's** report
  (`/domains/$domainId/runs/$runId`).
* The whole row is the hit target (`cursor-pointer`, hover highlight), routed via TanStack Router.

**The one exception — going inside a test.** Tests that have a full page
([checks/*.mdx §6.1](./checks/overview.mdx)) show a small chevron next to their status cell; pressing
that button goes **inside that test** (e.g. `/domains/$id/spf`) instead of opening the whole run
report. Row controls (§4.3/§4.4) likewise capture their own clicks. Everything else in the row —
including the other cells — falls through to the whole-row navigation.

## 7. Top-right controls

Two controls, stacked top-right: the **Run checks** button on top, the **scheduled-checks toggle**
directly below it.

### 7.1 Run checks button (top)

Kicks off a manual run for **every** monitored domain — one **run per domain**, fanned out in
parallel through the scan runner ([Progress UI §4](./progress_ui.mdx)).

* **API:** `POST /api/audit/run` (all domains); a single domain is `POST /api/audit/run/:domainId`.
* **States:** idle (`Run checks`, `RefreshCw` icon) → running (`Running…`, spinner, disabled) →
  done (queries invalidate; Table 1 refreshes, new rows appear at the **top of the Runs table**).

### 7.2 Scheduled-checks toggle + chevron (below)

An on/off switch for recurring checks, with a **chevron on its right**:

* **Toggle** reflects/sets whether recurring checks are enabled ([Scheduled checks](./scheduled_checks.mdx)).
* **Chevron** (`ChevronRight`) navigates to the scheduled-checks configuration page
  (`/settings/scheduling`). Navigation only; it never flips the toggle.

---

## 8. Empty & loading states

* **No domains:** Table 1 is replaced by a dashed card — "No domains yet. Add your first domain →"
  (links to `/domains`). The Runs table is hidden.
* **Never run:** domains exist but no run yet — Table 1 shows all-gray never-run cells and a helper
  line inviting **Run checks**; the Runs table shows "No runs yet."
* **Loading:** skeleton rows sized to each table's columns, so there is no layout shift.
* **Refetch after a run:** existing rows stay visible until new data replaces them.

## 9. Acceptance criteria

1. The brand header shows **"Email Delivery Hero"** as **normal text** — large and extra-bold, but
   `tracking-normal` with regular single spaces between the words (no letter tracking, no extra
   word gaps).
2. The vocabulary is **Run → Test → Sub-test**: a run is per-domain with `startedAt` and
   `finishedAt` date-times; the six locked tests are SPF, DKIM, DMARC, Blacklists,
   DNS & Infrastructure, Spam & Content; sub-tests are the individual checks inside a test.
3. **Table 1 (Domain health):** one row per monitored domain — Domain column, six test columns
   (newest run's roll-ups), and trailing row controls: a **▶ play button** that runs checks for
   **just that domain** (spinner while running, no navigation) and a **⋮ triple-dot menu** (Run
   checks now / Open newest report / Edit domain / Scheduled on-off / Delete domain).
4. **Table 2 (Runs):** one row per run, newest `startedAt` first — **Date** column
   (`YYYY-MM-DD HH:mm`), **Domain** column, the six test columns for that run, and trailing row
   controls: a **› right chevron** that opens the run and a **⋮ triple-dot menu** (Open report /
   Run checks again / Delete run).
5. **Every row in both tables is clickable**: Table 1 rows open the domain's **newest** run report;
   Runs rows open **that run's** report. Row controls and per-test chevrons capture their own
   clicks and never trigger row navigation.
6. Pressing a test cell's chevron goes **inside that test** (its full page) — the one exception to
   whole-row navigation.
7. The backend keeps a **run history** (`runId`, `startedAt`, `finishedAt`, findings; 50 newest per
   domain) and serves `GET /api/audit/runs` (+ `/:runId`); the Runs table renders from it.
8. Status cells follow the locked four-color mapping and short metric text rules (§5).
9. The **Run checks** button and **scheduled toggle + chevron** behave per §7.
10. Empty/never-run/loading states render per §8 with no layout shift.
